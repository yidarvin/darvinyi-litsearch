The Impact of AI on Developer Productivity: Evidence from GitHub Copilot

A controlled trial with 95 professional developers recruited on Upwork: told to build an HTTP server in JavaScript as fast as possible, the group with GitHub Copilot finished 55.8% faster than the group without it — the field-experiment counterpart to the benchmark leaderboards, and the paper everyone cites for "AI makes programmers faster."

Sida Peng · Eirini Kalliamvakou · Peter Cihon · Mert Demirer  |  Microsoft Research & GitHub & MIT Sloan

// the gap

What problem is this solving?

By early 2023, GitHub Copilot had been in technical preview for over a year, and the marketing pitch — "an AI pair programmer" — was already conventional wisdom. But the evidence behind it was thin: acceptance-rate telemetry on individual suggestions, survey data on how developers felt about the tool, and small usability studies watching a handful of people write short snippets. None of that answers the actual economic question — does a professional developer, working under normal incentives, finish real work faster with the tool than without it? — because none of it was a controlled trial with a measured outcome and a control group.

That gap mattered because the object being studied, Codex (the model underneath Copilot), had already posted strong numbers on code-generation benchmarks like HumanEval. But benchmark accuracy on isolated problems doesn't establish that a tool changes how fast a human actually ships something, especially once you account for the overhead of reading suggestions, deciding whether to accept them, and correcting the ones that are wrong. The paper's contribution is narrow and deliberately so: one standardized task, a randomized treatment, and a precise, pre-specified way to measure "faster."

// how it works

The experiment

The authors recruited software developers through Upwork, a freelancing platform, via a job posting advertised as a paid coding study. The experiment ran May 15 – June 20, 2022 — deliberately concluded right before Copilot became generally available, so every participant was working with a tool that was still novel to them. 166 offers were sent and 95 accepted; on signing the Upwork contract, participants were randomly split into a treated group (45, given Copilot access plus a 1-minute how-to video) and a control group (50, no Copilot, but otherwise free to use anything a developer would normally reach for — search engines, Stack Overflow, their own tooling).

The task itself was standardized: implement a small HTTP server in JavaScript — parse a request string, serve files from a content/ directory, and return the right status codes (200/400/404/405) with correct headers. Participants worked inside a personal GitHub Classroom repository seeded with the task description and a skeleton codebase, shown below. A hidden 12-check test suite ran on every push; the timestamp of repo creation to the timestamp of the first push that passed all 12 checks was recorded as that participant's task completion time. Payment was tied to speed (a sliding scale from $160 for finishing within 30 minutes down to $20 for anything past 150 minutes), which gave every participant — treated or not — the same incentive to go fast, a design choice that keeps the comparison to "does the tool help," not "who tried harder."

Two outcome measures were pre-defined: task success (did the submission pass the test suite at all) and task completion time (for those who did). An exit survey then asked participants to self-estimate the productivity gain or loss Copilot gave them, and separately, the highest monthly price at which they'd want to be notified about Copilot's release — a proxy for willingness to pay. The control group answered the same two questions after watching Copilot's 1-minute demo video, so both groups' self-estimates are comparable.

// what they found

The results

Conditioning on completing the task, the treated group's average completion time was 71.17 minutes versus 160.89 minutes for control — a 55.8% reduction (95% CI: 21–89%; t-test p = 0.0017). Four outliers ran past 300 minutes, and all four were in the control group; the authors report the result is robust to dropping them. Figure 6 below shows why the gap is so visible even by eye: the treated group's completion times pile up in the first 30–60 minute bin, while the control group's distribution is far more spread out, with a long tail running out past 500 and even 800 minutes.

Task success — did the submission pass the hidden test suite at all — tells a quieter story: the treated group's success rate was 7 percentage points higher than control's, but the estimate is not statistically significant, with a 95% confidence interval of [−0.11, 0.25]. In other words, the headline effect here is about speed, not about whether the task got done correctly — the paper doesn't find (or claim) a precise, significant lift in success rate to go with it.

Cutting the speed gain by participant characteristics (Table 1, a Horvitz–Thompson-transformed regression of the individual treatment effect on covariates), the authors report that less experienced developers, developers with more daily coding hours, and older developers (ages 25–44) benefited more from Copilot. The clearest of these is the age effect (coefficient −74.55, SE 33.52, p = 0.0303) and hours-per-day (coefficient −11.70, SE 4.74, p = 0.0168); the experience effect is directionally consistent but only marginally significant (coefficient 8.23, SE 4.36, p = 0.0629). None of the other covariates tested — employment status, income bracket, education, or language preference (Java, Python) — reached significance.

What participants themselves thought

The exit survey adds a nice sanity check: on average, both treated and control participants estimated Copilot would give roughly a 35% productivity gain — a real underestimate against the 55.8% actually measured. Willingness to pay moved in the same direction as the objective result: the treated group's average stated "interested" price for Copilot was $27.25/month versus $16.91/month for control, a statistically significant gap at the 95% level, which the authors read as indirect evidence that the treated group's experience with the tool during the task genuinely shaped how much they valued it.

// does it hold up?

The eval read

The internal validity of the trial is solid for what it measures: real random assignment, a pre-specified outcome, an automated (not self-reported) completion-time measure, and a payment scheme that gives both arms equal incentive to hurry. The 95% CI of [21%, 89%] is wide — this is a genuinely small trial — but the point estimate and the visual gap in Figure 6 are hard to explain away as noise.

The generalizability problem is bigger, and it starts with the task itself. A single-file HTTP server with a fixed, well-specified status-code contract is close to a best case for a Codex-era code model: it's exactly the kind of boilerplate, frequently-seen-in-training-data pattern that autocomplete tools are strongest at. The paper studies "does Copilot speed up a well-scoped, self-contained, greenfield task," not "does Copilot speed up the messier, larger, legacy-codebase work most professional engineers spend most of their time on" — and 55.8% has nonetheless become the number people quote for the latter. The authors are honest about this limitation in their own discussion section, but it hasn't stopped the number from traveling further than the task that produced it.

The sample is also a specific slice of the developer population, not a random one: Upwork freelancers, disproportionately based in India and Pakistan, median income under $20,000/year despite an average 6 years of coding experience — a population selected for being available for a short paid gig, not matched to (say) engineers at a typical US tech employer, where Copilot's real-world adoption is concentrated. And attrition is real: 166 offers were sent, only 95 accepted, and the paper reports that "thirty-five developers from both the treated and control groups completed the task and survey" — worded ambiguously enough that it's unclear whether that's 35 total or 35 per arm, but either way a meaningful share of the randomized 95 never produced a completion-time observation at all, and the paper doesn't characterize whether that drop-off was balanced across arms.

Two more points worth flagging. First, code quality is out of scope entirely — the authors say so directly, noting AI assistance "can increase code quality... or it can reduce quality," and this study measures neither; a submission that barely limps past the 12-check test suite counts the same as a clean one. Second, this is vendor research: three of the four authors are Microsoft/GitHub employees studying their own product, pre-general-availability, with no independent replication cited. That doesn't invalidate the randomization, but it's the kind of thing that should temper how much weight the specific "55.8%" carries versus the qualitative direction of the result.

baselines

Clean RCT with real random assignment and an automated outcome; the control group had normal access to search/Stack Overflow, so this is "Copilot vs. status-quo tooling," not "AI vs. nothing."

contamination

Not a leaderboard-style contamination risk, but a construct-validity one: a small, self-contained HTTP-server task is close to a best case for Codex-era autocomplete, likely inflating the speed gain relative to messier real-world engineering work.

what's missed

Code quality and correctness beyond "passes the hidden tests," generalization to large/legacy codebases, and a precise accounting of who dropped out between the 95 randomized and the ~35-per-group who finished (the paper's own wording leaves it unclear whether that's 35 total or 35 per arm).

would raise confidence

A larger trial across varied, less Copilot-friendly tasks; an independent (non-Microsoft/GitHub) replication; and a code-quality or security assessment of the accepted submissions, not just pass/fail.

// bottom line

The first controlled trial to put a number on AI pair-programming productivity, and a genuinely careful one on its own narrow terms — but the "55.8% faster" headline is a best-case number from a small, boilerplate-friendly task and a freelancer sample, run by the vendor. Treat the direction (Copilot measurably speeds up this kind of task) as the durable finding, and the specific magnitude as an upper bound, not a universal constant.

// takeaways

What to remember

  1. 55.8% faster, in a controlled trial. 95 Upwork developers randomized to Copilot-or-not on a standardized HTTP-server task; the treated group averaged 71.17 minutes to completion versus 160.89 minutes for control (95% CI: 21–89%).
  2. Speed moved, correctness didn't (measurably). The treated group's task success rate was only 7pp higher than control's and not statistically significant — this is a speed result, not a demonstrated quality or correctness result, and code quality wasn't assessed at all.
  3. A best-case task and sample. A small, well-specified, boilerplate-heavy coding task is close to ideal conditions for a Copilot-era model, and the Upwork freelancer sample skews toward a different population than typical enterprise engineers — reasons to treat 55.8% as an upper bound rather than a general constant.
