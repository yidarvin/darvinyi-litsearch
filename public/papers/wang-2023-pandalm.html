PandaLM: An Automatic Evaluation Benchmark for LLM Instruction Tuning Optimization

Distill GPT-3.5’s response comparisons into a LLaMA-based judge model, filter out the position-biased half, and you get PandaLM: an open, offline, reproducible evaluator that PandaLM-70B uses to edge past GPT-4 on a 1K human-labeled test set — and that the authors then point at 80 hyperparameter configurations per model to pick better instruction-tuning recipes than Alpaca’s defaults, with zero API calls.

// the gap

What problem is this solving?

Instruction-tuning a foundation model means picking a hyperparameter recipe — optimizer, learning rate, epoch count, scheduler — and the only way to know which recipe is best is to evaluate the resulting model. In mid-2023 that evaluation step had no good options. Objective metrics like accuracy and F1 on downstream tasks (via lm-eval) don’t capture the subjective qualities that separate two competent instruction-tuned models — conciseness, clarity, adherence to instructions, comprehensiveness, formality. Model-based similarity metrics like BERTScore and MAUVE score closeness to a reference text, not which of two responses is actually better, and their scores aren’t interpretable to a practitioner deciding between checkpoints.

That leaves two practical options, and the paper argues both are broken for this use case. Human evaluation is reliable but expensive, slow, and inconsistent across annotators — impractical when you need to compare dozens of hyperparameter configurations. API-based LLM evaluation (GPT-3.5/GPT-4-as-judge) is faster, but the paper flags three specific costs: closed-source opacity (you can’t inspect or improve the judge), uncontrolled versioning (the API model changes under you, breaking reproducibility), and the risk of data leakage — sending your instruction-tuning candidates’ outputs to a third-party API is a non-starter when the training data itself may be sensitive.

PandaLM’s answer is to train a dedicated, open-source judge LLM that runs entirely offline: no API calls, no data leaving the local environment, and a fixed set of weights that never silently changes underneath a research pipeline.

// how it works

The method

PandaLM is a judge LLM fine-tuned from LLaMA. Given an instruction, an input, and two candidate responses, it outputs three things: which response wins (1, 2, or Tie), a natural-language evaluation reason, and a reference response it considers ideal. That third output — generating its own reference answer rather than only comparing — is what lets PandaLM diagnose problems like logical fallacies, unnecessary repetition, grammatical errors, and missing context, not just declare a winner.

Building 300K training triples without human labels

Human-labeling millions of comparison triples wasn’t feasible, so the training data is built by distillation. Five similarly-sized open foundation models — LLaMA-7B, Bloom-7B, Cerebras-GPT-6.7B, OPT-7B, and Pythia-6.9B — are each instruction-tuned on the same Alpaca 52K data with Alpaca’s own hyperparameters. Pairs of their responses to Alpaca-52K instructions are then judged by GPT-3.5-turbo, following a self-instruct-style prompting setup, to produce the (result, reason, reference) triples.

Raw GPT-3.5 judgments are noisy, so two filters are applied before training. First, hand-crafted rules strip out invalid evaluations. Second — and more important — every comparison is run twice with the response order swapped; if GPT-3.5 flips its verdict when the same pair is reordered, that’s evidence of position bias, and the sample is dropped rather than kept. What survives filtering is 300K training samples, seq2seq fine-tuned with a standard cross-entropy next-token objective (no separate classification head) — on 8×A100-80GB with DeepSpeed ZeRO Stage 2, bfloat16, AdamW at a 2e-5 learning rate with cosine schedule, 2 epochs, batch size 2/GPU with 8-step gradient accumulation, inputs truncated to 1024 tokens.

A human-labeled test set built the same way, checked by three annotators

To measure whether PandaLM actually tracks human judgment, the authors build a held-out test set sampled from self-instruct’s human-evaluation instructions (drawn from Grammarly, Wikipedia, National Geographic, and roughly a hundred other apps and sites) — genuinely human-written contexts, not model-generated ones. Each of three independent human annotators labels every instance; samples where the annotators disagree too much are dropped to keep per-annotator Inter-Annotator Agreement (Cohen’s Kappa) above 0.85. The final filtered set has 1,000 instances (down from 2,500 raw), split 422 wins for response 1, 472 for response 2, and 105 ties, with final pairwise kappas of 0.85, 0.86, and 0.88 among the three annotators.

The results

On the 1K human-labeled test set, PandaLM is scored against GPT-3.5 and GPT-4 by treating the human labels as ground truth and computing accuracy, precision, recall, and F1.

PandaLM-70B edges out GPT-4 on accuracy (0.6687 vs 0.6647) and beats it more clearly on F1 (0.6923 vs 0.6180), driven by higher precision. PandaLM-7B, run at a fraction of the compute, lands below both GPT-3.5 and GPT-4 on every metric but stays in the same neighborhood. Ablating the reasoning and reference-response outputs and training PandaLM-7B on only the win/lose/tie label collapses performance to 0.4725 accuracy / 0.3152 F1 — evidence that forcing the model to explain and produce a reference answer, not just pick a winner, is what makes it work.

Using PandaLM to actually pick hyperparameters

The paper’s real deliverable is downstream: PandaLM-7B is used to search 80 hyperparameter configurations per base model (5 epoch checkpoints × 4 learning rates × 2 optimizers × 2 schedulers), run as 800 pairwise tournament comparisons, to pick a winner for each of the five foundation models. That winner is then compared against the same models tuned with Alpaca’s default recipe — using the same training data for a fair comparison — and judged independently by GPT-3.5, GPT-4, and human annotators.

The domain-shift experiments (legal LSAT questions, biomedical PubMedQA and BioASQ) tell a similar story with GPT-4 standing in for the gold label: PandaLM-7B lands well above the ~33% random-guess floor for a three-way win/lose/tie classification (e.g. 0.6154 accuracy on PubMedQA), and PandaLM-70B improves further (0.7692 on PubMedQA, 0.7727 on BioASQ) — evidence the judge generalizes past the Alpaca-instruction distribution it was trained on.

// does it hold up?

The eval read

The headline claim — “PandaLM-70B surpasses GPT-4’s performance” — is real but thin. On the metric that matters most (accuracy against human labels), PandaLM-70B beats GPT-4 by 0.0040: 0.6687 vs 0.6647. That is a genuine win on the paper’s own test set, but it’s a margin, not a rout, and it disappears entirely for the model most people would actually run: PandaLM-7B trails both GPT-3.5 and GPT-4 on all four metrics (0.5926 accuracy vs. 0.6296 and 0.6647). The abstract’s “PandaLM-7B offers performance comparable to both GPT-3.5 and GPT-4” is generous framing for a model that’s 4–7 accuracy points behind both.

There’s also a ceiling problem the paper states itself but doesn’t fully grapple with: the three human annotators only agree with each other at a Cohen’s Kappa of 0.85–0.88. The paper explicitly proposes 0.90 agreement with humans as an aspirational target above that human-human baseline. PandaLM-70B’s 0.6687 accuracy isn’t even the same metric as inter-annotator kappa, but the qualitative point stands — the model still disagrees with the human majority far more than the humans disagree with each other, so “surpasses GPT-4” should be read as “narrowly ahead of another imperfect judge,” not “solved evaluation.”

The training data itself is a second layer of distillation risk: every comparison in the 300K-sample training set is a GPT-3.5 judgment, filtered only for position-order consistency, not for correctness against any ground truth. PandaLM is therefore learning to imitate GPT-3.5’s preferences (whatever biases GPT-3.5 has toward length, style, or ordering beyond the one bias explicitly filtered for) and is then evaluated against a different, human-labeled distribution — a train/test mismatch the strong domain-shift results partly offset, but don’t eliminate.

baselines

Fair and current for mid-2023 — GPT-3.5-turbo and GPT-4 were the obvious proprietary judges to beat. The domain-shift experiments (LSAT, PubMedQA, BioASQ) substitute GPT-4 as the gold label instead of humans “due to constraints in time,” which is disclosed but means those specific numbers measure agreement with GPT-4, not with ground truth.

contamination

Training labels are 100% GPT-3.5-distilled, filtered only for position-bias consistency (order-swap agreement), not for correctness. The 1K human test set is drawn from a different pool (self-instruct’s human-eval instructions) than the Alpaca-52K training instructions, which is a reasonable train/test split but not a leakage audit.

what’s missed

No calibration analysis of when PandaLM is confidently wrong; no test against instruction-tuned models larger or more capable than the 6.7B–7B training pool (LLaMA-7B, Bloom-7B, Cerebras-6.7B, OPT-7B, Pythia-6.9B) it was built and validated on. “Privacy protection” is asserted (the paper frames it loosely via federated-learning principles) but never empirically tested — it’s a property of running locally, not something measured.

would raise confidence

Human (not GPT-4) gold labels for the domain-shift sets; a length-bias and position-bias audit of PandaLM’s own outputs the way MT-Bench and AlpacaEval later ran for GPT-4-as-judge; evaluation against newer/larger instruction-tuned models to test whether the judge generalizes past its 2023, ~7B-scale training pool.

// bottom line

PandaLM is a genuinely useful proof that a small, open, offline judge model — distilled from a proprietary one and filtered for a specific bias — can match proprietary judges closely enough to be useful for the task the paper actually cares about: ranking hyperparameter configurations, where GPT-3.5, GPT-4, and human judges all agree PandaLM’s picks beat Alpaca’s defaults. Treat the “beats GPT-4” framing as a narrow, 70B-only, half-a-point win rather than a general claim about judge quality, and treat PandaLM-7B — the version most people will actually deploy — as comparable to GPT-3.5 at best, not GPT-4.

// takeaways

What to remember

  1. The recipe: distill GPT-3.5 pairwise judgments over five instruction-tuned 7B-class models, filter out samples where the verdict flips under response-order swap, and fine-tune LLaMA on what survives (300K triples) to output a winner, a reason, and a reference answer — not just a label.
  2. The result is a genuine but narrow win: PandaLM-70B edges past GPT-4 in accuracy against 1K human labels (0.6687 vs. 0.6647) and more clearly on F1; PandaLM-7B stays below both GPT-3.5 and GPT-4 on every metric, despite abstract language suggesting rough parity.
  3. The actual payoff is offline hyperparameter search: pointed at 80 configurations across five base models, PandaLM-selected hyperparameters beat Alpaca’s defaults by every judge that checked — GPT-3.5, GPT-4, and human — with zero API calls and no risk of leaking training data to a third party.
