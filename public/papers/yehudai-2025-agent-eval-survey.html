A Survey on Evaluation of LLM-based Agents

The first field-wide map of how we test LLM-based agents — five perspectives spanning core capabilities to full generalist leaderboards — built from a 172-paper reference list, and unusually for a survey, still being revised: this v2 folds in benchmarks posted as late as 2026.

Asaf Yehudai, Lilach Eden, Alan Li, Guy Uziel, Yilun Zhao, Roy Bar-Haim, Arman Cohan, Michal Shmueli-Scheuer

// the gap

What problem is this solving?

By early 2025, LLM-based agents — systems that plan, call tools, and act across multi-step interactions with an environment rather than just answering a prompt — had spawned dozens of benchmarks with no shared organizing frame. A capability benchmark for tool use, a web-navigation environment, a software-engineering suite, and a "generalist" leaderboard were each built by a different group, for a different purpose, with no common vocabulary for what they were actually measuring or how their design choices compared. As the authors put it, evaluating an agent has to go beyond scoring LLM text output to assess "an agent's capacity for sequential decision-making and operation within dynamic environments" — a qualitatively different measurement problem than the static QA-style evaluation that surveys like Chang et al. (2023) had already mapped for plain LLMs.

The gap, in other words, wasn't a missing benchmark — agent benchmarks already existed in abundance — it was a missing taxonomy connecting them: which core LLM abilities (planning, tool use, memory, self-reflection) do agent tasks actually draw on; how do application-specific benchmarks (web, coding, science, dialogue) differ in what they test; what does "generalist" agent evaluation even mean; what structural dimensions (data curation, environment dynamicity, interface, metric, safety) cut across all of them; and what tooling exists for developers to evaluate agents during, not just after, development. The paper positions itself as the first survey to answer all five of those questions in one place.

// how it works

The method

Like any survey, the "method" is the taxonomy: the authors organize the literature into five perspectives, each its own section, populated with the specific benchmarks and frameworks that instantiate it.

Agent capabilities (§2). Four core abilities evaluated in isolation or as part of a full workflow: planning & multi-step reasoning (PlanBench, FlowBench, Natural Plan), function calling & tool use (BFCL, ToolBench, ComplexFuncBench — plus the newest wave, Scale's MCP-Atlas and Tool-Decathlon, sourcing tools from real MCP servers), self-reflection (LLM-Evolve, LLF-Bench — flagged as still lacking any standardized benchmark), and memory (StreamBench, MemBench, MemoryAgentBench, spanning episodic/semantic/procedural memory types).

Application-specific agents (§3). Four domains: web agents (from WebShop's simplified shopping simulation through Mind2Web's offline real-website traces to WebArena's dynamic, fully-functional sandboxes); software-engineering agents, where SWE-bench Verified's human-filtered 500-sample subset became "the de facto benchmark," and where the newest entrants push toward professional-grade difficulty — SWE-Lancer's 1,400 real Upwork tasks with over $1M in total payouts, and SWE-bench Pro's 1,865 human-verified tasks across 41 repositories, on which model performance "remains below 25% Pass@1"; scientific agents, decomposed across ideation, experiment design, code generation, and peer review; and conversational agents, from τ-Bench's retail/airline tool-use dialogues to τ²-Bench's added Telecom domain and compositional task generator.

Generalist agents (§4). Two complementary strategies for evaluating agents that aren't tied to one application: benchmarks that inherently demand a wide capability span (GAIA's real-world questions requiring reasoning, multimodality, and web browsing together; its 2025 successor Gaia2's mobile-environment tasks adding ambiguity, noise, and multi-agent collaboration; full computer-use environments like OSWorld and AppWorld), versus unifying many task-specific benchmarks under one harness (AgentBench, the Holistic Agent Leaderboard). The authors note this second, unifying approach doesn't yet support evaluating the same agent harness across every benchmark environment — the gap that emerging frameworks like Harbor and Exgentic are just starting to close.

The results

Section 5 turns from cataloguing to analysis, cross-cutting the benchmarks already reviewed along five structural dimensions — data curation, environment, interaction interface, metric, and safety — and Table 1 makes the pattern concrete across eight representative benchmarks:

Read down the right-most column: seven of the eight benchmarks the survey picked to represent the state of the art carry no explicit safety dimension at all — only τ-Bench does, and even there it's checking policy adherence within one narrow customer-service domain, not adversarial robustness. That's the paper's own headline evidence for the gap it names in §7.2: "current benchmarks lack sufficient focus on safety, trustworthiness, and policy compliance."

Table 2 does the same cross-cut for developer-facing evaluation frameworks (LangSmith, Langfuse, Google Vertex AI, Arize AI, Galileo, Patronus AI, LangChain's AgentEvals, Databricks Mosaic AI) against six capabilities. Every framework except AgentEvals supports stepwise assessment and continuous monitoring; only three (Arize, Patronus AI, Mosaic AI) support synthetic data generation; and only three (Google Vertex AI, LangSmith, and — via a narrower graph-alignment mode — AgentEvals) support trajectory-level assessment rather than judging the final response or individual steps in isolation.

Two trends and five gaps close out the synthesis. Trends: evaluation is shifting toward realistic, high-effort tasks (SWE-Lancer's real freelance payouts, SWE-bench Pro's multi-file, multi-hour edits) and toward "live," continuously-refreshed benchmarks (the BFCL v1→v3 progression, the SWE-bench→Verified→Pro lineage) built to outrun saturation. Gaps: (1) coarse pass/fail metrics that hide where in a trajectory an agent actually failed; (2) no standard accounting for cost/efficiency (tokens, API spend, latency) alongside capability; (3) over-reliance on static, human-annotated data that can't scale; (4) thin safety/policy-compliance testing, as Table 1 itself demonstrates; and (5) benchmarks that conflate the backbone LLM's capability with the agent harness/scaffold around it, which Harbor and Exgentic are only just beginning to disentangle.

// does it hold up?

The eval read

Judge this the way you'd judge any synthesis paper, not a benchmark paper: there's no original experiment to audit for leakage or unfair baselines, so the question is whether the map still matches the territory — and, unusually, whether the authors kept redrawing it. They did: the arXiv record shows a v2 dated 23 Apr 2026, thirteen months after the original March 2025 posting, and the body text itself has absorbed 2026-vintage entries — Scale's MCP-Atlas and Li et al.'s Tool-Decathlon in function-calling, Zhang et al. (2026) on long-horizon planning, He et al. (2026) on episodic memory, Harbor and Bandel et al.'s Exgentic in generalist-agent tooling. That's a meaningfully different posture than a frozen taxonomy paper like Chang et al. (2023), whose "maintained GitHub tracker" turned out to be a reading list, not a re-drawn tree; this survey's own tree has visibly grown.

That said, the paper's structure has the same load-bearing weakness every benchmark-of-benchmarks survey has: its own Table 1 samples 8 benchmarks to represent dozens catalogued elsewhere in the text, and the choice of which 8 make the comparison table is not derived from any stated selection rule — it reads as illustrative, not exhaustive, and a reader can't tell whether the 7-of-8-lack-safety pattern would hold if the table covered 20 benchmarks instead of 8. Likewise the "current trends" claims (toward realism, toward live benchmarks) are asserted narratively from the authors' reading of the literature rather than measured — contrast this with Chang et al.'s own Figure 2, an actual monthly-count growth curve (0→33 papers/month) that quantified the claim it was making. A survey whose central argument is "the field is moving in direction X" is more convincing with a citation-count time series behind X than with an assertion, even a well-read one.

The authors are candid about the rest, and it's worth taking them at their word: the Limitations section states plainly that "some very recent or forthcoming developments might not have been included," that benchmark/framework selection "is subject to the breadth of the field" and prioritizes what "illustrate[s] key trends," and that the "Future Directions" discussion "inherently involve[s] a degree of foresight and interpretation" whose "relative importance...will continue to evolve." That's an honest hedge, not a rigor claim — this is a reading guide to the field circa its last revision, and the value is in the taxonomy and the gap list, not in any of its trend claims being falsifiable.

baselines

N/A — a literature synthesis, not an experiment; there's nothing to run against a held-out baseline. The relevant question is representativeness of what's cited, and coverage is visibly current (2026 citations present) but self-selected, with no stated inclusion/exclusion criterion.

contamination

Not applicable in the usual sense; the closer analogue is whether the survey itself is exhaustive. It isn't and doesn't claim to be — the Limitations section says as much — but readers citing "the" agent-eval survey should know the taxonomy reflects the authors' reading, not a systematic literature-search protocol with inclusion criteria reported.

what's missed

Any quantitative backbone for its own "current trends" claims (contrast Chang et al.'s growth-curve figure); a stated rule for which benchmarks made the two comparison tables versus the dozens that didn't; and, per the paper's own Table 1, safety/robustness evaluation remains thin across the very benchmarks chosen to represent the state of the art.

would raise confidence

A reproducible, quantified version of the "trends" section — e.g. a benchmark-release timeline or citation-growth chart, the same instrument Chang et al. used — plus an explicit note on how benchmarks were selected for Tables 1 and 2 versus the wider text.

// bottom line

This is the right paper to cite for "what does agent evaluation look like as a field" — five perspectives, a real taxonomy tree, and a genuinely useful cross-cutting dimension analysis (Table 1's safety column alone is worth the read). It is a weaker source for "is agent evaluation getting better," since that claim is asserted rather than measured. Its willingness to actually revise itself into 2026 is the most concretely reassuring thing about it as a living reference.

// takeaways

What to remember

  1. Five perspectives — agent capabilities, application-specific benchmarks, generalist agents, cross-cutting benchmark dimensions, and developer frameworks — give agent evaluation the same kind of shared taxonomy Chang et al. (2023) gave plain LLM evaluation, but for the harder problem of sequential, tool-using decision-making.
  2. The paper's own Table 1 is its sharpest finding: seven of the eight benchmarks it picked to represent the field's state of the art (SWE-bench Verified, SWE-Lancer, Mind2Web, WebArena, PaperBench, AppWorld, GAIA) have no safety dimension at all — only τ-Bench does.
  3. Unlike most taxonomy surveys, this one kept moving: a v2 revision in April 2026 folded in benchmarks and frameworks (MCP-Atlas, Tool-Decathlon, Harbor, Exgentic) that didn't exist at first posting — a rare example of a survey's "living document" promise actually being kept, even if its trend claims remain qualitative rather than measured.
