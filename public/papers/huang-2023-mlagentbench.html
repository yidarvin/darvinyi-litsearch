MLAgentBench: Evaluating Language Agents on Machine Learning Experimentation

The first benchmark for LM agents that run the full ML experimentation loop — read the files, edit the training code, execute it, read the metric, iterate. Across 13 tasks the best agent (Claude 3 Opus) succeeds 37.5% of the time, but swings from 100% on a canonical dataset to 0% on a Kaggle challenge released after its training cutoff.

Qian Huang, Jian Vora, Percy Liang, Jure Leskovec · Stanford University

What problem is this solving?

Progress in machine learning is driven by experimentation: a researcher picks a method (an architecture, a learning rule), runs an experiment, reads the validation number, and revises. It is a loop that demands prior knowledge, working code, and the judgment to interpret a result and decide what to try next. The obvious question once language models could write and run code: can an agent close that loop on its own?

Prior automation only nibbled at the edges. Neural Architecture Search and AutoML optimize within a fixed, pre-specified search space — they don't read a task description, write novel code, and decide what experiment to run. The 2023 wave of LM agents (AutoGPT, generative agents, Voyager playing Minecraft, SayCan for robotics) showed open-ended capability, but — as the authors put it — it is "hard to evaluate the performance and reliability of these agents, especially over a long horizon of complex interactions." Concurrent agent benchmarks measured adjacent skills: AgentBench wraps LMs in fixed simple agents across diverse environments, WebArena scores web interaction, ARA probes high-stakes autonomy. None of them put an agent in front of a real ML task with starter code and an automatic grader and asked: can it actually move the metric?

MLAgentBench is the first benchmark built for exactly that. It frames ML experimentation as a set of well-scoped, executable tasks with clear goals and an automatic evaluator, chosen to sit at the intersection of four properties the authors argue prior agent evals lack together: containability, complexity, evaluability, and practical usefulness.

// how it works

The method

Every task in MLAgentBench is a triple: a task description ("Given a training script train.py, improve the current model accuracy… save per-class probabilities to submission.csv"), a set of starter files (data, data/metric descriptions, and usually a simple baseline model in diverse frameworks — PyTorch, TensorFlow, JAX, Keras), and an evaluator that assigns a raw score to the agent's final submission (e.g. test-set accuracy). The agent works in a real workspace directory and is graded only on the artifact it leaves behind.

The environment runs over time steps t = 1 … T. The state st is the set of files in the workspace; the agent's memory mt is the history of past observations and outputs. At each step the agent constructs a prompt and queries the LM for the next reasoning trace and action:

rt, at = Agent(st−1, mt−1) ,   mt = (o<t, r<t) ,   rt, at = LM(pt) // the prompt pt = action descriptions + task description + the last 3 steps (r, a, o)

The agent can take 12 actions. Nine are primitive: List / Read / Write / Append / Copy Files, Inspect Script Lines, Undo Edit Script, Execute Script, and Final Answer (which shuts the environment down). The other three are compound actions backed by their own LM calls: Understand File (retrieve content relevant to a query), Edit Script and Edit Script Segment (rewrite a file from a natural-language instruction and return a diff).

Thinking before acting

The heart of the design is the forced response format. Before emitting an action the LM must fill five fields, each borrowed from a prior agent idea and aimed at a specific failure:

  • Reflection — what the last observation means / how to debug an error (from Reflexion).
  • Research Plan and Status — a detailed, running plan that doubles as an interpretable log of what's been done (from AutoGPT).
  • Fact Check — for every claim in the plan, is it confirmed by an observation or hallucinated? This directly targets the most common early failure: the model editing a file and then asserting an accuracy gain without ever running it.
  • Thought — the reasoning for the next action (from ReAct).
  • Action + Action Input — the tool name and a JSON argument.

An agent succeeds on a task if it improves the performance metric by at least 10% over the starter-code baseline. Efficiency is tracked separately as wall-clock time and total input/output tokens. The 13 tasks are deliberately spread across difficulty and recency to probe generalization and dampen contamination: Canonical (CIFAR-10, IMDb, ogbn-arxiv), Classic Kaggle (house-price, spaceship-titanic), Kaggle Challenges (four contests launched Aug 2022–May 2023, e.g. Parkinson's-disease progression, FathomNet, feedback, identify-contrails), Recent Research (CLRS algorithmic reasoning, BabyLM — train an LM on 10M words), and Code Improvement (speed up LLaMA-7B inference; vectorize a convolution).

// what they found

The results

Across GPT-4 (0613), GPT-4-turbo (0125), Claude v1.0 / v2.1 / v3 Opus, Gemini Pro, and Mixtral — 8 runs each, ≤50 actions / 5 hours per run — the Claude v3 Opus agent is the strongest on success rate, building a passing ML model on 37.5% of task-runs on average. There is a clear generational climb inside each model family. But the headline number hides enormous spread: Claude v3 Opus goes 100% on house-price and 0% on BabyLM, parkinsons-disease, fathomnet, and vectorization.

Two numbers in that table fight each other. Claude v3 Opus wins on success rate (37.5%), but GPT-4 wins on average improvement (41.3%) — because that average is dominated by a single task, identify-contrails, where GPT-4 posts a +143.3% gain. Success rate, the more robust metric, is the one to trust.

The recency cliff is the paper's sharpest finding. Scores are highest on well-established datasets the models have surely seen (ogbn-arxiv 87.5%, house-price 100%) and collapse on the recent Kaggle challenges launched after the training cutoff — parkinsons-disease and fathomnet sit at 0% across every model, baseline included. The agents extrapolate poorly to genuinely novel tasks.

Against off-the-shelf scaffolds, the authors' agent wins clearly with GPT-4-turbo — 26.0% vs AutoGPT's 2.9% and LangChain-ReAct's 1.0% — and stays ahead with Claude v3 Opus (37.5% vs AutoGPT 13.5%, LangChain 33.7%, where the simpler LangChain is surprisingly competitive because it doesn't break the submission format). On efficiency, GPT-4-turbo is the cheapest, spending 51.0% fewer tokens than the average agent by finishing and submitting early. A full sweep of the benchmark once with GPT-4-turbo costs ~6M tokens (~$60) — but at a 26% success rate, the expected cost to actually accomplish a task balloons to ~$231. Running longer generally degrades the final metric (the agent regresses), with Claude v3 Opus the only exception.

// does it hold up?

The eval read

MLAgentBench's framing is honest and its agent comparisons (AutoGPT, LangChain-ReAct) are the right 2023 baselines. The soft spot is the metric. "Success" is binary — beat the starter baseline by an arbitrary 10% — which says nothing about whether the improvement came from a sound method or a lucky learning-rate bump, and nothing about run-to-run reliability behind the averaged 8 runs. Worse, four tasks (house-price, spaceship-titanic, imdb, fathomnet) ship no baseline, so "improvement" is measured against a trivial random predictor (e.g. 0.5 accuracy for imdb). The paper's own margin note flags the hole: "for tasks that don't have a baseline, how do you compute improvement? technically any non-zero improvement is infinite percent increase?" Some of that 100%-on-house-price shine is an easy reference point, not agent prowess.

The recency design is a genuine strength and a confound at once. Recent Kaggle and 2023 research datasets do mitigate contamination — but the benchmark then reads novelty and difficulty off the same axis. A 0% on parkinsons-disease could mean the task is uncontaminated, or just under-specified and hard; the high canonical scores partly reflect solutions sitting in pretraining. The benchmark can't separate "the agent can't generalize" from "this task is simply harder."

baselines

Right agent baselines for 2023 (AutoGPT, LangChain-ReAct), and the scaffold beats them decisively on GPT-4-turbo (26.0 vs 2.9 / 1.0). But the per-task "baseline" is the starter code, and for four no-baseline tasks it's a trivial random predictor — so the 10% threshold is ill-defined there.

contamination

Explicitly designed against, via post-cutoff Kaggle and 2023 datasets — but this conflates uncontaminated with harder, and canonical highs (ogbn-arxiv 87.5, house-price 100) likely lean on memorized solutions.

what's missed

A binary "moved the metric 10%" ignores solution quality, experimental soundness, and reproducibility; only 13 tasks × 8 runs; the 2023-era frontier (Claude v3 Opus, GPT-4-turbo) and its 37.5% headline are already stale.

would raise confidence

Report variance / CIs over the 8 runs; a baseline normalization that handles no-baseline tasks; a held-out set that varies novelty independent of difficulty; human-expert success as an anchor; trace-level grading of whether the method was sound, not just whether the number rose.

// bottom line

MLAgentBench was the first credible testbed for "can an LM agent run the ML experimentation loop," and its honest finding — strong on memorized canonical tasks, ~0% on fresh ones, with hallucination and long-horizon planning as the named failure modes — set the agenda that MLE-bench and the AI-Scientist line built on. Read the 37.5% as a feasibility signal, not a capability measurement.

// takeaways

What to remember

  1. The first benchmark to evaluate LM agents on the full ML experimentation loop — read → edit → execute → iterate — with an automatic metric-improvement evaluator over 13 tasks spanning canonical datasets to post-cutoff Kaggle challenges.
  2. The best agent (Claude 3 Opus) reaches 37.5% average success but ranges 100%→0%; the cliff between canonical and recent-Kaggle tasks is the story, and hallucination + long-horizon planning are the named failures. The Fact Check and Research Plan fields are the design's antidote.
  3. Reliability, not peak capability, is the bottleneck: at ~26% success the expected cost to actually finish a task is ~$231 even though one full sweep is ~$60 — and a binary "10% over baseline" metric hides solution quality and run-to-run variance.
