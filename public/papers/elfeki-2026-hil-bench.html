HiL-Bench (Human-in-Loop Benchmark): Do Agents Know When to Ask for Help?

Frontier coding agents solve fully-specified tasks at 75–89% and collapse to 4–24% the moment a few critical details go missing — not because they can't code, but because they don't notice what they don't know. HiL-Bench withholds that information and measures whether the agent asks.

Trinh, Elfeki, Luo, Luu, Hunt, …, Castillo, Liu (Scale AI)

// the gap

What problem is this solving?

An experienced engineer handed a vague ticket does not start coding. They resolve what they can and, when a requirement admits dozens of equally plausible readings and guessing wrong wastes hours, they ask before committing. The authors call this judgment selective escalation: recognizing, mid-task, that a gap cannot be closed by exploration or inference alone and must be surfaced to someone with more context. Frontier agents already ship the machinery for it — Claude Code exposes an AskUserQuestion tool, Codex and Cursor have interactive modes — yet they rarely fire it at the right moment. Faced with an unclear spec, they fill the gap with a confident assumption and produce plausible but wrong output. As Karpathy put it, agents "don't ask humans, lack the right context, and try to one-shot everything."

This silent, confident wrongness is a prime suspect behind the >90% failure rate reported in enterprise agent pilots. And current benchmarks are blind to it — worse, they select against catching it. SWE-bench and its descendants, HumanEval, BIRD-SQL: all hand the agent a complete, unambiguous spec and reward execution correctness only. An agent that silently assumes its way past a missing requirement and gets lucky scores identically to one that would have asked. The result is a misleading loop: high scores drive deployment, deployment surfaces the judgment failures the benchmark never tested, and the pilot-failure rate persists because the evaluation and the failure mode live in different worlds.

// the bet

The bottleneck is not raw capability — every model evaluated can code and can escalate. It is judgment: knowing when to act autonomously and when to ask. HiL-Bench is the first benchmark designed to isolate and measure that single skill.

// how it works

The method

HiL-Bench takes well-specified tasks from established benchmarks and turns them into judgment challenges. It draws from two domains where agents are actually being deployed: SWE-Bench Pro for repository-scale software engineering (Python, Go, JavaScript, TypeScript) and BIRD for text-to-SQL over real databases. Crucially, tasks are pre-filtered so that top models reach ~85% pass@3 with full information "put back" — guaranteeing that any drop under blocked conditions reflects judgment, not a capability ceiling.

For each task, trained domain-expert annotators inject 3–5 realistic blockers — pieces of critical information removed or obscured from the spec and stored in a hidden blocker registry. There are three types, mirroring distinct production failures: missing information (42% — a required value simply absent, e.g. an unspecified parser fallback, or which of seven rating columns is the "overall" one), ambiguous requests (36% — multiple valid interpretations, e.g. handle version epochs "appropriately," or which countries count as "the Middle East"), and contradictory information (22% — two requirements that cannot both hold, e.g. a query asking for "northern California schools" that then names southern ones).

Every blocker must clear seven quality criteria, any one violation rejecting it: realism, criticality (it must prevent correct completion), objectivity (a single unambiguous resolution), vast search space (can't be guessed or brute-forced — common defaults like "3 retries, port 8080" are invalid blockers), independence (resolving one never reveals another), no contamination (the answer exists only in the registry, not the repo/schema/tests), and non-contrived. Each task survives 5–6 rounds of independent human audit plus automated structural checks.

Progressive discovery

The design decision that separates HiL-Bench from prior clarification benchmarks: blockers surface through execution and exploration, not upfront inspection. The agent must begin working, hit a gap it cannot resolve, ask a targeted question, incorporate the answer, and continue — the incremental uncertainty-resolution cycle of real engineering. This kills the degenerate strategy of front-loading all questions before starting. A spec-only ablation confirms it bites: give Claude Opus 4.6 the task and ask_human() but strip its environment tools, and blocker recall falls from 61% to 11% — the gaps only become visible after you've touched the schema or the code.

The ask_human() oracle and the Ask-F1 metric

Agents ask via a tool, ask_human(question) → str, backed by a frozen Llama-3.3-70B-Instruct acting as a semantic judge: it compares the question against the registry's trigger phrasings and returns the exact resolution if the question targets a registered blocker, else the fixed string "irrelevant question". That yields a binary, reproducible signal per question (validated at 97% precision / 91% recall against human-labeled pairs). Tasks are admitted only if they pass two automated gates — necessity (≤5% pass without the tool) and sufficiency (~90% pass with all resolutions handed over).

Because both failure directions are costly — never asking yields confident garbage; asking about everything is slower than doing the work yourself — the core metric is Ask-F1, the harmonic mean of question precision (fraction of questions that are relevant) and blocker recall (fraction of blockers resolved):

Prec = |Qrel||Q|,   Rec = |Baddr||B|,   Ask-F1 = 2 · Prec · RecPrec + Rec // the harmonic mean architecturally defeats question spam

The harmonic mean is the point: an agent that hits 80% recall by firing 50 questions per task (8% precision) scores a dismal 14.5%. Spamming your way to recall is structurally impossible. And since no HiL-Bench task can be passed without resolving every blocker, recall directly governs pass rate while precision keeps the human-agent collaboration viable. The final dataset is 300 tasks (150 SWE, 150 SQL), 1,131 blockers (avg 3.77/task), split into 200 public and 100 private held-out tasks.

// what they found

The results

The central finding is a large, universal judgment gap. Under full information, models reach 86–91% pass@3 on SQL and 64–88% on SWE. Once they must decide when to invoke ask_human(), the best model reaches only 38% on SQL and 12% on SWE; Ask-F1 averages 40.5% (SQL) and 37.4% (SWE). Near-zero "No Tool" baselines confirm the tasks genuinely require clarification — so the collapse is judgment, not capability. On the precision–recall plane, no model lands in the well-judged quadrant, and no model exceeds 50% recall on SWE.

The decomposition shows models fail in structurally distinct, model-level ways — stable signatures that persist across both domains, classified from 3,600+ failure traces:

  • GPT-5.4 / GPT-5.3 Codex — confident execution on wrong beliefs. Accuracy-dominant: on SQL, 73–93% of tool-use failures are the right tool with wrong parameters, and 76–88% of logic failures are wrong beliefs applied to reasoning. The pattern is barely altered by ask_human() — they so rarely detect a gap that the escalation option goes unused.
  • Claude Opus 4.6 — uncertainty detection without resolution. In 45% of alignment failures the agent explicitly recognizes in its trace that it is stuck, then submits anyway. It's the only model that verbalizes a task is infeasible — but detection doesn't convert to action: it is completion-heavy (82% of tool-use failures), exploring extensively without ever executing the critical step, burning 2–5× the tokens of any other model.
  • Gemini 3.1 Pro — domain-sensitive, externally correctable. Largest swing between domains. On SQL it has the highest logic self-assessment failure rate but responds strongly to grounding — tool-use completion failures drop 38pp once ask_human() is available (inverting to 82.4% accuracy errors). On SWE that correctability vanishes.

The takeaway from the failure analysis: help-seeking tools reshape failure topology, they don't uniformly fix it. Selective escalation is a skill layered on top of each model's existing failure signature, not a switch capability alone can flip.

Judgment is trainable

If models fail because judgment is undertrained rather than untrainable, a reward signal for ambiguity detection and targeted escalation should help. The authors decompose Ask-F1 into a shaped reward — a dense per-step term (+0.3 if a question targets a registered blocker, −0.1 if irrelevant or duplicate) shaping precision, and a terminal coverage term (|Bdiscovered|/|B|, gated to be 0 if nothing was asked) shaping recall — and fine-tune Qwen3-32B with LoRA via SkyRL (120 train / 30 held-out tasks per domain).

Three findings land: the shaped reward improves both precision and recall on held-out tasks; improved Ask-F1 tracks improved pass@3 in lockstep (validating that help-seeking quality governs completion); and cross-domain transfer is positive — train only on SQL, gain on SWE. The model doesn't learn SQL- or SWE-specific heuristics for when to ask; it learns to detect unresolvable uncertainty and act on it. Notably, on the full combined leaderboard even the newest frontier model, GPT-5.5, tops out at just 29% combined pass@3 / 53% Ask-F1 — still nowhere near the well-judged corner.

// does it hold up?

The eval read

The benchmark's central design choice is its strongest move: by pre-filtering source tasks to ~85% pass@3 with full information restored, HiL-Bench genuinely isolates judgment from capability, so the collapse to single digits can't be hand-waved away as "the task was just hard." The model panel is current and broad (GPT-5.3/5.4/5.5, Claude Opus 4.6/4.7, Gemini 3.1 Pro, plus open GLM-5.1, Kimi-k2.6, Grok-4.20, Minimax-M2.5), and the Ask-F1 metric is a real contribution — the harmonic mean structurally closes the "spam questions for recall" loophole that inflates detection-only evaluations. The headline is clean and damning.

But several things temper the trainability claim and the realism of the setup. The RL result rests entirely on a single 32B open model (Qwen3-32B) — impressive that judgment moves at all, but there's no evidence a frontier model can be trained to close the gap, so "judgment is trainable" is shown at small scale only. More fundamentally, the ask_human() oracle is an idealized collaborator: it always answers correctly, instantly, and for free. Real humans are slow, noisy, and sometimes wrong, and the benchmark measures whether to ask, never how to ask a confused stakeholder or recover from a bad answer. With single-slot, single-round minimal ambiguity, the authors themselves note the reported rates are a floor — which cuts both ways: it makes the collapse more alarming, but it also means pass@3 under a never-lying oracle likely overstates real-world recoverability.

baselines

Strong and current — frontier + open models, all under one SWE-Agent scaffold. Source tasks deliberately tuned to ~85% full-info pass@3 to isolate judgment. The RL side, though, is one 32B model; no frontier-scale training demonstration.

contamination

Actively defended: uses SWE-Bench Pro (not the memorized Verified), "no contamination" is a hard blocker criterion, commit history is squashed, and the vast-search-space rule blocks guessing. Risk: 200/300 tasks are public, so the public split will erode over time; only the 100 private tasks stay clean.

what's missed

The oracle never lies, never stalls, never charges — so cost/latency of asking, noisy answers, multi-round and multi-blocker clarification all go untested. The failure taxonomy leans on an LLM judge (Sonnet-4) reported at κ=0.928 self-agreement, with human calibration only via spot-checks — softer than the ask_human judge's annotated 97/91 validation.

would raise confidence

Train a frontier model on shaped Ask-F1 to show the ceiling actually moves; report human (not self) agreement for the failure judge; stress-test with a noisy/wrong oracle and explicit ask-cost; track the private leaderboard over time to watch for contamination drift.

// bottom line

HiL-Bench is the first benchmark to cleanly isolate selective-escalation judgment with a spam-proof metric, and it lands an alarming, well-controlled result: agents that ace fully-specified work fall to 4–24% when they must notice what they don't know, and that failure is a stable per-model fingerprint rather than noise. The trainability evidence is promising but thin, and the never-lying oracle means the numbers are a generous floor on a harder real-world problem — which only sharpens the indictment of today's confidently-wrong agents.

// takeaways

What to remember

  1. The judgment gap is enormous and universal. 75–89% pass@3 with full information collapses to 4–24% when agents must decide whether to ask — despite having a clarification tool the whole time. The wall is knowing what you don't know, not capability.
  2. Ask-F1 is spam-proof by construction. As the harmonic mean of question precision and blocker recall, hitting 80% recall via 50 questions (8% precision) scores just 14.5%. No frontier model — not even GPT-5.5 — reaches the well-judged quadrant.
  3. Failure has a model-level fingerprint. GPT executes confidently on wrong beliefs (76–93% accuracy failures); Claude detects uncertainty but submits anyway (45% self-assessment failures); Gemini is externally correctable, but only on SQL. Stable across domains — these are baked-in behaviors, not task artifacts.
  4. Judgment is trainable. Shaped Ask-F1 RLVR lifts Qwen3-32B's Ask-F1 +28pp (SQL) / +17pp (SWE) with pass@3 moving in lockstep, and gains transfer across domains — the model learns to detect unresolvable uncertainty, not domain-specific tricks.
