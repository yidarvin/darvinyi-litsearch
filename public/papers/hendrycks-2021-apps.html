Measuring Coding Challenge Competence With APPS

Can a model turn an arbitrary natural-language problem statement into working Python? APPS poses 10,000 coding-challenge problems across three difficulty tiers and grades models the way a judging site grades humans — by executing their code on hidden test cases. In 2021 the best model solved under 4% of the easy ones.

Hendrycks, Basart, Kadavath, Mazeika, Arora, Guo, Burns, Puranik, He, Song, Steinhardt

What problem is this solving?

By 2021, language models were posting strong numbers on reading comprehension, commonsense, and even law-and-medicine knowledge — but nobody had a rigorous way to ask whether they could actually write code. The code-generation benchmarks that existed measured something narrower and easier. CodeXGLUE's CONCODE generated Java from short docstrings; SPoC converted line-by-line pseudocode into C++; others did code translation between languages. In each, the model was handed something already close to code, and the output was scored with text-overlap metrics like BLEU.

That setup misses the thing that makes programming hard. A real engineer reads an unrestricted natural-language specification — often long, sometimes with edge-case caveats and mathematical notation — and has to devise an algorithm, pick data structures, and produce a program that is not just plausible-looking but correct. APPS argues that two gaps need closing at once: the task (long natural-language spec → general-purpose program, not pseudocode → code), and the metric (functional correctness checked by execution, not BLEU, which the paper shows is sometimes anticorrelated with whether the code runs).

APPS — the Automated Programming Progress Standard — was built as a deliberately hard, contamination-aware, difficulty-graded yardstick for exactly this, so that progress on program synthesis from natural language could be tracked objectively as models scaled.

// how it works

The method

APPS is 10,000 problems manually curated from open coding sites — Codewars, AtCoder, Kattis, and Codeforces — where programmers already pose and judge challenges. Several graduate and undergraduate authors polished the set over six months: custom HTML parsers per source preserved LaTeX, lists, and formatting; equation images were converted to LaTeX via the MathPix API; problems relying on figures were dropped; and near-duplicates were removed with tf-idf + SVD cosine-similarity dedup. Problems are self-contained and long — averaging 293.2 words, versus the 14.7-word docstrings of prior Java benchmarks.

The data splits evenly into 5,000 train / 5,000 test, and ships with 131,777 test cases and 232,421 human-written ground-truth solutions — over an order of magnitude more reference solutions than comparable datasets. Each test problem carries on average 21.2 test cases. Crucially, the problems are stamped with one of three difficulty tiers, so a single score doesn't blur a one-line warm-up with an Olympiad-grade challenge:

  • Introductory — solvable by a programmer with 1–2 years of experience without complex algorithms.
  • Interview — harder, data-structure-and-algorithm problems of the kind asked in technical interviews.
  • Competition — the level of Codeforces / USACO / collegiate contests.

Grading by execution, two ways

Because the objective is correct programs, APPS does not score text overlap. It runs the generated code against the test bank and reports two metrics. Test Case Average is the mean fraction of a problem's test cases that pass — a lenient view that still gives credit to code that handles the common case but misses a corner. Strict Accuracy is the fraction of problems for which every test case passes — the eventual gold standard.

TestCaseAvg =  1P Σp  1Cp Σc 𝟙{ eval(codep, xp,c) = yp,c } // fraction of test cases passed, averaged over problems
StrictAcc =  1P Σp  Πc 𝟙{ eval(codep, xp,c) = yp,c } // the product over test cases — one failure zeroes the problem

The models evaluated are GPT-2 (0.1B and 1.5B) and GPT-Neo 2.7B, fine-tuned on the APPS training set, plus GPT-3 175B probed few-shot (its weights being closed, it could not be fine-tuned). Since GPT-2 was never pretrained on code, the authors first further-pretrained it on 30 GB of Python scraped from GitHub repos with more than one star — decontaminated by removing any repo matching common-exercise keywords or sharing function signatures with APPS starter code.

// what they found

The results

Programming is very hard for 2021 models — but no longer at zero. The best system, fine-tuned GPT-Neo 2.7B, passed 14.68% of test cases on Introductory problems and reached 3.90% strict accuracy there; on Competition problems every model scored 0.00% strict accuracy. Accuracy falls monotonically with difficulty and rises with both fine-tuning and model size — and tellingly, the jump from GPT-2 1.5B to GPT-Neo 2.7B is larger than the jump from 0.1B to 1.5B.

The most counter-intuitive result: few-shot GPT-3 175B is beaten by fine-tuned models two orders of magnitude smaller. GPT-3 passed just 0.57% of introductory test cases and solved only 3 of 5,000 problems outright — two introductory and one interview. Part of this is mechanical: roughly 59% of GPT-3's introductory solutions had outright syntax errors, where GPT-Neo's rate was about 3%.

Two more findings shaped how the field would measure code afterward. First, sampling more candidates helps a lot: with beam search width 5, evaluating all five beams (Top-5) lifts GPT-Neo's introductory test-case average from 14.7% to 19.9% and strict accuracy from 3.9% to 5.5% — "almost 20% of test cases given five attempts." Second, BLEU is a broken proxy for code: it rises with difficulty even though accuracy falls, and ranks GPT-2 0.1B (BLEU 26.8 / 29.7 / 30.2 across tiers) above the far stronger GPT-Neo 2.7B (27.1 / 29.1 / 29.3) — exactly backwards. The paper also previews where this goes next: it cites the contemporaneous Codex 12B (Chen et al.), which on these same introductory problems reaches 4.14% top-1 / 9.65% top-5, and 25% when scaled to 1,000 samples — better, but still far from solved.

// does it hold up?

The eval read

As a benchmark design, APPS aged extraordinarily well — execution-graded, difficulty-tiered, natural-language-spec code evaluation is now the default, and the SWE-bench / BigCodeBench / Code Llama lineage all build on it. But the 2021 evaluation inside the paper has soft spots worth naming, and they are exactly the kind a careful reader should not wave through.

Start with the headline contrast — "fine-tuned 2.7B beats few-shot 175B." It is real, but it is apples-to-oranges by construction: GPT-3 was probed few-shot only (its weights were closed), while the small models were fine-tuned on 5,000 in-distribution problems. And GPT-3's collapse is partly a formatting artifact — the authors themselves note that improper indentation of otherwise-functioning code triggered syntax errors, and that ~59% of its introductory outputs failed to parse. So the comparison says less about parameter count than the framing suggests; a fine-tuned GPT-3 was simply never run.

The sharpest confound is asymmetric decontamination. The paper's strongest model, GPT-Neo, was pretrained on the Pile, whose GitHub slice was not run through APPS decontamination — while the GPT-2 models were decontaminated. The authors anticipate the memorization worry and argue against it (performance tracks difficulty rather than being uniform), which is a fair point, but it doesn't fully neutralize the fact that the leaderboard's leader had a data-cleanliness advantage its rivals did not.

Then the metrics. Test Case Average flatters. "Passes ~15–20% of test cases" reads as progress, but strict accuracy — actually solving the problem — is ~4–5.5% on the easiest tier and identically 0.00% on Competition for every model, which means the hardest third of the benchmark produced no measurable signal in 2021 (a floor effect, not a measurement). And the strict metric has a small leak the authors honestly surface: among problems with only 2 test cases that GPT-Neo "passed," 8 of 12 were false positives (small output spaces like YES/NO let a wrong program guess both). They bound the overall false-positive rate under 1% because most problems have many tests — but the thin-test problems are a genuine soft edge in the gold metric.

baselines

GPT-3 175B is few-shot-only against fine-tuned small models — and its score is depressed by parse failures, not just weak reasoning. Sampling is limited to beam-5 (Top-5), not the temperature-sampled pass@k that Codex would standardize months later.

contamination

GPT-2 was decontaminated against APPS; the winning GPT-Neo's Pile/GitHub data was not. The difficulty-tracking argument mitigates but doesn't erase the asymmetry.

what's missed

Test-case average overstates competence relative to strict accuracy; Competition sits at 0.00% strict (no signal); thin 2–3-test problems admit false positives (8/12 on 2-test problems).

would raise confidence

Fine-tune a large model; proper pass@k with sampling; decontaminate Neo's pretraining; thicker test suites on small-output problems; per-tier confidence intervals on strict accuracy.

// bottom line

The 2021 numbers are a snapshot of a field at near-zero, and several of the within-paper comparisons are confounded enough that you shouldn't over-read them. But the instrument — long NL specs, execution grading, three honest difficulty tiers, hundreds of thousands of tests — is the part that mattered, and it held up: APPS became a load-bearing benchmark for the coding-model era it predicted.

// takeaways

What to remember

  1. APPS reframed code evaluation around functional correctness. 10,000 problems, 131,777 test cases, 232,421 human solutions, graded by execution rather than BLEU — and the paper directly demonstrates BLEU ranking a worse model above a better one.
  2. Difficulty tiers turned one number into three honest ones. Introductory / Interview / Competition cleanly separated "models are starting to code" (Intro 3.9% strict) from "models cannot yet" (Competition 0.00% strict).
  3. It called the trajectory before the tools arrived. Fine-tuning beat few-shot, more samples beat one, syntax errors were already vanishing — and the cited Codex 12B (4.14%→25% with 1,000 samples) showed code-specialized scale was the next lever. APPS became the substrate the SWE-bench / Code Llama / BigCodeBench line was built on.
