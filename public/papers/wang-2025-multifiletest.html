MultiFileTest: A Multi-File-Level LLM Unit Test Generation Benchmark and Impact of Error Fixing Mechanisms

Existing unit-test benchmarks hand a model one function or one file. This one hands it a whole small project — and asks how much of the failure is genuine inability versus a two-line import bug a human could fix in seconds.

Yibo Wang, Congying Xia, Wenting Zhao, Jiangshu Du, Chunyu Miao, Zhongfen Deng, Philip S. Yu, Chen Xing · UIC / Salesforce Research / Scale AI

// the gap

What problem is this solving?

Generating unit tests is one of the most concrete, immediately useful things you can ask a coding LLM to do — and one of the easiest to over-rate, because almost every benchmark grades it on toy inputs. HumanEval and TestEval test a model against a single function. ClassEval scopes it to a single class. TestGenEval stays inside a single file. In all of these, the code under test fits in one screen and has no dependencies the model has to chase down.

Real software does not look like that. A function in stock.py imports a class from structure.py that itself depends on validate.py. To write a test that even runs, the model has to resolve the right import paths, set up the test environment, and reason across files — exactly the part that single-file benchmarks erase. The one prior benchmark that touched multi-file unit testing, DevBench, was built for general software-development breadth, not test generation: only ~16 of its projects are genuinely multi-file for unit testing, the dataset quality is uneven, and many of its projects are hard to even trace back to source.

MultiFileTest (originally released as ProjectTest) fills that exact slot: a benchmark designed for unit-test generation, where cross-file dependencies are a guaranteed property of every project — and the first to pair that with a systematic study of what breaks and how much error-fixing recovers.

// how it works

The method

The dataset is 60 projects — 20 each in Python, Java, and JavaScript — hand-curated from GitHub under three rules: appropriate size (2–15 files, under 1,600 lines of code so the whole project fits in a context window without truncation), real inter-file dependencies, and reliable provenance (high stars/forks, permissive licenses). Projects extracted from larger codebases are reworked to be self-contained — import paths and naming adjusted so they run standalone. Projects average ~4.9 files (Python 6.10, Java 4.79, JavaScript 4.00). That deliberately keeps the test at "reason across a few files" and not "manage a million-token repo" — the authors argue retrieval and chunking would otherwise become confounds.

Each model gets the entire project plus a language-specific, zero-shot prompt (temperature 0) and must emit a full test suite. Suites are then run for real — pytest for Python, JUnit+JaCoCo for Java, Jest for JavaScript — and scored on four interdependent metrics:

ER = |Xer||X| CRx = |Txcor||Tx| Covx = covered(x)total(x) // ER = executability rate (suites that compile/run) · CR = correctness rate (tests that pass) · Cov = line / branch coverage

The metrics are chained: if a suite has an executability error, none of its tests run, so correctness and coverage collapse to zero for that whole project. That coupling is the paper's lever. Rather than report only the raw numbers, MultiFileTest evaluates every model in three scenarios:

  • Vanilla — tests exactly as the model emitted them. Raw capability.
  • Manual fixing — CS PhD annotators apply minimal, mechanical fixes (averaging 2–6 lines/project) to clear executability and "cascade" errors, following an objective per-fix checklist, while preserving all test logic and assertions. This forces ER to 100% and exposes the potential hiding behind trivial bugs.
  • LLM self-fixing — the model is handed the error messages and the conversation history and asked to repair its own tests. Can it do what the human annotator did?

Two error classes drive the gap between vanilla and fixed. Executability errors stop a suite cold (a hallucinated import path → ModuleNotFoundError, mismatched parentheses, missing Java package declarations). Cascade errors let tests run but knock out many at once from a single root cause (one missing import numpy, an unmocked file read). Both are "easy" — and both, left in place, make a capable model look incapable.

A fourth metric, unique contribution, measures the coverage each test adds that no other test in the suite already covers — a guard against models that inflate coverage by dumping many redundant tests.

// what they found

The results

Across 11 frontier models — 7 closed (GPT-4-Turbo, GPT-3.5-Turbo, GPT-o1, GPT-5-mini, Gemini-2.0-Flash, Gemini-3.0-Pro, Claude-3.5-Sonnet) and 4 small open ones (CodeQwen1.5-7B, DeepSeek-Coder-6.7B, CodeLlama-7B, CodeGemma-7B) — vanilla performance is mediocre. The best overall model, Gemini-3.0-Pro, manages a 77% correctness rate on Python but only 62% on Java; GPT-o1 tops JavaScript at 87% but trails on Java (41%). Expertise is sharply language-dependent, and Java is hardest — its stricter syntax sinks executability rates, and four models (CodeQwen1.5, CodeLlama, CodeGemma) score a flat 0 on Java because they can't produce a single runnable test.

The headline: a lot of "failure" was a two-line bug

When annotators clear executability and cascade errors — averaging just 2–6 lines per project — executability is forced to 100% and correctness jumps everywhere. The gains are largest exactly where vanilla was worst: Java benefits most. Models that scored 0 on vanilla Java leap to 60 (CodeQwen1.5, +60), 57 (CodeGemma, +57), and 36 (CodeLlama, +36) in correctness. Strong closed models climb too — Python GPT-o1 to 89 (+29), Claude-3.5-Sonnet to 92 (+28), Gemini-3.0-Pro to 90 (+13). Crucially, rankings reorder after fixing: CodeQwen1.5 goes from worthless to GPT-4-Turbo-parity on Java. The vanilla leaderboard was partly measuring "who avoids trivial import bugs," not "who writes good tests."

Self-fixing: closed models can, small open models collapse

Asked to repair their own tests from the error trace, most closed models self-fix effectively — Gemini-3.0-Pro reaches 91% correctness on Python (vs. 90% with human fixing), Claude-3.5-Sonnet 86%. But self-fixing lags human fixing almost everywhere (the parenthesized deltas in Table 4 are mostly negative), because the model can't guarantee executability the way a human can: post-self-fix executability rates sit at 60–95%, not 100%. The small open models break down — DeepSeek-Coder's Java executability falls to 20% (−80), CodeLlama's Python to 5% (−95) — often because they emit prose "instructions for fixing" instead of corrected code, or run out of context. One revealing exception: on JavaScript, CodeQwen1.5 self-fixes better than human fixing (+23 CR) — its vanilla failure was misreading the prompt and emitting no tests at all, which a human minimal-fix can't repair but an error message ("no tests found") can prompt the model to fix itself.

Even the winners write redundant tests

Unique-contribution (Table 5) is low across the board: models pad coverage with overlapping tests. GPT-o1 has higher coverage than GPT-5-mini but lower unique contribution (6.75 vs. 15.10) — it buys coverage with volume (36 tests/project vs. 13), not efficiency. The persistent, cross-language errors are all about not understanding codebase structure: hallucinated functions/classes, missing required dependencies, and — after fixing — expected-vs-actual value mismatches traced to weak reasoning about what the code should return.

// does it hold up?

The eval read

The three-scenario design is the paper's real contribution and it is genuinely sharp: separating raw output from "potential after a 2-line fix" is exactly the right move for a metric (executability) that's brutally all-or-nothing. The reordering of rankings after manual fixing is a result you can't get any other way, and it should make anyone reading a vanilla test-gen leaderboard suspicious. But several things keep the headline numbers from being load-bearing on their own.

First, scale and statistical power. Twenty projects per language is small, and every cell in Tables 2–4 is an average over those 20. A "+13 vs. +29" difference, or a 4-point CR gap between two models, sits well inside the noise you'd expect from 20 heterogeneous projects — and the multi-run variance check (Table 8) is run for one model on one language, not the leaderboard that the conclusions rest on. Second, and more fundamental: the <1,600-LOC cap deliberately removes the hardest part of real multi-file testing — long-context navigation, retrieval, knowing which of fifty files matter. The authors are explicit and reasonable about why (avoid confounds), but it means the benchmark measures cross-file reasoning in miniature; "multi-file" here is 2–15 files that fit in one prompt, not a production repository.

Third, the manual-fix scenario is the centerpiece and the least reproducible part. "Minimal, mechanical, 2–6 lines, objectively verifiable" is a good-faith protocol with a checklist — but whether a given fix counts as "mechanical" (vs. quietly rewriting a broken test) is a human judgment applied by SE PhD students, and the inter-annotator reliability of that boundary is never measured. Since the entire "potential" story rides on those edits, an agreement study would materially raise confidence. Fourth, contamination is unaddressed. The projects are high-star public GitHub repos — precisely the distribution most likely to be in pretraining — and there's no decontamination check, held-out/fresh split, or memorization probe. The self-contained reorganization (renamed imports, restructured files) may perturb memorized content, but that's a side effect, not a measured defense, so vanilla numbers carry an unknown amount of leakage.

Finally, on what the metrics certify: "correct" means a test passes against the reference implementation, which rewards tests that are true but says nothing about whether they're strong. There's no mutation score or fault-detection signal (the way ULT/PLT report), so a model can score high CR with assertions that can't catch a real bug; unique-contribution helps on redundancy but is still coverage-derived. And the model roster spans wildly different eras and sizes (GPT-3.5-Turbo next to Gemini-3.0-Pro; 7B open models against frontier closed ones) under single-shot zero-shot prompting — so "open models fail" is partly a size-and-vintage story, and none of this reflects how test-gen is actually deployed (agentic, iterative, tool-using).

baselines

Sensible benchmark lineage (HumanEval → ClassEval → TestGenEval → DevBench), and Table 15 makes the niche clear. But the model roster is a grab-bag of eras/sizes under zero-shot single-shot prompting — no agentic/iterative scaffold, so external validity to real test-gen workflows is limited.

contamination

Not addressed. High-star public GitHub repos are prime training data; no decontamination, fresh split, or memorization probe. The self-contained rewrite perturbs but doesn't measure leakage, leaving vanilla numbers with unknown inflation.

what's missed

No test-strength metric (mutation / fault detection) — "correct" rewards passing, not bug-catching. Manual-fix scenario has no inter-annotator agreement study despite carrying the central claim. 20 projects/language is thin for the model-vs-model deltas drawn from it.

would raise confidence

Per-cell variance across the full leaderboard; an inter-annotator agreement number for "mechanical" fixes; a contamination probe or a held-out fresh-repo split; a mutation/fault-detection metric; and at least one agentic baseline to anchor the zero-shot ceiling.

// bottom line

A genuinely useful diagnostic, not yet a definitive leaderboard. The vanilla-vs-fixed framing convincingly shows that much of LLM test-gen "failure" on multi-file code is trivial, recoverable executability bugs — and that small open models still can't self-recover. Trust the qualitative story (Java is hardest; a few lines unlock big gains; self-fixing trails human fixing) more than the exact rank order, which rests on 20 projects per language, a contamination-blind sample, and a human-fix protocol whose reproducibility isn't quantified.

// takeaways

What to remember

  1. Multi-file is the missing axis. 60 self-contained projects (20 each in Python/Java/JavaScript, <1,600 LOC) make cross-file dependency reasoning a guaranteed property — the part single-function and single-file test benchmarks erase.
  2. Separate raw output from potential. Evaluating vanilla, human-fixed, and self-fixed suites shows that a 2–6 line mechanical fix can swing correctness from 0 to 60 — so vanilla leaderboards conflate "can't write tests" with "wrote a hallucinated import." Java, the hardest language, gains the most from fixing.
  3. Self-fixing is a capability cliff. Closed models repair their own tests nearly as well as humans (Gemini-3.0-Pro 91% CR on Python); small open models collapse (CodeLlama Python executability −95), often emitting prose instead of corrected code.
  4. Read it as a diagnostic, not a verdict. 20 projects/language, no contamination control, no test-strength metric, and an unquantified human-fix protocol mean the qualitative findings travel further than the exact rankings.
