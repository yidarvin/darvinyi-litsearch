ScienceAgentBench: Toward Rigorous Assessment of Language Agents for Data-Driven Scientific Discovery

102 tasks lifted straight out of 44 peer-reviewed papers in four sciences, each reduced to "write one self-contained Python program" — and the best agent, given three tries, still only solves 34.3% of them.

// the gap

What problem is this solving?

By late 2024, a wave of papers had started claiming language agents could automate science end-to-end. The most prominent, Sakana AI's "The AI Scientist" (Lu et al., 2024), described a system that generates ideas, runs experiments, and writes up papers with no human in the loop — a claim the ScienceAgentBench authors call out by name as exactly the kind of assertion that needs scrutiny before anyone believes it. The trouble is that evaluating a claim like "fully automates research" only at the end-to-end level — e.g., having an LLM judge grade a generated paper — tells you almost nothing about which individual steps the agent can actually do versus which it is bluffing through or getting lucky on.

At the same time, the benchmarks that could isolate individual steps didn't fit the job. Software-engineering benchmarks like SWE-bench test GitHub-style bug fixes, not scientific data analysis. ML-oriented benchmarks like MLAgentBench or ML-Bench sit closer but pull from Kaggle competitions or generic GitHub repos rather than the heterogeneous, messy data — cell images, molecular graphs, geospatial layers, EEG time series — that actually shows up in bioinformatics, computational chemistry, geographic information science, or psychology research. And none of the existing options had done serious work on two problems that matter a lot for LLM-based evaluation specifically: agents taking shortcuts (e.g., reading ground-truth test labels directly instead of training a model) and data contamination (the benchmark's own public code and data having leaked into a model's pretraining corpus).

ScienceAgentBench's answer is to build a benchmark of individual, verifiable tasks — one per essential step of a real data-driven discovery workflow — sourced directly from the code and data behind actual peer-reviewed publications, and to engineer the evaluation harness specifically to catch the failure modes that matter for agents, not just for humans.

// how it works

The method

Nine graduate-student annotators across four disciplines — Bioinformatics, Computational Chemistry, Geographical Information Science, and Psychology & Cognitive Neuroscience — each start from a peer-reviewed paper that released its code and data under a permissive license, then adapt one self-contained, documented code example into a benchmark task. Every task is reduced to the same shape: given a natural-language instruction, a dataset (with directory listing and content preview), and optional expert-provided domain knowledge, the agent must produce one Python program file that an evaluator can run independently.

The pipeline that gets from "110 candidate tasks" to "102 final tasks" is deliberately heavy: four discarded outright for excessive runtime or brittle environment setup, then a full round of validation by nine subject-matter experts (senior PhD students and professors) who check realism, revise 41 instructions, and cut three more tasks as unrepresentative; then a second annotator-verification pass that refines 29 annotations and drops one more task whose result couldn't be reproduced due to randomness. What survives is 102 tasks, each with four components shown concretely in the ClinTox drug-toxicity example below.

Two contamination-mitigation tricks are built directly into the data: for most datasets, five test-set data points are removed at random, so an LLM that recites a memorized public data-loading routine will silently produce misaligned results and fail; and for model-development tasks, the dataset is re-split and the test labels are replaced with dummy placeholders (e.g., -1) visible only at evaluation time, so an agent that tries to shortcut by reading ground-truth labels directly — a failure mode the authors observed OpenHands actually doing — gets caught rather than credited.

Evaluation itself runs each generated program in a per-task conda environment (auto-configured via pipreqs + pip-tools) and scores it on four axes: Valid Execution Rate (VER, did it run and save the right file), Success Rate (SR, did the output meet the task's specific success criteria — a binary pass/fail gated on VER), CodeBERTScore (CBS, embedding similarity to the gold program, snapped to 1.0 whenever SR=1), and API Cost in USD. Figure-producing tasks are additionally graded by GPT-4o as a judge (borrowing the prompting approach of prior figure-quality-judging work), averaged over 3 samples for stability. On top of all this, a fine-grained, five-stage rubric (Data Loading → Data Processing → Modeling/Visualization → Output Formatting → Output Saving) — drafted by GPT-4o and refined by the domain expert per task — is used for a supplementary human evaluation of the best agent's outputs, assigning partial credit even to programs that ultimately fail the binary success criteria.

Three agent frameworks are tested per LLM: Direct Prompting (one-shot code generation, no execution), OpenHands CodeAct v1.9 (a generalist tool-use framework with a Python interpreter, bash shell, and web browser), and a re-implemented Self-Debug (execute → read the error → revise, with early exit if two consecutive attempts produce the same program). Each task gets three independent runs per configuration, and the run reported is chosen by a fixed tie-break order: maximize SR, then VER, then CBS, then minimize Cost.

// what they found

The results

Across five LLMs (Llama-3.1-Instruct 70B/405B, Mistral-Large-2, GPT-4o, Claude-3.5-Sonnet), the best result comes from Claude-3.5-Sonnet with Self-Debug: 32.4% SR without expert knowledge, 34.3% SR with it. Direct Prompting alone caps out far lower — Claude-3.5-Sonnet solves only 16.7% (no knowledge) / 20.6% (with knowledge) without any execution feedback, confirming that letting an agent see and react to its own runtime errors is doing most of the work: Self-Debug very nearly doubles Claude's SR versus Direct Prompting (16.7 → 32.4, a 1.94× multiplier) at a fraction of OpenHands's cost.

That cost gap is stark and is one of the paper's sharper findings: without expert knowledge, Claude-3.5-Sonnet + Self-Debug solves 10.8 more percentage points of tasks than Claude-3.5-Sonnet + OpenHands CodeAct (32.4% vs. 21.6% SR) while spending 17× less in API fees ($0.057 vs. $0.958 per task on average). OpenHands's larger action space and tool surface (bash, file navigation, a web browser) doesn't pay for itself here — for four of five LLMs, Self-Debug's simpler execute-and-revise loop beats OpenHands outright; GPT-4o is the sole exception, apparently because it is comparatively better at using OpenHands's browser and bash tools. With expert knowledge added, Self-Debug still leads Direct Prompting by a wide margin on the same axes (SR 20.6 → 34.3, a 13.7-point/1.67× gain; VER 41.2 → 86.3, a 45.1-point/2.09× gain).

Expert-provided domain knowledge itself is a mixed bag rather than a clean win: within Self-Debug, it reliably improves SR and CBS (Claude-3.5-Sonnet's SR rises 32.4 → 34.3, CBS 86.4 → 87.1), but it also depresses VER for several agents — e.g., Claude-3.5-Sonnet's own VER dips 92.2 → 86.3 once knowledge is added — because knowledge sometimes points the agent at unfamiliar, more specialized tools that it then misuses with hallucinated API calls, rather than sticking to the basic, execution-safe libraries (rdkit, sklearn) it would otherwise default to.

For context, OpenAI o1-preview is reported separately rather than compared head-to-head, since its extra inference-time reasoning tokens and different hyperparameters make the comparison unfair by the authors' own framing: o1-preview reaches 42.2% SR (no knowledge) with Self-Debug — the single highest number in the paper — but at more than 10× the API cost of the other models ($0.636 vs. $0.057 for Claude).

The supplementary rubric-based human evaluation of all 102 Claude-3.5-Sonnet + Self-Debug + knowledge programs reinforces the same story from a different angle: data loading and data processing ratings cleanly separate successful from failed programs (almost all successes get a near-perfect data-loading score, while 25% of failures score below 50), but output-formatting and output-saving ratings show essentially no difference between the two groups — meaning when Claude fails, it is failing on substance (processing the data correctly, building a working model), not on cosmetic formatting.

// does it hold up?

The eval read

The core design is genuinely careful: sourcing tasks directly from peer-reviewed code (rather than synthesizing them or scraping arbitrary GitHub repos) buys real scientific authenticity, and the two contamination countermeasures — dropping test points and swapping in dummy labels — are concrete, low-cost, and specifically targeted at the two failure modes (memorized data loaders, label-reading shortcuts) that plague LLM-agent benchmarks built from public code. That the authors caught OpenHands doing exactly the shortcut behavior they designed against is a good sign the mitigation is pointed at a real problem, not a hypothetical one.

That said, a few things are worth weighing carefully. First, scale: 102 tasks across 4 disciplines is enough to support the paper's headline claims but is thin for the discipline- and sub-task-level breakdowns in Figure 3 — some cells (e.g., "Feature Selection: 2 tasks", "Data Selection: 1 task" in the taxonomy) are single- or double-digit samples, so per-category error rates should be read as suggestive rather than statistically solid. Second, the rubric-based evaluation — one of the paper's three headline design pillars — is explicitly not automated: it's a manual, two-rater human process applied only to the 102 outputs of a single agent configuration (Claude-3.5-Sonnet + Self-Debug + knowledge), and the authors themselves flag "automating this rubric-based evaluation approach, such as developing an LLM-based judge" as future work rather than something this paper delivers. That's an honest disclosure, but it means the "rigorous graded evaluation" pillar rests more on the VER/SR/CBS/Cost battery than on the rubric, and the rubric numbers can't be reproduced at scale by a future user of the benchmark without building that judge themselves.

Third, the "10× more expensive but only 8 points better" o1-preview framing is fair, but it also means the paper never gives a clean apples-to-apples comparison against the frontier of what LLM agents can do — o1-preview is reported and then set aside, so the reader is left extrapolating whether more recent, cheaper reasoning models would blow past 34.3% entirely. Given how fast this space moves, that's a benchmark that will likely saturate its five-model baseline table quickly even if the underlying 102 tasks remain hard.

baselines

Five contemporary (mid-2024) LLMs × three agent frameworks is a reasonable sweep for its time, with o1-preview separated out for a fair reason (fixed hyperparameters, extra inference-time reasoning compute) — but that separation also means no frontier-vs-frontier comparison is offered, and the table will age fast.

contamination

Two concrete, targeted mitigations (test-point removal; dummy test labels) directly built into the dataset, motivated by an observed real shortcut (OpenHands reading ground-truth labels) — one of the stronger contamination stories among code-generation agent benchmarks of this era.

what's missed

The rubric-based fine-grained evaluation — a headline design pillar — is manual, two-rater, and applied to only one agent configuration; it isn't a reusable or automated part of the benchmark yet. Per-discipline/sub-task error breakdowns also rest on small cell counts (single digits in places).

would raise confidence

An automated LLM-judge version of the rubric evaluation applied across all agent/model configurations (which the authors flag as their own next step), plus a rerun against 2025-era reasoning models to see how much of the 34.3% ceiling was model-generation-specific versus task-specific.

// bottom line

A well-constructed, contamination-aware benchmark that backs up its headline number (34.3%) with a real execution harness and a directly relevant comparison table against eight prior agent/code benchmarks — but its most novel-sounding claim, rubric-based graded evaluation, is disclosed by the authors themselves as manual and not-yet-automated, so treat the VER/SR/CBS/Cost battery, not the rubric, as the part of the evaluation design that's actually load-bearing today.

// takeaways

What to remember

  1. 102 tasks distilled from 44 peer-reviewed papers across Bioinformatics, Computational Chemistry, Geographical Information Science, and Psychology/Cognitive Neuroscience — each reduced to "produce one self-contained Python program" and graded on execution + task-specific success criteria, not just text similarity.
  2. Execution feedback matters far more than framework complexity: Self-Debug (simple execute-revise loop) beats both Direct Prompting and the more elaborate OpenHands CodeAct for 4 of 5 LLMs, and does it at up to 17× lower API cost than OpenHands.
  3. Even the best 2024-era setup (Claude-3.5-Sonnet + Self-Debug + expert knowledge) only reaches 34.3% success — and the paper explicitly positions this as a rebuttal to claims (like "The AI Scientist") that LLM agents can already automate scientific discovery end-to-end.
