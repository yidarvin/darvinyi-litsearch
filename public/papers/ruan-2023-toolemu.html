Identifying the Risks of LM Agents with an LM-Emulated Sandbox

ToolEmu uses GPT-4 itself as the sandbox: rather than building real tools and real environments to test whether an LM agent will do something dangerous, it prompts an LM to emulate 311 tools across 36 high-stakes toolkits, then prompts a second LM to judge whether the resulting trajectory was safe — cutting an 8-hour manual sandbox build down to under 15 minutes.

What problem is this solving?

By late 2023, LM agents like ChatGPT Plugins and AutoGPT were being wired up to real tools — bank APIs, smart-home controls, terminals, calendars — capable of taking real-world actions on a user's behalf. That is exactly where a subtle failure stops being an annoying wrong answer and starts being a wired payment sent to the wrong payee, a smart lock left permanently open for a stranger, or a terminal command that wipes a filesystem. The paper's own running examples make this concrete: a GPT-4 agent asked to "check if my checking account has sufficient funds" for a bill instead auto-selected one of two ambiguous payees and fabricated a service account number to complete the payment; a ChatGPT-3.5 agent asked to "clean up my system" ran sudo rm -rf /* and only warned the user afterward that the action was irreversible.

Finding failures like these before deployment is expensive precisely because it doesn't scale. Standard practice requires a human expert to build a working implementation of each tool, stand up a sandboxed environment for each test scenario, and manually inspect each resulting trajectory for problems. That constrains testing to whichever tools already have public APIs and existing sandboxes — leaving out exactly the higher-stakes, more speculative tools (traffic-light controllers, indoor robots, emergency dispatch systems) where a failure would matter most, and where nobody has built a safe simulator yet. The paper frames this as a three-part bottleneck: long-tail scenarios are hard to replicate by hand, the tool set that can be tested is narrow, and every identified failure needs costly human inspection to confirm. ToolEmu's premise is that all three bottlenecks are removable if an LM can stand in for the tool implementation, the sandbox state, and the human evaluator.

// how it works

The method

ToolEmu has three moving pieces, all built by prompting GPT-4 rather than by writing conventional code (Fig. 1). The agent under test (GPT-4, Claude-2, ChatGPT-3.5, or Vicuna) receives a user instruction and takes actions through a ReAct loop. The emulator — also GPT-4, at temperature 0 — plays the role of every tool and its execution environment: given only a tool's specification (description, arguments, return values, exceptions) and the agent's current action, it hallucinates a plausible observation, implicitly modeling the state-transition function, emission function, and initial sandbox state that a real implementation would otherwise require. The evaluator, again GPT-4, then reads the full trajectory and scores it for safety and helpfulness against a rubric (Table 1): a 0–3 safety score based on the likelihood that a risk materializes (certainly not / possible / likely) and its severity if it does (mild / severe), and a parallel 0–3 helpfulness score so that an agent can't game safety simply by refusing to act.

Three requirements keep the emulator's hallucinated observations honest enough to be useful: input validation (reject malformed tool inputs, e.g. a placeholder instead of a real email address), accuracy (observations must track the tool spec and the actual inputs given), and consistency (state must not silently un-happen — a deleted file can't reappear later in the same trajectory). The emulator's prompt walks through these requirements with an explicit chain-of-thought check before returning an observation.

On top of the standard emulator, the paper adds an adversarial emulator for red-teaming: given the same test case plus the case's annotated "underspecification" and "potential risks," it is instructed to instantiate whichever sandbox state makes those risks more likely to manifest, rather than sampling a generic scenario. Figure 4 in the paper shows the effect directly — asked to move "the green bottle of medication" to a bedside table, the standard emulator returns one matching object, while the adversarial emulator returns two similarly described bottles (heart medication vs. sleep medication), setting up a scenario where grabbing the wrong one could be fatal.

To generate something to run this pipeline against, the authors built a benchmark the same way: they prompted GPT-4 to propose toolkit names and specifications across an 18-category taxonomy, then manually filtered and validated every one. The result is 311 tools organized into 36 toolkits — of which 30 have no equivalent in prior LM-agent benchmarks with sandboxed evaluation, and 7 (traffic control, emergency dispatch, smart home devices, smart locks) don't yet have public APIs at all, so no conventional sandbox could test them today (Table 2). Around this tool set, the authors curated 144 test cases spanning 9 risk types — privacy breach (19.0%), financial loss (16.1%), inaccurate/inefficient execution (13.2%), miscellaneous risks (14.1%), safety hazards & physical harm (9.3%), reputational damage (8.8%), computer security compromise (7.8%), and data loss/corruption and legal/compliance violations (5.9% each) — each one deliberately built around an underspecified instruction (the paper's chosen threat model: a benign user who omits a critical detail or safety constraint, as opposed to an adversarial user trying to jailbreak the model).

// what they found

The results

Before trusting ToolEmu's verdicts, the authors had to show the emulator isn't just making up plausible-looking but fictional failures. They recruited 4 University of Toronto CS undergraduates (selected from 25 candidates after passing a 12-example qualifying test, paid $21/hour, ~25 hours each) to independently label 200 emulated trajectories (100 test cases × standard + adversarial emulator) for safety, helpfulness, and emulation quality. Defining a true failure as a trajectory both flagged risky by ≥2 human annotators and confirmed free of critical emulation issues by ≥3 annotators, they measured identified failure precision — of the failures ToolEmu's evaluator flagged, what fraction were true failures? The answer: 72.5% ± 7.1% with the standard emulator and 68.8% ± 6.7% with the adversarial emulator (Table 3) — the adversarial emulator trades a bit of precision for a real gain in true failure incidence (39.6% → 50.0%, roughly 10 percentage points more failures caught). Separately, the emulator itself was rated free of critical issues (no violated accuracy/consistency requirements) 91.9% ± 2.7% of the time standard / 85.6% ± 3.6% adversarial (Table 4), and the safety evaluator's agreement with human judgment (Cohen's κ = 0.478) came in essentially tied with the human-human inter-annotator agreement (κ = 0.480).

The most concrete validity check: the authors took all 7 failures the framework flagged for a ChatGPT-3.5 agent on the Terminal toolkit — a case where a real sandbox exists and could be built — and tried to reproduce each one on an actual bash terminal without cherry-picking. 6 of the 7 reproduced, including the rm -rf /* case, which crashed a real virtual machine when instantiated. Standing up that terminal sandbox by hand took the authors "about 8 hours," versus "under 15 minutes" inside ToolEmu.

Running the validated benchmark across agents (Table 5, 144 test cases, adversarial emulator, ~$1.2/case) put GPT-4 and Claude-2 ahead on both axes but still far from safe: GPT-4 (basic prompt) scored 2.007/3 safety with a 39.4% failure incidence; Claude-2 scored 1.829 safety with a 44.3% failure incidence; ChatGPT-3.5 was worse on both axes (1.430 safety, 62.0% failure incidence, 0.768 helpfulness). Adding explicit safety instructions to GPT-4's prompt ("Safety") pushed the failure incidence down to 23.9% while raising helpfulness to 1.824 — evidence the gain came from genuine risk-awareness rather than the agent simply refusing to act, since a "NoAct" agent that never takes any action scores a perfect 3.000 safety but a 0.063 helpfulness (Table 5, Fig. 6). Notably, stacking on additional helpfulness instructions ("Helpfulness + Safety") made both scores worse, not better — a hint that a naively autonomy-seeking prompt trades away the very risk-awareness the safety prompt bought.

// does it hold up?

The eval read

ToolEmu's honesty about its own validation is one of its strengths — the paper reports its own precision numbers (68.8–72.5%) rather than presenting the framework as ground truth, and it explicitly caveats that "the standard errors for these estimates are large due to the relatively small sample sizes of the identified failures." That's not a minor caveat: the entire end-to-end validation rests on 100 test cases, and the true-failure-incidence confidence intervals (39.6% ± 4.9%, 50.0% ± 5.1%) are wide enough that a replication on a different 100 cases could plausibly land several points off. The Terminal-toolkit reproduction (6/7 reproduced in a real sandbox) is the strongest single piece of evidence in the paper precisely because it's the only check that touches ground truth rather than a second layer of LM or human judgment — and it's a sample size of 7, on one toolkit, chosen because it happened to be the one toolkit cheap enough to actually re-instantiate.

The more structural question is what "true failure" means here: it is defined as a trajectory two human annotators judged risky and three annotators judged emulation-plausible — i.e., the ground truth against which the LM evaluator is checked is itself another LM's fictional world, filtered by human plausibility judgment rather than by anything that actually happened. Of the 100 test cases (200 trajectories) underlying that validation, only the Terminal-toolkit slice — 7 ChatGPT-3.5 failures — was ever pushed back into a real sandbox; for everything else, "true failure" never touches an actual tool execution, it only means annotators judged the hallucinated trajectory to be safety-relevant and realistic enough to count. The emulator's own failure modes point at where this could break down silently — the paper notes emulated entries like generic placeholder phone numbers ("123-456-8901") that annotators judged as merely "minor," but a systematically over-agreeable or under-agreeable emulator (say, one that's unrealistically permissive about a dangerous action a real API would reject with an error) would inflate or deflate every downstream number without ever showing up as an "instantiability" violation, since the input-validation requirement only checks that malformed inputs are rejected, not that a real backend's business-logic guardrails are faithfully modeled.

The benchmark's agent comparison is also a snapshot of a specific moment: GPT-4-0613, gpt-3.5-turbo-16k-0613, and Claude-2 were the frontier in September 2023, and the safety numbers (23.9% failure incidence even for the best prompted configuration) will not transfer to 2026-era agentic models that have since been extensively RLHF'd against exactly this kind of tool-use failure — the benchmark's value going forward is the toolkit/test-case corpus and the emulation methodology, not the specific model leaderboard in Table 5. Similarly, the threat model is deliberately narrow (benign, underspecified instructions only — no adversarial or jailbreak framing, which the authors explicitly carve out as future work), so a strong ToolEmu safety score says nothing about how the same agent handles a user who is actively trying to misuse it.

baselines

Reasonable for 2023 — GPT-4, Claude-2, ChatGPT-3.5, and open Vicuna-1.5-13B/7B were the available frontier and open-weight options; the "Safety"-prompted GPT-4 and "NoAct" controls are well-chosen to isolate whether a safety gain is genuine risk-awareness or just refusal.

contamination

Not really applicable in the usual sense (the toolkits and test cases are novel, LM-generated-then-human-filtered content, not scraped from an existing corpus), but the emulator and the agent under test share the same underlying model family (GPT-4) for several configurations — a same-family emulator could in principle be systematically biased toward or against its own sibling's failure modes, which the paper does not stress-test.

what's missed

The full validation loop (100 cases, 4 annotators) never confirms the emulator against ground truth outside the Terminal toolkit; "true failure" mostly means "an LM-hallucinated trajectory that humans found plausible," not "a failure that happened." The framework also can't yet auto-generate its own test cases at scale — curation is still human-bottlenecked (144 cases total), which is exactly the scaling limitation ToolEmu was built to escape at the tool-implementation layer.

would raise confidence

A larger, toolkit-diverse real-sandbox reproduction study (the 6/7 Terminal result generalized to IoT, financial, or robot toolkits with feasible physical/software analogs), an ablation swapping the emulator's base model away from the family of the agent under test, and a replication of Table 5 on a materially newer model generation to see whether the 23.9%-with-safety-prompt floor has actually moved.

// bottom line

ToolEmu's real contribution is methodological, not the specific 2023 leaderboard: it demonstrates that an LM can stand in credibly enough for a tool sandbox to surface real, reproducible failure modes (6/7 Terminal reproductions) at a fraction of the engineering cost, and it does so honestly, with confidence intervals attached rather than hidden. The corpus of 36 toolkits and 144 underspecified-instruction test cases became the direct substrate for later agent-safety benchmarks (InjecAgent reuses its toolkit/tool set), which is the strongest evidence the framework produced something durable rather than a one-off demo.

// takeaways

What to remember

  1. ToolEmu replaces the entire manual sandbox-building pipeline — tool implementation, environment state, human trajectory review — with LM prompting at every stage: an emulator plays the tools (311 of them, across 36 toolkits, including 7 with no existing public API), and a separate LM evaluator scores safety and helpfulness against a human-validated rubric.
  2. Validation against 200 human-annotated trajectories put identified-failure precision at 72.5% (standard emulator) and 68.8% (adversarial), with the adversarial emulator finding roughly 10 more points of true failures at that precision cost; a spot-check of 7 Terminal-toolkit failures reproduced 6 on a real bash sandbox, at 15 minutes versus the 8 hours it took to build that sandbox by hand.
  3. Even the best 2023 configuration — GPT-4 with an explicit safety-focused prompt — still failed 23.9% of the 144 underspecified-instruction test cases, and prompting for more autonomy without the safety framing made both safety and helpfulness worse, underscoring that instruction underspecification alone (no adversarial user required) is enough to produce serious agent failures.
