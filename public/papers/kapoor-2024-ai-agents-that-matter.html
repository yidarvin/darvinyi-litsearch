AI Agents That Matter

Three do-nothing-clever baselines — retry, warming, escalation — match or beat three published state-of-the-art coding agents on HumanEval at up to 55× less cost. A methodology audit of how agent benchmarks measure, and mismeasure, progress.

// the gap

What problem is this solving?

By the middle of 2024, AI agent research had exploded: over a dozen agent benchmarks had shipped in roughly a year, spanning web interaction, programming, and tool use, and each new "state of the art" agent leaned on progressively more elaborate architecture — debugging loops, tree search over generation paths, verbal self-reflection. Kapoor, Stroebl, Siegel, Nadgir, and Narayanan argue that agent evaluation copied its methodology wholesale from language-model evaluation without accounting for what makes agents different. Agents wrap a stochastic model that can be queried repeatedly rather than scored on a single forward pass; they cost dramatically more to run than one model call (the SWE-Agent authors capped every run at $4, which the paper notes "translates to hundreds of thousands of language model tokens"); and they act on external, changing environments — a web page, a command line — rather than static input/output pairs. None of the leaderboards driving the field report cost alongside accuracy, freeze a held-out test set pitched at the right level of generality, or separate the needs of a model developer (who wants a scientific, cost-agnostic comparison) from a downstream developer (who is making a procurement decision and cares about dollars).

The paper's central empirical worry is that this gap lets "accuracy" be manufactured for free. Because language models are stochastic even at temperature zero, an agent can raise its measured accuracy simply by calling the underlying model more times and keeping the best of many tries. The paper cites Li et al.'s AlphaCode result as the extreme case: accuracy on a competitive-programming benchmark rises from close to 0% zero-shot to over 15% with 1,000 retries and over 30% with a million retries, on the same underlying model. Coding benchmarks make this shortcut especially easy, because they typically ship example test cases an agent can use to check its own work before submitting — an agent can keep sampling from the same model until a candidate solution passes, with no architectural cleverness at all. If a leaderboard doesn't hold cost constant, "state of the art" can be bought with brute-force resampling rather than earned with better agent design, and nobody reading the leaderboard would know, because none of them report the bill. The paper organizes its response around five concrete failure modes — unmeasured cost, conflated model/downstream evaluation, missing or mis-leveled holdout sets, ignored human supervision, and irreproducible evaluation code — and builds one empirical demonstration for each.

Before any of that, the paper has to say what it means by "agent" in the first place, since no single definition commands consensus in the field. Rather than propose a new one, the authors identify three clusters of properties that make an LLM-based system read as more agentic: the complexity of its environment and goals (multi-stakeholder settings, long time horizons, goals pursued without step-by-step instruction read as more agentic); how much user supervision its interface requires (natural-language instruction and autonomous action on a user's behalf, with less supervision reading as more agentic); and its system design (tool use such as web search or code execution, and planning patterns such as reflection or subgoal decomposition, with dynamic, LLM-driven control flow reading as more agentic than a fixed pipeline). This framing matters for what follows: it's precisely along these axes that agent evaluation diverges hardest from language-model evaluation — more environment complexity means more room for the kind of distribution shift a benchmark's holdout set has to anticipate, and less user supervision means an evaluation harness, rather than a human in the loop, is the only check on a wrong or unsafe action.

// how it works

The method

This is not a paper proposing a new agent; it is a paper proposing a way to look at existing ones. The recurring device is a two-dimensional Pareto frontier — accuracy on one axis, dollar cost on the other — so a "SOTA" claim can be checked against what it cost to buy, and any agent with a competitor that is simultaneously more accurate and cheaper is disqualified from the frontier.

For the cost-control argument, the authors re-evaluate three complex, open-source coding agents drawn from the HumanEval leaderboard on PapersWithCode — LDB (which debugs its own generated code against runtime traces), LATS (which tree-searches over alternative generation paths), and Reflexion (which verbally self-reflects on failed attempts before retrying) — the only agents on that leaderboard, at the time of the analysis, that shared their code publicly. (AgentCoder, the leaderboard's actual top entry, was excluded because neither a working code link nor a paper link was available when the authors ran their analysis.) Against these three, they run three deliberately unintelligent baselines with no agent architecture: Retry re-invokes the same model at temperature zero up to five times until the problem's example tests pass; Warming does the same but raises the temperature across attempts (0, then 0.3, then 0.5) to increase the odds some retry succeeds; Escalation starts with a cheap model (Llama-3-8B) and climbs a cost ladder — Llama-3-70B, then GPT-3.5, then GPT-4 — only when a cheaper model's output fails the example tests. Every agent and baseline is run five times on all 164 HumanEval problems (using the LDB paper's modified version of the benchmark, which supplies example tests for all 164 tasks; the original ships tests for only 161), and both mean accuracy and mean total dollar cost are reported with 95% confidence intervals.

The second contribution turns the same lens into a design tool rather than just a diagnostic. The authors split an agent's lifetime cost into a one-time fixed cost (tuning its prompt and few-shot examples) and a recurring variable cost (tokens billed on every subsequent call), and argue that spending more upfront on optimization can lower the bill that scales with usage:

Total cost(N) = Cfixed + Cvariable × N // N = number of times the agent is run; the crossover between two agents' cost curves marks the usage volume at which the cheaper-per-call design starts winning overall

They demonstrate this by modifying DSPy — a framework for compiling declarative language-model pipelines — on the HotPotQA multi-hop question-answering benchmark. Five agent designs share the same retrieval stack (ColBERTv2 over a 2017 Wikipedia dump) and are scored on whether they retrieve every ground-truth supporting document for a question: an uncompiled baseline with no optimization; formatting instructions only; DSPy's own few-shot and random-search optimizers; and the authors' joint-optimization variant, which runs the Optuna hyperparameter search over temperature, number of few-shot examples, specific examples chosen, and formatting instructions to find agent configurations that jointly maximize accuracy and minimize the token count of the prompt — evaluated on 100 HotPotQA training samples and tested on 200 held-out evaluation samples, on two backbones (GPT-3.5 and Llama-3-70B).

// what they found

The results

The HumanEval result is blunt: none of the three complex agents beats the simple baselines, and most cost far more to tie. Warming (GPT-4) reaches 93.2% mean accuracy (92.1–93.9 across five runs) for $2.45 total; Retry (GPT-4) reaches 92.0% (91.4–92.7) for $2.51. LATS (GPT-4), the most expensive complex agent tested, reaches 88.0% for $134.50 — roughly 55× the cost of Warming for about five fewer points of accuracy. LDB (GPT-4), the strongest complex agent in the comparison (marked with an asterisk because the original LDB paper never evaluated this exact configuration), reaches 93.3% (92.1–94.5) for $6.36 — statistically indistinguishable accuracy from Warming at 2.6× the price. Zero-shot GPT-4 with no agent scaffolding at all already reaches 89.6% (87.8–90.9) for $1.93, sitting almost on the frontier. As the paper puts it: "there is no significant accuracy difference between our warming strategy and the best-performing agent architecture," and none of the papers proposing these agents test their designs against anything as simple as retrying the same model.

On HotPotQA, joint optimization delivers the promised cost cut at matched accuracy. For GPT-3.5, the joint-optimization agent reaches 0.509 retrieval accuracy (0.475–0.54) at a variable cost of $0.174 per 100 inferences, versus DSPy's own random-search optimizer at 0.495 (0.50–0.485) for $0.376 or DSPy few-shot at 0.47 (0.46–0.48) for $0.384 — a 53% lower variable cost at equal-or-better accuracy. For Llama-3-70B, joint optimization reaches 0.601 (0.59–0.62) for $0.374 per 100 inferences against DSPy random search's 0.617 (0.60–0.63) for $0.643 and DSPy few-shot's 0.622 (0.615–0.63) for $0.661 — a 41% lower cost for comparable (marginally lower) accuracy. That saving isn't free: the optimization search itself costs a fixed $2.714 (GPT-3.5) or $3.844 (Llama-3-70B). The authors calculate the joint-optimized agent only becomes cheaper than DSPy's default pipeline in total after 1,275 inferences (GPT-3.5) or 1,332 inferences (Llama-3-70B) — past which, they note, "the variable cost is by far the dominant term... as an agent might be used millions of times."

The NovelQA case study (Section 4) makes the model-vs-downstream distinction concrete with a cost gap the benchmark's own leaderboard hides. Re-evaluating GPT-4 on NovelQA's multiple-choice subset, retrieval-augmented GPT-4 (embedding each novel with text-embedding-3-large and retrieving ten 1,000-character chunks per question) scores 67.89% accuracy for $52.80 total; long-context GPT-4, fed the entire novel as input every time, scores 67.81% for $99.80 — accuracy is a coin flip apart, and RAG is already far cheaper. The gap widens once the workload looks like real use: NovelQA's leaderboard evaluates every question about a novel in one batched pass, which understates long-context's true recurring cost. Priced per single question against a novel, long-context costs $1.115 versus RAG's $0.051 — a cost ratio of roughly 21.86× — while the leaderboard's own batched accounting shows RAG costing only "half as much," which the authors call a tenfold underestimate of the real gap. A parallel proxy failure shows up in industry pricing: Mistral's launch materials for Mixtral 8x22B used active-parameter count as a cost stand-in, under which Mixtral 8x7B looks competitive with Llama 2 13B — but on the Anyscale endpoint the authors checked (June 2024), Mixtral 8x7B actually costs about twice as much to serve.

Section 5's generality taxonomy turns "agents overfit benchmarks" into a checkable claim. The core principle: the more general a benchmark aims to be, the more its holdout set must differ from its training distribution, as Table 1 of the paper lays out.

Surveying 17 agent benchmarks against this taxonomy (Table A4, drawing on AgentBench, AgentBoard, OpenDevin, and the ICLR 2024 LLM-agents workshop), the authors find 7 of the 17 have no held-out set at all and no stated plan to add one, and — per the table above — only 5 of the 10 that do have a holdout hold it out at the generality level the benchmark itself targets. The WebArena case study makes the risk concrete: STeP, the top public agent on WebArena's leaderboard, scores 35.8% accuracy — more than double the strongest baseline agent in the original WebArena paper and over 10 points above the next-best agent — achieved partly through policies hardcoded to WebArena's specific task set, such as reading the current Reddit-clone URL and appending a literal /user/user_name suffix to reach a profile page. That policy is brittle by construction (it breaks the moment the site's URL structure changes) but scores perfectly on WebArena's fixed, never-refreshed task set, precisely because WebArena — despite aspiring to be domain-general — ships no held-out tasks to catch it.

Section 6's reproducibility audit reruns released agent code directly and confirms every discrepancy by correspondence with the original authors. The gaps are large: LATS (GPT-4) was reported at 94.4% accuracy but reproduced at a mean of 88.0% (0.823–0.915); LATS's authors confirmed they had silently dropped a HumanEval problem after an execution error and evaluated on a subset of test cases for others, which alone explains roughly 3 percentage points of the inflation. Zhong et al.'s LDB paper reported its own zero-shot GPT-4 baseline at 75.0% — the reproduced mean is 89.6% (0.878–0.909), a 14.6-point gap that makes LDB's improvement over that baseline look considerably larger than it is. WebArena's Reddit-clone environment imposes a rate limit that silently fails posting actions: this affected 2 of 129 Reddit tasks in the original WebArena evaluation and 30 of 129 in Sodhi et al.'s STeP evaluation, and STeP's own logs (Listing 2) show the agent correctly reasoning it "cannot post more, wait a while before trying again" on a rate-limited task — which the evaluation harness nonetheless scored as solved, with reward 1.0. LATS and STeP each silently dropped a handful of tasks their agents failed (1 and 8, respectively) without disclosing it in the original papers.

The same section ties these bugs to a structural cause: agents are too expensive to evaluate more than a handful of times, so nobody checks the variance. SWE-bench alone comprises over 2,000 programming tasks, and the SWE-Agent authors capped spending at $4 per task — meaning a single full evaluation run of SWE-Agent on the entire benchmark could cost over $8,000. At that price, running an agent enough times to compute a meaningful confidence interval is rarely realistic, which the authors tie directly to what they found when they reran agents themselves: several published accuracy scores exceeded the maximum of the five reproduction runs the authors performed, and some reported baselines fell below the minimum of the five runs they measured — variance that a single official run, with no error bars, simply cannot reveal.

// does it hold up?

The eval read

This paper's subject is evaluation methodology itself, which means the fair question isn't only "is the agent good" but "is the critique's own evidence sound." On that score it is uneven, and instructively so: its most rigorous claims and its most sweeping claims rest on very different kinds of evidence.

The reproducibility audit (Section 6, Tables A6–A7) is the strongest section precisely because it makes no generalization claim at all — it reruns released code, finds concrete bugs (a silently dropped test case, a rate-limited environment scoring failed tasks as solved, an underreported baseline), and confirms each one by direct correspondence with the original authors, who are thanked by name in the acknowledgments. That is about as solidly sourced as an evaluation critique gets.

The cost-controlled-baselines claim — the paper's most-cited result — rests on thinner ground. Three retry-based strategies are compared against three complex agents on one benchmark (HumanEval), at one snapshot of GPT-3.5/GPT-4-turbo pricing (April 2024). The agent sample is a convenience sample, not a survey: the authors explicitly excluded AgentCoder, the actual top entry on the HumanEval leaderboard at the time, because a working code or paper link wasn't available when they ran their analysis — so the "SOTA" being outperformed is whichever complex agents happened to have public repos, not the state of the art itself. HumanEval is also a heavily-trained-on, largely-saturated benchmark for GPT-4-class models by 2024 — exactly the contamination risk the paper itself treats as serious when it writes, in Section 5, that benchmark overfitting is "a much more serious problem" for agents than training-data contamination. Yet the paper never checks whether GPT-4's near-frontier zero-shot score (89.6%) on HumanEval is itself partly a memorization artifact rather than evidence that agent complexity doesn't help. The authors are honest that this limits the finding — "it is possible that System 2 techniques will be useful for harder programming tasks than those represented in HumanEval, such as SWE-bench" — but that qualifier is one sentence, while the finding it qualifies is the one the paper is now cited for.

The generality taxonomy has a similar shape. The headline count — 7 of 17 benchmarks with no holdout, only 5 of 10 with a holdout that's pitched at the right level — depends on the authors' own single-rater coding of each benchmark's intended generality, which they candidly flag as a judgment call ("we made our best guess based on how it was presented in the paper") with no independent second coder reported. And the NovelQA and WebArena case studies that anchor Sections 4 and 5 are each single-run demonstrations: the NovelQA cost comparison was run once, by the authors' own admission, "due to the high cost of running the evaluation" — the exact "can't afford repeated runs, so no error bars" failure mode Section 6 diagnoses as an industry-wide problem. The paper's own case study exhibits the disease it is describing, and says so.

To its credit, the paper does test its own robustness in at least one place. Appendix A.2 reruns the HumanEval baseline-vs-agent comparison with the June 2023 versions of GPT-3.5 and GPT-4 instead of the April 2024 turbo models the main analysis uses, and reports "substantially similar results" — complex agents still fail to clear the simple baselines, even though the June 2023 GPT-4 is markedly more expensive than its April 2024 successor, which the authors use to explain why absolute costs shift even as the accuracy ranking doesn't. That's a genuine, if partial, answer to the "is this just a pricing-snapshot artifact" worry raised above — it does nothing to address the contamination question, since both model vintages have plausibly seen HumanEval solutions during training, but it does show the accuracy-side finding isn't purely an artifact of one specific model release.

baselines

Three open-source, code-available complex agents (LDB, LATS, Reflexion) against three retry-style baselines, on one benchmark; the actual top HumanEval leaderboard entry at the time, AgentCoder, was excluded for lack of accessible code, so the "SOTA" being beaten is a convenience sample rather than the frontier.

contamination

HumanEval is widely treated as saturated and likely memorized by GPT-3.5/4-class models. The paper calls agent-benchmark overfitting worse than training contamination elsewhere, but never audits whether its own headline Pareto result is partly a contamination artifact.

what's missed

Dollar-cost frontiers are pinned to specific April/May 2024 model snapshots (gpt-4-turbo-2024-04-09, gpt-3.5-turbo-0125); the paper's own Section 4 argues inference cost "come[s] down over time," so which agent is cost-optimal is a snapshot, not a durable ranking — even with the authors' companion price-recalculator webapp.

would raise confidence

Repeating the baseline-vs-agent comparison on a harder, actively maintained, less-contaminated benchmark (SWE-bench, which the paper names but doesn't test); a second independent coder for the Table A4 generality labels; more than one run of the NovelQA and STeP case studies.

// bottom line

Treat the parts of this paper built from rerunning code and talking to the original authors — the reproducibility bugs, the cost-accounting recommendations — as close to established fact; they're about as rigorously sourced as an evaluation critique gets. Treat "agent complexity rarely buys accuracy" as the load-bearing but comparatively under-tested hypothesis it is: three baselines beating three convenience-sampled agents on one contamination-prone benchmark, at one moment's API prices — a claim the paper itself hedges as possibly not holding on harder tasks, even as it has become the field's default citation for "don't bother building a complex agent."

// takeaways

What to remember

  1. On HumanEval, three baselines with zero agent architecture matched or beat three published "SOTA" agents: Warming (GPT-4) hit 93.2% accuracy for $2.45, while LATS (GPT-4) hit 88.0% for $134.50 — roughly 55× the price for lower accuracy.
  2. Jointly optimizing cost and accuracy (instead of accuracy alone) cut DSPy's per-inference cost on HotPotQA by 53% for GPT-3.5 and 41% for Llama-3-70B at matched accuracy — but the fixed cost of that optimization only pays for itself after roughly 1,275–1,332 inferences.
  3. Cost proxies built for model evaluation actively mislead downstream deployment decisions: NovelQA's batched-question leaderboard understates RAG's real cost advantage by roughly 10×, and Mistral's active-parameter proxy made Mixtral 8x7B look cheaper than a model it actually cost twice as much to serve.
  4. Of 17 surveyed agent benchmarks, 7 have no held-out test set and no stated plan to add one; WebArena's top public agent, STeP, hit 35.8% partly via task-specific hardcoded policies — like a literal Reddit URL suffix — that a proper holdout would have caught.
  5. Rerunning released agent code surfaced undisclosed bugs across the board: LATS silently evaluated on a subset of test cases (inflating reported accuracy by roughly 3 points), and LDB's own paper underreported its GPT-4 baseline at 75.0% versus a reproduced 89.6% — a reminder that a leaderboard number is only as trustworthy as the script that produced it.
