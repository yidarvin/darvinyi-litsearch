OpenAI Gym

A common reset/step interface wrapped around a growing suite of RL tasks — classic control, Atari, board games, algorithmic sequences, and MuJoCo-based continuous control — plus a website for sharing results. A four-page whitepaper that became the substrate nearly the entire deep-RL literature measured itself against for the next half-decade.

Greg Brockman, Vicki Cheung, Ludwig Pettersson, Jonas Schneider, John Schulman, Jie Tang, Wojciech Zaremba · OpenAI

// the gap

What problem is this solving?

By mid-2016, deep RL was moving fast — as the paper puts it, "general algorithms such as policy gradients and Q-learning can achieve good performance on difficult problems, without problem-specific engineering" — but the infrastructure for measuring that progress was fragmented. The Arcade Learning Environment (ALE) had turned Atari 2600 games into RL problems, and "recently the RLLab benchmark for continuous control" had done the same for simulated robotics; beyond those two, the paper points to a long tail of prior benchmark collections — RLPy, RL-Glue, PyBrain, RLLib, and the 2014 RL competition among the papers it cites for "a survey on other RL benchmarks." Each had grown up independently, with its own APIs, its own conventions for reporting results, and no shared way to tell whether an algorithm that "won" on one lab's task suite would do anything at all on another's.

The paper's stated fix is not a new algorithm or a new task — it is to combine "the best elements of these previous benchmark collections, in a software package that is maximally convenient and accessible": one common interface, wrapped around a diverse and deliberately growing collection of environments, backed by a companion website (gym.openai.com) where results are posted, compared, and — the paper is explicit about this framing — peer-reviewed rather than treated as a leaderboard to win.

// how it works

The method

Gym formalizes the RL loop the standard way — an agent situated in an environment, formalized as a POMDP, interacting over episodes — and then makes one pointed design choice: it standardizes only the environment's side of the interaction, not the agent's. "We have chosen to only provide an abstraction for the environment, not for the agent... to maximize convenience for users and allow them to implement different styles of agent interface" — either an "online learning" style that updates on every (observation, reward, done) as it arrives, or a "batch update" style that collects experience and updates later. What every environment guarantees, regardless of agent style, is the interaction pattern from the paper's own code listing:

ob = env.reset()  // sample initial state
a = agent.act(ob)
ob, rew, done, info = env.step(a)  // repeat until done == True // the paper's own listing (lightly condensed) — env.step returns the next observation, the reward, a boolean episode-complete flag, and a debug-info dict

Three more design decisions shape what "a common interface" actually means in practice. First, the paper deliberately steers evaluation toward sample complexity, not just final performance — final performance alone can always be bought with "arbitrary amounts of computation," so Gym encourages measuring how many episodes it takes to cross a per-environment performance threshold, not only where an agent tops out. Second, strict versioning: "if an environment changes, results before and after the change would be incomparable," so any behavioral change to an environment must ship as a new version number — the initial Cartpole-v0 becoming Cartpole-v1 if its dynamics or reward ever change. Third, every environment is instrumented with a Monitor by default, logging every step and reset, optionally recording video, and producing the learning-curve data that gets posted to the website — turning "please describe your setup" from a request into something environments do automatically.

The initial beta shipped five environment families, each wrapping an existing simulator or engine rather than reinventing one: classic control and toy text (small RL-literature staples like CartPole and MountainCar); algorithmic tasks such as adding multi-digit numbers or reversing a sequence, memory-dependent tasks whose difficulty scales with sequence length; Atari, exposing classic games with screen pixels or RAM as observations "using the Arcade Learning Environment"; board games, 9×9 and 19×19 Go with the open-source Pachi engine as the opponent; and 2D and 3D robot control, built on "the MuJoCo physics engine, which was designed for fast and accurate robot simulation," with several tasks "adapted from RLLab." The paper notes that by the time of writing, Box2D-based and VizDoom-based (Doom) environments had already been added on top of the initial release — the collection growing exactly the way the design was meant to allow.

// what they found

The results

This is a four-page systems whitepaper, not an experiments paper — there is no table of algorithm scores, no baseline sweep, no held-out test set to report. The paper's own measure of success is process, not a number: whether researchers actually adopted the interface, contributed environments, and posted comparable results to the companion website rather than each re-inventing their own. By that measure it landed about as hard as a benchmark-infrastructure paper can: it sits at roughly 5,340 citations (Semantic Scholar, as of this writing) — a figure that still understates its real footprint, since the great majority of Atari- and MuJoCo-based deep-RL papers published from 2016 onward simply run Gym as their environment layer rather than discuss it beyond a citation.

The environment suite grew exactly the way the design was meant to allow: within the graph this page joins, Gym's release sits downstream of DQN, TRPO, and A3C — all three cited directly in the paper's own reference list as the deep-RL results Gym was built to make comparable — and upstream of every continuous-control paper that followed. PPO (2017), for instance, cites Brockman et al. directly, alongside Todorov et al.'s MuJoCo paper, for the exact HalfCheetah/Hopper/Walker2d-style task suite it reports on — the pattern this node was added to the graph to make explicit: later continuous-control papers overwhelmingly reach MuJoCo's physics through Gym's wrapper, and cite the wrapper, rather than citing the underlying engine on its own.

// does it hold up?

The eval read

There's no claimed result to interrogate here, so the right critique is about what "a common interface" actually promises versus what it delivered. The paper gets the easy half right: a shared reset()/step() contract genuinely made environment code interchangeable across labs, and its strict-versioning design — bump the version number the moment an environment's behavior changes — was a real, foresighted attempt to head off exactly the failure mode that hit the field anyway: incomparable results quietly reported as if they were comparable.

Because in practice, version drift became one of the ecosystem's defining reproducibility headaches — not because the versioning idea was wrong, but because it only covers an individual environment's own behavior, not the software stack underneath it. Henderson et al.'s widely cited "Deep Reinforcement Learning that Matters" (2018) found that headline continuous-control results were highly sensitive to implementation details and, pointedly, to which Gym/MuJoCo release an experiment ran against — sometimes flipping which algorithm looked best. Gym's own MuJoCo bindings changed too, moving from the third-party mujoco-py wrapper to DeepMind's official mujoco Python package, quietly shifting contact dynamics and reward scaling underneath task names (HalfCheetah-v2 vs -v3 vs -v4) that looked stable on the surface. Atari access broke for a non-technical reason entirely: around Gym v0.21 (2021), pip install gym[atari] stopped bundling the Atari ROMs over licensing concerns, and users had to separately opt in through the accept-rom-license/AutoROM path just to run the same games the paper shipped five years earlier. And the core step() contract itself made a breaking change in Gym v0.26 (2022), splitting the single done flag into terminated and truncated — the right fix for a genuine ambiguity (did the episode end because the agent failed, or because a time limit ran out?) that nonetheless forced every downstream RL library to add a compatibility shim or pin an old Gym version.

By the time all of that had accumulated, OpenAI itself had largely stopped maintaining the project: Gym went, in the Farama Foundation's own description, "wholly unmaintained" by late 2020. In early 2021 OpenAI handed stewardship to Farama, which forked the project as Gymnasium — a drop-in-compatible successor that carries the same interface philosophy forward and is where the terminated/truncated fix, ongoing maintenance, and most new environment integrations now actually live. That succession is, if anything, the strongest evidence the paper's central bet paid off: the interface mattered enough that a nonprofit was founded specifically to keep maintaining it once the original authors moved on, rather than the field reverting to per-lab environment wrappers.

baselines

None in the usual sense — this is an infrastructure whitepaper with no learned agent and no head-to-head comparison table. The implicit comparison is against the fragmented status quo (ALE, RLLab, RL-Glue, PyBrain, RLLib, the 2014 RL competition) the paper explicitly positions itself as unifying.

contamination

Not applicable to a benchmark-suite paper directly, but the interface it created enabled a field-wide monoculture: nearly a decade of continuous-control "wins" measured on the same handful of Gym/MuJoCo task instances, where engine-version and wrapper-library differences could masquerade as algorithmic progress (Henderson et al., 2018).

what's missed

The paper's own versioning safeguard covers an environment's behavior changing, not the dependency stack underneath it — the mujoco-py→mujoco binding switch, the 2021 Atari ROM-licensing break, and the 2022 done→terminated/truncated API change all sat outside what "bump Cartpole to v1" was designed to catch.

would raise confidence

A documented mapping of which published results ran against which Gym/MuJoCo/ALE release — the kind of version provenance that Gymnasium's stricter release discipline and versioning tooling now try to enforce going forward, but that the original ecosystem left largely to individual authors to disclose (or not).

// bottom line

The actual bet — that a shared, versioned software interface would do more for reproducible RL research than any single lab's benchmark collection — paid off completely: the interface, not any one environment, is what survived, first as Gym and now as its Farama-maintained fork Gymnasium. What the paper didn't and couldn't anticipate is that the interface's own dependency stack — physics bindings, ROM licensing, the step-API contract — would become exactly the kind of silent version drift its own "bump the version number" rule was designed to prevent at the environment level.

// takeaways

What to remember

  1. A common interface, not a single benchmark. The contribution isn't any one environment — it's the versioned reset()/step() contract wrapped around a deliberately growing, heterogeneous suite: classic control, algorithmic sequence tasks, Atari via ALE, Go via the Pachi engine, MuJoCo-based 2D/3D robot control adapted in part from RLLab, and later Box2D and VizDoom environments.
  2. It's the reason later papers cite Gym instead of MuJoCo. Continuous-control papers that build on MuJoCo overwhelmingly reach it through Gym's wrapper — PPO, for one, cites Brockman et al. directly alongside Todorov et al.'s MuJoCo paper for the exact task suite it benchmarks on.
  3. Version drift found exactly the gaps the versioning scheme didn't cover. The mujoco-py→mujoco binding switch, the 2021 Atari ROM-licensing break, and the 2022 done→terminated/truncated API change all caused real reproducibility pain outside what "bump the environment version" was designed to catch — eventually prompting a full stewardship handoff to the Farama Foundation's Gymnasium fork.
