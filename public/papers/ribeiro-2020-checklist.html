Beyond Accuracy: Behavioral Testing of NLP Models with CheckList

A hand-authored matrix of linguistic capabilities × test types that turns "92% accurate" into a map of exactly which basic behaviors a model still gets wrong — and finds a shipping commercial sentiment API failing simple negation 54.2% of the time.

// the gap

What problem is this solving?

NLP evaluation has run on held-out accuracy since train/validation/test splits and leaderboards (SQuAD, GLUE) became the standard: report one number, rank models by it. The trouble is that held-out test sets are drawn from the same distribution — and share the same biases — as the training data, so an accuracy figure can badly overestimate how a model behaves on real, out-of-distribution language. And even when the number is trustworthy, a single aggregate statistic gives a practitioner no traction on where the model is failing or how to fix it. A number of point solutions exist — robustness-to-noise checks, adversarial perturbations, fairness audits, logical-consistency probes, hand-built challenge datasets — but each one interrogates a single task or a single capability. None of them hands a practitioner a systematic, reusable checklist of what to test before calling a model "done," the way software engineers have unit tests and regression suites for exactly that purpose. CheckList's premise is that NLP evaluation is missing behavioral testing: validating a system's input/output behavior against a comprehensive list of capabilities, independent of the model's internals, the same way software QA treats a program as a black box.

// how it works

The method

CheckList operationalizes behavioral testing as a matrix. Rows are general linguistic capabilities — the paper suggests at least ten as a starting point: Vocabulary+POS, Taxonomy (synonyms/antonyms), Robustness (typos, irrelevant changes), NER, Fairness, Temporal, Negation, Coreference, Semantic Role Labeling, and Logic — explicitly "not exhaustive," since users are expected to add capabilities specific to their own task or domain. Columns are three test types, borrowed from software testing: a Minimum Functionality Test (MFT) is a small set of simple, hand-labeled examples that isolate one behavior (e.g. "That is a private aircraft." → neutral) — like a unit test, it's designed to catch a model taking a shortcut instead of actually mastering the capability. An Invariance test (INV) applies a label-preserving perturbation (swap a location name, introduce a typo) and checks that the prediction doesn't change. A Directional Expectation test (DIR) applies a perturbation with a known expected direction of change (append "You are lame." to a tweet and check that predicted sentiment doesn't go up). INV and DIR need no ground-truth labels at all — they test relationships between predictions before and after a perturbation, so they can run directly on unlabeled, naturally-occurring data.

failure rate = test cases where prediction ≠ expected behaviortotal test cases in the cell // the one metric CheckList reports per capability × test-type cell — no aggregation across cells into a single score

To make filling in this matrix tractable at scale, the open-source tool (released at github.com/marcotcr/checklist) provides templates: generalize a concrete example like "I didn't love the food." into "I {NEGATION} {POS_VERB} the {THING}.", supply fill-in lexicons for each slot, and generate the full Cartesian product of test cases automatically. To help populate those lexicons without requiring the user's own creativity for every slot, CheckList also offers masked-language-model suggestions: mask part of a template ("I really {mask} the flight.") and RoBERTa (Liu et al., 2019) proposes candidate fill-ins ({enjoyed, liked, loved, regret, ...}), which the user filters into positive/negative/neutral lists and reuses across many subsequent tests. The tool ships general-purpose perturbations out of the box too — character swaps to simulate typos, name and location swaps for NER tests, common protected-group adjectives for fairness checks — plus visualizations for inspecting and sharing test suites across models and teams.

// what they found

The results

The authors instantiate the full matrix on three tasks, each tested against systems that were, by their own benchmark, "solved" or near human-level. For sentiment analysis: three commercial paid APIs — Microsoft's Text Analytics, Google Cloud's Natural Language, and Amazon's Comprehend — plus BERT-base and RoBERTa-base finetuned on SST-2 (92.7% and 94.8% accuracy). For Quora Question Pairs (QQP) duplicate-question detection: the same two finetuned models, at 91.1% and 91.3% accuracy — both above reported human accuracy on the benchmark. For machine comprehension: BERT-large finetuned on SQuAD (93.2 F1). Every one of these systems has severe, specific holes CheckList uncovers. Simple negation templates like "The food is not poor." (expected positive/neutral) fail on Google's API 54.2% of the time and Amazon's 29.4%; failure climbs to near 100% across all three commercial APIs when the negation sits at the end of the sentence ("I thought the plane would be awful, but it wasn't.") or has neutral content between the negation and the sentiment word. BERT and RoBERTa fail simple negation MFTs despite scoring 91.5% and 93.9% accuracy on the negation-containing subset (18% of instances) of the SST-2 validation set itself — the benchmark number simply never surfaces the gap. Swapping a location name in a tweet moves Google's prediction 20.8% of the time and Amazon's 14.8%; swapping a person's name moves them 15.1% and 9.1%. On QQP, BERT and RoBERTa — both above human accuracy on the leaderboard — fail a modifier-insertion test ("Is Mark Wright a photographer?" vs. "...an accredited photographer?" should not be duplicates) 78.4% and 78.0% of the time, an antonym-invariance test ("more optimistic" ≟ "less pessimistic") 69.4% and 100% of the time, and a coreference substitution test (he/she) 79.0% and 96.6% of the time. On machine comprehension, BERT-large/SQuAD fails a before/after temporal MFT 82.9% of the time and a coreference MFT 100% of the time, and shows an explicit gender bias: given the template "{P1} is not a {PROF}, {P2} is." with {PROF}=doctor, it picks the man as the doctor 89.1% of the time when {P1} is a male name — but only 3.2% of the time with genders swapped — and for {PROF}=secretary it wrongly picks the man only 4.0% of the time versus the woman 60.5% of the time.

Two user studies test whether this actually helps practitioners, not just the authors. In a roughly five-hour session, the team responsible for Microsoft's own production sentiment model — already tested extensively with in-house benchmarks and years of customer-driven fixes — used CheckList to brainstorm about 30 new tests (implementing ~20), surfacing previously unknown, actionable bugs and proposing capabilities the authors hadn't tested themselves (sentiment in camel-cased hashtags like "#IHateYou", implicit negation like "I wish it was good"). In a controlled study, 18 practitioners (8 industry, 10 academia) had two hours to write Minimum Functionality Tests for RoBERTa finetuned on QQP, split evenly into three conditions — Unaided (no guidance, simulating current practice), Cap. only (given the capability list), and Cap.+templ. (also given the template/fill-in tooling):

Full CheckList (Cap.+templ.) users wrote more than twice as many tests and found almost three times as many severity-≥3 bugs as the unaided control, in the same two hours — the paper's headline user-study claim. A follow-up round with an independent rater who hadn't written any tests re-scored the same bugs' severity and obtained "nearly identical aggregate results" to the original self-reports.

// does it hold up?

The eval read

CheckList's central bet is that a hand-curated matrix of capabilities and test types can serve as a comprehensive checklist of what a model "should" get right — but the matrix is exactly as complete as its designers' imagination, which is the same structural limit every fixed evaluation taxonomy eventually runs into. The paper is unusually candid about this itself: the ten suggested capabilities are explicitly "not exhaustive, but a starting point for users, who should also come up with additional capabilities that are specific to their task or domain." A capability nobody thought to name — some interaction effect, some domain-specific failure mode outside Vocabulary/NER/Negation/etc. — simply never gets a cell in the matrix, and a clean CheckList report gives no signal that anything is missing; a fully green checklist can look complete while silently omitting the one failure that actually matters. The related-work section draws this boundary even more explicitly, stating CheckList "cannot be directly used for non-behavioral issues such as data versioning problems... labeling errors, annotator biases... worst-case security issues... or lack of interpretability" — it only ever reports on the behaviors someone remembered to write a test for.

The evidence for sustained real-world adoption is thinner than the citation count (1,469 and climbing) suggests. Both user studies are single-session demonstrations run by the authors themselves: the Microsoft team's five-hour workshop and the 18-participant two-hour QQP study show that CheckList helps you find bugs quickly with fresh enthusiasm and hands-on author support, not that a team keeps a CheckList suite running in CI a year later, or that bugs found this way actually get fixed and stay fixed — the paper reports the Microsoft team merely "plan[ned] to fix" the bugs in the next model iteration and "indicated they would definitely incorporate CheckList into their development cycle," both stated intentions rather than a follow-up measurement. Consistent with that, the tool's own GitHub release (marcotcr/checklist) has accumulated a respectable ~2,000 stars but its most recent commit predates this write-up by more than two years — a pattern more consistent with a paper that gets cited as the canonical origin of "behavioral testing for NLP" than with a tool teams actively maintain against new model releases. Writing a good capability list and template set per task is itself expert, creative labor, and that authoring cost is a real barrier to the kind of everyday, per-project use neither user study was designed to measure.

baselines

Reasonable for 2020: three commercial sentiment APIs (queried 11/2019, with the paper noting "similar results" on a 04/2020 recheck) plus standard BERT/RoBERTa finetunes on SST-2, QQP, and SQuAD. The commercial-API snapshot is necessarily frozen — all three products have shipped many updates since, so the exact failure rates are a 2019–2020 artifact, not a claim about today's APIs.

contamination

Not really at risk in the usual leaderboard sense — CheckList's test cases are authored or template-generated specifically to probe behaviors, not drawn from data a model could have memorized. The more relevant version of the concern is reflexive: the suggested capability list was itself shaped by failures the authors already expected to find, which is a softer form of the same fixed-taxonomy problem.

what's missed

Whatever capability isn't on the list. The ten suggested capabilities are an explicit "starting point," and the paper's own related-work section disclaims data-versioning bugs, annotator bias/labeling error, worst-case adversarial robustness, and interpretability as entirely out of scope.

would raise confidence

A longitudinal case study — does a team that adopts CheckList still run it a year later, and do fixes made in response to CheckList failures move real production error rates — rather than two single-session, author-facilitated demonstrations with n=6 practitioners per condition.

// bottom line

CheckList's core empirical claim holds up well six years on: benchmark accuracy alone hides severe, easily-triggered failures — a model that's "solved" SST-2 or surpassed human accuracy on QQP can still fail basic negation, coreference, or a modifier swap most of the time. But it inherits the one problem no fixed taxonomy escapes — it only surfaces failures somebody thought to write a test for — and the paper's evidence that this changed how teams evaluate models day-to-day, versus becoming a widely-cited idea more praised than practiced, is largely absent from the paper itself, and, judging by a now-dormant GitHub repo, from the years since.

// takeaways

What to remember

  1. CheckList reframes evaluation as a matrix — linguistic capabilities (Vocabulary, NER, Negation, Coreference, SRL, …) crossed with three test types (Minimum Functionality, Invariance, Directional Expectation) — turning "is the model good" into "which specific behaviors does it get wrong."
  2. Every system tested — three commercial sentiment APIs, finetuned BERT/RoBERTa, and a SQuAD-finetuned BERT-large — looked "solved" or near human-level on its standard benchmark, yet failed simple negation, NER-invariance, or coreference tests at rates from roughly 15% up to 100%.
  3. In a controlled study, giving practitioners CheckList's capability list and templates (vs. no guidance) more than doubled the number of tests written and nearly tripled the number of severity-≥3 bugs found in the same two hours — real evidence that structured behavioral testing beats ad hoc manual testing.
  4. Its core limitation is inherent to any hand-authored taxonomy: it can't test a capability nobody thought to name, a limit the paper itself acknowledges, and the evidence for sustained day-to-day adoption (versus being a widely-cited but lightly-used idea) is thin.
