Gorilla: Large Language Model Connected with Massive APIs

A retrieve-aware finetuned LLaMA-7B that writes ML API calls more accurately than zero-shot GPT-4 — and APIBench, the benchmark that first made tool use measurable.

Shishir G. Patil, Tianjun Zhang, Xin Wang, Joseph E. Gonzalez · UC Berkeley · Microsoft Research

// the gap

What problem is this solving?

An LLM's knowledge is frozen in its weights; the world's tools are not. The obvious fix is to let the model call tools — but in early 2023 that meant hand-injecting a handful of well-documented APIs into the prompt. That doesn't scale to the real target: millions of overlapping, constantly-changing cloud APIs that can't all fit in a context window and whose documentation is updated faster than any model is retrained.

And the frontier models of the day were bad at it. Asked to pick the right API for a task, GPT-4 would confidently invent a model that doesn't exist; Claude would reach for the wrong library. The paper names two distinct failure modes: an error (calling a real-but-wrong API) and a hallucination (invoking an entirely imagined tool). The second is the dangerous one — it looks plausible and fails silently.

There was also no way to measure the problem. Tool-use papers like Toolformer and HuggingGPT demonstrated capability anecdotally but offered no systematic benchmark. Gorilla's contribution is two-sided: a method (retrieval-aware finetuning) and the missing measuring stick (APIBench + an AST-matching metric) to score it.

// how it works

The method

Gorilla has three moving parts: a benchmark dataset (APIBench), a retrieve-aware finetuned model, and an AST-based evaluation metric.

APIBench — scrape three model hubs

The authors exhaustively scraped model cards from three hubs into 1,645 API calls: 95 from Torch Hub and 696 from TensorFlow Hub v2 (both exhaustive), plus the top-20 most-downloaded models per domain from HuggingFace (which hosts ~200k models, most poorly documented) for 925 entries. Each model card becomes a JSON object with fields like domain, api_call, api_arguments, performance, and description — a schema meant to generalize beyond ML to RESTful APIs.

Self-Instruct the instructions

For each API, GPT-4 generates 10 synthetic user instructions via Self-Instruct — given three in-context examples and the API doc, told to write realistic use cases without naming the API. The only hand-written data is 18 seed examples (six per hub). The result is 16,450 {instruction, API} pairs, which become user→agent chat conversations for finetuning.

Retriever-aware finetuning

The signature idea. Gorilla is a finetuned LLaMA-7B (5 epochs, lr 2e-5 cosine, batch 64, 8×A100). The twist: during training, half the time the prompt is augmented with "Use this API documentation for reference: <retrieved_doc>" using the golden (oracle) doc. This teaches the model to read the second half of the prompt to answer the first half — so at test time it can absorb an updated doc and adapt to API changes without retraining. At inference Gorilla runs in two modes: zero-shot (query only) or retrieval (BM25 or GPT-Index fetches the top-1 doc first). No further prompt tuning either way.

AST sub-tree matching — how do you grade an API call?

Unit tests don't work here: dozens of models can classify an image, so there's no single right answer to test against. Instead, Gorilla parses the generated code into an abstract syntax tree and checks whether the API call is a sub-tree of a reference API in the database, matching only the arguments that matter (e.g. repo_or_dir and model, ignoring optional ones like pretrained=True). This gives a crisp, executable definition of hallucination:

hallucination  :=  AST(call)  ⊄  any API in database // a call that matches no known API sub-tree = an imagined tool. A wrong-but-real API is an "error", not a hallucination.
// what they found

The results

In the zero-shot setting (no retriever), the lightly-finetuned Gorilla-7B beats every baseline — including GPT-4 — at writing correct API calls, while hallucinating far less. The paper reports Gorilla is 20.43% more accurate than GPT-4 and 10.75% better than ChatGPT overall in zero-shot, with the gap over the base LLaMA as large as 83%.

Three findings worth keeping

Test-time adaptation works. Because of retriever-aware training, swapping in an updated doc lets Gorilla react to API changes — upgrading an FCN backbone from ResNet-50 to ResNet-101, or following a model registry move from pytorch/vision to NVIDIA/DeepLearningExamples — without any retraining.

Retrieval is a double-edged sword. Surprisingly, "augmenting an LLM with retrieval does not always lead to improved performance, and can at times hurt." Finetuning with the oracle retriever beats finetuning without it by 12.37% (TorchHub) and 23.46% (HuggingFace) — but at eval time a real retriever leaves a big gap to the oracle: GPT-Index costs ~29.20% accuracy and BM25 ~52.27%.

RLHF may buy truthfulness. Across every hub and retriever setting, GPT-3.5 hallucinates less than GPT-4 — which the authors read as a hint that RLHF, not scale, is what nudges a model toward not inventing tools.

// does it hold up?

The eval read

Gorilla is a genuinely seminal paper — it gave the field its first systematic tool-use benchmark and a clean, executable definition of hallucination. But the headline ("a 7B model beats GPT-4") rests on an evaluation that is narrower than the framing suggests, and the marquee adaptation result leans on an idealized retriever.

"Functional correctness" is a name-match, not execution. AST sub-tree matching checks the API name and a hand-picked subset of arguments — not whether the call runs or returns the right output. The authors explicitly "haven't tested the execution result of the code" and de-prioritize the code field. So the metric measures did you select and parameterize the right API, a useful but soft proxy. The execution-graded code benchmarks that followed and cite Gorilla — BigCodeBench, LiveCodeBench — grade by actually running tests, a much stronger bar.

The HuggingFace column isn't apples-to-apples. By the authors' own statement, on HuggingFace "for all the models except Gorilla, we only check if they can provide the correct domain names" — reducing the baselines' task to multiple choice while Gorilla must emit the full call. That handicaps Gorilla, not the baselines, so the win is if anything understated — but it means the precise HuggingFace gaps (71.68 vs GPT-4's 19.80) measure different tasks per model and shouldn't be read as a clean head-to-head.

It's a specialist vs un-adapted generalists. Gorilla is finetuned on Self-Instruct data drawn from the same 1,645 APIs and the same GPT-4 pipeline as the test set; the holdout splits instructions, not APIs, so every target API is seen at training. The baselines are zero-shot, never adapted to this distribution or output format — LLaMA-7B's 0% (≈100% "hallucination") is largely a format-following artifact, which inflates the visual contrast. "Finetuned 7B beats zero-shot GPT-4 on a narrow in-distribution task" is the expected outcome, not a surprise.

The adaptation story needs an oracle. The retrieval and test-time-change results are strongest with the Oracle retriever (100% relevant doc); with real retrievers the gains shrink or reverse (BM25 ~52% below oracle), and the doc-change demo (Fig. 6) is a couple of hand-built examples, not a measured distribution. Several with-retriever "wins" are within noise — Gorilla 61.82 vs GPT-3.5 60.21 (TorchHub, GPT-Index), 47.46 vs 47.34 (HuggingFace) — and there are no seeds or error bars anywhere.

baselines

GPT-4 (gpt-4-0314), GPT-3.5 (gpt-3.5-turbo-0301), Claude (claude-v1), LLaMA-7B — current for May 2023, but all zero-shot and un-adapted to the API output format, compared against an in-distribution finetuned specialist.

contamination

Low classic leakage, but train and test share the same 1,645 APIs and the same GPT-4 Self-Instruct generator; the holdout splits instructions, not APIs, so every target API is seen during training.

what's missed

AST match scores API name + selected args, not execution; HuggingFace is scored differently per model; retrieval/adaptation gains shown mainly with an oracle retriever; every task is a single, single-line call — no composition, argument-value reasoning, or multi-step use.

would raise confidence

Execution-grounded grading, symmetric metrics across models, a realistic retriever measured at scale, multi-API / argument-value tasks, a real-user query distribution, and seeds + error bars.

// bottom line

A strong proof-of-concept and a field-defining benchmark: a small retrieve-aware finetune can out-call zero-shot GPT-4 on structured API selection, and APIBench made tool use measurable. But the win is a specialist-vs-generalist comparison on a name-match proxy, and the test-time-adaptation selling point rests on an oracle retriever — so it's evidence that the recipe is promising, not that retrieval-augmented finetuning solves real-world API use.

// takeaways

What to remember

  1. Tool use is an API-selection problem. Gorilla reframes "use a tool" as "emit the correct, fully-qualified API call," and shows a 7B finetune can beat zero-shot GPT-4 at it (≈+20.43% overall) while sharply cutting hallucinated, imagined APIs.
  2. AST sub-tree matching made hallucination measurable. Parse the generated call into a syntax tree and check it's a sub-tree of a known API — a hallucination is a call that matches nothing in the database. Cheap and crisp, but a name/argument proxy, not execution.
  3. Retriever-aware training is the durable idea — and the fragile one. Training with the doc in-context lets the model adapt to changed APIs at test time, yet real retrievers (BM25/GPT-Index) can hurt vs zero-shot, so the clean gains depend on an oracle. The lineage runs straight to today's function-calling and MCP benchmarks.
