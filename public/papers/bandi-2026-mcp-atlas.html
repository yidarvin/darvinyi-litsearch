MCP-Atlas: A Large-Scale Benchmark for Tool-Use Competency with Real MCP Servers

1,000 expert-written tasks run against 36 production MCP servers and 220 real tools, scored by an answer-centric claims rubric that credits any valid tool path. Across 20 frontier models the agentic frontier is no longer bottlenecked by tool mechanics: 63.3% of failures are cognitive — premature stopping, faulty synthesis — and a leading reasoning model (o3 Pro) lands near the bottom at 44.5% because it issues no tool call at all on 40% of its failed tasks.

// the gap

What problem is this solving?

The Model Context Protocol has become the standard wiring through which LLM agents discover servers, enumerate tools, pass typed parameters, and consume results. But evaluating those agents credibly is hard, and the authors argue that real deployments demand three things at once: realistic multi-step workflows that branch and cross servers; breadth across authentic MCP servers rather than mocked stubs; and structured, reproducible claim-level scoring that is disentangled from an agent's verbosity or style.

Every prior MCP benchmark, they show, nails one or two of these and compromises on the third. There is a hard tension between rigor and scale. Benchmarks that insist on programmatic ground-truth verifiers stay small — MCP-Universe (231 tasks), MCPMark (127), Toolathlon (108) — because hand-authoring custom verifiers is expensive, leaving too few tasks for statistically robust comparisons. Benchmarks that chase scale buy it with shortcuts: MCPEval reaches 676 tasks through automated generation that can erode task quality; MCP-Bench and LiveMCPBench adopt holistic LLM-as-judge scoring that introduces style bias (a verbose answer and a terse-but-correct answer can score differently); MCPVerse and MCP-RADAR widen server coverage but lean partly on synthetic or mocked implementations, weakening how well results predict live behavior.

A second, subtler gap is tool discovery under noise. Most suites hand the agent a clean tool set, so the task reduces to executing a known API. Real deployments don't: the agent must first figure out which of many plausible tools is relevant. MCP-Universe named this the "unknown-tools" challenge; MCP-Atlas builds it directly into every task by mixing the 2–8 required tools with a larger pool of semantically similar distractors.

// how it works

The method

MCP-Atlas is a 1,000-task suite built on 36 production MCP servers exposing 220 real tools, spanning five environment buckets: Basic (32% of tasks — search, maps, weather), Productivity (22% — filesystem, notion, slack, arxiv), Coding (22% — git, github, code executors, cli), Analytics (12% — airtable, mongodb), and Financial (12% — twelvedata, alchemy). Every server is a real implementation hit at a live endpoint, so tasks inherit authentic rate limits, pagination boundaries, schema-version mismatches, and transient error codes — exactly the failure conditions mocked environments smooth away. Servers run in sandboxed, version-pinned Docker containers, restarted between tasks to prevent state leakage.

Tasks built to force discovery and orchestration

Each task exposes a tool set of 6–37 tools (mean 15.2), of which only 2–8 are actually required (mean 4.1); the remaining ~11 per task are distractors drawn from semantically similar categories. Prompts are single-turn natural language that never name a server, tool, or parameter — the model must infer the whole tool chain from task semantics. And the tasks genuinely demand orchestration: 98.6% require tools from two or more servers (mean 2.55 servers — 51.7% need exactly two, 38.2% three, 8.7% four or more), with later calls parameterized by earlier outputs.

Claims-based scoring: grade the answer, not the trajectory

Instead of matching a gold trajectory or asking a judge for a holistic verdict, each task defines a claims list: a set of atomic, independently verifiable factual statements grounded in tool outputs (mean 4.7 claims, range 1–23). An LLM judge scores each claim independently as fulfilled (1.0), partially_fulfilled (0.5), or not_fulfilled (0.0). Task coverage is the mean claim score, and a task passes at coverage ≥ 0.75.

Crucially, scoring is answer-centric. A separate reference trajectory (mean 9.8 tool-call steps) is recorded per task, but it is used only for solvability checks and post-hoc failure diagnosis — never for pass/fail. So any alternative tool path that yields the same factual claims earns full credit. The judge is invoked once per claim (a 5-claim task issues 5 independent evaluator calls), which avoids loss-of-context when scoring many claims against one long response. To decouple evaluator bias from model skill, three judges score every task: Gemini 3.1 Pro Preview (primary), GPT-5.4, and Claude Opus 4.6.

An 11-mode failure taxonomy

Every failed task (coverage < 0.75) gets an automated diagnosis assigning a primary failure mode from an 11-category taxonomy split into two families. Tool-call failures: malformed parameters, wrong tool, no tool use, failed error recovery. Cognitive failures: task misunderstanding, faulty synthesis, response misparsing, early termination, hallucinated facts, logical errors, constraint violations. This is what turns a leaderboard into a diagnostic instrument: it separates "couldn't call the tool" from "called the tools, then mishandled what came back." Quality is maintained through five layers (expert trajectory execution, domain review, leakage review, automated claim/trajectory checks, random-sample audit), at a cost of ~4.6 hours of expert labor per task, and the 1,000 tasks are split 500/500 into a released public set and a held-out private set.

// what they found

The results

Across 20 frontier models from six providers, the leaderboard shows a clean three-tier structure. A top cluster of three sits within overlapping confidence intervals: Muse Spark (82.2%), Claude Opus 4.7 (79.1%), and Gemini 3.1 Pro Preview (78.2%). A mid-tier of eight models spans 67.6–76.8%. A long tail of nine models falls below 65%, bottoming out at Claude Haiku 4.5 (40.2%). Two facts stand out: the open-weight GLM-5.1 reaches 75.6% — the strongest open model, statistically tied with GPT-5.5 and entering a band that used to be proprietary-only; and o3 Pro, a top math/coding reasoner, finishes near the bottom at 44.5%, because it produces no tool calls on 40.1% of its failed tasks.

The bottleneck has moved past calling tools

Diagnosing roughly 6,900 failures across the 20 models, 63.3% are cognitive and only 36.7% are tool-related. The benchmark isn't mainly testing whether a model can issue a valid tool call — for strong models that's largely solved. The dominant failure is what happens after tool access: understanding the task, deciding whether enough evidence is in hand, and synthesizing the answer. Tool-call breakdowns still dominate the tail — o3 Pro (57.6% tool-related, 40.1% of failures with zero tool calls), GPT-5.1 (50.4%), Claude Haiku 4.5 (46.7%) — but they no longer explain most frontier errors.

Among stronger models the story is an evolving frontier. Early termination — stopping before all claims are covered — is the signature failure for one subgroup: Gemini 3.1 Pro Preview (42.8% of its failures), GPT-5.4 (41.4%), Claude Opus 4.6 (36.3%). But the newest checkpoints almost eliminate it: GPT-5.5 and Claude Opus 4.7 terminate early on just 3.6% and 3.8% of failures. The errors don't vanish — they migrate downstream into faulty synthesis (19.8% for GPT-5.5, 25.0% for Opus 4.7). Agents increasingly persist through the whole workflow, then miscombine or misreport the evidence they gathered.

Accuracy isn't free — and reasoning-heavy configs are dominated

Plotting pass rate against mean wall-clock time per task exposes a Pareto frontier of six models, from Claude Haiku 4.5 (27 s, 40.2%) through GLM-4.7 (42 s, 58.1%), Claude Opus 4.5 and Gemini 3 Pro Preview (~50 s, ~70%), Claude Opus 4.7 (71 s, 79.1%), up to Muse Spark (121 s, 82.2%). Several models are strictly dominated: o3 Pro is the slowest in the set (194 s) yet among the least accurate (44.5%); GPT-5.4 Mini is both slower and worse than the full GPT-5.4; and GLM-5.1, despite its strong pass rate, sits off the frontier at ~149 s — roughly twice the wall-clock cost of the Opus models for comparable accuracy.

The three independent judges agree closely. The primary Gemini judge is the most generous (mean pass rate 65.4%), GPT-5.4 the strictest (62.3%), and Claude Opus 4.6 intermediate (63.9%); per-model scores span only 2.1–4.6 percentage points across judges, and rankings are broadly preserved. The few rank swaps that occur are concentrated in the crowded top cluster — Claude Opus 4.7, for instance, ranks 2nd under the Gemini judge but 5th under the Claude judge — which the authors read as evaluator uncertainty rather than real capability gaps.

// does it hold up?

The eval read

MCP-Atlas is the largest real-server MCP benchmark to date, and its most credible contribution isn't the leaderboard — it's the diagnosis. The claim that two-thirds of frontier failures are cognitive, and that the very newest checkpoints trade early-termination errors for synthesis errors, is the kind of mechanism-level finding that survives even if the absolute numbers wobble. But the absolute numbers carry several asterisks worth reading carefully.

It grades models with models. Every score — both the claim rubric and the failure diagnosis — comes from an LLM judge, so the authors rightly caution that pass rates should be read modulo a 2–5 point band. The three-judge protocol is a real mitigation and ranks are stable, but two things temper it: the headline leaderboard uses the most generous of the three judges (Gemini 3.1 Pro, mean 65.4% vs GPT-5.4's 62.3%), and that primary judge is itself one of the evaluated models, ranking third on its own leaderboard. The authors check for self-preference and find the Claude judge doesn't favor Claude — a good test — but a vendor benchmark whose top numbers come from a judge that is also a contestant deserves the caveat stated up front.

The public/private gap is exactly the shape exposure would take. With the 500-task public split released (prompts, claims, and reference trajectories), most models score higher on public than private tasks, by −0.8 to +9.6 pp — and the gaps cluster by provider: OpenAI +8.4 pp on average (GPT-5.4 at +9.6), Google +5.2, Anthropic +4.8, open-source +3.8, with Kimi K2.5 the lone model scoring higher on private (−0.8). The authors are careful to frame this as a split-sensitivity "audit signal," not proof of contamination, since publishing trajectories can inflate public scores by other routes. Fair — but a consistent, provider-clustered positive gap of up to ~10 points is unresolved, and the released numbers are the ones the public can verify. The 500-task held-out split is the right structural defense for the forward leaderboard.

Several contestants aren't externally checkable. The leaderboard is topped by internal or pre-release checkpoints — Muse Spark (a Meta model), Claude Opus 4.7, Gemini 3.1 Pro Preview, GPT-5.5, GLM-5.1 — evaluated against a May 2026 snapshot of live servers. That makes independent reproduction hard on two fronts: the models may not be publicly available, and the servers drift (rate limits, auth schemes, schema versions change without notice, and the authors concede a few tasks may become unsolvable or trivial over time). Container pinning bounds but doesn't eliminate this. Add the provider-specific native tool-calling adapters (anthropic-native, openai-responses-api) — published best practice, but not an identical harness across vendors — and cross-provider comparisons inherit a fairness caveat.

What the headline metric still misses. Pass@0.75 is a threshold on a real-valued coverage score; the 0.50↔0.75 Spearman correlation is high (ρ = 0.965), so rankings are robust to the choice, but the threshold flatters the top (Muse Spark drops only 89.6→82.2 from pass@0.50 to @0.75, while Haiku falls 55.8→40.2). And a residual claim-extraction error band means even a perfect agent can't reach 100%, since some claims are recoverable only under ambiguous prompt readings. Finally, the setting is deliberately narrow: single-turn (no clarification or dynamic replanning), English-only, capped at 100 tool calls per task.

baselines

20 models across six providers under matched task-level conditions — strong breadth. But several top entries are internal/pre-release checkpoints, and the native tool-calling adapters differ across vendors, so the harness isn't byte-identical between providers.

contamination

500-task public split released; public-minus-private gaps run up to +9.6 pp and cluster by provider (OpenAI +8.4). Framed as an audit signal, not proven leakage — but unresolved. A 500-task private split guards the forward leaderboard.

what's missed

Single-turn only (no clarification/replanning), English-only, 100-call budget; all scoring by LLM judges within a 2–5 pp cross-judge band; a claim-extraction error band caps even a perfect agent below 100%.

would raise confidence

A human-graded sample to anchor the LLM judges; a distractor ablation isolating tool discovery from execution; multi-turn clarification tasks; and published re-scoring runs as the live servers drift.

// bottom line

Take the leaderboard with a vendor-eval, LLM-judge, drifting-server, partly-pre-release-model asterisk — but take the diagnosis to heart. MCP-Atlas's durable result is that for frontier agents, tool mechanics are largely solved; the next gains live in claim-aware stopping and trajectory-grounded verification of the final answer against the evidence collected.

// takeaways

What to remember

  1. Scale, realism, and structure at once. 1,000 expert-authored tasks over 36 live MCP servers and 220 real tools, with answer-centric claims scoring that credits any valid tool path — and distractors baked in so 98.6% of tasks demand genuine tool discovery across two or more servers.
  2. The frontier is no longer tool-mechanics-bound. Of ~6,900 diagnosed failures, 63.3% are cognitive, not tool-call errors. Top models fail by stopping early or mis-synthesizing evidence; the newest checkpoints (GPT-5.5, Opus 4.7) nearly eliminate early termination but shift the error into faulty synthesis (up to 25%).
  3. Reasoning skill ≠ tool-use policy. o3 Pro, strong on math and code, lands at 44.5% — slowest and near-worst — because it issues no tool call on 40.1% of its failed tasks. Meanwhile open-weight GLM-5.1 cracks the top band at 75.6%, narrowing the proprietary lead on agentic tool use.
