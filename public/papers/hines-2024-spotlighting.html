Defending Against Indirect Prompt Injection Attacks With Spotlighting

Three prompt-engineering techniques — delimiting, datamarking, and encoding — that mark where untrusted data begins and ends, cutting GPT-family models' susceptibility to indirect prompt injection from over 50% to below 2% attack success, with no measured hit to task accuracy for the recommended datamarking default.

// the gap

What problem is this solving?

Large language models are built to consume a single, boundary-less stream of tokens. When an application concatenates a system prompt with untrusted external content — a web page, a retrieved document, an email — the model has no structural way to tell where the trusted "code" (its instructions) ends and the untrusted "data" it's supposed to merely process begins. Indirect prompt injection (XPIA) exploits exactly that blind spot: an attacker who cannot talk to the model directly instead plants instruction-shaped text inside a data source the model will later read. Because LLMs are "eager" to follow anything that looks like an instruction, the model can mistake the attacker's text for a command from the legitimate user — hijacking the session with the user's own credentials while the user remains an unwitting bystander.

This is a different, and in the authors' view harder, problem than the more familiar direct jailbreak (what the paper calls UPIA — user prompt injection attacks), where a user directly tries to subvert the model's own safety training. XPIA's language distribution is different: a typical attack is a small, sometimes perceptually invisible, instruction buried inside an otherwise long, benign document. Prior defenses for the direct case — alignment tuning, safety fine-tuning — don't transfer cleanly, because the exploit here isn't the model producing harmful content, it's the model being structurally unable to tell whose instructions it's following. The paper builds directly on Yi et al.'s BIPIA framework for measuring the problem and sets out to test whether purely prompt-side, no-retraining-required defenses can close the gap. As the baseline chart below shows, with no defense at all the vulnerability is severe and inconsistent across models and tasks — GPT-3.5-Turbo and text-davinci-003 in particular are hijacked well over half the time on a summarization task with no defense present.

// how it works

The method

Spotlighting is a family of three prompt-engineering techniques, each pairing an (optional) transformation of the untrusted input with an explicit instruction in the system prompt telling the model how that transformation should be interpreted. All three are "black-box" — no fine-tuning, no model access beyond the prompt — which is the paper's whole point: something a development team can ship today.

Delimiting is the simplest instantiation and mostly a restatement of prior work (Yi et al.): wrap the untrusted block in special marker tokens (e.g. << … >>) and tell the model never to obey instructions between them. Datamarking goes further by interleaving a special token throughout the entire input rather than just at its edges — e.g. replacing every whitespace character with ^, so "In this manner Cosette traversed…" becomes "In^this^manner^Cosette^traversed…" — giving the model a continuous, not just boundary, signal of provenance. Encoding is the strongest variant: the entire untrusted block is transformed with a well-known reversible encoding (base64, ROT13, binary) before being placed in the prompt; sufficiently capable models can implicitly decode it while still working with its content, but the transformation makes the block visually and structurally unmistakable as untrusted data rather than instructions.

The authors are explicit that not all instantiations are equally safe to ship. A static delimiter or a naive datamarking token (interleaved only at whitespace, so an attack string with no spaces slips through unmarked) can be trivially defeated by an attacker who has learned the system prompt — they simply craft input containing the same marker to escape it. The recommended fix is to make the marking dynamic: generate a fresh random k-gram marking token from a character pool of size N before every model call, so a leaked system prompt only ever exposes a token that will never be reused.

Pr[adversary guesses the marking token] = 1Nk // N = size of the character pool the k-gram token is drawn from, k = token length

Encoding has an analogous subtlety: a naive choice like ROT13 is bidirectional and trivially subverted — an attacker can pre-compute the ROT13 encoding of their desired plaintext attack and embed that, so the "defended" decode step hands the model exactly the attack they intended. The paper's recommendation is a one-way transformation such as base64 that the attacker cannot invert in their favor while still being cheaply decodable by the model.

// what they found

The results

All experiments use black-box GPT-family models — text-davinci-003, GPT-3.5-Turbo (June 2023), and GPT-4 (June 2023) at temperature 1.0 — measured against a synthetic corpus of 1,000 documents containing a simple keyword-payload attack (the injected text instructs the model to abandon its task and emit a single canary keyword). The headline number: across techniques and tasks, spotlighting reduces Attack Success Rate from greater than 50% to below 2%, with encoding as the strongest single technique.

Delimiting alone is the weakest of the three: with GPT-3.5-Turbo the baseline ASR of roughly 60% is only cut by about half by adding delimiters, and the authors don't recommend it in practice — "this kind of defense could be easily subverted by an attacker who gains knowledge of our system prompt." Datamarking is a much larger jump: on a document summarization task it takes GPT-3.5-Turbo's ASR to 3.10% and text-davinci-003's to 0.00% (Figure 4). On a document Q&A task, datamarking reduces ASR to 8.0% for GPT-3.5-Turbo, 1.0% for GPT-4, and 0.00% for text-davinci-003 (Figure 5, below).

Encoding pushes furthest still: with GPT-3.5-Turbo, encoding brings ASR to 0.0% on summarization and 1.8% on Q&A (Figure 6, below) — the best result in the paper. The catch is capability-dependence, covered in the critique section: encoding only works cleanly with high-capacity models.

On the utility side, the paper checks whether these transformations quietly break the underlying task. Across four benchmarks — SQuAD Q&A, IMDB Sentiment, SuperGLUE Word-in-Context, and SuperGLUE BoolQ — datamarking showed no detrimental effect on task performance. Encoding is a different story: GPT-4 held up on most of the four benchmarks (though it still lost roughly 10 points of accuracy on the Word-in-Context task), while GPT-3.5-Turbo's task accuracy dropped noticeably across the board, since it more often fails to decode cleanly or hallucinates during decoding. The paper's recommendation follows directly from this asymmetry: use datamarking as a safe default across the model family, and reserve encoding for high-capacity models where it can be validated not to hurt the task at hand. A separate appendix experiment adds a handful of few-shot examples demonstrating how to spot and refuse an injection to the system prompt, which also drove GPT-3.5-Turbo's ASR below 5% — but the authors flag this result themselves as fragile, since it's hard to keep the few-shot examples from correlating with (i.e., "leaking the label" for) the test attacks.

// does it hold up?

The eval read

The reductions are real and the authors are unusually candid about spotlighting's own weak points — but the entire evaluation is against a single, static, non-adaptive attacker. The 1,000-document corpus is built from "variations on a simple keyword payload attack": the injected text tells the model to forget its task and emit one canary word. That's a clean, easy-to-score signal (the paper's own Appendix walks through exactly how ambiguous cases are resolved), but it is also the easiest class of injection to catch — a single, syntactically obvious command with no attempt to blend into the surrounding text, disguise itself, or survive the datamarking/encoding transform itself. Nowhere in the paper is spotlighting tested against an attacker who knows a marking or encoding defense is in place and optimizes against it.

That gap is not hypothetical — the paper's own "Adversary Considerations" section reasons qualitatively about exactly this: a static delimiter or whitespace-only datamarking token can be bypassed by an attacker who has learned the system prompt, and a naive ROT13 encoding can be subverted by pre-computing the ROT13 of the desired plaintext attack. The authors propose fixes (dynamic per-call marking tokens, one-way encodings like base64) but never actually run an attacker that tries these bypasses, or a learned/optimization-based trigger search of the kind used elsewhere in the same red-teaming literature (e.g. GCG-style suffix search) against the marked or encoded prompt. The defense is evaluated only against the threat model it was designed to catch.

The utility evaluation has a similar asymmetry: datamarking's no-detrimental-effect claim is checked across four benchmarks, but only on GPT-3.5-Turbo — GPT-4 and text-davinci-003 (the two other models used throughout the ASR experiments) are absent from the task-utility table, so the claim that datamarking costs nothing in task performance is really only demonstrated for one model in the family. And because the strongest technique (encoding) is explicitly capability-gated — the paper recommends it only "with the highest-capacity models (e.g. GPT-4)" — the headline abstract number (<2% ASR) is not achievable everywhere the technique might be deployed; teams running smaller or open-weight models get datamarking's weaker (but still substantial) protection.

baselines

Compared only against no defense and a plain system-prompt instruction telling the model not to take instructions from the document. No comparison to concurrent structured-input defenses; the paper explicitly frames itself as extending Yi et al.'s BIPIA work rather than competing against a broader defense literature.

contamination

Not really applicable in the usual train/test sense — this is a prompt-engineering defense with no training step — but the 1,000-document attack corpus is synthetic, self-authored, and (as far as the paper states) unreleased, so results can't be independently replicated on the exact same test set.

what's missed

Any adaptive attacker — one that knows spotlighting is deployed and crafts input to survive the transform, rather than a fixed keyword-canary payload. Also missing: task-utility numbers for datamarking on GPT-4/text-davinci-003, and any agentic or multi-turn setting (the eval is single-shot summarization/Q&A).

would raise confidence

An adaptive-attack evaluation — even something as simple as hand-crafting injections that survive whitespace-based datamarking, or running an optimization-based trigger search against the encoded prompt — plus a released attack corpus and a full model × technique task-utility matrix.

// bottom line

Spotlighting is a well-executed, deployable answer to the easy version of indirect prompt injection — a fixed, unsophisticated keyword-payload attacker — and the authors' own discussion of how each technique could be broken is more rigorous than most defense papers bother to be. But "more rigorous discussion" isn't "more rigorous evaluation": none of the bypasses the paper itself describes are ever actually run. The paper this site pairs it with next, Neural Exec, is exactly the missing half of this picture — an optimization-based trigger search that treats a defense's marking/encoding transform as just another obstacle to route around, which is precisely the adaptive attacker spotlighting's own evaluation never faces.

// takeaways

What to remember

  1. Spotlighting is a family of three prompt-engineering defenses of increasing strength — delimiting, datamarking, encoding — that give an LLM a continuous "provenance signal" so it can distinguish trusted instructions from untrusted external data, cutting a keyword-payload indirect-injection attack from over 50% to below 2% success at its strongest.
  2. The techniques trade generality for effectiveness: datamarking is the recommended default, reducing ASR across all three tested models with no measured task-accuracy cost on GPT-3.5-Turbo (the only model checked for utility), while encoding is the strongest defense but only recommended for high-capacity models like GPT-4 — weaker models like GPT-3.5-Turbo lose task accuracy when forced to decode the transformed input.
  3. The evaluation is entirely against a static, non-adaptive keyword-canary attacker. The paper is candid that naive delimiters and ROT13-style encodings can be bypassed by an attacker who knows the defense is in place, but never actually runs that attacker — a gap that optimization-based trigger search (as in Neural Exec) is built to exploit.
