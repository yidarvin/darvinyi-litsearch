The BiGGen Bench: A Principled Benchmark for Fine-grained Evaluation of Language Models with Language Models

Most generation benchmarks score a response against one generic yardstick — "helpfulness" — or against 12 fixed domain criteria. BiGGen Bench instead writes a bespoke scoring rubric for every one of 765 test instances (e.g. "does the rationale substitute the variable x,y,z multiple times to reduce the value 560 while solving the problem?"), evaluates 103 frontier LMs across nine capabilities with five different judge LMs, and shows that this instance-level granularity correlates with human judgment better than either coarse-grained or domain-specific criteria.

// the gap

What problem is this solving?

Generation benchmarks that use LLM judges have converged on a small set of abstract evaluation criteria — "helpfulness," "harmlessness," "relevance," "level of detail" — as a catch-all proxy for response quality (the family the authors trace through Chatbot Arena/MT-Bench, AlpacaEval, and ChatEval). The problem: a single abstract criterion cannot capture what actually distinguishes a good response from a bad one on a specific instance. Judging a math rationale for "helpfulness" tells you almost nothing about whether it correctly substitutes variables to shrink an equation; judging a Theory-of-Mind response for "helpfulness" tells you nothing about whether it correctly inferred what a character in a story would do next. The paper's own worked example (Figure 1) is a math word problem — given three positive integers x, y, z satisfying x²+y²+z²=560, find xyz — where a coarse-grained rubric can only ask whether the response is "logically sound"; it has no way to check whether the model actually performed the specific variable-substitution steps the problem requires.

A second, related gap is coverage bias: most generation benchmarks disproportionately probe instruction-following (IFEval, FollowBench, InfoBench) while capabilities like theory of mind, multi-step tool use, or planning get comparatively little dedicated evaluation. And the benchmark most similar in spirit to this one, FLASK, only gets partway there — it assigns each instance one of 12 pre-defined domain-specific criteria (bottom-up, sampled from existing benchmarks), which is finer than "helpfulness" but still a fixed vocabulary applied after the fact, not a criterion designed for that specific instance. BiGGen Bench's premise is that human interviewers know precise, instance-tailored questions produce more diagnostic answers than generic ones (citing Cannell et al. 1981 and Patton 2002 on structured-interview design) — and that LLM judges should be given the same kind of tailored criteria, not a fixed rubric recycled across every instance.

// how it works

The method

BiGGen Bench is built top-down through a fixed hierarchy: capability → task → instance → evaluation criteria. The authors first fix nine capabilities to test — instruction following, grounding, planning, reasoning, refinement, safety, theory of mind, tool usage, and multilingualism — then design 77 tasks under those nine capabilities (roughly 5–11 tasks per capability), and finally write 765 individual test instances across those tasks. Every instance ships with a system message, an input, a reference answer, and a 5-point Likert scoring rubric whose five score descriptions are written specifically for that instance rather than reused. Figure 1 shows the contrast directly on the xyz example: instead of a generic "is this logically sound?" criterion, the instance-specific rubric spells out exactly what a score of 1 through 5 requires — does the response substitute variables once, twice, and correctly recover x=4, y=12, z=20 to compute xyz=960?

Construction ran as a four-step human-in-the-loop pipeline. Step 1: eighteen coauthors, each responsible for one capability, hand-crafted 25 instances across five tasks each; ten native-speaking annotators (Korean, Kazakh, Bengali, Spanish, Indonesian, German, French, Arabic, Russian, Thai) separately wrote ten instances each across ten multilingual tasks. The first author then pruned confusing or unchallenging tasks, cutting 23 tasks and leaving 77 tasks with 385 hand-written instances. Step 2: GPT-4-0125 expanded that seed set, using the human-written instances per task as in-context demonstrations (not zero-shot generation) and, for diversity, generating five candidates per new instance and keeping the one with the lowest BERTScore similarity to existing instances, repeated five times — bringing the total to 770 instances across the same 77 tasks. Step 3: the eighteen coauthors cross-validated instances they had not written (each reviewing two capabilities, ten tasks, fifty instances), revising anything one reviewer flagged and discarding anything both reviewers rejected as a bad task fit, difficulty, reference answer, or rubric — removing 5 instances and landing on the final 765. Step 4: for four of the response LMs (six for multilingual), the authors collected human Likert ratings via a three-stage recruit → qualify → annotate pipeline, which becomes the ground truth used later to validate the LLM judges themselves.

At evaluation time, each instance's system message and input go to a Response LM: post-trained (instruction-tuned/RLHF) models get zero-shot prompting, while pre-trained base models get the URIAL prompt (cross-task 3-shot demonstrations), specifically to avoid "accidental translation" — base models answering in English regardless of the requested language. Because pre-trained LMs can't be fairly zero-shot prompted for multilingual tasks, only post-trained models are scored on that capability. The response is then handed to an Evaluator LM, which reads the instance-specific rubric and produces a single 1–5 Likert score in a direct-assessment format (not pairwise ranking, chosen so a new response LM can be scored without re-comparing it against every existing one), using the prompt template from the authors' own prior work, Prometheus. In total the benchmark evaluates 103 frontier response LMs — 28 pre-trained base models from roughly 1B to 141B parameters, 61 post-trained open chat models, and 14 proprietary models — using five evaluator LMs: GPT-4-Turbo-2024-04-09, GPT-4-1106, Claude-3-Opus, and two variants of the open-source Prometheus-2 8x7B (a base version and Prometheus-2-BGB 8x7B, further trained on GPT-4-1106's own feedback specifically for this benchmark).

// what they found

The results

Base-model performance scales smoothly and log-linearly with parameter size: across the 28 pre-trained models, GPT-4-Turbo's average score regresses against log-scaled parameter count with a correlation coefficient of 0.68 and R² = 0.47 (p = 1.64e−5). Chat models scale far less predictably — 61 post-trained models show correlation 0.47 and R² = 0.22 — which the authors read as evidence that post-training quality, not just parameter count, is doing a lot of the work for instruction-tuned models. Breaking this down by capability (Table 1 in the paper), base-model scaling is tightest for Tool Usage (R²=0.62) and Reasoning (R²=0.51), and weakest for Theory of Mind (R²=0.34).

The base-vs-chat performance gap shrinks as models get larger, but at very different rates per capability: fitting a linear mixed-effects model, every capability shows a statistically significant negative interaction between model size and the base–chat gap (all p<0.001), meaning bigger base models close more of the ground on their chat counterparts. That closing is slowest for Refinement (coefficient −0.05), Reasoning (−0.07), and Grounding (−0.07) — capabilities the authors argue depend on post-training itself, not just scale — and fastest (i.e., gap narrows most) for Theory of Mind (−0.14).

The central validity check is whether instance-specific criteria actually track human judgment better than the alternatives. Using 3,236 human ratings collected over the 765 instances (4 response LMs per instance, 6 for multilingual), GPT-4-Turbo-2024-04-09 reaches the highest single-evaluator average Pearson correlation with humans at 0.623; taking a majority vote across all five evaluator LMs edges it out slightly at 0.627. The open-source Prometheus-2-BGB 8x7B — fine-tuned on GPT-4-1106's own judgments specifically for this benchmark — reaches 0.577 on its own and 0.607 with self-consistency decoding (N=3 or N=5 samples), putting it ahead of Claude-3-Opus (0.578) and roughly matching GPT-4-1106 (0.597) despite being a fraction of the size and fully open. The one consistent soft spot across all five judges is Theory of Mind (0.371–0.533) and Tool Usage (0.412–0.551) — every evaluator LM correlates with humans noticeably worse on these two capabilities than on the other seven.

The granularity ablation (Figure 6, above left) is the paper's headline result: for both GPT-4-Turbo and Prometheus-2-BGB, instance-specific criteria produce higher correlation with human scores than either coarse-grained (MT-Bench-style helpfulness) or domain-specific (FLASK-style, 12 fixed criteria) rubrics — and, notably, Prometheus-2-BGB using instance-specific criteria out-correlates GPT-4-Turbo using coarse-grained criteria, i.e. rubric granularity matters more than judge scale here. Surprisingly, domain-specific criteria score lower than plain coarse-grained criteria in this setup, which directly contradicts FLASK's own reported findings; the authors attribute this to construction order — FLASK maps 12 fixed criteria onto instances after the fact, while BiGGen Bench designs criteria for instances that were built to need them.

Finally, on verbosity bias — the well-documented tendency of LLM judges to reward longer answers — the authors regress response length (in tokens) against assigned score across all 78,795 GPT-4-Turbo judgments spanning the 103 response models: correlation coefficient 0.050, R²=0.049, p=2.69e−42. Statistically significant given the sample size, but the practical effect size is close to negligible, which the authors read as evidence that the detailed per-instance rubric and direct-assessment format keep the benchmark largely free of length gaming.

// does it hold up?

The eval read

The core claim — instance-specific criteria beat coarse-grained and domain-specific criteria — rests on a genuinely well-designed ablation (same evaluator LMs, same instances, only the rubric granularity changes) and the effect is consistent across both a proprietary judge (GPT-4-Turbo) and an open one (Prometheus-2-BGB), which is reassuring. But several parts of the pipeline deserve scrutiny before taking the 103-model leaderboard numbers at face value.

First, roughly half of the final 765 instances are LLM-generated, not human-written: Step 1 produced 385 hand-crafted instances, and Step 2 used GPT-4-0125 to expand that to 770 (later trimmed to 765) using the human instances only as in-context seeds. The rubrics that make this benchmark's headline claim work are therefore, for about half the dataset, machine-authored and then human-reviewed — not human-authored from scratch. That's a defensible way to scale a benchmark, but it means the "instance-specific" advantage is partly measuring how well GPT-4 can write good rubrics when shown examples, which is a different claim than "human-crafted instance-specific criteria beat generic ones."

Second, the quality-control step is thin relative to the scale of the claim: Figure 3 shows that cross-validation was done by exactly two reviewers per instance, and an instance was only revised (not independently re-verified) when a single reviewer flagged it. 94.6% of instances were rated "Good" task fit, 87.9% "Good" reference-answer quality, and 96.5% "Good" rubric quality — solid numbers, but they describe agreement between two people, not a broader annotator pool, and 4.4% of reference answers were rated "Bad" by exactly one of the two reviewers (a further 0.3% rated "Bad" by both).

Third, Theory of Mind and Tool Usage are flagged by the authors themselves as capabilities where every evaluator LM, including majority voting across all five, correlates comparatively poorly with humans (0.37–0.55 vs. 0.6–0.7+ elsewhere) — which means the leaderboard rankings on exactly those two capabilities should be trusted least, even though they're reported with the same apparent precision as the rest.

Fourth, this is a static, publicly released benchmark with reference answers and rubrics posted on GitHub and HuggingFace — the standard staleness/contamination risk applies to every model trained after mid-2024, and the paper (evaluating 103 models as of publication) already reads as a snapshot rather than a living leaderboard for anyone comparing against 2025–26 frontier models, notwithstanding the hosted leaderboard the authors maintain separately.

baselines

The ablation against MT-Bench-style coarse-grained and FLASK-style domain-specific criteria is a fair, controlled, same-instance comparison — the paper's strongest piece of evidence.

contamination

Not addressed directly; it's a static public benchmark with rubrics and reference answers openly released, so the usual leakage risk grows for any model trained after the paper's release.

what's missed

No error bars or per-instance variance are reported for the 103-model leaderboard scores themselves — only for the regression/correlation statistics — so it's unclear how much of a model-to-model score gap is noise versus signal.

would raise confidence

A larger, independent (not coauthor) annotator pool for the quality-control pass; a breakdown of the granularity ablation's advantage restricted to only the human-written 385 instances, to isolate the effect from GPT-4-generated rubrics; and dedicated follow-up work on why Theory of Mind and Tool Usage evaluation lag for every judge tested.

// bottom line

The instance-specific-criteria result is real and well-isolated by the ablation, and the human-correlation validation (0.62 for the best single judge, matched by a small open-source model with self-consistency) is a genuinely useful contribution to LLM-as-a-judge methodology. The 103-model leaderboard itself should be read with the caveats above in mind — thin two-reviewer quality control, machine-generated rubrics for roughly half the instances, and known weak spots in Theory of Mind and Tool Usage — rather than as a precise ranking.

// takeaways

What to remember

  1. Rubric granularity matters more than judge scale: instance-specific evaluation criteria correlate with human judgment better than coarse-grained or FLASK-style domain-specific criteria, for both a small open evaluator (Prometheus-2-BGB 8x7B) and GPT-4-Turbo — and, surprisingly, FLASK's 12 fixed domain criteria underperform plain coarse-grained criteria in this setup.
  2. Base-model scaling is smooth and predictable (corr 0.68, R²=0.47), but chat-model improvement is not (corr 0.47, R²=0.22) — post-training quality explains a lot of the variance that parameter count alone doesn't, and the base→chat gap closes slowest for Refinement, Reasoning, and Grounding.
  3. An open-source judge fine-tuned on GPT-4 feedback (Prometheus-2-BGB 8x7B) reaches human-correlation parity with GPT-4-1106 and Claude-3-Opus once boosted with self-consistency decoding — but Theory of Mind and Tool Usage remain the two capabilities every evaluator LM, proprietary or open, struggles hardest to judge accurately.
