Progress over Points: Reframing LM Benchmarks Around Scientific Objectives

A position paper with a working demo: stop scoring models on already-solved puzzles and instead measure progress on objectives that are themselves scientific targets — instantiated as a NanoGPT-speedrun environment where an evolutionary agent loop set a new training-time record.

Jin, Hendryx, Nath

// the gap

What problem is this solving?

Progress in large language models is almost always tracked through static evaluations of already-solved problems — competition math and coding challenges, multiple-choice academic exams (MMLU, GSM8K, MATH, GPQA), and curated replication tasks like PaperBench. These were invaluable for charting whether models could acquire basic reasoning skills. But the field's reflex when a benchmark saturates has been to build a bigger, harder collection of the same kind of problem: even Humanity's Last Exam positions itself as the "final" academic benchmark by assembling thousands of frontier questions across subjects.

Every one of these shares a ceiling the authors want to call out. The problems are already solved, the answers are closed-ended, and topping the leaderboard means matching a known key — not producing new knowledge. So "doing well on the benchmark" optimizes for narrow point-gains, not scientific advancement. The authors argue this inadvertently constrains the kinds of advances we can even measure or incentivize.

Even the recent wave of "open-ended" benchmarks doesn't escape this. DSBench and MLE-bench evaluate agents on Kaggle-style ML engineering, but each task has an upper bound on its score. PaperBench, EXP-Bench, and the Automated LLM Speedrunning Benchmark test whether systems can reproduce prior scientific advances — science-flavored, but still judged against a known result. LLM-SRBench's equation-discovery tasks are typically rearrangements of existing equations under controlled data. The gap, then: there is no benchmark whose objective is an open scientific frontier, where beating the benchmark is the science.

// how it works

The method

The core proposal is a progress-oriented benchmark: a problem environment whose objective is itself a target of scientific progress, so that achieving state of the art on it advances the field. As a concrete first instance, the authors wrap the community NanoGPT speedrun (Keller Jordan's modded-nanogpt): the challenge of training a GPT to ≤ 3.28 cross-entropy validation loss on FineWeb, on a single 8×H100 node, as fast as possible. Since June 2024 that record has fallen from 45 minutes to under 3 minutes, driven by genuine algorithmic advances — the Muon optimizer, precision changes — that routinely generalize beyond the task. That generalization is exactly why it makes a good discovery substrate: improvements to NanoGPT training tend to be improvements to language modeling.

The environment standardizes the data slice, reference model, and training harness, and then emits a rich reward surface rather than a single score: validation loss and total training time (the headline pair), plus step-level incremental metrics, code-section profiling (forward pass, loss backward, optimizer steps, data loading — total/average times, % of run, call counts), the 15 most time-consuming CUDA kernels and CPU ops, and overall token throughput and peak memory. The intent is that an optimizer — model or agent — has fine-grained signal to act on, not just a final number.

The evolutionary loop

To drive discovery they fork the open-source OpenEvolve (an AlphaEvolve-style evolutionary coding system). Four components run in a cycle: a database D of past programs and their metrics, using island evolution with periodic elite migration to keep quality and diversity; a prompt sampler that compiles a set of top programs T and a set of diverse non-elite programs into one prompt and asks the model for changes as aider-style search/replace blocks; an evaluator; and a pluggable language model. Each cycle samples a parent program, prompts the LM for an improved child, evaluates it, and stores it with its feedback.

The creative engine is two-stage meta-prompting: rather than ask for a code edit directly, the system first asks the LM for a natural-language idea sketch emphasizing novelty, then re-prompts it to implement that sketch as search/replace blocks. Decoupling idea generation from code generation is meant to lower cognitive load and raise creativity. The evaluator runs a fast error-catcher first — candidates are briefly executed, and compile/runtime errors are sent back to the LM for fixing up to Nfast times before a full run — and the database is ordered by a single scalar score:

sc = tstep · ℓval // database ranking score: step-average training time × final validation loss — lower is better

Anti-gaming is the crux

A flexible optimization environment that lets a model rewrite arbitrary code is wide open to reward hacking — trivial programs that maximize the signal without doing the intended work. Two guards make it a real benchmark. First, runtime injection of evaluation parameters: the environment overrides any model edits to the exact training and validation data slices and the validation sequence length at evaluation time. Second, immutable core logic: it injects its own cross-entropy loss (so a model can't swap in a trivial, non-equivalent loss) and freezes the training-document and causal masks (so tokens can't attend across documents or to future tokens). The authors are candid that this bans a whole class of legitimate loss-function innovations as a necessary trade-off, and flag verifying functional equivalence as future work.

// what they found

The results

The headline run instantiates the system with o3 over 90 iterations, seeded from the August 2025 leading NanoGPT code (branching factor 10, elite archive size 20, Nfast=3, with meta-prompting disabled for the first 20 iterations to build a strong parent pool). Across roughly 900 programs it set a new state-of-the-art training time of 172.68s at 3.280 validation loss — beating the 175.2s current record (3.2797 loss) and the 176.7s previous SOTA. The paper frames this ~2.5s cut as "improving upon the previous record by 3 seconds," and notes the margin "mirrors the previous human-set gains."

The record-setting change was concrete and small: intelligently casting down the precision of optimizer operations on the large hidden-layer weight matrices — keeping Muon's hidden matrices in bf16 while leaving the Adam-optimized tensors in fp32. Meta-prompting mattered: turning it on (the red line in the figure) sharply lowered the rate of successful programs but greatly raised the quality of the ones that worked — and was necessary for the SOTA discovery. The persistent downward trend line and low meta-prompt success rate are read as evidence of headroom for stronger models or longer runs.

Comparison as a diagnostic, not the goal

The new paradigm still allows model comparison — but the authors insist it is "a means, not the end." Running the same setup for 50 iterations each on three more frontier models, Claude 4 (Sonnet) posts the fastest time at 173.28s, ahead of GPT-5 Thinking (174.48s) and Gemini 2.5 Pro (174.78s) — all three beat the 175.2s current record. Yet the endpoint ranking is hedged by trend reading: GPT-5 Thinking's downward trend suggests it would pull ahead with more iterations; Gemini's flat trend suggests weaker idea generation; and Claude struggles to produce acceptable programs consistently (its run is dominated by buggy candidates) yet still scores well. Every model shows a spike in buggy-program rate right after iteration 20, when meta-prompting turns on.

Finally, the authors highlight two emergent ideas that compiled and ran (though neither is shown to set a record): GPT-5 proposed a sliding-window-attention scheme where heads share a small core of KV blocks and round-robin the rest into complementary "stripes" — same coverage, fewer operations; Gemini 2.5 proposed an AdaptiveComputeBlock that routes tokens by a difficulty predictor through subpaths ranging from small MLPs to full attention, with token memory and a learnable compute budget. "Not groundbreaking by themselves," but offered as a sign that the open-ended environment surfaces novel ideas at all.

// does it hold up?

The eval read

This is a workshop position paper with a single proof-of-concept demo, and the empirical claims should be read at that weight — which, to the authors' credit, is roughly how they present them. The reframing is the contribution; the numbers are a pilot.

The biggest issue is sample size and variance. The SOTA rests on one 90-iteration o3 run; the cross-model comparison on one 50-iteration run each. There are no repeats, no error bars, and no confidence interval on a ~1.4% (2.52s) time delta — and evolutionary search is stochastic. 172.68 vs 175.2 vs 174.48 vs 173.28 could plausibly sit inside run-to-run noise, and the "3 seconds" headline rounds a 2.52s table delta up. The comparison is also confounded by iteration budget: o3 got 90 iterations while the other three got 50, so the "Claude 4 is strongest" ranking isn't iteration-matched — which is exactly why the paper retreats to trend-line storytelling, but that storytelling is then doing a lot of un-quantified work.

It's also worth being precise about what was discovered. The one record-setting change is a bf16 precision cast on Muon's hidden matrices — and precision tuning is a well-trodden lever in the speedrun community, so whether this is "novel algorithmic discovery" or competent precision search is debatable. The two ideas showcased as genuinely novel (attention striping, the AdaptiveComputeBlock) are explicitly only shown to compile and run, not to improve loss or time, so the "emergence of novel ideas" claim is qualitative. And the entire premise — that speedrun discoveries "often generalize" to language modeling — is asserted from history (Muon), not demonstrated here: this paper's own discovery is never evaluated outside the 3.28-loss / FineWeb / 8×H100 setting.

baselines

Measured against the live NanoGPT records (176.7s previous SOTA, 175.2s current); cross-model comparison spans current frontier models (o3, GPT-5 Thinking, Claude 4, Gemini 2.5 Pro) — but o3 runs 90 iterations vs 50 for the rest, so the ranking is not iteration-matched.

contamination

Not the usual answer-key leakage — the objective is open-ended (beat a time). The real integrity risk is reward hacking, which is why the environment must inject data slices, the loss, and the causal masks at runtime; the authors concede those guards are incomplete (no functional-equivalence check yet).

what's missed

A single-run ~1.4% delta with no variance reported, on one frozen reference snapshot and one hardware setup; the two "novel" ideas are unverified for any speedup; and there is no test that the discovery transfers beyond NanoGPT.

would raise confidence

Repeated runs with error bars; iteration-matched model comparison; an ablation isolating the precision cast's contribution; at least one discovery shown to transfer to a different model/dataset; and a released, de-anonymized environment.

// bottom line

The reframing is the real contribution — "progress on the benchmark is progress on the science" is a genuinely useful lens, and an anti-gaming-wrapped NanoGPT speedrun is a credible first instance. The empirical demo (one o3 run shaving 2.52s via a bf16 cast; four models within ~1.5s of each other) is a proof of concept, not evidence — and is presented honestly as such, with comparisons cast as "a means, not the end." Judge it as a well-argued position paper with an encouraging pilot, not as a measured capability result.

// takeaways

What to remember

  1. Progress-oriented benchmark = an environment whose objective is itself an open scientific target, so topping the leaderboard advances the field — vs static benchmarks of already-solved, closed-ended problems (MMLU/GSM8K/MATH/HLE) that can only measure point-gains.
  2. The instance: the NanoGPT speedrun (≤3.28 validation loss on FineWeb, single 8×H100, fastest wins), wrapped with a standardized data slice and harness, rich profiling + hardware telemetry, and runtime anti-gaming injection of data slices, the loss function, and the causal masks.
  3. The demo: an AlphaEvolve-style evolutionary loop (an OpenEvolve fork) with o3 over 90 iterations cut the record to 172.68s via a bf16 precision cast on Muon's hidden matrices; two-stage meta-prompting trades success rate for quality and is what enabled the discovery.
  4. Comparison-as-diagnostic: Claude 4 (173.28s) > GPT-5 Thinking (174.48s) > Gemini 2.5 Pro (174.78s) at 50 iterations each — but the intended signal is the trend lines, not the endpoints; model ranking is explicitly a means, not the goal.
  5. Read it at workshop weight: single runs with no variance on a ~1.4% delta, an iteration-mismatched comparison, a frozen reference snapshot, and unverified "novel" ideas — a strong position paper plus an encouraging pilot, not a settled empirical result.
