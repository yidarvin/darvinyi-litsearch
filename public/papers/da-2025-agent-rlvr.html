Agent-RLVR: Training Software Engineering Agents via Guidance and Environment Rewards

RLVR works for math because the reward is dense enough to learn from. Drop it into a multi-step SWE agent and the signal goes nearly silent — a strong open model solves under 10% of tasks, so almost every rollout returns zero. Agent-RLVR breaks the silence with a teacher: train-time guidance that nudges failing agents onto passing trajectories, manufacturing the preference pairs that vanilla RLVR never gets to see.

// the gap

What problem is this solving?

Reinforcement Learning from Verifiable Rewards (RLVR) is the recipe behind the current reasoning models — OpenAI's o1, DeepSeek-R1 — and it works beautifully under two conditions: the reward is checkable (a math answer is right or wrong; a unit test passes or fails) and it is reachable (the model lands on a correct answer often enough that the gradient carries signal). Math and competitive programming satisfy both, so RLVR trains cleanly.

Agentic software-engineering tasks satisfy the first condition and badly violate the second. Resolving a real GitHub issue means reading a natural-language bug report, navigating an unfamiliar repository, editing across multiple files, and passing a hidden test suite — a multi-turn problem hard enough that Qwen-2.5-72B-Instruct solves only 9.4% of SWE-Bench Verified (pass@1). With a single rollout succeeding under one time in ten, the vast majority return reward 0 and the reward landscape is too sparse for conventional RLVR to climb: there is almost nothing to prefer. Worse, the signal is expensive — each task is its own Docker container that has to be built and executed to yield a single scalar reward, so even the sparse positives are slow to collect.

That is the gap Agent-RLVR targets: how do you make verifiable-reward RL effective when correct trajectories are both rare and costly to find? The authors' answer is not a new RL algorithm — it is a way to manufacture reward signal where there was none.

// how it works

The method

Agent-RLVR keeps the verifiable reward (unit tests) but adds agent guidance: a teacher that, after the agent fails a task, hands it targeted hints and lets it retry. The retry succeeds often enough to create the positive trajectories that the unguided policy could not produce — and those become the winning half of a preference pair. Inspired by how a junior engineer learns from a tech lead, the guidance comes in three flavours:

  • Plan — a high-level series of steps, or a pointer to the crux of the problem, derived from the issue description. (Following PlanSearch's finding that an explicit plan widens generation diversity.)
  • Environment feedback — given the stack trace and failing test, the teacher explains, specifically and actionably, what to fix.
  • Environment interaction — a pointer to the file(s) and location(s) that need changing (à la Learn-by-Interact), de-risking the brittle localization phase.

Crucially, guidance is a train-time-only scaffold. At test time the trained model is evaluated exactly like any other — no hints — so the gains have to be baked into the weights, not leaked through the prompt. To keep costs down, the "teacher" here is an external LLM (claude-3-7-sonnet) standing in for a human annotator.

The training loop

One iteration of Agent-RLVR (Algorithm 1) runs in three stages:

  1. Initial attempt. The agent — Qwen-2.5 driving a stripped-down Agentless scaffold (localization + repair only; the expensive test-generation patch-selection step is removed) — attempts each task. The environment runs the unit tests and assigns reward 1 (all pass) or 0.
  2. Generate guidance. For every failed task, the teacher produces guidance and the agent reattempts with it. A newly-passing guided trajectory is paired with that task's earlier failing (unguided) trajectory to form a preference pair.
  3. Update policy. A 20% subset of positive trajectories is used for SFT (to adapt the model to the task format and set the reference policy), then DPO over the preference pairs performs the RLVR step.

The reward keeps a KL leash to the reference policy, and DPO optimizes the preference directly — where the "winner" yw passes all unit tests and the "loser" yl fails one or more:

ℒDPO(θ) = −𝔼(x, yw, yl) log σ ( β [ log πθ(yw|x)πref(yw|x) − log πθ(yl|x)πref(yl|x) ] ) // preference pairs come straight from unit-test outcomes — no learned reward model in the loop

The whole thing runs offline: because guided trajectories are collected up front, the environments can be executed asynchronously, sidestepping the infrastructure burden of online rollouts. The paper is explicit that "experiments in this paper describe one iteration cycle" — this is a single offline round, not a converged online run.

The dataset

The training set is 817 SWE environments — 593 problems from SWE-Gym plus 219 self-collected — across 27 repositories, with each problem packaged as a full Docker runtime plus its test suite. The authors deliberately exclude every repository in the SWE-Bench test set to avoid contamination. Sampling 16 trajectories per problem yields 8,186 trajectories total, of which 4,093 are positive. Models come from the Qwen-2.5 family (14B Coder, 32B Coder, 72B Instruct); SFT runs 5 epochs at lr 1e-5, DPO runs 1 epoch at lr 1e-6. Evaluation is on SWE-Bench Verified using the authors' official harness on Modal.

// what they found

The results

The headline: Agent-RLVR lifts Qwen-2.5-72B-Instruct from 9.4% → 22.4% pass@1 on SWE-Bench Verified — a 2.4× relative jump — using just 817 training environments and a single offline DPO round. Since the base model barely performs the task at all, the model is learning to be a SWE agent almost entirely through the environment reward signal. The effect scales roughly linearly with model size, and it beats a strong rejection-sampling SFT baseline (Agent-SFT, 20.8%).

The reward-model bonus. The same guidance-augmented RLVR data is good for more than policy training: an RM (Qwen-2.5-Coder-32B) trained on it, used to rerank k=32 generated patches at test time, pushes pass@1 from 22.4% to 27.8% "while incurring minimal overhead." That is test-time compute, on a different axis from the policy improvement.

Guidance is the load-bearing ingredient. Re-running the loop but excluding guidance-augmented data from DPO drops performance across the board, and the gap widens with k: pass@1 19.8% → 22.4%, pass@32 34.2% → 38.4% (Figure 2). The interpretation is that guidance-trained models sample more diverse and more accurate generations. Mechanically, guidance also just gets more trajectories over the line: successful rollouts rise from 138 (16.9%) to 165 (20.3%) on average, and the empty-patch rate falls from 8.71% to 7.20% (Table 2).

In context. The authors are careful to frame the absolute number honestly. 22.4–27.8% trails the frontier scaffolds (though it edges out the trained SWE-Gym 32B at 20.6%), but it is achieved with a far smaller training set and a simpler scaffold (SWE-RL trains on 11M instances; SWE-Gym on ~2k; Agent-RLVR on 817):

One nice piece of nuance from the appendix: while guidance helps RL, it actively hurts SFT — SFT trained with guidance scores 16.8 vs 20.8 without (Table 4). Preference learning can exploit the contrast between a guided success and an unguided failure; supervised fine-tuning just overfits to the hint prompt.

// does it hold up?

The eval read

The internal comparisons are clean and well-controlled: base vs RLVR vs SFT all use the same 817 tasks and the same scaffold, the SFT baseline is grid-searched rather than strawmanned, and pass@1 is reported with standard deviations (±3.4–3.8). But the honest read is more measured than the headline. The full pipeline (instruct-tuning on 20% of positive trajectories plus one DPO epoch) lifts the base model +13 points (9.4% → 22.4%) — yet the paper's own rejection-sampling SFT baseline already reaches 20.8%, so the gain attributable to the RL/DPO step over that strong SFT baseline is only +1.6 points. Most of the lift is reproducible without RL at all.

The load-bearing caveat is the teacher. Guidance is generated by claude-3-7-sonnet — a frontier model far stronger than the 72B student (for scale, the paper's own Table 3 benchmarks the closely-related Claude-3.5-Sonnet at 50.8% on this exact suite; it does not report the 3.7 teacher's score). So a meaningful slice of the gain is distillation of a frontier model's hints, not pure self-play RL from environment rewards. The paper never ablates teacher strength (a same-size or weaker teacher), so "RL from verifiable rewards" and "guided distillation from Claude" can't be cleanly separated here. The framing as a generic RLVR recipe slightly oversells what was demonstrated: RLVR bootstrapped by a stronger model's guidance.

On contamination they do the right things and are candid about the residual risk. Training repos are explicitly disjoint from the SWE-Bench test set; but they flag that Verified instances (public GitHub issues/PRs) may overlap the base model's pretraining, and that there is essentially only one human-validated SWE-agent test set — no second benchmark to triangulate. (6 of the 500 Verified instances wouldn't compile in their harness yet are kept in the denominator for consistency — a wash across methods, but worth noting.)

baselines

Internal baselines (base / SFT) are fair and controlled — but the headline obscures how strong the SFT baseline is: rejection-sampling SFT alone hits 20.8%, leaving the RL/DPO step worth only +1.6 (→22.4%). The external Table 3 is explicitly apples-to-oranges (stronger scaffolds): the trained 72B (22.4%) edges out the trained SWE-Gym 32B (20.6%) but trails the rest by ~8 (SWE-Fixer, 30.2%) to ~28 (Claude-3.5, 50.8%) points.

contamination

Train/test repos are disjoint by construction. Residual risks acknowledged: possible pretraining overlap on public GitHub data, and a single human-validated test set with no second benchmark to cross-check.

what's missed

No ablation isolating teacher strength, so guided-distillation vs pure-RL is conflated. Results come from a single offline SFT+DPO cycle (not online PPO/GRPO, which the authors expect would be stronger), greedy pass@1, and Python-only across 27 repos. The 27.8% is test-time reranking over 32 generations, not a better policy.

would raise confidence

A weaker/same-size teacher ablation; an online-RL run to test the "lower bound" claim; a second held-out suite (SWE-Bench Pro — which already cites this — or a non-Python benchmark); a held-out-repo split to probe memorization.

// bottom line

A clean, well-scoped demonstration that what kills RLVR for agents is reward sparsity, not the RL algorithm — and that cheap train-time guidance restores enough signal to more than double a mid-tier open model's SWE-Bench score from one offline round. The absolute numbers trail frontier systems and the gains lean partly on a frontier teacher, but as a recipe for bootstrapping agentic RL from a weak base it's convincing and refreshingly honest about its own limits.

// takeaways

What to remember

  1. RLVR's failure in agentic settings is a reward-sparsity problem, not an algorithm problem. With the base model's pass@1 at only ~9% (and even 16-shot rollout collection clearing just ~17% of problems, Table 2), there is almost no preference signal to learn from; agent guidance manufactures signal by helping failing agents finish, then learning from the guided-success vs unguided-failure contrast.
  2. The numbers: 9.4% → 22.4% pass@1 on SWE-Bench Verified from 817 environments and one offline SFT+DPO cycle, then → 27.8% with a reward model reranking 32 patches at test time. Guidance is load-bearing — the with/without gap widens with k (19.8→22.4 at k=1, 34.2→38.4 at k=32).
  3. Guidance helps RL but hurts SFT (16.8 vs 20.8), and the hints come from a frontier teacher (Claude 3.7) — so the result is as much guided distillation as pure self-improvement. And mind the attribution: the +13-point lift is the whole SFT+DPO pipeline — the RL step alone adds only +1.6 over a plain rejection-sampling SFT baseline (20.8%).
