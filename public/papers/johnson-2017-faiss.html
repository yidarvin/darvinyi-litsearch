Billion-Scale Similarity Search with GPUs

A GPU-native k-selection primitive (WarpSelect) plus a GPU computation layout for product-quantized (IVFADC) indexing that builds a billion-vector k-NN graph in under 12 hours — the open-source engine, later released as FAISS, that today's dense-retrieval and RAG stacks quietly depend on to search millions to billions of embeddings.

Jeff Johnson, Matthijs Douze, Hervé Jégou · Facebook AI Research

// the gap

What problem is this solving?

By 2017, similarity search over high-dimensional embeddings — image descriptors, word2vec-style vectors, CNN features — had become a core operation across ML systems, but doing it at hundreds of millions to billions of vectors was still essentially a CPU-bound problem. Exact search costs O(ℓ) per query against a collection of size ℓ, which doesn't scale; the standard fix, product-quantization indexes such as IVFADC, compress each vector into a short code and restrict the search to a subset of "probed" inverted lists, trading a controlled amount of accuracy for orders-of-magnitude memory savings. GPUs are arithmetically well-suited to the batched distance computations this requires, but the step that actually bottlenecked every prior GPU attempt was k-selection — finding the k smallest of up to millions of per-query distances. K-selection is inherently serial and data-dependent (heaps, quickselect, radix-select all require multiple passes or irregular memory access), which is exactly what GPU SIMD/warp execution handles badly. As a result, existing GPU similarity-search systems were limited to binary codes, small datasets, or brute-force-only search; the single prior system built for billion-scale quantized search on GPU, from Wieschollek et al., is the paper's direct point of comparison. No published system could build or query a billion-vector index end-to-end on GPU without running out of memory or giving up unacceptable amounts of speed to a slow k-selection step.

// how it works

The method

The paper makes two contributions that compose into what became the FAISS library. The first is WarpSelect, a k-selection primitive that keeps all of its working state in GPU register memory rather than shared or global memory, needs only a single pass over the input, and requires no cross-warp synchronization. Each of a warp's 32 lanes maintains a small "thread queue" of the t smallest values it has personally seen; an incoming value is rejected in O(1) if it's larger than the lane's current worst kept value, so most elements never touch anything more expensive. Only when a warp-ballot instruction detects that some lane's thread queue has gotten better than the shared warp-wide k-th best value does the warp pay for a full merge — done with an odd-size Batcher-style sorting/merging network operating on register-held lane-stride arrays — between all 32 thread queues and the running "warp queue" of the k smallest values overall. The choice of thread-queue depth t (2, 3, 4, or 8, depending on k) balances the cost of these two regimes.

The second contribution is a GPU computation layout that plugs WarpSelect into both exact and approximate search. For exact (brute-force) search, the L2 distance reduces to a matrix multiplication XYᵀ via the identity below, computed with cuBLAS GEMM; the paper fuses the k-selection step directly onto the GEMM output so the tiled distance matrix D′ is written once and read once, instead of being written, read, and re-read by a separate selection pass. For approximate search the paper targets IVFADC, the two-level product-quantization index from Jégou et al.: a coarse quantizer q₁ buckets each database vector into one of |C₁|≈√ℓ inverted lists (trained by k-means), and a fine quantizer q₂ encodes the residual as b sub-vector codes of 256 reproduction values each (so b bytes/vector, b up to 64 in general, capped at 48 or 96 on the GPU depending on float precision). The key algebraic move — reusing precomputed per-query lookup tables across every inverted list a query probes — is what turns scanning a list of PQ codes into cheap table lookups rather than repeated arithmetic.

‖x − q(y)‖² = query-independentterm 1 + dist. to q₁(y)term 2 − indep. of listterm 3 // IVFADC's squared distance splits into three precomputable terms (Eq. 11); only term 1 depends on both the query and the probed inverted list, so terms 2–3 are computed once per query and reused across every list it probes

Multi-GPU scaling is handled two ways: replication, where R GPUs each hold a full copy of the index and split the incoming queries (near-linear speedup); and sharding, where S GPUs each hold ℓ/S of the data, each processes the full query batch against its shard, and a final round of k-selection merges the partial results. The two compose into S×R GPUs total, and the paper's largest experiments use exactly this combination.

// what they found

The results

On the k-selection microbenchmark alone (batches of nq=10,000 rows, k=100 or k=1000, array length ℓ up to 128,000), WarpSelect is 1.62× faster than the prior fgknn GPU library at k=100 and 2.01× faster at k=1000 at the largest tested ℓ, while sustaining 55% of theoretical memory-bandwidth-limited peak at k=100 (dropping to 16% of peak at k=1000, since bigger k means bigger merge/sort networks). On exact search over SIFT1M, fusing k-selection with the GEMM-based distance computation reaches 85% of peak possible performance; removing the fusion (one extra pass over D′) makes the identical algorithm at least 25% slower, and a naive thrust::sort_by_key baseline is more than 10× slower than either GPU comparison method.

On GPU k-means over MNIST8m (8.1M images, 784-d), the paper's implementation is more than 2× faster than BIDMach's GPU k-means — 140s vs. 320s at 256 centroids, 316s vs. 735s at 4096 centroids on one GPU — and reaches 84s/100s on 4 GPU replicas (a near-linear 3.16× speedup at 4096 centroids). The headline billion-scale numbers: on SIFT1B (1 billion 128-d SIFT descriptors), at matched memory (8 bytes/vector) and matched hardware, the paper reports R@10 = 0.376 in 17.7 µs/query versus Wieschollek et al.'s reported R@10 = 0.35 in 150 µs/query — the paper's "8.5× faster" claim, at higher accuracy, not just matched accuracy. On DEEP1B (1 billion CNN descriptors), using 4 GPUs (2 shards × 2 replicas) with 20-byte OPQ-preprocessed codes, the system reaches R@1 = 0.4517 in 0.0133 ms/query, versus the Deep1B paper's own single-CPU-thread baseline of R@1 = 0.45 in 20 ms/query — roughly three orders of magnitude faster per query, though (as the authors note) on very different hardware.

The paper's flagship application is k-NN graph construction: a 10-NN graph over 95 million YFCC100M images (ResNet features, PCA'd to d=128) is built — indexing plus exhaustive search — to >0.80 accuracy (10-intersection@10) in 35 minutes on 1×4 GPUs. A 10-NN graph over the full 1-billion-vector DEEP1B is built in just 6 hours at lower quality and about half a day (matching the abstract's headline "under 12 hours" figure) at higher quality, on 4 Titan X GPUs; doubling to 8 Maxwell M40s only buys a sub-linear ~1.6–1.7× speedup. For scale, the largest prior CPU k-NN graph construction the authors cite (NN-Descent, 128-server cluster) took 108.7 hours for just 36.5 million 384-d vectors, and extrapolating a prior GPU brute-force approach to Deep1B's size would take an estimated 200 days.

// does it hold up?

The eval read

The paper is admirably candid about the price of approximation: its own best DEEP1B configuration reaches R@1 = 0.4517 — meaning the single truest nearest neighbor is only recovered for the query about 45% of the time, even after paying for multi-GPU sharding and a 20-byte-per-vector code. The SIFT1B headline result is R@10 = 0.376 — under 4 in 10 queries have their true nearest neighbor land anywhere in the top 10 returned. Speed and memory improve dramatically over exact search, but "billion-scale search" here means "billion-scale approximate search with a real, non-trivial miss rate," and the paper reports that trade-off rather than hiding it (Figure 5's whole point is to show the recall/build-time frontier explicitly). The more interesting question for 2026 is what happens once this recall/latency profile — measured entirely on 128-d SIFT descriptors and one CNN-embedding set (Deep1B/YFCC100M, ~80–120 dimensional) — gets inherited wholesale by systems working in a completely different embedding regime. Every dense-retrieval paper already in this graph (DPR, ANCE, ColBERT, Contriever) drops a FAISS index — usually IVF or IVF-PQ — into its retrieval pipeline over 768-dimensional transformer sentence embeddings, a geometry with different variance structure and anisotropy than the image descriptors this paper benchmarks against. None of those papers re-derive or re-report FAISS's own recall@k on their embedding distribution before shipping it; they treat the ANN step as a black box whose approximation error is assumed negligible relative to the retriever's own ranking error. That assumption is plausible but is rarely checked, which means a genuine failure mode — an index configuration that silently drops relevant passages before the retriever or reranker even sees them — could hide inside "retrieval accuracy" numbers that are actually bottlenecked by the ANN layer underneath, not the learned model on top of it.

baselines

Wieschollek et al. 2016 (the only other GPU system built for billion-scale quantized search at the time) and BIDMach (GPU k-means) — reasonably current for a 2017 systems paper, but narrow. There is no wall-clock comparison against a CPU IVFADC implementation at billion scale, and the Deep1B CPU baseline (R@1 = 0.45 in 20 ms/query) is lifted from a different paper's different codebase and hardware — a comparison the authors themselves flag as cross-platform.

contamination

Not applicable in the train/test-leakage sense — this is a systems benchmark, not a learned model. The nearest analog: SIFT1M/1B ground truth comes from brute-force search on the same descriptor set (a clean gold standard), whereas the Deep1B R@1 comparison mixes ground truth and hardware from an entirely separate paper.

what's missed

Every recall number in the paper comes from two homogeneous, low-dimensional domains (SIFT descriptors and one CNN-embedding set). The library's largest modern user base — dense text retrieval and RAG — indexes much higher-dimensional transformer embeddings, and no downstream paper in this graph re-validates FAISS's recall/latency trade-off for that regime before adopting it.

would raise confidence

Recall/latency curves on modern high-dimensional text embeddings rather than only SIFT/Deep1B; an ablation isolating WarpSelect's own contribution from the rest of the engineering (kernel fusion, tiling, multi-GPU sharding); and an explicit caveat that these specific numbers describe the 2017 IVFADC-on-GPU code path, since the FAISS library has since added substantially different index types (e.g., graph-based indexes) that this paper's benchmarks don't cover.

// bottom line

FAISS's own benchmarks are honest about the accuracy/speed dial — R@1 as low as 0.45 at billion scale is reported plainly, not buried. But every recall curve here is validated on image-descriptor data. Every dense-retrieval paper in this graph that reaches for faiss.IndexIVFPQ is effectively borrowing a recall guarantee measured on SIFT vectors and betting it holds for BERT-family embeddings; that's a defensible bet, but it is a bet, and it is rarely re-checked.

// takeaways

What to remember

  1. The real contribution is WarpSelect, a register-resident, single-pass k-selection primitive for GPUs — not a new index algorithm. IVFADC and product quantization are Jégou et al.'s prior work, re-engineered here for GPU execution.
  2. The paper's "8.5× faster" headline is a matched-memory, matched-hardware comparison against exactly one prior GPU system (Wieschollek et al.) on SIFT1B — not a comparison against CPU FAISS, nor against every ANN approach available at the time.
  3. Recall is a tunable, sometimes low, dial: the paper's own best-latency DEEP1B configuration hits R@1 = 0.4517. The authors report this candidly, but it means "billion-scale search" is billion-scale approximate search with a real miss rate, not a free lunch.
  4. This is why "search a billion vectors" became a solved sub-problem for ML systems: FAISS now underwrites the ANN backend of the dense-retrieval and RAG pipelines already in this graph (DPR, ANCE, ColBERT, Contriever), even though none of them re-validate the approximation quality reported here against their own embedding distribution.
