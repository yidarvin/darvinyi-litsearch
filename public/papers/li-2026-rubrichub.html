RubricHub: A Comprehensive and Highly Discriminative Rubric Dataset via Automated Coarse-to-Fine Generation

Rubrics are how you give open-ended generation a verifiable reward — but the good ones are hand-written, narrow, and too coarse to push a top model. RubricHub manufactures them instead: a fully automated pipeline that grows ~110k question–rubric pairs across five domains, then a two-stage rubric-rejection-sampling + RL recipe that takes a base Qwen3-14B to 69.3 on HealthBench — past GPT-5's 67.2.

Sunzhu Li, Jiale Zhao, Miteto Wei, Huimin Ren, Yang Zhou, Jingwen Yang, Shunyu Liu, Kaike Zhang, Wei Chen — Li Auto / Zhejiang U. / CUHK-Shenzhen / NTU

// the gap

What problem is this solving?

Reinforcement Learning with Verifiable Rewards (RLVR) cracked math and code because those domains hand you a ground-truth checker: the answer is right or it isn't. Most real queries — write a poem, explain a diagnosis, draft a brief — are open-ended, with no answer key. The field's workaround is the rubric: decompose "is this good?" into a weighted checklist of concrete, gradeable criteria, and let an LLM judge each one. A rubric turns a fuzzy preference into something that looks like a verifiable reward.

The catch is that the rubrics themselves are the bottleneck. The paper names three failure modes in the existing rubric supply. (i) Manual expertise: the high-quality rubrics — HealthBench's physician-written criteria, PaperBench's expert checklists — cost expensive human labor and don't scale. (ii) Narrow domain breadth: the automatically-generated sets that do exist (e.g. Rubrics-as-Rewards) are confined to a few specialized domains, useless for a general-purpose model. (iii) Low discriminability: the criteria that get generated automatically tend to be coarse and generic — "is it on-topic? is it coherent?" — so a superficially plausible answer and a genuinely excellent one both score near the top. That ceiling is fatal for training: a criterion every rollout already passes contributes no gradient.

Figure 1 makes the discriminability point concrete. Ask for a short autumn poem; a serviceable answer comes back. A coarse rubric ("related to autumn", "coherent structure") scores it a 1.0 — the model "can't learn anything." A fine-grained rubric adds a criterion the poem fails — "conveys emotion through imagery instead of directly stating feelings" — and the score drops to 0.6, surfacing a specific weakness to optimize against. RubricHub's whole thesis is that the second kind of rubric can be produced automatically, at scale, in every domain.

// how it works

The method

The core is a fully automated Coarse-to-Fine Rubric Generation pipeline (Figure 2a) that runs over a cleaned corpus of ~110k open-ended queries in three stages, each targeting one of the three failure modes above.

Stage 1 — Response-Grounded & Principle-Guided Generation. Generating a rubric from the query alone invites "rubric drift" — criteria that are generic, hallucinated, or unmoored from what a real answer looks like. So the generator is conditioned on a reference response (response grounding, anchoring criteria to concrete output) and constrained by a fixed set of meta-principles Pmeta — Consistency & Alignment, Structure & Scope, Clarity & Quality, Reasoning & Evaluability — producing a candidate rubric Rcand.

Stage 2 — Multi-Model Aggregation. Any single model bakes its own blind spots into a rubric. The pipeline synthesizes parallel candidate sets from heterogeneous frontier models (GPT-5.1, Gemini 3 Pro Preview), pools them, then distills the pool with an aggregation prompt Pagg into a compact, de-duplicated base rubric Rbase — broad coverage, single-source bias removed.

Stage 3 — Difficulty Evolution. A base rubric captures correctness but saturates: top models max it out and lose their gradient. So the pipeline picks a pair of consensus-high reference responses, and an augmentation prompt Paug mines the discriminative nuances that separate "excellent" from "exceptional," producing additive criteria Radd. This is where a generic "Is the code correct?" hardens into "Does the code handle the edge case in O(n)?". The final rubric merges the two: Rfinal = Rbase ∪ Radd.

The resulting dataset, RubricHub (~110k query–rubric pairs), spans five domains with very different criterion densities (Figure 3): Medical 27.1% (avg 31 criteria/query), Science 27.1% (28), Instruction Following 20.9% (5), Writing 15.9% (32), and Chat 9.0% (25). Queries are mined from existing open-ended sets — II-Medical, RaR-science, ResearchQA, MegaScience, IFTRAIN, LongWriter / LongWriter-Zero / DeepWriting-20K / LongAlign, WildChat-1M and LMSYS-1M — then filtered for length and formatting. Crucially, even a top model like Qwen3-235B averages only ~0.6 on these rubrics, confirming the evolved criteria stay non-saturated and leave headroom to optimize.

Turning a rubric into training signal

Two post-training stages consume the rubrics (Figure 2b). RuFT (Rubric-based Rejection-Sampling Fine-Tuning): for each query, sample K candidate answers, score each by the fraction of weighted criteria it satisfies (normalized by the rubric's max), discard any below a threshold τ, and keep the single best as an SFT target — a curated 30K-instance cold start. RuRL (Rubric-based RL): a unified grader G assigns each criterion a binary pass/fail bi (an LLM judge for semantic criteria, a rule for verifiable ones), and the dense reward is the weight-normalized sum, optimized with DAPO in the verl framework, separately per domain.

The results

The headline: post-training the base Qwen3-14B (not the released Instruct) with RuFT→RuRL reaches 69.3 on HealthBench, edging the frontier GPT-5 (67.2) despite being a fraction of the size — a +46.5 jump from the 22.8 base score, and +22.6 over Qwen3-14B's own Non-thinking Instruct release (46.7). The stage ordering is strictly monotonic everywhere: Base < RuFT < RuRL < RuFT→RuRL.

The most dramatic single number is general chat: on ArenaHard V2, Qwen3-14B climbs from 5.2 (base) to 74.4 (+69.2), beating GPT-4.1 (49.1) and DeepSeek V3.1 (62.4). Across the full nine-benchmark sweep the RuFT→RuRL 14B lands at IFEval 92.6 (+43.1), IFBench 51.4, WritingBench 79.4, CreateWriting-V3 70.4, LLMEval-Med 83.2, ResearchQA 86.2, and GPQA-Diamond 58.5. It beats the larger Baichuan-M2-32B in 4 of 5 domains, and surpasses GPT-4.1 and DeepSeek V3.1 on both IFEval and ArenaHard V2.

Is the win from the rubrics, or just from more RL? The cleanest control swaps only the rubric source. Taking the public Rubrics-as-Rewards (RaR) medical questions and re-generating their rubrics with RubricHub's pipeline lifts HealthBench from 47.7 → 62.1 with everything else held fixed; adding the full RubricHub dataset reaches 66.2, and the full RuFT→RuRL pipeline 69.3 (Figure 5). The same ladder holds on ResearchQA (76.7 → 82.5 → 85.5 → 86.2) and, more modestly, GPQA-Diamond (52.1 → 53.9 → 58.4 → 58.5).

The ablations back the three-stage design. Each generation component adds monotonically on HealthBench / LLMEval-Med: naïve single-model generation 60.9 / 71.7 → +Principle-Guided & Response-Grounded 63.8 / 74.1 → +Multi-Model Aggregation 65.0 / 75.6 → +Difficulty Evolution 66.2 / 79.5. On the RuFT side, raising rejection-sampling candidates from 1 to 12 lifts the selected training-set rubric score from 63.5 to 79.5 and HealthBench from 43.6 to 48.8. One revealing knob: positive-only criteria beat positive-plus-penalty (HealthBench 66.2 vs 63.2), which the authors attribute to graders being unreliable on negative criteria.

// does it hold up?

The eval read

The engineering is clean and the gains are large, but the headline — "a 14B model surpasses GPT-5 on HealthBench" — has to be read for what it actually measures. The entire loop is model-authored and model-graded. The rubrics are written by GPT-5.1 and Gemini 3 Pro, the rewards are assigned by a gpt-oss-120B judge, and the flagship benchmark (HealthBench) is itself a rubric benchmark scored by an LLM. So "69.3 > 67.2" doesn't establish that the model is medically better than GPT-5; it establishes that RuFT→RuRL learned to satisfy an LLM grader's rubric extremely well. That is precisely the regime where reward hacking lives — and notably, the follow-up that cites this paper (Mahmoud et al., "Reward Hacking in Rubric-Based RL") shows rubric-RL buys completeness while shedding conciseness, relevance, and factual correctness, because rubric weight is overwhelmingly presence-based.

Two design choices make that worry concrete rather than hypothetical. First, the positive-only criteria win (Figure 6) is explained by the grader being bad at penalties — i.e. the reward is tuned toward what the judge can reliably reward, which is exactly the presence/completeness axis that inflates rubric scores without inflating quality. Second, the human-agreement evidence (Figure 7: Cohen's κ up to 0.80, F1 0.91 across 940 criteria) validates that the grader matches humans on judging individual criteria — it says nothing about whether the trained model's answers are better by human preference. There is no human head-to-head of the final model against GPT-5 anywhere in the paper.

Two more cracks. The frontier comparisons aren't apples-to-apples: several proprietary scores — including GPT-5's HealthBench 67.2, and the Gemini-3 / WritingBench / CreateWriting numbers — are marked †, "reported from official blogs, technical reports, or leaderboards," not run under the paper's harness. And the SOTA story is domain-shaped: the win is on open-ended, model-graded benchmarks (HealthBench, ArenaHard V2, WritingBench), while on the one genuinely verifiable science benchmark, GPQA-Diamond, the model reaches 58.5 — a real gain, but far below Gemini 3 Pro's 90.8 and GPT-5's 85.7. RubricHub lifts most exactly where grader-preference optimization helps most.

baselines

A strong cast — Gemini 3 Pro, GPT-5.1, GPT-4.1, DeepSeek V3.1, Rubicon, Baichuan-M2-32B, RaR — but many proprietary numbers are reported "†" values, not re-run, so "beats GPT-5" isn't measured under one harness.

contamination

Training queries are mined from WildChat / LMSYS / II-Medical / ResearchQA / RaR; no leakage analysis between those query pools and the HealthBench / GPQA / ArenaHard eval sets.

what's missed

Every headline metric is LLM-graded. No human evaluation of the final model's answers; positive-only, presence-based rubrics bias the reward toward completeness over conciseness, relevance, and factual correctness.

would raise confidence

A blinded human head-to-head of the trained 14B vs GPT-5; an explicit reward-hacking probe (length, sycophancy, cross-family judges); seeds / CIs; and verifiable-domain gains (GPQA) that match the open-ended ones.

// bottom line

A genuinely useful, scalable recipe for manufacturing discriminative rubrics, with real and large post-training gains on open-ended tasks. But the win is reported as a single run with no variance, and the whole signal lives inside a rubric-write → rubric-grade → rubric-train loop — so "surpasses GPT-5 on HealthBench" is best read as "learned the grader's rubric preferences very well," a claim a cross-family human evaluation has yet to confirm.

// takeaways

What to remember

  1. Difficulty evolution is the load-bearing idea. Pushing rubrics past "is it correct?" to "what separates exceptional from excellent?" keeps even a 235B model at ~0.6, preserving a gradient where coarse rubrics saturate — and it's the last ablation rung (65.0 → 66.2 HealthBench).
  2. A clean cold-start-then-RL template. RuFT (rubric rejection-sampling SFT) → RuRL (DAPO on rubric reward) is a tidy two-stage recipe, and ArenaHard V2 going 5.2 → 74.4 shows how much latent capability a good open-ended reward can unlock from a base model.
  3. Read the headline as a grader result. 69.3 > GPT-5's 67.2 is a model-graded rubric number from a model-written rubric — impressive, but it sits squarely in the loop where reward hacking hides, and the very paper that cites this one formalizes how rubric-RL trades correctness for completeness.
