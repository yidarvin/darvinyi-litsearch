LHAW: Controllable Underspecification for Long-Horizon Tasks

A dataset-agnostic synthetic pipeline that turns any well-specified long-horizon task into controllably underspecified variants — stripping out information across Goal, Constraint, Input, and Context, and validating each variant by what agents actually do — then measures whether frontier agents know when to stop and ask for clarification, and how much it costs.

Pu, Lee, Sehwag, Lee, Zhu, Maurya, Raghavendra, Xue, Denton — Scale AI

// the gap

What problem is this solving?

A long-horizon agent executes dozens to hundreds of sequential actions across multiple stages with minimal supervision. That setting inverts the economics of a chat assistant. In conversation, asking a clarifying question is cheap and often the point; in an autonomous workflow every human interruption carries real cost along three axes the paper names: interruption (a supervisor pays a context-switch), latency (the workflow blocks pending a reply), and trust (an agent that asks constantly erodes confidence). The interesting question is no longer whether an agent can execute a well-specified task — it is whether the agent can detect when the spec is insufficient and decide whether clarification is worth its price.

Existing benchmarks miss this tension from both sides. Long-horizon execution suites — TheAgentCompany, SWE-Bench Pro, MCP-Atlas — assume the task is fully specified and only measure whether the agent can do it; they never test whether it notices missing information. Clarification benchmarks — ClariQ, AmbigQA, QuestBench, UserBench — live in short-context QA, where a question costs essentially nothing. Nothing systematically injects and grades controlled ambiguity inside a real, expensive, multi-step workflow. The failure that results is concrete: an agent that silently assumes a missing parameter can run an entire incorrect workflow with cascading consequences, while one that asks about everything is just as broken.

// two failure shapes, only one in scope

The paper distinguishes semantic ambiguity (a request admits multiple valid interpretations, each leading to a different complete answer) from underspecification (a request is missing critical information and is simply unsolvable until that information is acquired). LHAW targets underspecification — and specifically the outcome-critical kind, where the missing detail provably causes the agent to fail rather than merely lowering quality.

// how it works

The method

LHAW (Long-Horizon Augmented Workflows) is a synthetic pipeline that takes a well-specified benchmark task and manufactures underspecified variants by removing information along a four-part taxonomy: Goal (the deliverable or success criterion — "Write the summary." → what summary, for whom, how long?), Constraint (rules, thresholds, methods — "List top customers." → top by revenue or count?), Input (which data source, file, or system — "Use the latest dataset." → which one, stored where?), and Context (domain knowledge, jargon, implicit requirements — "Send the report to the team." → which team, email or chat?). The taxonomy wasn't imposed a priori; it was reverse-engineered from the failure modes observed across 33 TheAgentCompany tasks.

The pipeline runs in three phases. Phase 1 — Segment Extraction & Scoring: an LLM reads the well-specified prompt (optionally with grounding data such as the golden trajectory or evaluation checkpoints) and identifies atomic, removable information segments, tagging each by dimension, sub-dimension, and exact text span. It also scores each segment on a 3-level scale for two properties — criticality (would removal cause failure?) and guessability (can the agent infer it from context?) — and ranks them by a priority score.

priority = criticality × (1 − guessability) // each score ∈ {0.0, 0.5, 1.0} — rank highest the segments that matter and can't be guessed

Phase 2 — Candidate Generation: for each segment (or combination) the pipeline applies a removal strategy — delete (remove the span entirely), vaguify (replace it with vague language, e.g. "the file"), or genericize (replace with a generic phrase, e.g. "an appropriate format"). Difficulty is dialed up by removing more segments at once, which forces the agent to spot several blockers before it can succeed.

Phase 3 — Empirical Agent Trials is the methodological heart of the paper. Rather than trust an LLM's prediction that a variant is ambiguous, LHAW runs the variant through real agent rollouts and classifies it by what actually happened. Given n trials, c successes, and the set S of unique terminal states observed, a variant is labeled:

// ambiguity, defined by execution not intuition

Outcome-Critical — c = 0: the agent always fails without the missing information (a true blocker).
Divergent — c > 0 and |S| > 1: variable outcomes; sometimes the agent guesses right.
Benign — c > 0 and |S| = 1: the agent reliably infers the missing detail and always succeeds.
New Task — c = 0 and |S| = 1: every trial converges to the same non-target outcome — i.e. the removal accidentally created a different but valid task. An LLM judge catches these and filters them out, so "Find users aged 18–34" → "Find users" never sneaks in as underspecification.

Applied across three benchmarks, this yields the 285-variant LHAW benchmark: 85 from TheAgentCompany's OwnCloud subset (13 base tasks, 40 outcome-critical / 30 divergent / 15 benign), 100 from SWE-Bench Pro (75 unique tasks, 31 / 34 / 35), and 100 from MCP-Atlas (1:1 with tasks, 50 / 30 / 20). Tasks are pre-filtered so the reference models are already competent under full specification — that way a performance drop reflects ambiguity sensitivity, not general inability. To grade clarification behavior, LHAW exposes an ask_user tool (an MCP-hosted LLM user simulator) that can see the original spec plus ground truth for the removed segments, and is policy-constrained to answer only questions about those segments.

// what they found

The results

First, the pipeline works as a difficulty dial. On MCP-Atlas, removing a single segment drops pass@3 from a ~100% baseline to 44.7–60.5%; removing two drops it to 35.5–43.5%. And the outcome-critical label is a real blocker, not a quality nudge: on the n=50 MCP-Atlas outcome-critical subset, Opus-4.5 collapses from a 1.00 baseline to 0.14 underspecified, and GPT-5.2 from 1.00 to 0.14; on SWE-Bench Pro's n=31 outcome-critical set, GPT-5.2 falls from 0.72 to a flat 0.00. Benign variants, by contrast, barely move (0.85–1.00), confirming agents really do infer those on their own.

Second, the ask_user tool helps meaningfully but rarely fully. With clarification on tap, Opus-4.5 recovers MCP-Atlas to 78% pass@3 (from 47%) — still short of its 100% baseline. Across nearly every model and benchmark, clarification buys back a large chunk of the lost performance yet stops below the original ceiling. The lone exception is GPT-5.2 on SWE-Bench Pro, where a +22.0-point recovery reaches 91% (and 1.00 on the outcome-critical subset). Notably, Sonnet-4.5 posts the highest underspecified SWE-Bench Pro pass@3 (79.0%) while gaining nothing from ask_user — it would rather guess than ask.

Third — and this is the most interesting axis — clarification style is a stable per-family fingerprint. The paper introduces Gain/Q (performance gain divided by questions asked) to separate "asks a lot" from "asks well." GPT-5.2 over-clarifies everywhere: the highest Ask% in every domain (97% on MCP-Atlas, 1.73 questions per trajectory), which yields the biggest absolute gains but the worst per-question efficiency (Gain/Q just 0.07 on MCP-Atlas, 0.04 on TheAgentCompany). Models that ask selectively extract more: Gemini-3-Pro records the single best Gain/Q of any pairing (0.24 on TheAgentCompany) while engaging ask_user in only 18% of trials, and Sonnet-4.5 is the most efficient on MCP-Atlas (0.17). On the failure-mode side, the most common defect is question quality (compound questions, flagged in 0.85–0.97 of trials with questions); the Gemini families are the most prone to under-clarifying (31%). Tellingly, when a task ultimately fails, the under-clarification rate jumps to 48% (from 26%) — pointing at the agent's own question quality, not the simulated user, as the reason it can't fully recover.

Fourth, the paper probes the cost side by appending a persona to the system prompt — a Supervisor ("waiting to answer… do not guess"), a Standard Assistant, or a Busy Executive ("only interrupt… if proceeding would lead to a factual failure"). Agents adjust: asking the Supervisor lifts checkpoint score to 64.9% but at a low 0.09 Gain%/Q over 266 questions, while asking the Busy Executive yields a smaller 56.0% checkpoint score but a higher 0.13 Gain%/Q over just 124 questions — fewer, better-targeted questions. The encouraging signal is the clean gradient in how much agents ask: from 0.99 ask_user calls per trial on outcome-critical tasks facing a Supervisor down to 0.22 on benign tasks facing a Busy Executive. The intuition about when clarification is optional vs required clearly exists — even if, as the recovery gap shows, the execution doesn't always follow.

Finally, an ablation on agentic scaffolding (GPT-5.2 on TheAgentCompany) finds the simplest strategy, CodeAct, wins on overall pass@3, but heavier strategies pay off precisely where it's hardest: on outcome-critical tasks with the user available, Reflexion and Plan & Execute reach 17.9% and 18.4% and a ~4-point checkpoint edge over ReAct/CodeAct. Plan & Execute also asks the most (80% of trials) — explicit planning surfaces more uncertainty.

// does it hold up?

The eval read

The headline strength is genuine: ambiguity is defined by observed terminal-state divergence, not by an LLM's linguistic guess, and the New-Task filter (c=0, |S|=1) is a smart guard against the failure mode that sinks most synthetic-ambiguity work — accidentally producing a different-but-valid task and calling it "ambiguous." That operationalization is the paper's real contribution. The caveats are about how heavily the whole apparatus leans on models, and about one oracle in the loop.

The most important one: the user simulator is an oracle. It holds the original specification and the ground truth for every removed segment and is told to answer up to that original prompt. That is an upper bound on the value of clarification — a real user neither knows the canonical answer nor responds noiselessly. So the gap between "underspecified" and "+ask_user" understates real-world difficulty, and the finding that agents still don't recover to baseline despite a near-perfect helper is actually the more damning reading: even handed the answer on request, frontier agents leave points on the table because their questions are bad. Worth remembering, too, that recovery is reported as pass@3 — three independent trajectories — so "recovered" can mean one lucky run out of three.

Second, the entire ambiguity signal is model-generated and model-judged. Gemini-3-Flash performs segment extraction and variant generation; an LLM decides the New-Task filter; the user simulator is an LLM; and the seven-way failure-mode taxonomy is scored by LLM-as-judge. A different extraction model would yield a materially different benchmark, and the per-dimension and per-class numbers inherit whatever blind spots that one model has. Nothing here is verified against human-labeled ambiguity — which the authors themselves name as future work (bridging to human-in-the-loop benchmarks).

Third, the Goal/Constraint/Input/Context split is confounded with explorability. The paper concedes the pipeline operates only at the prompt level and that agents can sometimes recover missing facts by poking the environment (listing a folder, reading a config). It absorbs this via repeated sampling — inferable removals land as "benign" — but that means an Input ambiguity ("which file?"), often resolvable by an ls, is being compared against a Goal ambiguity that no amount of exploration recovers. Some of the per-dimension spread is therefore about how discoverable each dimension is, not how intrinsically hard.

baselines

Appropriately current — Opus-4.5, Sonnet-4.5, Gemini-3-Pro/Flash, GPT-5.2. But "baseline competence" filtering selects tasks these very models can already solve, so the benchmark is a moving target keyed to one model generation, and the difficulty curves are partly an artifact of which tasks survived filtering — the absolute numbers aren't a capability ceiling.

contamination

Low risk by construction: variants are synthetic perturbations of existing tasks and grading runs each benchmark's native, execution-based harness. The exposure is the opposite kind — the pipeline is LLM-end-to-end, so a quirk of Gemini-3-Flash's extraction propagates into the labels.

what's missed

The oracle user simulator caps clarification value above what real users provide; pass@3 framing lets a single lucky trajectory count as recovery; and scope is underspecification only — semantic ambiguity (conflicting context, multiple valid readings), the messier real case, is explicitly out.

would raise confidence

A noisy / non-oracle user simulator; human-validated ambiguity labels; a second independent extraction model to show the benchmark isn't a Gemini-3-Flash artifact; and an explicit cost-calibrated value-of-information metric so "over/under-clarification" is scored against an optimum rather than described as a behavior gradient.

// bottom line

LHAW is the first principled, execution-grounded way to manufacture and grade controlled underspecification in long-horizon agent tasks, and its core result — frontier agents detect missing information unreliably and, even with a near-oracle user on tap, recover only partway because their questions are weak — is important and credible. The honest discount is that the entire ambiguity signal is LLM-generated and LLM-judged, and that the helper is an upper-bound oracle, so the real-world clarification gap is almost certainly wider than these numbers show.

// takeaways

What to remember

  1. Ambiguity by execution, not intuition. LHAW labels a variant outcome-critical / divergent / benign from the spread of terminal states across n agent trials — with a New-Task filter to discard perturbations that merely create a different valid task — rather than asking an LLM to predict ambiguity. That's the paper's central, transferable idea.
  2. Frontier agents recover only partway, even with help. Outcome-critical variants near-fully break performance (MCP-Atlas Opus 1.00→0.14; SWE-Bench Pro GPT-5.2 0.72→0.00); ask_user buys back a lot (Opus →0.78) but rarely the full baseline. Only GPT-5.2 on SWE-Bench Pro fully recovers (+22.0 → 91%).
  3. Clarification style is a model fingerprint. GPT-5.2 over-clarifies (97% Ask%, lowest 0.07 Gain/Q); Gemini under-clarifies (31%); Gemini-3-Pro does the most with the least (0.24 Gain/Q at 18% Ask%). Agents do scale questions to task stakes and user cost (0.99→0.22 calls/trial), so the judgment exists — the execution lags.
