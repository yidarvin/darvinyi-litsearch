DROP: A Reading Comprehension Benchmark Requiring Discrete Reasoning Over Paragraphs

SQuAD-style span extraction had been effectively solved. DROP put a live adversarial baseline in the crowdsourcing loop so questions had to survive being unanswerable by BiDAF — forcing systems to resolve references across a passage and then add, subtract, count, or sort. The best 2019 baseline reached 32.7% F1 against 96.4% expert human performance.

// the gap

Span extraction stopped being a hard problem

By early 2019, the reading-comprehension leaderboard had become a solved-and-done story. Systems were "matching humans on the most popular datasets for the task" — SQuAD, CNN/Daily Mail — because those tasks reduce to one operation: find the span of text that answers the question. That operation is a good test of surface-level passage-question alignment, but it doesn't require a system to actually do anything with the information it finds. A model that locates the sentence "…was sold by Robert Lehrman for $16.3 million, well above its $12 million high estimate" has solved SQuAD; it has not answered the paired DROP question — how many more dollars the painting sold for than the estimate — which needs subtraction to reach the answer, $4.3 million.

The semantic parsing literature had the opposite problem: datasets like WikiTableQuestions ask genuinely compositional questions (find rows, filter, count, compare) but only over clean structured tables, not raw paragraphs. Nobody had combined compositional question semantics with actual paragraph understanding. The authors also note a related but distinct trend — QuAC, HotpotQA, TriviaQA, and others were adding complexity through conversational state, multi-hop retrieval, or knowledge-source integration — but DROP deliberately avoids all of that: single passages, independent questions, no retrieval. The only added complexity is that the question demands a symbolic operation over what the passage says, which the authors argue is "necessary means to force more comprehensive passage understanding," not a distraction from it.

The construction method itself targets this gap directly: DROP is built with an adversarial baseline in the crowdsourcing loop. Workers wrote questions against a live BiDAF reading-comprehension model running in the background; a question could only be submitted if BiDAF got it wrong. That converts "make it hard" from a vague instruction into an operational filter — the resulting dataset is calibrated against a genuine 2018-era model's blind spots, not against workers' intuitions about difficulty.

// how it works

Adversarial crowdsourcing, then a numerically-aware model

Passages come from Wikipedia, selected for having "a narrative sequence of events, particularly with a high proportion of numbers" — the authors found NFL game summaries and history articles especially fruitful, and additionally pulled any passage containing at least twenty numbers, yielding about 7,000 candidate passages. On Amazon Mechanical Turk, each worker saw five passages and had to write at least twelve question-answer pairs total, primed with five example categories drawn from the semantic parsing literature: addition/subtraction, minimum/maximum, counting, selection, and comparison. Every question had to beat the live BiDAF baseline to be accepted. Answers were restricted to three types — a span from the passage or question, a date, or a number (only when the question named a unit, e.g. "how many yards") — which keeps evaluation tractable.

Dev and test questions were independently re-answered by at least two more workers (or marked invalid — 0.7% of the data, discarded). Overall inter-annotator agreement was Cohen's κ = 0.74, breaking down to 0.81 for numbers, 0.65 for dates, and 0.62 for spans. The result: 96,567 question-answer pairs over 6,735 passages, split by passage 80/10/10 into 77,409 train / 9,536 dev / 9,622 test questions (5,565 / 582 / 588 passages), collected for a Mechanical Turk budget of $60k.

A manual annotation of 350 sampled questions gives the reasoning-type breakdown: Subtraction 28.8%, Selection 19.4%, Comparison 18.2%, Count 16.5%, Addition 11.7%, Sort 11.7%, Set-of-spans 6.0%, Other 6.8%, Coreference resolution 3.7%, Other arithmetic 3.2% (categories can co-occur within one question, so they don't sum to 100%). Automatic NER tagging of the training-set answers found 66.1% are NUMBER, 12.2% PERSON, 9.4% OTHER, 7.3% OTHER ENTITIES, 3.5% VERB PHRASE, 1.5% DATE. On average, answering a question required considering 2.18 spans in the passage, at an average distance of 26 words apart, with 20% of sampled questions needing at least 3 spans — a real multi-span aggregation demand, not single-hop lookup.

NAQANet: giving QANet an arithmetic head

None of the standard span-extraction baselines can produce a number that isn't copied from the text, so the authors built NAQANet — a numerically-aware extension of QANet with four parallel output heads sharing the same passage/question encoder: (1) a passage-span head, identical to standard extractive QA; (2) a question-span head, since some DROP answers live in the question itself; (3) a count head, a 10-way softmax classifier over integers 0–9 conditioned on the passage representation; and (4) an arithmetic-expression head, which extracts every number mentioned in the passage and assigns each one a learned sign (+, −, or 0), so the selected numbers can be summed into the final answer. A categorical answer-type gate picks which head to trust at inference time. Because DROP supplies only the answer string, not which mechanism produced it, training is weakly supervised: the objective marginalizes over every possible execution (span match, count value, or sign assignment) that evaluates to the correct answer, searching only 2-number addition/subtraction to keep the space tractable.

ptype = softmax(FFN([hP, hQ])) // categorical gate over {passage span, question span, count, arithmetic} — picks the answer mechanism at test time
// what they found

32.7% F1 for the best baseline, 47.0% for NAQANet, 96.4% for humans

Three families of baseline were tested. Heuristic checks for annotation artifacts — a majority-answer predictor, question-only, and passage-only models — all scored near zero (test F1 1.44 / 8.59 / 2.26), showing DROP has little exploitable shortcut signal. Semantic parsers ported from WikiTableQuestions-style pipelines (parsing the passage into a table via syntactic dependencies, Open IE, or semantic role labeling, then searching for a logical form) topped out around 11% F1 (SRL: 11.45 test F1) — badly hampered by the information-extraction step, which the authors trace separately below. SQuAD-style extractive readers did best among prior architectures: BiDAF 27.49, QANet 28.36, QANet+ELMo 29.67, and BERT 32.70 test F1 — the number quoted in the abstract as "the best systems." For context, BERT was state-of-the-art on SQuAD 1.1 at 84.7% EM at the time; on DROP it drops by more than 50 absolute F1 points.

NAQANet's own ablation shows where its gain comes from: adding only the question-span head actually hurts relative to plain QANet+ELMo (28.18 vs 29.67 test F1); adding the count head brings it to parity with BERT (32.75); the arithmetic (add/subtract) head is the real driver, jumping to 42.96; and the complete model reaches 44.07 EM / 47.01 F1 on test (46.20 EM / 49.24 F1 on dev) — a 14.3-point absolute F1 gain over the best prior baseline. Human performance, measured by the six authors collectively answering 560 test questions under the same metric, is 94.09% EM / 96.42% F1.

A breakdown by answer type (dev set) is revealing: on Numbers (61.94% of answers), NAQANet scores 44.2 F1 against BERT's 14.8 — BERT has no way to produce a computed number, so it can only guess a nearby span. On Date (1.57% of answers) the relationship flips: BERT edges ahead, 42.8 vs 35.5 F1, since NAQANet has no date-arithmetic head at all. An error analysis of 100 wrong NAQANet predictions found the dominant failure modes were arithmetic operations (51%), counting (30%), domain/common-sense knowledge (23%), coreference (6%), or a combination of types (40%) — confirming the model's remaining errors track precisely the reasoning types it wasn't built to handle.

On the semantic-parsing side, the paper diagnoses its own baseline's weakness: the information-extraction pipeline produced any usable logical form for only 34% of training examples with the SRL and syntactic-dependency schemes, and just 25% with OpenIE; and on a 60-question sample of SRL output (the best-performing of the three), only 25% of the resulting tables actually contained the information needed to answer the question. Of the logical forms found for that same 60-question sample, only 8 (13%) were judged non-spurious (i.e., actually reflected the question's real semantics rather than accidentally computing the right number).

// does it hold up?

The eval read

DROP's headline gap — 32.7% vs 96.4% — is a legitimate and influential result: it correctly diagnosed that span-extraction had plateaued for the wrong reason (models were good at alignment, not reasoning) and it's the direct ancestor of every "does the model actually compute, or just retrieve" benchmark that followed. But several parts of the evaluation setup inflate the drama of that gap beyond what the architecture comparison alone would show.

The most consequential issue is buried in a single sentence: "we also omitted the training questions which cannot be answered by a span in the passage (45%)." BiDAF, QANet, QANet+ELMo, and BERT are all pure span extractors, so nearly half of DROP's training signal — every question whose answer is a count or a computed number that doesn't appear verbatim in the text — was simply unavailable to them. NAQANet, by contrast, is trained (weakly supervised) on the full set, including exactly the examples the other baselines couldn't learn from. Some of the 32.7% → 47.0% F1 jump is therefore a training-data availability gap dressed up as an architectural one, not purely "NAQANet reasons better than BERT." The paper is transparent about this omission, but the abstract's framing ("the best systems only achieve 32.7% F1... we present a new model that... achieves 47.0% F1") doesn't foreground it.

The 96.4% human ceiling comes from "the authors collectively answering 560 questions from the test set" — not an independent crowd or expert panel. The paper argues this is fairer than holding out one gold annotation (which would underestimate human performance against multiple valid phrasings), and that's a reasonable methodological point, but six people who designed the taxonomy of reasoning types are not a neutral upper bound; they know exactly which operations DROP rewards and how the answer format is normalized. An arm's-length human baseline, on the same 350-question sample used for the reasoning-type annotation, would be a stronger reference.

Contamination gets no discussion at all — reasonably, since the paper predates the contamination-auditing era by several years — but it matters retroactively. DROP's passages are lifted verbatim from Wikipedia NFL game summaries and history articles: exactly the kind of high-frequency, high-quality web text that ends up memorized by any model pretrained on Common Crawl or Wikipedia dumps. There is no held-out or refreshed test set, so a 2026-era zero-shot DROP number on a model with web-scale pretraining is confounded by potential memorization in a way the 2019 baselines never had to worry about. Relatedly, the adversarial filter calibrates difficulty against one specific model (BiDAF, 2018) at one point in time; it is not a moving target the way iteratively-filtered datasets like SWAG's AF or WinoGrande's AFLITE are, so "hard for BiDAF" has drifted further from "hard for a 2026 model" than the authors could have anticipated.

baselines

BiDAF/QANet/QANet+ELMo/BERT are legitimate 2019 SOTA extractive readers, but they were trained on 55% of DROP's questions (span-answerable only) — the other 45% were unavailable to them. NAQANet trains on the full set. The 32.7 → 47.0 F1 gain conflates a training-data asymmetry with an architectural one.

contamination

Not discussed — pre-dates the audit era. Passages are verbatim Wikipedia NFL/history text, exactly the kind of content later web-scale pretraining absorbs; there's no refresh mechanism, so zero-shot DROP scores on modern LLMs carry an unaudited memorization risk.

what's missed

NAQANet's own numeracy is deliberately narrow — count(0–9) and 2-number add/subtract only, by the authors' own description a "very limited set of operations." Comparison-as-computation, multiplication, division, and multi-step chains are out of scope even for the "complete" model, so 47% F1 doesn't mean 47% of DROP's own reasoning taxonomy is solved.

would raise confidence

An independent (non-author) human baseline instead of the paper's own six authors; per-reasoning-type confidence intervals rather than one aggregate F1; and a contamination check or living test split for any evaluation run against post-2019 pretrained models.

// bottom line

DROP correctly identified that extractive QA had plateaued on the wrong skill and forced the field to treat discrete/numerical reasoning as a first-class reading-comprehension capability — that diagnosis held up and shaped a decade of harder benchmarks. But the headline 32.7-vs-96.4 gap is partly an artifact of the eval setup: baselines missing 45% of the training signal, an author-sourced human ceiling, and a static test set with no contamination control. NAQANet's own fix only closes a narrow slice (counting and two-number arithmetic) of the reasoning space DROP itself names — a pattern ("the benchmark outlives the model built to solve it") that recurs across nearly every dataset in this lineage.

// takeaways

What to remember

  1. DROP is 96,567 crowdsourced questions over ~7,000 Wikipedia passages (NFL summaries, history articles), built with a live BiDAF baseline in the loop so workers could only submit questions the model got wrong. The best 2019 extractive baseline (BERT) scored 32.7% F1 against 96.4% human performance.
  2. The paper's own NAQANet model — QANet extended with count(0–9) and 2-number add/subtract heads — reached 47.0% F1, but part of that 14.3-point gain reflects a training-data asymmetry: the SQuAD-style baselines had to drop the 45% of training questions that aren't span-answerable, while NAQANet trained on all of them.
  3. Semantic-parsing baselines scored only ~11% F1, less because compositional parsing is hard in principle than because the information-extraction pipeline feeding it produced usable structured tables for barely a quarter of examples — a confound the paper documents but that muddies any "neural vs. symbolic" comparison drawn from the results table.
