Describe, Explain, Plan and Select: Interactive Planning with Large Language Models Enables Open-World Multi-Task Agents

A Minecraft agent that turns every planning failure into the next plan's training signal: a Descriptor logs what happened, an Explainer diagnoses why a sub-goal failed, the Planner rewrites the plan around that explanation, and a learned Selector reorders parallel sub-goals by how many steps they actually take from here — together nearly doubling zero-shot success across 71 open-world tasks.

// the gap

What problem is this solving?

Hierarchical LLM planners — an LLM decomposes a task into sub-goals, a low-level controller executes each one — had already worked reasonably well on tabletop manipulation and text-adventure environments. The authors first check whether that success transfers to a genuinely open world by running Inner Monologue, a strong existing planner, on Minecraft, ALFWorld, and a tabletop environment side by side. Even handed a ground-truth oracle low-level controller, Inner Monologue's success rate is 99% on Tabletop and 42% on ALFWorld but only 23% on Minecraft; swap in a realistic learned controller and Minecraft collapses further to 9%. Open worlds are a different regime, and the paper pins the gap on two specific failure modes rather than a single vague "it's harder."

Challenge #1 — complex sub-task dependency. A tabletop rearrangement task is usually a handful of consecutive sub-goals; mining a diamond in Minecraft requires at least 13 sub-goals executed in the correct order (chop wood → craft planks → craft a table → craft tools → mine stone → smelt iron → mine at depth, and so on), so the probability that an LLM emits a flawless plan in one shot drops fast as the chain lengthens. Challenge #2 — state-dependent feasibility. Even a logically valid plan can be inefficient or dead-on-arrival given the agent's actual state: to craft a bed, slaughtering a sheep for wool is normally fastest, but if no sheep or village is reachable within three minutes of gameplay, the agent instead needs to kill a spider for string. A planner that ignores where the agent currently stands in the world will confidently propose routes that are unreachable.

// how it works

The method

DEPS keeps the same LLM in three roles — Planner, Explainer, and (implicitly) the target of a Descriptor's reports — and adds a small trained Selector network. The loop starts like any zero-shot LLM planner: given a task instruction, the Planner decomposes it into an initial sequence of sub-goals. Whenever the goal-conditioned controller fails to complete a sub-goal, the Descriptor — a rule-based module over Minecraft's symbolic game state (inventory, biome, GPS) — summarizes what happened as a short text description d_t. Rather than just handing that description back as raw feedback (what Inner Monologue and similar predecessors do), DEPS first prompts the same LLM as an Explainer to locate the specific error in the previous plan — for instance, that a goal needs an iron pickaxe the agent hasn't crafted yet, or that a craft step needs more planks than are currently in inventory — using a handful of chain-of-thought demonstrations. Only then does the Planner re-plan, conditioned on both the description and its own explanation.

d_t = fDESC(s_{t−1})  →  e_t = fEX(d_t)  →  P_t = fLM(CONCAT(p_{t−1}, d_t, e_t))  →  g_t ~ fS(P_t, s_{t−1}) // describe → explain → re-plan → select, repeated until the task finishes or the horizon runs out

The second piece answers Challenge #2. When a plan contains parallel sub-goals with no strict ordering between them (e.g. chop an oak tree or a birch tree or an acacia tree — any satisfies "get wood"), a naive planner has no way to prefer whichever is actually closest. DEPS trains a small horizon-predictive Selector (HPS) — a goal-conditioned CNN sharing the controller's backbone — to predict h_t(g) := T_g − t, the number of remaining steps to complete each candidate goal g from the current state, by minimizing a cross-entropy loss against ground-truth horizons in offline trajectories. The Selector then picks the candidate with the lowest predicted horizon as the current goal. The authors motivate learning this rather than using a vision-language model like CLIP directly: an oak tree in the current camera frame can look semantically "close" to CLIP even when a canyon lies between the agent and the tree, so raw visual similarity is a poor proxy for actual traversal cost.

// what they found

The results

The main benchmark is Minecraft Task101: 71 tasks from the MCU SkillForgeChain suite, grouped into eight meta-groups MT1 (basic, e.g. "make a wooden door") through MT8 (the single ObtainDiamond challenge task), each run 30 times with the same fixed controller and the same OpenAI models (text-davinci-003 for GPT/CoT/IM, code-davinci-002 for the code-style planners including DEPS) across every method for a controlled comparison. DEPS beats every baseline on every meta-group. Averaged over all eight groups, DEPS reaches 48.56% success — nearly double Code-as-Policies' 25.77%, the strongest prior baseline, and well ahead of Inner Monologue's 21.64%. Dropping just the Selector (the "DEP" ablation) already reaches 39.36% purely from describe-explain-replan, with the Selector adding the rest — up to a +2.7× success-rate jump on the efficiency-sensitive MT7 group.

The same describe-explain-replan design (without the Minecraft-specific Selector) transfers to non-open-ended domains: on ALFWorld, DEP's task success averages 76.0% across six task groups, versus 52.0% for a re-planning-augmented GPT baseline, 37.0% for the trained BUTLER agent, 30.0% for Inner Monologue, and 10.0% for one-shot GPT. The paper's other headline number is ObtainDiamond — mining Minecraft's hardest single item from scratch, long a benchmark task for the community. DEPS reaches 0.59% success within a 12,000-step (about 10-minute) budget, framed by the authors as "on-par" with two specialized prior systems: VPT (≈20% after pre-training on ~70k hours of human demonstrations plus RL fine-tuning) and DreamerV3 (2%, trained from scratch with a world model in an easier, modified environment). DEPS needs neither — it is the same zero-shot prompted planner used everywhere else in the paper.

// does it hold up?

The eval read

The comparisons that matter — DEPS vs. GPT/CoT/IM/CaP, and DEP vs. DEPS — are genuinely controlled: same controller, same underlying OpenAI model family, same prompt-construction effort per method, 30 runs each. That is real experimental discipline for an embodied-agent paper. Two things temper the headline "nearly doubling" framing, though. First, every baseline is self-reproduced by the DEPS authors ("we reproduce them to conform to the Minecraft specification") rather than run by the original authors or ported verbatim, so some of the gap could reflect implementation effort rather than a fundamental limitation of Inner Monologue or Code-as-Policies. Second, the per-task variance is large relative to the means — e.g. GPT on MT1 reports 25.85±24.8, a standard deviation nearly as large as the mean itself — which the paper's AVG-column framing smooths over.

The ObtainDiamond framing is the sharpest place to push. The text calls DEPS's 0.59% "on-par performance in this grand challenge" — but by the numbers the paper itself cites, VPT reaches 20% and DreamerV3 reaches 2% — both above DEPS's 0.59%, just via very different resource budgets (70k hours of human demonstrations, or training a world model from scratch, versus zero-shot prompting a frozen LLM). "On-par" is doing real persuasive work there; a more literal read is "closes some of the gap to specialized RL agents without their training cost, but still trails both." The ablation in the appendix (Table 4) is also worth noting for what it reveals about compute assumptions: the biggest jumps (e.g. MT1: 28.6% at round 0 → 79.8% at round ∞, a +51.2-point swing) come from letting DEPS re-plan an unbounded number of rounds, where "∞" is itself capped only by "the maximum tokens of LLMs" — a real but externally-imposed ceiling, not evidence the loop converges on its own.

baselines

Fair within the paper (identical controller + LLM family across all methods) but self-reproduced by the DEPS authors rather than run from the original codebases, and all run on now-deprecated text/code-davinci models — the gap to a modern LLM planner (GPT-4-class) is untested here.

contamination

Not the usual static-benchmark leakage risk — this is an interactive Minecraft/ALFWorld environment, not a held-out text set — but the 71-task eval suite is the authors' own curated subset of MCU SkillForgeChain, not an independently released split.

what's missed

The AVG columns hide per-task variance nearly as large as the mean (e.g. 25.85±24.8), and the "on-par" ObtainDiamond claim elides that both cited specialist baselines (VPT 20%, DreamerV3 2%) numerically beat DEPS's 0.59%.

would raise confidence

An independent re-run of the baselines (not authors' own reproductions), a modern-LLM version of DEPS, and per-task variance reported alongside the AVG headline rather than folded into it.

// bottom line

Describe-explain-then-replan, plus a learned selector for parallel sub-goals, is a real and carefully ablated improvement over one-shot-plan-plus-generic-feedback planners — the DEP vs. DEPS vs. baseline comparisons are controlled and the gains are large. But the paper's own favorable framing ("nearly doubling," "on-par") outruns what the raw numbers show once you account for self-reproduced deprecated-model baselines, wide per-task variance, and an ObtainDiamond result that two cited specialist systems still beat outright.

// takeaways

What to remember

  1. Having the LLM explicitly explain why a sub-goal failed before re-planning — not just feeding back pass/fail — is what fixes accumulating errors on long-horizon open-world tasks: DEP alone (no selector) already reaches 39.36% average success across 71 Minecraft tasks, vs. 25.77% for the strongest prior baseline.
  2. A small trained horizon-predictive Selector, which learns to predict remaining steps-to-completion rather than relying on visual similarity (CLIP/MineCLIP), adds the rest of the gain — up to a 2.7× success-rate jump on the hardest, most parallel-goal-heavy task group.
  3. Even so, ObtainDiamond — Minecraft's flagship hard task — comes out to 0.59% success; interactive LLM planning narrows but, by the paper's own cited numbers, does not close the gap to specialized RL agents like VPT (20%, after 70k hours of human demonstrations) or DreamerV3 (2%).
