τ2-Bench: Evaluating Conversational Agents in a Dual-Control Environment

τ-bench's sequel gives the simulated user tools too — a shared, dynamic world where both agent and user can act, not just talk. Frontier LLMs that clear up to 79% pass^1 on retail/airline collapse to 34-49% on the new telecom domain, and simply removing the user (letting the agent control everything) recovers 18-25 points of that gap.

Victor Barres, Honghua Dong, Soham Ray, Xujie Si, Karthik Narasimhan

// the gap

What problem is this solving?

Every conversational-agent benchmark that came before this one — τ-bench, AgentBench, ToolSandbox, WebShop — is what the authors call a single-control environment: the AI agent is the only party that can call tools and change the state of the world, while the (simulated) user is limited to communicating preferences and goals in natural language. In τ-bench specifically, the user's understanding of the environment is baked into carefully-written instructions that keep the task solvable along a single path — useful for controlled evaluation, but a poor model of real customer-support interactions, where the user often has to do something (restart the phone, check a status bar, toggle a setting) as part of resolving their own issue.

That single-control assumption hides an entire class of agent failure. An agent that reasons well but explains poorly, or that can't tell when a user has misunderstood an instruction, looks identical to a strong agent under single-control scoring — because the environment never gives the user the power to get something wrong. τ2-bench's premise is that testing this requires literally handing the user a matching toolset and a piece of the same shared, mutable database, and measuring what happens to task success once the agent has to guide, not just decide.

// how it works

The method

τ2-bench formalizes dual-control interaction as a Decentralized Partially Observable Markov Decision Process (Dec-POMDP), defined by the tuple (S, {Ai}, {Oi}, T, R, U, M) over two players, i ∈ {agent, user}. The state space S = Sworld ⊗ Shistory splits into the underlying agent/user database states and a log of every action, observation, and message. Each player's action ai is either a tool call against their own database or a natural-language message drawn from the shared message space M — and only one player acts per turn. The transition function T maps a joint action to a new state and a joint observation; a reward function R : S → [0,1] scores task success from the final database state; and the instruction space U holds both the user's scenario and the domain policy the agent must follow.

The new telecom domain instantiates this: the agent's CRM tools see customer/line records, the user's tools see a mocked phone (signal strength, airplane-mode state, data toggle, battery level), and a task like "mobile data isn't working" can require the agent to diagnose the cause via its own tools and talk the user through checking or flipping settings only the user can reach. Building a domain: five stages, mostly LLM-assisted

Each domain (telecom is the new one; retail and airline carry over from τ-bench) is built in five stages: (1) prompt an LLM to draft a Product Requirements Document defining the agent's database schema and tool functions, generate implementations plus unit tests, and manually refine until tests pass; (2) repeat the same process for the user's database and tools; (3) programmatically generate tasks from atomic subtasks, each defined by {finit}, {fsol}, and {fassert} function sets (e.g. an init that sets airplane_mode = True, a solution that calls toggle_airplane_mode(), an assertion that checks connectivity is restored) — mutually-exclusive subtasks are grouped, and a composite task picks at most one subtask per group; (4) LLM-draft a domain policy document from the curated tasks; (5) manually refine everything jointly. The telecom domain ends up with 15 atomic subtask groups across three user intents of increasing complexity — service_issue, mobile_data_issue, mms_issue — which compose into 2,285 possible tasks, of which 114 are subsampled for a balanced distribution over intents and subtask counts. τ2-bench also adds optional user personas (None / Easy / Hard) that shape how technically capable the simulated user acts.

Task success is checked with up to five criteria — DB state check, status assertions, natural-language assertions (e.g. "the agent diagnosed the cause of the issue"), communication-info check, and action matching against the known solution steps — though telecom specifically scores only via its assertion functions. Every experiment reuses pass^k from τ-bench: the fraction of k independent trials (temperature 0, run four times per task) that succeed, which punishes agents that solve a task inconsistently rather than reliably.

// what they found

The results

Four models were evaluated as the agent — gpt-4.1, gpt-4.1-mini, o4-mini, and claude-3.7-sonnet — against a gpt-4.1 user simulator throughout. Telecom is uniformly the hardest domain: gpt-4.1's pass^1 drops from 74% (retail) / 56% (airline) to 34% (telecom); the other three models land closer to 50% on telecom (o4-mini 42%, gpt-4.1-mini 44%, claude-3.7-sonnet 49% — on par with its own airline score of 50%). But reliability degrades faster on telecom as k increases: claude-3.7-sonnet's telecom pass^4 falls to 25%, a steeper decline than its airline pass^4 of 36%.

Isolating why: an ablation over reasoning vs. coordination

To separate "the agent can't reason about the domain" from "the agent can't guide a user," the paper runs telecom in three modes: Default (the normal dual-control setup), No-User (the agent gets a ticket summary and controls every tool itself, including the user's), and Oracle Plan (the agent is handed the exact sequence of tool calls needed, agent and user actions both). For gpt-4.1: Default 34% → No-User 52% → Oracle-Plan 73%. For o4-mini: Default 42% → No-User 67% → Oracle-Plan 96%. That is, removing the user entirely recovers an 18-point pass^1 gain for gpt-4.1 and a 25-point gain for o4-mini — the paper's clearest evidence that decentralized coordination, not raw domain reasoning, is the dominant bottleneck in the telecom results.

Two more breakdowns sharpen the picture. Performance falls off steeply with task length: in Default mode, gpt-4.1 and o4-mini both trend toward zero pass^1 past roughly seven required actions, though the No-User gap narrows from about 0.3-0.4 points at short horizons to under 0.2 at long ones — length hurts both settings, coordination is not the only bottleneck at scale. And difficulty is intent-dependent: gpt-4.1's pass^1 falls from 66% on service_issue (independently resolvable) to 30% on mobile_data_issue to 18% on mms_issue (both of which require first ruling out an underlying service issue) — a difficulty hierarchy baked into the domain design.

The user simulator is measurably more reliable, on a small sample

The paper's fourth contribution is a tightly-coupled user simulator — constrained by the tools and observable state available to it rather than leaning on natural-language prompting alone. Manually annotating gpt-4.1-vs-gpt-4.1 interaction traces (two annotators per conversation, tagged for critical vs. benign errors) gives telecom an 16% total error rate with 6% critical errors, against 40% (12% critical) for retail and 47% (13% critical) for airline under the original τ-bench user-simulation approach. The telecom sample is 50 conversations (vs. 100 for airline, 50 for retail) — small enough that a single reclassified conversation moves the reported rate by two points.

// does it hold up?

The eval read

The headline claim — dual control is measurably harder than single control, and the gap is mostly coordination rather than reasoning — is well-supported by the paper's own ablation: it isn't just a lower aggregate number, it's the same models scoring 18-25 points higher the instant the user's tools are handed to the agent instead. That's a clean, internally-controlled experiment. What's thinner is everything around the edges: model coverage, sample sizes on the reliability claims, and how far a 114-task, single-new-domain benchmark should be trusted as a general "dual-control" measurement rather than a telecom-specific one.

baselines

Four models, all released within months of the paper (gpt-4.1, gpt-4.1-mini, o4-mini, claude-3.7-sonnet-20250219) — all frontier, all closed-weight. No open-weight model, no older/weaker model for a difficulty floor, and no non-function-calling agent architecture is tried. The user simulator is fixed to gpt-4.1 for every agent, which keeps agent-to-agent comparisons fair but means every absolute pass^k number is entangled with one specific simulator's quality — a better or worse user simulator would shift every reported score.

contamination

Tasks are procedurally composed from atomic building blocks rather than static text, which blunts classic memorization risk, but the paper is explicit that code and data are public on GitHub as of the "Preprint. Under review." posting (June 2025) and evaluated against frontier models trained around the same period — the paper does not discuss whether any evaluated model's training data could already include the public repository.

what's missed

No inter-annotator agreement figure (e.g. Cohen's κ) is reported for the manual user-simulator error labeling that produces the headline 16%-vs-40-47% reliability claim, and the telecom sample is only 50 conversations. No real-human user study is run anywhere in the paper — every reported number measures an agent against an LLM-simulated user, not a person, so "guiding a user" here really means "guiding a same-model-family simulated user," which the paper itself flags by naming the expert-novice gap and cross-domain generalization of the tool-augmented user simulator as open future work.

would raise confidence

A reported inter-annotator agreement statistic for the error-rate claims; a held-out or adversarially-perturbed task split to stress leakage/overfitting to the specific atomic-subtask templates; at least one open-weight model and one real-human-user pilot as external validity checks on the simulated-user results.

// bottom line

The dual-control ablation is the paper's strongest result and survives scrutiny — it isolates coordination as the bottleneck with a same-model, same-task comparison, not just a lower headline score. The reliability claims around the user simulator (16% vs. 40-47% error rates) and the domain-generalization claims (does dual control hurt equally hard outside telecom?) rest on thinner evidence — a 50-conversation manually-annotated sample with no reported annotator agreement, and a single new domain. Treat telecom's 34-49% pass^1 numbers as a real, reproducible floor for current frontier agents under dual control, but not yet as a fully hardened benchmark the way τ-bench's retail/airline numbers have become.

// takeaways

What to remember

  1. τ2-bench generalizes τ-bench from single-control (only the agent has tools) to dual-control (both agent and user act via tools on a shared, dynamic world), formalized as a Dec-POMDP, plus a compositional task generator and a new telecom domain (114 tasks sampled from 2,285 possible compositions across 15 atomic subtask groups).
  2. Dual control is a real, measurable difficulty jump: frontier models that clear 50-79% pass^1 on τ-bench's retail/airline drop to 34-49% on telecom, and the paper's Default/No-User/Oracle-Plan ablation shows that simply removing the user recovers 18 points (gpt-4.1) to 25 points (o4-mini) of pass^1 — strong evidence that guiding a user, not domain reasoning, is the current bottleneck.
  3. Tightly coupling the user simulator to tools and observable state (instead of free-form NL prompting) is reported to cut error rates roughly in half-to-third — telecom's 16% (6% critical) vs. retail's 40% (12%) and airline's 47% (13%) — though that comparison rests on a 50-conversation manually annotated sample with no reported inter-annotator agreement, and every result in the paper measures agents against a simulated, not a real, user.
