Megatron-LM: Training Multi-Billion Parameter Language Models Using Model Parallelism

A transformer layer is mostly matrix multiplies. Split them across GPUs the right way and the whole layer needs just two all-reduces per pass — in vanilla PyTorch, no compiler. That trick breaks the single-GPU memory wall and trains an 8.3B-parameter GPT-2 on 512 V100s at 76% scaling efficiency.

Mohammad Shoeybi, Mostofa Patwary, Raul Puri, Patrick LeGresley, Jared Casper, Bryan Catanzaro · NVIDIA

// the gap

What problem is this solving?

By 2019 the scaling story was clear: bigger transformers — BERT, GPT-2 — kept advancing NLP, and the obvious next move was to keep growing them. The obstacle was brutally physical. To train a model with standard tooling, the whole model has to fit on one accelerator. A 32 GB V100 runs out of room well before a couple of billion parameters once you count what training actually stores: the weights, the gradients, and — with Adam — two extra optimizer moments per parameter, plus the activations needed for backprop.

Data parallelism doesn't help with this. Replicating the model across workers and splitting the batch raises throughput, but every worker still holds a full copy of the model — the memory ceiling is untouched. To go bigger you have to split the model itself across devices.

Frameworks for that existed, but each came with a tax. GPipe sliced the model into pipeline stages — but pipelines have idle "bubbles" and need optimizer changes to stay consistent. Mesh-TensorFlow could express general distributed tensor computation — but you had to rewrite your model in its language and lean on a custom compiler. Both asked you to leave your existing PyTorch code behind. The gap Megatron-LM targets is a model-parallel method that (1) actually breaks the per-GPU memory wall, (2) is simple — a handful of communication primitives inserted into a normal PyTorch transformer, no new compiler — and (3) is efficient and composes with the data and pipeline parallelism you already use.

// how it works

The method

Megatron-LM introduces intra-layer model parallelism — today called tensor parallelism. Instead of cutting the network into stages (pipeline parallelism) or replicating it (data parallelism), it shards the weight matrices inside each layer across a group of GPUs. The whole art is choosing how to split each matrix so the partial results line up with almost no communication.

The MLP block: split columns, then rows

A transformer's feed-forward block is two GEMMs around a GeLU: Y = GeLU(XA), then Z = Dropout(YB). There are two ways to shard the first matmul, and only one is good. If you split A by rows (and X by columns), each GPU computes a partial sum X₁A₁ + X₂A₂ — but GeLU is nonlinear, so you must synchronize and add before applying it. That's an all-reduce in the middle of the block. Split A by columns instead and GeLU applies independently to each shard — no sync. Then split the second matrix B by rows so it consumes the sharded GeLU output directly, and sum the partials with a single all-reduce at the block's exit.

[Y1, Y2] = [GeLU(X·A1), GeLU(X·A2)]    A split by columns → no sync before GeLU
Z = Dropout( g( Y1·B1 + Y2·B2 ) )    B split by rows → one all-reduce // f and g are conjugate operators: f = identity (forward) / all-reduce (backward); g = all-reduce (forward) / identity (backward). Each is a few lines of PyTorch autograd.

The self-attention block gets the same treatment for free, because attention heads are already independent. Partition the query/key/value projections column-wise so each head's matmul lives entirely on one GPU; compute attention locally; then row-split the output projection so it consumes the parallel result without a gather. One all-reduce forward, one backward.

Two all-reduces per layer — and nothing else

Stack the two blocks and a complete model-parallel transformer layer needs exactly four collective operations: two all-reduces in the forward pass and two in the backward pass — one pair for attention, one for the MLP. Everything else is engineered to avoid communication. Layer normalization, dropout, and the residual adds are duplicated on every GPU in the group rather than sharded-and-broadcast — recomputing them redundantly is cheaper than moving tensors around. The input embedding is sharded along the vocabulary dimension; the (tied) output embedding is fused with the cross-entropy loss, so the group communicates b×s scalar losses instead of b×s×v logits — a large saving when the vocabulary is ~51,200.

Composing with data parallelism

Because tensor parallelism is orthogonal to data parallelism, you use both at once. Megatron keeps each model-parallel group inside a single server — 8 GPUs wired by NVSwitch at 300 GB/s — and forms data-parallel groups across servers. The 8.3B model runs as 8-way tensor × 64-way data parallelism = 512 GPUs. None of this needs a compiler: the whole scheme is a few synchronization primitives dropped into an ordinary PyTorch transformer, which is exactly why it spread.

// what they found

The results

The scaling baseline is deliberately strong: a 1.2B-parameter GPT-2 on a single V100 sustains 39 TeraFLOP/s — 30% of the GPU's theoretical peak, which is already a hard number to beat. Scaling up to the 8.3B model on 512 GPUs (8-way tensor parallel), the application sustains 15.1 PetaFLOP/s — 76% of linear scaling relative to that single-GPU baseline. The weak-scaling efficiency degrades gracefully: 100% → 95% → 82% → 77% as tensor parallelism grows 1→2→4→8, and 96% → 83% → 79% → 74% once 64-way data parallelism is layered on to reach 512 GPUs.

Scaling also paid off in accuracy. Evaluated zero-shot, the GPT-2 models improve monotonically with size and the 8.3B model sets state of the art on both datasets it reports:

A science nugget: layernorm placement

On the BERT side, the team hit the puzzle ALBERT had also reported — naively growing BERT past BERT-Large (336M) degrades it. They traced it to architecture, not capacity: reordering the layer normalization and residual connection (architecture (b) below — essentially a pre-LN transformer) removes the training instability and reaches a lower loss. With that one change, downstream accuracy improves monotonically all the way to a 3.9B BERT. Held-out validation perplexity falls 1.58 → 1.30 → 1.16 for the 336M / 1.3B / 3.9B models, and Megatron-3.9B sets development-set SOTA across MNLI (91.4/91.4), QQP (92.7), SQuAD 1.1 (95.5/90.0) and SQuAD 2.0 (91.2/88.5), with a RACE test score of 89.5 single / 90.9 ensemble versus the prior ALBERT ensemble's 89.4.

// does it hold up?

The eval read

The engineering is genuinely load-bearing, but the headline numbers flatter a little and the science claim deserved a cleaner test. Start with the scaling figure. "76% scaling efficiency" is weak scaling: the problem grows with the GPU count (~1B params per GPU), so it answers "can I train a proportionally bigger model efficiently," not the harder "can I train this model faster" (strong scaling), where the per-layer all-reduces eat a larger share of each step. And the baseline itself runs at 30% of peak FLOPs, so 76% of that is roughly ~23% absolute hardware utilization at 8.3B — excellent for 2019, but the percentage reads better than the MFU.

The "512 GPUs" headline also conflates two axes. Tensor parallelism is only 8-way, and deliberately confined inside one DGX-2H (NVSwitch, 300 GB/s); the 512-GPU run is 8-way tensor × 64-way data. The paper itself notes that models beyond ~16B params would need inter-node model parallelism — i.e. the tensor-parallel dimension does not scale past a single node's NVLink domain. So this isn't evidence tensor parallelism scales to hundreds of GPUs; it's evidence it scales to ~8–16 within a node, after which you lean on data and pipeline parallelism. Memory isn't fully solved either: activations still grow with batch×sequence and the method leans on activation recomputation on every layer, and the optimizer-state redundancy across data-parallel replicas is left untouched (the gap ZeRO/FSDP later filled).

On evaluation, credit where due — they ran an 8-gram contamination check (WikiText103 test ≤10.8% overlap with training, LAMBADA ≤1.4%), though they note WikiText103's own train/test overlap is already 9.09%, and the web corpus (OpenWebText, RealNews, CC-Stories) keeps leakage a live risk for web-derived test sets. The decoder wins are also narrow: zero-shot perplexity and cloze on two datasets, the metrics that track the training objective most directly, with no broad downstream eval. Finally the layernorm result — the paper's real scientific claim — is shown via loss curves and downstream accuracy on a single rearrangement, without an ablation isolating which ingredient (pre-LN ordering vs. the 1/√(2N) residual scaling they also apply) does the work; concurrent results on pre-LN transformers reached related conclusions, so "first to report for BERT" is a narrow novelty, and the 3.9B BERT was "still training" at submission.

baselines

Strong single-GPU anchor (39 TFLOP/s, 30% of peak). But no head-to-head throughput comparison against GPipe or Mesh-TensorFlow — the frameworks it argues it improves on.

contamination

8-gram overlap checked (WikiText103 ≤10.8%, LAMBADA ≤1.4%) — reasonable, but web-corpus leakage and a tokenization-"adjusted" WikiText103 perplexity soften the LM SOTA.

what's missed

The headline is weak-scaling with only 8-way node-local tensor parallelism; absolute MFU ≈ 23%; the GPT-2 eval is narrow zero-shot LM; no ablation isolating the layernorm-reorder effect.

would raise confidence

Strong-scaling numbers at fixed model size; cross-node tensor-parallel scaling; a controlled ablation of the pre-LN reorder vs. residual rescaling.

// bottom line

A deliberately simple idea — shard each transformer layer's GEMMs so the whole layer needs only two all-reduces, in plain PyTorch — that broke the single-GPU memory wall and became the default "tensor parallelism" axis of nearly every large model since. The scaling headline is weak-scaling and node-local, and the layernorm finding deserved a sharper ablation, but the engineering was the point: GPT-3, LLaMA, BloombergGPT, FlashAttention and HELM cite it because they are built on it.

// takeaways

What to remember

  1. Tensor parallelism in one trick: column-split the first GEMM and row-split the second so a transformer block needs only two all-reduces per pass — and it's a few lines of PyTorch, not a new compiler. This became the "TP" axis of 3D (data × tensor × pipeline) parallelism.
  2. The f/g conjugate pair is the whole mechanism. Identity-forward/all-reduce-backward and its conjugate place the only synchronization at the block boundaries; fusing the output projection into cross-entropy turns logit-sized communication into scalar-sized.
  3. The scaling is node-local weak scaling. 8-way tensor parallelism inside an NVSwitch box × 64-way data parallelism = 512 GPUs at 15.1 PFLOP/s and 76% efficiency; pushing past ~16B params needs pipeline / inter-node parallelism — the paper says so itself.
  4. Architecture, not just compute. Simply scaling BERT degrades it; reordering layernorm and the residual (pre-LN) is what let BERT grow to 3.9B and keep improving — a reminder that "make it bigger" assumes the architecture stays stable at scale.
