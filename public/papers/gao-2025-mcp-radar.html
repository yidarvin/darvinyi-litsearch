MCP-RADAR: A Multi-Dimensional Benchmark for Evaluating Tool Use Capabilities in Large Language Models

The Model Context Protocol turned tool use into a standard interface — so MCP-RADAR builds the first benchmark around it: 507 tasks across six domains, graded on objective answer-, operation-, and token-efficiency metrics. The headline across ten frontier models: they can usually fire a syntactically valid tool call, but they routinely pick the wrong tool for the job.

Xuanqi Gao, Siyi Xie, Juan Zhai, Shiqing Ma, Chao Shen · Xi'an Jiaotong University & UMass Amherst

// the gap

What problem is this solving?

LLMs stopped being passive text generators the moment they could call external tools — search the web, run a calculator, send an email, edit a file. The plumbing under that shift is the Model Context Protocol (MCP), Anthropic's open standard that replaced the combinatorial "N × M" mess of bespoke, per-tool integrations with a single protocol for dynamic tool discovery and invocation. MCP was adopted across the industry in under a year — and yet no benchmark measured whether models actually use it well.

The existing evals split into two camps that both miss tool use. Knowledge benchmarks (MMLU, BIG-bench) and instruction-following tests measure what a model knows or whether it complies — not whether it can drive a tool. The tool-use benchmarks that do exist suffer two flaws the authors name directly: (1) they struggle to separate genuine tool-driven problem-solving from a model reciting memorized answers, and (2) they lean on synthetic or simulated environments that don't capture the messiness of real tool interactions. MCP-RADAR is positioned as the first benchmark built specifically for the MCP paradigm — and, importantly, one whose metrics are objective and quantifiable rather than resting on subjective human ratings or a single binary success flag.

// how it works

The method

MCP-RADAR is a three-stage pipeline: assemble an MCP Pool of tools, run ten models through 507 tasks, and score them on three objective metrics. The 507 tasks span six domains and split into two fundamentally different task archetypes.

Two task types

Precise Answer tasks have a single, verifiable ground-truth value — a number, an expression, a specific noun. They cover Math (120 tasks, 4 tools) and Web search (94 tasks, 2 tools), curated and adapted from established academic datasets rather than generated from scratch. Crucially, the authors decontaminate them: every candidate query was first handed to Gemini 2.5 Flash, and anything the model could already answer without tools was discarded — isolating problems that genuinely require invoking something external. Grading looks only at the final answer (exact match); the tool path is not scored, because a correct answer can be reached many ways.

Fuzzy Match tasks judge the operation, not a text answer: success means invoking the right tool with the right arguments. They cover Email (119, 17 tools), Calendar (28, 4 tools), File management (91, 13 tools), and Terminal (63, 9 tools), generated programmatically from per-tool templates (five tasks per template). Multi-tool variants chain the three most-used tools in each domain, with a strictly-defined invocation sequence so the solution stays unique and verifiable. Each task carries one ground-truth tool name plus arguments, and the model's final decisive tool call must match it.

Real tools, sandboxed actions

The 49 MCP tools mix real, open-source MCP servers — calculator-mcp-server for math, duckduckgo-mcp-server for search, community servers for files and terminal — with high-fidelity mocks for Email and Calendar (EmailMCP, CalendarMCP) that replicate the interfaces of GmailMCP and macOS CalendarMCP but talk to a controlled local database pre-seeded with 100 emails and 50 calendar entries. That keeps email, file, and terminal actions reproducible and safe — nothing touches a live system — while preserving realistic tool schemas.

Three objective metrics

Performance is reported along up to three axes. Result Accuracy (RA) is a binary exact-match: did the final answer (Precise) or final tool call's name and parameters (Fuzzy) match the ground truth. Dialogue Turn Success Rate (DTSR), defined only for Fuzzy tasks, is the fraction of interaction turns that produced a well-formed, applicable tool call — a measure of step-by-step execution skill that is independent of whether the final answer was right. Computational Resource Efficiency (CRE) captures token cost, max-min normalized across models so it can be compared on a single radar.

DTSR = # successful tool invocations# total interaction turns // a "successful" invocation selects an applicable tool with correctly-formatted parameters

Ten models were run through OpenRouter with a cap of K = 10 interaction rounds (exceeding it counts as failure): six closed-source — GPT-5, GPT-4o, Gemini 2.5 Flash, Gemini 2.5 Pro, Claude Sonnet 4, Claude 3.7 Sonnet — and four open-source — Qwen3-235B-A22B, DeepSeek-V3-0324, DeepSeek-R1-0528, and Llama-4-Maverick.

// what they found

The results

On Precise Answer tasks, closed-source models lead — but the size of the lead depends entirely on the domain. In math, Gemini 2.5 Flash tops the table at 61.2% with GPT-5 just behind at 60.7%, while open-source trails badly: DeepSeek-V3 at 28.7% and Llama-4 at just 12.8%. Web search is the great equalizer — and a wall for everyone. Every one of the ten models scored under 30%; Gemini 2.5 Pro was best at 29.8%, Llama-4 worst at 0.8%. The closed-vs-open gap that yawns on math collapses here: 20.7% average for closed models versus 10.8% for open — a margin under ten points.

On Fuzzy Match tasks, accuracy tracks task complexity. Simple operational domains score high — Gemini 2.5 Pro hits 82.5% on both Email and Calendar — while domains needing precise sequential logic crater: the best File-management accuracy is 59.6% and the best Terminal accuracy 59.9% (both Gemini 2.5 Pro). The radar charts below make the trade-off visible: leading closed models are well-rounded; some open models (notably Qwen3) reach competitive accuracy but burn far more tokens to get there.

The signature finding: syntax without semantics

The most telling result is a gap within a single model. On File management, GPT-4o reached an 84.5% DTSR but only 43.8% accuracy — a 40.7-point chasm. The model emits well-formed, applicable tool calls turn after turn, then fails to select the semantically appropriate tool to actually solve the task. This pattern — fluent execution, poor tool selection — recurs across models and domains and is the paper's central diagnosis: today's LLMs have a superficial grasp of what a task truly requires.

A targeted experiment drives the point home. When the authors gave models a hint about how many tools a multi-tool task needs, accuracy barely moved — a 2.5%–5% improvement. The bottleneck is not recognizing that a tool is needed; it is deciding which tool to invoke and how to parameterize it correctly.

An error breakdown explains the two task types' very different failure profiles. On Precise Answer tasks, the dominant failure is Tool Omission — the model trusts its own parametric knowledge instead of calling a tool (the single largest error category) — followed by Faulty Reasoning. On Fuzzy Match tasks, failures flip to execution: Inaccurate Tool Invocation (picking the wrong tool) is the most common error, with Parameter Errors close behind — malformed arguments like placing a recipient's email address in the subject line.

// does it hold up?

The eval read

MCP-RADAR's model roster is genuinely current — GPT-5, Claude Sonnet 4, Gemini 2.5, DeepSeek-R1, Llama-4, Qwen3 — and its decontamination step is a real attempt to isolate tool use from recall. But several design choices mean the precise numbers should be read as directional rather than definitive.

Start with the decontamination filter. Using Gemini 2.5 Flash to discard any query it can answer tool-free is sound in spirit, but it is a single-model gate: it can't guarantee a stronger model (GPT-5) won't still solve a "tool-required" math problem from memory, and it biases the retained set toward Flash's specific blind spots. There's a mild circularity worth noting — Flash then tops the Math leaderboard (61.2%) on a set whose membership was defined by Flash's own inability to solve it tool-free.

The bigger lever is exact-match operation grading. Fuzzy-Match RA demands the final tool call's name and every argument equal a single canonical ground truth. Real operational tasks admit many valid forms — different file paths, equivalent email phrasings, alternate but correct tool orderings — and single-reference exact match under-counts correct-but-different solutions. That likely inflates the very DTSR-vs-RA gap the paper headlines: some of the "wrong" final calls behind GPT-4o's 43.8% may be valid alternatives the grader rejected. The paper grants this non-uniqueness for Precise Answer (it grades only the final value) but enforces a rigid single answer for operations.

The CRE metric is under-specified. The prose calls it "a measure of the computational cost," yet the radar charts treat a longer (higher) CRE edge as superior, and max-min normalization makes it a relative, dataset-bound score with no absolute token counts reported. The direction is genuinely ambiguous, and "efficiency" can't be compared across the paper's own tables, let alone to outside work. Add that the headline "507 tasks" doesn't reconcile with Table 1's domain counts (which sum to 515), domains are lopsided (Email 119 vs Calendar 28), multi-tool subsets are tiny (Calendar: 8), and metrics are single-run binaries with no seeds or confidence intervals — so rankings separated by a point or two (GPT-5 60.7 vs Gemini-Flash 61.2 on math) sit inside the noise.

baselines

Ten up-to-date frontier models (GPT-5, Claude 4, Gemini 2.5, DeepSeek-R1, Llama-4, Qwen3) — a strong, current roster. But no human or oracle-tool upper bound to anchor what these accuracies mean.

contamination

Actively addressed — a Gemini 2.5 Flash filter strips queries solvable without tools — but the gate is single-model and slightly circular (Flash then leads the very set it defined).

what's missed

Single-reference exact-match grading penalizes valid alternative tool paths/parameterizations; no error bars or seeds; the CRE "efficiency" axis has an undefined direction and no absolute token numbers.

would raise confidence

Multi-reference operation grading, multiple seeds + CIs, absolute token/cost figures beside CRE, a human/oracle upper bound, and a live-tool validation slice to test the "high-fidelity mock" claim.

// bottom line

The lasting contribution is conceptual: a clean separation of "can the model fire a valid tool call" (DTSR) from "did it pick the right one" (RA), and the robust, well-supported finding that frontier models are far better at the former. The exact-match operation grading and the under-specified CRE axis leave the exact rankings soft — but the qualitative diagnosis, syntactic tool competence atop semantic tool confusion, holds.

// takeaways

What to remember

  1. The gap that matters isn't if a tool is needed but which one. Models reliably emit well-formed tool calls — GPT-4o hit 84.5% DTSR on file management — yet pick the wrong tool often enough to drop task accuracy to 43.8%, a 40.7-point syntax-vs-semantics gap. Telling the model how many tools to use barely helps (+2.5–5%).
  2. Web search is the universal wall: all ten frontier models scored under 30% (best: Gemini 2.5 Pro at 29.8%), and the closed-vs-open chasm that's wide on math (Gemini Flash 61.2% vs Llama-4 12.8%) shrinks to under ten points here.
  3. Two metrics worth borrowing: DTSR (valid-call rate per turn) and an objective exact-match operation check decouple execution skill from tool selection — even if MCP-RADAR's own single-reference grading and ambiguous CRE axis leave room to sharpen them.
