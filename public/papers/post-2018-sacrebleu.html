A Call for Clarity in Reporting BLEU Scores

Everyone quotes “the” BLEU score as if it were one number. It is not: BLEU is a family of metrics whose hidden tokenization and casing knobs swing the result by up to 1.8 points — often more than the gain a paper is claiming. SacreBLEU locks those knobs and prints a version string so two papers' numbers can finally be compared.

Matt Post · Amazon Research, Berlin

What problem is this solving?

Machine translation runs on a single number. A new model is “better” because its BLEU went up. But Post points out an embarrassing fact the field had been living with for fifteen years: there is no such thing as “the” BLEU score. BLEU (Papineni et al., 2002) is a parameterized metric, and the same system output can be scored to wildly different numbers depending on choices that almost nobody writes down.

Two layers of hidden choices sit underneath any reported BLEU. The first is the metric's own parameters — the number of references, the length-penalty calculation for multi-reference sets, the maximum n-gram order, and the smoothing of zero-count n-grams. The second, and the one that does the real damage, is preprocessing of the reference translation: tokenization, normalization, lowercasing, compound-splitting. Because BLEU is a precision metric over n-grams, changing how the reference is tokenized changes the very set of n-grams the system is graded against. Score two systems against differently-processed references and the comparison is simply meaningless — even though both numbers wear the same “BLEU” label.

The problem is compounded by silence. Papers rarely state which tokenizer they used, whether scores are cased or uncased, or how many references the test set carries. Post tries to reverse-engineer the configuration of seven influential MT papers and mostly cannot — the foundational Transformer paper (Vaswani et al., 2017) is logged as “user or userlc (unclear).” Even the dataset itself is ambiguous: the heavily-used WMT’14 English–German test set exists in two versions (2,737 vs. 3,004 sentences), and researchers routinely fail to say which one they scored on.

// the core claim

BLEU scores from different papers are not comparable unless they share the same reference preprocessing — and the variation this introduces is frequently larger than the improvements those papers report. A field that compares models by BLEU has, quietly, been comparing apples to subtly different apples.

// how it works

The fix: take the reference away from the user

The conceptual key is a distinction Post draws sharply: user-supplied vs. metric-internal tokenization. When people say a score is “tokenized” or “detokenized,” they are confusing themselves — for the purpose of computing BLEU, both the system output and the reference are always tokenized. What actually varies is who does it: the researcher's own pipeline, or the scoring code. The only way scores become comparable across papers is if the reference is processed by the metric, identically, every time.

The pipeline below is the whole argument in one picture. The user is allowed to do whatever they like to their own system output — preprocess it, decode it, postprocess it back into detokenized text. But the user must never touch the reference. The reference flows straight into the metric (the black box), which applies its own internal tokenization to both sides before scoring.

An old community already solved this

Post's model is not a new algorithm but a piece of social engineering, borrowed from the parsing community. Parser quality has been comparable across papers for nearly thirty years thanks to PARSEVAL and a single standard codebase, evalb, whose COLLINS.PRM parameter file freezes every awkward edge case (count the ROOT node? punctuation? — answered once, for everyone). MT had the raw materials — WMT's official mteval-v13a.pl uses metric-internal preprocessing — but every available script had a flaw: Moses' multi-bleu.perl demands user tokenization, mteval-v13a.pl demands clumsy XML wrapping, and even Nematus' detokenized variant still hands the reference to the user.

SacreBLEU

The deliverable is a small Python tool, sacrebleu, that puts a “protective layer” between the researcher and the reference. It (1) ships and downloads the standard test sets itself — all WMT 2008–2018 and recent IWSLT sets — so the user never handles the reference at all; (2) expects detokenized system output and applies WMT's own metric-internal preprocessing, reproducing the official scores; and (3) emits a compact version string that records every parameter, meant to be pasted directly into a paper.

BLEU+c.mixed+l.en-de+#.1+s.exp+t.wmt14+tok.13a+v.1.2.10 // mixed-case · en–de · 1 reference · exponential smoothing · WMT14 test set · ‘13a’ tokenization · sacrebleu 1.2.10

Usage is one line — cat output.detok | sacrebleu -t wmt14 -l en-de — and the restored 3,004-sentence WMT’14 set is reachable with -t wmt14/full, so even the two-datasets trap becomes an explicit, recorded flag rather than a silent one. The tool is Apache-2.0 and also exposes chrF via -m. Nothing here changes what BLEU measures; it changes what gets reported, so that the number is reproducible and self-documenting.

// what they found

The variation is bigger than the gains

To quantify the damage, Post takes a single system — the online-B submission from WMT’17 — and scores it four ways across twelve language arcs, varying only the reference preprocessing: basic (Moses tokenizer), split (compound-splitting), unk (rare words mapped to UNK), and metric (WMT's internal tokenization). The system is identical; only the bookkeeping moves.

The spread reaches 1.8 BLEU on German–English and averages roughly 1.0 across arcs — and that range deliberately excludes the most extreme column, unk. The single largest lever is casing — the paper flags it as “the biggest” single effect — yet many papers never say whether they report cased or case-insensitive BLEU at all. For perspective, a great many MT papers stake their contribution on gains well under one BLEU point. The reporting noise, in other words, routinely swamps the signal.

The number of references is its own trap

Beyond tokenization, simply changing how many reference translations a test set carries reshapes the scale entirely:

SettingBLEU
WMT’17 en–fi online-B, 1 reference22.04
WMT’17 en–fi online-B, 2 references25.25
NIST OpenMT ar–en / zh–en, 4 referenceshigh-40s → low-50s

The same system jumps 3.2 BLEU on English–Finnish purely by adding a second reference, and four-reference NIST test sets land in the high 40s and low 50s — numbers that, gathered under the single word “BLEU,” quietly teach the community to expect very different scores for different language pairs without anyone flagging why.

And nobody documents which knobs they turned

Post's attempt to recover the configuration of seven well-cited benchmark papers is the rhetorical punchline: in most cases the setup cannot be determined from the paper at all.

Seven landmark systems, and no two demonstrably comparable — including the Transformer, whose reported BLEU underpins much of modern NLP. That is the case for sacreBLEU stated as evidence rather than argument.

// does it hold up?

The eval read

This is a rare paper whose entire content is an evaluation critique, so the fair way to read it is to turn its own lens back on it. On its central claim it is unimpeachable and it won decisively: sacreBLEU and its version string became the de-facto standard for MT reporting, and the “ship the references, print the config” pattern became a template later metric tools followed. But it is worth being precise about what it fixed and what it left untouched.

First, the empirical core is thinner than its influence suggests. The headline numbers in Table 1 come from one system (online-B) on one year's test sets (WMT’17), with no confidence intervals and no second system to confirm the spread generalizes. The dramatic “1.8” is a single arc (de–en); the average is nearer 1.0. And the most alarming column — unk-masking the reference to inflate scores — is explicitly framed as a hypothetical (“This hypothetical scenario could easily happen”), not a documented case from a real paper, and the range rows set it aside as an out-of-band corruption rather than a legitimate tokenization choice. The argument is correct, but the quantification is a vivid illustration, not a survey.

Second, and more important, sacreBLEU standardizes BLEU's reporting while deliberately bracketing BLEU's validity. Post says so plainly — the paper is “not concerned with the shortcomings of BLEU as a proxy for human evaluation.” But that boundary has a cost: by making BLEU the clean, reproducible, citeable default right at the close of the 2010s, the tool helped entrench a string-overlap metric whose correlation with human judgment was already contested (the very Callison-Burch et al. 2006 critique it cites), exactly as learned metrics that correlate better with humans — BLEURT (2020), COMET, and others — were arriving. A reproducible bad proxy is still a bad proxy; comparability is necessary but not sufficient.

Third, the fix centralizes trust. “Take the reference away from the user” works by making sacreBLEU the canonical distributor of test sets — a single authority, and a single point of failure, for what the references even are. And the 2018 default tokenization (‘13a’) the paper recommends is, in hindsight, essentially English-centric; making BLEU truly comparable for Chinese, Japanese and other scripts later required language-specific tokenizers that sacreBLEU itself had to add in subsequent versions — a reminder that “metric-internal” is itself a parameter choice, not a neutral ground truth.

baselines

Not a model paper, so there are no baselines to be unfair to — but the comparison set for the variation claim is a single system on a single year; the spread is illustrated, not measured across systems.

contamination

The relevant analogue is the WMT’14 en–de two-dataset trap (2,737 vs. 3,004 sentences): sacreBLEU turns it into an explicit flag, but only if authors actually report the version string.

what's missed

Comparability, not validity. The number becomes reproducible; whether BLEU should be the number at all is out of scope — and standardizing it arguably slowed the migration to human-correlated metrics.

would raise confidence

Multiple systems and test-set years with confidence intervals on the variation; at least one documented (not hypothetical) case of unk-inflated reference scoring in the wild.

// bottom line

A small paper with outsized, lasting impact: it diagnosed a real, embarrassing source of irreproducibility and shipped the fix in the same breath. SacreBLEU made MT numbers comparable and self-documenting — a genuine win for scientific hygiene. Just remember what it standardized: the protocol for reporting BLEU, not the case that BLEU is what you should be reporting.

// takeaways

What to remember

  1. “The” BLEU score is a fiction. BLEU is a family of metrics; reference tokenization, casing and reference count move it by up to 1.8 points — routinely more than the gain a paper claims.
  2. The fix is to take the reference away from the user. SacreBLEU ships the test sets, applies metric-internal preprocessing, and prints a version string so any two scores can be compared and reproduced — PARSEVAL's evalb discipline, ported to MT.
  3. It standardized reporting, not validity. The number is now reproducible; whether a string-overlap metric should be the headline number is a separate question the paper deliberately leaves open — and arguably entrenched.
