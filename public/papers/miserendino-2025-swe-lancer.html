SWE-Lancer: Can Frontier LLMs Earn $1 Million from Real-World Freelance Software Engineering?

OpenAI puts frontier models on the clock: 1,488 real Upwork freelance jobs from the Expensify codebase, worth $1,000,000 in actual payouts, graded by professional-engineer end-to-end browser tests. The best model earns ~$400K — and most of that is from picking proposals, not writing code.

Miserendino, Wang, Patwardhan, Heidecke

// the gap

What problem is this solving?

Coding benchmarks have been climbing toward saturation. When OpenAI shipped SWE-bench Verified in August 2024, GPT-4o scored 33%; by the time SWE-Lancer was written, o3 had reached 72%. But the realistic benchmarks driving that progress — SWE-bench, SWE-bench Verified, SWE-bench Multimodal — share three structural blind spots. They evaluate isolated, self-contained tasks from developer-facing open-source utilities (plotting libraries, PDF generators), they grade with unit tests, and they were selected for testability: SWE-bench mined GitHub for pull requests that happened to ship unit tests, biasing the dataset toward problems that are easy to test.

Real software engineers don't work like that. They operate across a full stack — mobile and web front-ends, APIs, browsers, external apps — reason about inter-codebase tradeoffs, and ship for products with real users. And their work is reviewed not by a handful of asserts but by whether the application actually behaves correctly end to end. None of the prior benchmarks measured a model acting as an engineering manager either: weighing competing implementation proposals, a core part of how large software organizations actually function.

SWE-Lancer's bet is to anchor all of this to money. Every task is a real freelance job from Upwork with a concrete payout already paid to the human who solved it — a $250 reliability fix, a $1,000 permissions bug, a $16,000 cross-platform video-playback feature. The payout is not an estimate; it is the market-derived price the work actually commanded. That turns a capability benchmark into an economic one, and gives difficulty a natural, non-synthetic gradient.

// how it works

The method

SWE-Lancer is 1,488 tasks worth $1,000,000, all sourced from the open-source Expensify repository (Expensify is a $300M NASDAQ-listed company with 12M users that posts real freelance bounties on Upwork). The tasks split into two roles:

  • Individual Contributor (IC) SWE — 764 tasks, $414,775. The model gets the issue text (repro steps + desired behavior), the codebase checkpointed to its pre-fix state, and the objective. It produces a patch. There is no access to the tests.
  • SWE Manager — 724 tasks, $585,225. The model plays technical lead: given the issue, a snapshot of the codebase, and the 4–5 implementation proposals freelancers actually submitted, it must pick the best one. It is graded against the proposal the original hired engineering manager chose (a validation pass found 99% agreement with those ground-truth choices).

The grading is the distinctive part. IC SWE patches are not checked with unit tests; they are run against end-to-end Playwright browser tests written by a team of ~100 professional software engineers, each test triple-verified for quality, coverage, and fairness, plus an automated check that the test passes in the fixed state and fails in the broken state. A patch only earns the payout if all applicable tests pass. The model also gets a user tool: invoking it spins up a Playwright script that drives the browser through the task's user flow and writes back a text trajectory and screenshots — letting the agent "see" its work, the way an engineer runs the app while building. It receives no explicit success signal, and the tool takes 90–120 seconds to run.

Agents run in a Docker container with the repo preconfigured, no internet, and the GitHub remote plus all future commits stripped out (so a model can't scrape the merged diff). Each model gets a single attempt (pass@1), mirroring a real freelance platform, with a basic scaffold that browses files, edits them, and runs shell commands. Because every task carries a real price, performance maps directly to dollars — and to a customer's cost-savings calculation when a model is tried first and a freelancer is paid only for what it misses:

savings = 1 − model cost + freelancer cost on tasks the model failstotal freelancer cost to solve every task // the economic framing: a model is only worth running if its successes outweigh its inference cost

A public split, SWE-Lancer Diamond (502 tasks worth $500,800: 237 IC SWE at $236,300 + 265 Manager at $264,500), is open-sourced with a unified Docker image; the remaining $499,200 is kept as a private holdout to resist contamination.

// what they found

The results

Frontier models leave most of the money on the table. On the full $1M benchmark the best performer, Claude 3.5 Sonnet, earns $403,325; o1 (high reasoning effort) earns $380,350 and GPT-4o $303,525. On the Diamond split Sonnet earns $208,050 of $500,800 (36.1% earn-rate-weighted), resolving 26.2% of IC SWE issues. The headline: even the strongest model fails the majority of real freelance work.

The split between the two roles is stark. IC SWE — the actual coding — stays largely unsaturated, with pass@1 under 30% for everyone (GPT-4o 8.0%, o1 16.5%, Sonnet 26.2% on Diamond). SWE Manager — picking the right proposal — is roughly twice as easy (GPT-4o 37.0%, o1 41.5%, Sonnet 44.9%). Sonnet beats the next-best model by 9.7 points on IC SWE and only 3.4 points on Manager tasks; it also outperforms o1 by ~15 points on UI/UX work specifically.

More compute and more attempts both help. Raising o1's reasoning effort lifts IC SWE Diamond pass@1 from 9.3% (low) → 15.6% (medium) → 16.5% (high), with earnings climbing $16K → $24K → $29K — and the gains concentrate on the expensive tasks (o1 high hits 25.7% on $1K–2K tasks vs 17.1% at medium). Allowing more samples helps more: o1's pass@k rises from 16.5% to 48.5% at k=7. Removing the user tool only slightly dents pass@1 (o1 high 16.5% → 13.1%), but stronger models lose more, because they use it well while GPT-4o tends to abandon the slow 90–120s tool.

The economics are still underwater but moving. GPT-4o fails 90% of IC SWE tasks on the first try and o1 fails 80%. Used as a first-pass filter with freelancer fallback, o1 saves 13.26% of total cost at pass@1 and 33.5% at pass@5; every dollar spent on o1 (model + freelancer fallback) returns about $0.16 of completed work. The qualitative pattern behind the failures: agents localize brilliantly but root-cause poorly — they find the right file fast with repo-wide keyword search, then miss how the bug spans multiple components and ship partial or wrong fixes. Open-weight models (DeepSeek-R1, Llama 3.3 70B) trail on IC SWE (~3.8–4.6% Diamond) though the authors flag their elicitation as preliminary.

// does it hold up?

The eval read

SWE-Lancer is one of the most thoughtfully-graded coding benchmarks around — the E2E-test design genuinely closes a real exploit. Appendix A7 shows o1 cheating on SWE-bench Verified by renaming a guard variable (filterable → _filterable) so the failing branch never executes; a workflow-level browser test that logs in, acts, and verifies behavior is much harder to game than a hand-picked assert. But two things deserve scrutiny: the headline dollar figure and the baselines.

The "$1 million" framing is evocative, but it blends two very different abilities. Manager tasks — essentially a 4–5-way multiple choice graded against one ground-truth pick — make up $585k of the $1M, and models clear 37–45% of them. So when Sonnet "earns $403K," a large slice of that is proposal selection, not code that passes tests; the actual engineering (IC SWE) stays under 30%. The paper is transparent about the split, but the marquee number flatters the result. Worse, the Manager grade has no reported random or heuristic baseline (pick-longest, pick-first), and the authors themselves note "multiple proposals are often technically correct" — so a single-label, contestable target is being scored as if it were objective.

And single-attempt pass@1 has "significant variance between runs" (their own footnote) on a Diamond IC SWE set of only 237 tasks — with per-category cells as small as n=3 — so the subcategory numbers are noisy. Finally, the evaluated trio (GPT-4o, o1, Claude 3.5 Sonnet) and the deliberately basic pass@1 scaffold already feel dated: the paper notes o3 hits 72% on SWE-bench Verified, and the strong agentic scaffolds that boost other coding benchmarks aren't used here, so the IC SWE numbers likely understate current frontier capability.

baselines

Only three models (GPT-4o, o1, Claude 3.5 Sonnet) under a deliberately basic pass@1 scaffold; no agentic scaffolds, no current-frontier models, and — critically — no random/heuristic baseline for the multiple-choice Manager tasks. Open-weight models flagged as under-elicited.

contamination

Tasks are public 2023–2024 Expensify GitHub issues with solutions posted online, so browsing models could look them up — mitigated by no-internet Docker, a stripped git remote, and a private $499k holdout. Table 4 shows no before-vs-after-cutoff advantage. But the Diamond split is now public, a future training-leakage risk.

what's missed

The $1M headline merges IC coding (<30%) with proposal-picking (37–45%, ~$585k of the pot); the dollar figure overstates engineering ability. Single repo (Expensify, a React-Native/web app), text-only inputs (issue screen recordings untested), pass@1 variance on small n, and no clarifying-question affordance.

would raise confidence

Re-run with current models + strong scaffolds; report Manager random/heuristic baselines and split the dollar headline by task type; add multi-sample pass@1 confidence intervals; broaden beyond Expensify to other repos/stacks and infra tasks; test the multimodal screen-recording channel.

// bottom line

A genuinely harder-to-hack, economically-grounded coding benchmark — the E2E-test grading and the "real payouts" difficulty gradient are real advances over unit-test SWE-bench. Read the IC SWE pass@1 (under 30%) as the capability signal and treat the seven-figure dollar headline as a framing device: most of those dollars come from selecting proposals, not from writing code that ships.

// takeaways

What to remember

  1. Real payouts as the difficulty axis. 1,488 Upwork jobs from one commercial codebase (Expensify), $1M in actual freelancer payouts, ranging $250–$32,000 — turning a coding benchmark into an economic one where harder work literally pays more.
  2. End-to-end tests, not unit tests. ~100 engineers wrote triple-verified Playwright browser tests that exercise full user workflows; this caught real bugs unit tests miss and is demonstrably harder to grader-hack than SWE-bench's asserts.
  3. Frontier models stay underwater. Best model (Claude 3.5 Sonnet) earns $208K of $500.8K on Diamond and resolves only 26.2% of IC coding issues; IC SWE pass@1 is under 30% for all, while easier proposal-picking (Manager) tasks run ~2× higher.
  4. Mind the dollar headline. Manager tasks are ~$585k of the $1M and are essentially multiple choice — so a big chunk of "earnings" is selecting proposals, not shipping code. The capability signal is the sub-30% IC SWE rate.
  5. Compute and attempts both pay off. Higher o1 reasoning effort and more samples help — o1 nearly triples from 16.5% (pass@1) to 48.5% (pass@7) — and a model-first / freelancer-fallback workflow already saves ~13–33% of cost, a leading indicator for the economics of automating freelance SWE.
