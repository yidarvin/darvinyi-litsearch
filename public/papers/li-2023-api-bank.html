API-Bank: A Comprehensive Benchmark for Tool-Augmented LLMs

A runnable, three-tier benchmark — Call, Retrieve+Call, Plan+Retrieve+Call — grounded in 500 user interviews, built from 73 hand-implemented APIs and 314 annotated dialogues, plus a five-agent pipeline that cut synthetic training-data cost by 98%.

Minghao Li, Yingxiu Zhao, Bowen Yu, Feifan Song, Hangyu Li, Haiyang Yu, Zhoujun Li, Fei Huang, Yongbin Li

// the gap

What problem is this solving?

By early 2023, a wave of papers had shown LLMs can use tools — ReAct chains search calls into reasoning, Toolformer teaches a model to insert its own API calls, WebGPT browses for citations. But every demonstration was anecdotal: a handful of curated examples, no systematic way to compare one tool-augmented model against another, and no shared definition of what "good at tools" even means. The authors frame this as three unanswered questions: how effective are current LLMs at using tools, how do you make them better at it, and what's actually stopping them?

The benchmarks that did exist were narrow. The paper's own comparison table stacks API-Bank against six contemporaries — Toolformer's dataset (1 domain, 1 API), APIBench/Gorilla (90 domains, 1,645 APIs), ToolAlpaca (50 domains, 426 APIs), Qin et al.'s ToolBench (49 domains, 16,464 APIs), Xu et al.'s tool-manipulation set (8 domains, 232 APIs), and ToolQA (6 domains, 13 APIs) — and none of them combine multi-turn dialogue, multi-call-per-turn requirements, grading of both the API call and the generated response, and all three tool-usage abilities in one place.

To ground those abilities in something other than researcher intuition, the authors ran structured interviews with 500 users who wanted tool-augmented LLMs, and derived two axes from what came back: how many APIs a user hands the model (few vs. many, i.e. whether it must be retrieved first) and whether a request needs one API call or several chained together (i.e. whether it must be planned). The method

API-Bank has two halves that share a design: a manually-annotated evaluation system that answers "how good are LLMs today," and an automatically-generated training set that answers "can we make them better."

The evaluation system — 73 real, executable APIs

The team's engineers spent 98 person-days implementing 73 commonly-used APIs — weather, calendars, account management, search, even calls into other AI models like text-to-image — all runnable within one framework, with databases seeded with real entries so an API call actually changes system state. A special API Search tool lets the model retrieve candidate APIs by embedding-similarity when the pool is too large to fit in the prompt. On top of that system, annotators built 314 dialogues with 753 API calls spanning the three abilities, at an annotation cost of $8 per dialogue after discarding 21.5% of a larger annotated pool for quality. Every model is graded two ways: Accuracy on the predicted API call against the annotated ground truth (same database operation, same returned result), and ROUGE-L on the model's generated response text.

The training set — a five-agent factory instead of human annotators

Scaling the evaluation set's recipe to training-set size hit a wall: recruited annotators could only invent 100 APIs by hand, and a single LLM asked to generate a whole tool-use dialogue in one shot — domain, API, query, ability, call, and response, all at once — only produced usable output 5% of the time with ChatGPT and 25% with GPT-4. The fix is Multi-agent: decompose the same task into five specialized LLM agents, each handling one sub-step, so no single call has to satisfy every constraint at once.

The result: 2,138 APIs across 1,000 domains, distilled into 1,888 training dialogues with 4,149 API calls — at $0.1 per dialogue, a 98% cost reduction versus the $8/dialogue manual process. A separate quality check on 100 sampled training instances found a 94% availability rate for Multi-agent's output, an 89% improvement over a single-agent Self-Instruct baseline. The authors then fine-tune Lynx — Alpaca-7B continued for three epochs (batch 256, lr 2e-5) on this synthetic corpus — as the paper's proof that the generated data is actually useful for training, not just cheap.

// what they found

The results

On the 314-dialogue evaluation set, performance falls sharply as the required ability climbs from Call to Plan+Retrieve+Call for every model except GPT-4, and instruction-tuning matters more than raw scale: the 175B GPT-3 Davinci scores just 0.57% overall accuracy — barely above zero — while the far smaller, instruction-tuned Alpaca-7B (15.19%) and ChatGLM-6B (16.42%) clear double digits.

Fine-tuning on synthetic data closes most of the gap to GPT-3.5. Lynx-7B improves over its Alpaca-7B starting point by 26 points in Call accuracy and 0.41 in response ROUGE-L, landing at 39.58% overall — approaching GPT-3.5-turbo's 47.16% from a fraction of the parameters, though still 21 points behind GPT-4. On a head-to-head with a same-format Alpaca model fine-tuned on ToolAlpaca's training set instead, Lynx (6,184 samples) edges out the ToolAlpaca fine-tune (10,366 samples) on both accuracy (54.64 vs. 53.88) and ROUGE (39.80 vs. 39.75) — fewer synthetic examples, slightly better result.

GPT-4's 70% on the hardest tier is the outlier the paper leans on — a "nearly 50% improvement" over GPT-3.5's 22% that the authors attribute to superior planning and reasoning, similar to GPT-4's edge on math tasks. But error analysis (Table 6) shows GPT-4's dominant failure mode on this tier is failed API retrieval (67.86% of its errors) — it still can't reliably find the right tool, it just plans well once it does.

Where each model actually breaks

The per-model error breakdowns (Tables 4–6) tell three different stories. Alpaca-7B's top failure is simply "No API Call" (36.77%) — it often doesn't attempt one at all. After fine-tuning into Lynx, that collapses to 5.29%, but a new failure dominates: API Hallucination at 61.38% — inventing an API name it saw during training but that isn't in the current pool. GPT-4's errors are almost entirely retrieval failures (67.86%) plus unparseable multi-call formatting (17.86%) — a strong reasoner that still struggles to search a tool library correctly.

// does it hold up?

The eval read

API-Bank set a genuinely useful template — a runnable system instead of static text matching, a three-ability taxonomy grounded in real user interviews, and a training pipeline whose cost argument is concrete and checkable ($8 → $0.10/dialogue). But several of its measurements are thinner than the headline numbers suggest, and one is arithmetic: with only 50 test dialogues in the Retrieve+Call and Plan+Retrieve+Call tiers, every percentage point is exactly 2 dialogues — GPT-4's marquee 70.00% on the hardest tier is 35/50, and a single different rollout would move the score by 2 points. That's not disclosed as a confidence interval anywhere in the paper.

The comparison table is self-graded on competitors' behalf. Table 1's checkmarks for whether ToolAlpaca, the two ToolBench variants, ToolQA, and Gorilla's APIBench support "multi-turn," "multi-call," or "response" evaluation are assigned by the API-Bank authors reading those papers, not run against those benchmarks directly — and the table conveniently has API-Bank checking every box. Look closely and the "most comprehensive" claim rests on one column: Qin et al.'s ToolBench1 (the ToolLLM paper, concurrently released with over 16,000 APIs — nearly 8× API-Bank's 2,138) already matches API-Bank on multi-turn, multi-call, and all three abilities in the same table; the only box it doesn't check is "Response" evaluation. That's a real point of comparison, but it's a much narrower gap than "the most comprehensive benchmark for tool-augmented LLM available at present" implies.

ROUGE-L is a weak proxy for response quality. A correct assistant reply phrased differently from the reference will score low ROUGE even though it fulfilled the user's request — and there's no human evaluation of response quality anywhere to sanity-check what the automatic metric is actually rewarding. The paper itself leans almost entirely on the Accuracy (API-call-match) numbers in its narrative, which suggests even the authors treat ROUGE as secondary.

Training and test APIs are drawn from different generative processes, not held out from the same distribution. The 2,138 training APIs are LLM-invented (seeded only by examples from a public GitHub API list); the 73 evaluation APIs are hand-implemented and genuinely executable. The paper acknowledges "variations exist among them," which is honest, but it also means Lynx's gains partly reflect generalizing from imagined-API patterns to real ones — a harder and less well-characterized transfer than a standard train/test split within one distribution.

2026 contamination risk, unavoidable given the paper's age. API-Bank's dialogues and code have been public on GitHub since April 2023; every frontier model trained since has almost certainly seen this exact benchmark in pretraining. Any accuracy number reported for a post-2023 model against this benchmark today should be read as an upper bound at best, not a clean capability measurement — a caveat the paper obviously couldn't have anticipated but that matters for anyone citing its numbers now.

baselines

GPT-3 Davinci, GPT-3.5-turbo-0613, GPT-4-0613, ChatGLM-6B, Alpaca-7B — appropriate and current for mid-2023; no later open-source tool-tuned baselines (e.g. ToolLLaMA) since none existed yet.

contamination

Not a concern at publication (evaluation APIs are original implementations), but the public GitHub release means any model trained after April 2023 has likely seen this benchmark — a real issue for anyone re-running it today.

what's missed

Retrieve+Call and Plan+Retrieve+Call tiers rest on only 50 dialogues each, so headline percentages carry large, undisclosed uncertainty; ROUGE-L for response quality has no human-eval cross-check; the competitor comparison table is self-assessed, not independently reproduced.

would raise confidence

Error bars or multiple-seed evaluation on the small tiers, a human evaluation of response quality alongside ROUGE, and an independent reproduction of Table 1's feature checkmarks against the actual competing benchmarks.

// bottom line

A field-shaping benchmark whose taxonomy (Call / Retrieve+Call / Plan+Retrieve+Call) and cheap-synthetic-data recipe both had real staying power in later tool-use work. The core weakness is scale: the two hardest, most benchmark-defining tiers are graded on only 50 examples each, so precise numbers like GPT-4's 70.00% should be read as "roughly 2 in 3," not a stable three-significant-figure measurement.

// takeaways

What to remember

  1. A three-ability taxonomy grounded in user research. Call, Retrieve+Call, and Plan+Retrieve+Call came from 500 real user interviews, not researcher intuition — and the taxonomy outlived the specific benchmark, shaping how later tool-use papers frame difficulty.
  2. Five cheap LLM agents beat one expensive one at data generation. Decomposing "generate a whole tool-use dialogue" into five narrower agent roles (domain → API → query → call/response → verify) turned a 5%-usable-output problem into a 94%-usable one, cutting cost 98% ($8 → $0.10/dialogue) while still producing training data that out-performed a same-format human-curated alternative.
  3. Instruction-tuning beats scale for tool use — and reasoning still gates retrieval. A 175B un-tuned GPT-3 Davinci (0.57%) loses badly to instruction-tuned 6–7B models (15–16%); GPT-4's standout 70% on the hardest tier comes from strong planning despite its own errors still being dominated by failed API retrieval (67.86%), not planning mistakes.
