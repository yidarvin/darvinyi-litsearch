Reasoning with Language Model is Planning with World Model

RAP repurposes a single frozen LLM twice — once as a reasoning agent that proposes actions, once as an internal world model that predicts the resulting state — then searches the LLM's own simulated reasoning tree with Monte Carlo Tree Search instead of committing to one linear chain of thought.

Shibo Hao, Yi Gu, Haodi Ma, Joshua Jiahua Hong, Zhen Wang, Daisy Zhe Wang, Zhiting Hu · UC San Diego / University of Florida
venue EMNLP 2023 citations 1,074 topic Reasoning source ↗

// the gap

What problem is this solving?

By 2023, chain-of-thought prompting had made LLM reasoning look deceptively close to human reasoning: give the model a few worked examples and it produces a plausible step-by-step trace. But CoT and its relatives — self-consistency, least-to-most prompting — all still generate a reasoning path the same way: autoregressively, once, forward, with no lookahead and no backtracking. On problems that require actually planning ahead, that shows. The paper opens with a striking number: in a block-stacking planning task, GPT-3 "achieves a success rate of only 1%, compared to 78% for humans" (Valmeekam et al., 2022) — and LLMs "also struggle with complex tasks that require multiple steps of math, logical, or commonsense reasoning."

The authors trace the deficiency to one missing ingredient: LLMs "lack an internal world model to predict the world state ... and simulate long-term outcomes of actions." Humans planning a course of action explore alternatives, roll forward to anticipate consequences, and revise earlier steps when a path dead-ends. A single autoregressive CoT pass can do none of that — once a step is generated, it is never reconsidered, and there is no explicit notion of "state" separate from the text already written. Concurrent work had started to bolt search onto CoT — the paper describes Yao et al.'s Tree of Thoughts as "Concurrently to our work" applying "heuristic-based search, like depth-/breadth-first search" — but the paper's diagnosis is that none of these methods formally instantiate a world model, a state, and a reward into one unified framework — they add search without adding simulation.

// how it works

The method

RAP (Reasoning via Planning) casts LLM reasoning as a Markov decision process. At each step t the LLM, acting as a reasoning agent, samples an action at ~ p(a | st, c) from a state st using an in-context prompt c. The same LLM, now repurposed as a world model with a different prompt c′, predicts the next state: p(st+1 | st, at, c′). Chaining this produces a trace of interleaved states and actions (s0, a0, s1, …, sT) — critically, states are explicit objects the model reasons over, not just the accumulated text of a CoT trace. What state and action actually mean is task-specific: in Blocksworld a state is the current block configuration and an action is a move instruction (PICKUP/PUTDOWN/STACK/UNSTACK); in GSM8K math reasoning a state is the set of known intermediate variable values and an action is a sub-question that derives a new one; in PrOntoQA logical inference a state is the fact currently in focus and an action is which logical rule to apply next.

Each reasoning step is scored by a reward rt = r(st, at), built from up to four signals the paper introduces: the log-probability of the action (the LLM's own "instinct"); the confidence of the predicted state (resample the world model's answer several times and take the frequency of the majority answer — used in GSM8K); a self-evaluation signal, where the LLM is asked "Is this reasoning step correct?" and the next-token probability of "Yes" is used directly as a reward; and task-specific heuristics, such as comparing the current block state against the goal in Blocksworld. Which combination to use per task is chosen, in the paper's own words, "based on heuristics and our exploratory experiments," rather than derived automatically.

Given the world model and reward, RAP plans with Monte Carlo Tree Search (MCTS): each node is a state, each edge an action, and a running Q(s,a) estimates the expected future reward of taking action a in state s. One MCTS iteration has four phases — selection (walk down the existing tree using the UCT rule below to balance exploring under-visited children against exploiting high-value ones), expansion (sample d candidate actions from the leaf and predict their resulting states), simulation (roll out a lightweight policy to a terminal state to estimate the value of the new branch), and back-propagation (push the terminal reward back up to update every Q(s,a) on the path). Selection uses UCT (Kocsis & Szepesvári, 2006):

a* = argmaxa∈A(s) [ Q(s,a) + w·√ln N(s)N(c(s,a)) ] // N(s): visit count of s · c(s,a): child reached by taking a in s · w: exploration weight

After a fixed iteration budget (10 or 20 in the main experiments), RAP reads off the reasoning trace whose leaf was visited most often. For tasks needing only a final answer (math, logic) rather than a full trace (planning), the paper adds RAP-Aggregation: run several MCTS iterations, collect the multiple answers they produce, and pick the plurality answer — a search-time analog of self-consistency.

// what they found

The results

All main experiments use LLaMA-33B as the base LLM for both RAP and the baselines (chain-of-thought and least-to-most prompting, with self-consistency variants), so the comparison is same-model, prompting-strategy-only. Plan generation is evaluated on Blocksworld (Valmeekam et al.), split by minimum solution length into 30 two-step, 57 four-step, and 114 six-step cases (201 total):

CoT with LLaMA-33B "completely fails on harder problems," while RAP(20) reaches a case-weighted average success rate of ~64% across all 201 cases. The headline comparison — LLaMA-33B with RAP against GPT-4 with plain CoT — works out to a 33% relative improvement (RAP(20)'s 63.7% case-weighted average vs GPT-4-CoT's 48.0%), i.e. a much smaller open model, given search instead of a single forward pass, outscoring a far larger closed model given one shot. The qualitative case in Figure 4 shows why: CoT commits to stacking blocks toward the second sub-goal first, which blocks the first sub-goal from ever being satisfied, and there is no mechanism to undo it; RAP makes the identical wrong first move in some iterations but backtracks and explores the alternative branch that succeeds.

On GSM8K math word problems (4-shot for all methods), RAP without aggregation reaches 48.6% accuracy and RAP-Aggregation pushes it to 51.6%, against chain-of-thought's 29.4% (46.8% with self-consistency×10) and least-to-most's 25.5% (42.5% with self-consistency×10):

On PrOntoQA logical inference (500 sampled examples mixing 3/4/5-hop proofs), RAP reaches 94.2% answer accuracy and 78.8% full-proof accuracy, against CoT's 87.8%/64.8% and CoT+self-consistency's 89.8% answer accuracy (proof accuracy not reported for the SC variant) — a 14-point proof-accuracy gain and a 4.4-point answer-accuracy gain over the strongest CoT baseline. A follow-up run swaps in Llama-2 70B and the full 602-case Blocksworld set (vs. the 201-case subset above), split into "Easy" (demonstrations matched to the test case's plan length) and "Hard" (demonstrations drawn from the full pool regardless of length): RAP(10) reaches 65% success on Easy and 51% on Hard, against CoT's 8% and 5% — the gap holds up on a larger benchmark and a stronger, different-family base model.

Reward ablation

On Blocksworld, dropping all three rewards collapses success from 0.91 (all three combined) to 0.02, and using the action-likelihood reward alone — without the task-specific or self-evaluation signals — only reaches 0.46; on GSM8K, the state-confidence reward alone underperforms combining it with self-evaluation (0.490 vs 0.503 at RAP(10)+aggr). The paper reads this as evidence that "the combination of multiple rewards contributes to the performance," but it also means each task's best reward mix was found by trying combinations, not derived from the framework itself.

// does it hold up?

The eval read

Within the comparison the paper actually runs — same base LLM, same 4-shot prompting budget, prompting-strategy-only — the gains over CoT, self-consistency, and least-to-most are large, consistent across three structurally different tasks (block-stacking planning, arithmetic word problems, formal logical inference), and hold up when the backbone changes (Llama-2 70B) and the test set grows (602-case full Blocksworld). That is a real, well-replicated result. Two things undercut how far the headline claims can travel, though: an unquantified compute cost, and a "general framework" that leans on per-task hand-tuning.

The 33% relative improvement over GPT-4 with CoT is real arithmetic (RAP(20)'s case-weighted 63.7% vs. GPT-4-CoT's 48.0% across the 201 Blocksworld cases checks out against Table 1), but it compares a much smaller open model doing many LLM calls per problem against a much larger closed model doing one. RAP's own appendix reports the compute hardware ("4 × NVIDIA A5000 GPUs") but nowhere quantifies the actual multiplier — number of forward passes, tokens, wall-clock, or dollar cost — that MCTS adds over a single CoT pass, even though every iteration involves sampling d actions, predicting d resulting states, and rolling out a simulation to a terminal state. Tree of Thoughts, the paper's own cited concurrent search-based reasoning method, reports its inference overhead explicitly (dollar cost, token multiplier); RAP does not, which makes "RAP beats GPT-4" harder to read as a fair compute-matched claim rather than a compute-for-accuracy trade.

baselines

Controlled where it matters — CoT, least-to-most, and their self-consistency variants all use the same LLaMA-33B backbone and 4-shot prompt budget as RAP, so the within-model comparisons are clean. The GPT-4-CoT comparison, while arithmetically correct, mixes model scale with inference-time search budget and isn't compute-matched.

contamination

Not discussed. Blocksworld, GSM8K, and PrOntoQA are all pre-existing public benchmarks; the paper does not audit LLaMA-33B, Llama-2 70B, or GPT-4 pretraining data for overlap with any of them.

what's missed

No token/dollar/wall-clock cost for MCTS is ever reported. The reward function is a different hand-picked combination per task, chosen by "exploratory experiments" rather than a fixed recipe, so "a general framework" still requires per-domain reward engineering to reproduce. All three domains are small, clean, and symbolic (201 or 602 Blocksworld cases, 500 PrOntoQA examples) with deterministic, checkable transition rules — there is no open-ended or embodied evaluation, and the world model's own state-prediction faithfulness is never measured directly, only inferred from downstream task reward. No head-to-head against Tree of Thoughts or other concurrent search-based reasoning methods on a shared benchmark, despite both being framed as siblings.

would raise confidence

Report the actual inference multiplier (calls/tokens/$ /latency) against CoT and self-consistency at matched budgets; test on a larger or more open-ended planning environment than Blocksworld; measure world-model state-prediction accuracy against ground truth, not just final-task reward; and a direct comparison to ToT-style tree search on overlapping tasks.

// bottom line

The core idea — one frozen LLM standing in as both agent and world model, searched with MCTS instead of committed to a single CoT pass — is well-motivated and the within-model gains are large and reproduced across three different reasoning domains and two model families. But the framework's real inference cost is never quantified, its reward design needs per-task tuning, and every benchmark here is small and symbolic — so the paper's reasoning-is-planning thesis is demonstrated in a clean, low-noise sandbox, not yet in open-ended settings where a world model would be harder to fake.

// takeaways

What to remember

  1. One LLM, two roles. RAP repurposes the same frozen LLM as a reasoning agent (proposes actions) and a world model (predicts the resulting state), giving reasoning an explicit state to plan over instead of just an accumulating text trace.
  2. MCTS replaces committing to one chain. A four-phase Monte Carlo Tree Search (selection via UCT, expansion, simulation, back-propagation) explores and backtracks through the LLM-simulated tree; RAP-Aggregation majority-votes across iterations when only a final answer is needed.
  3. Gains are large but the compute bill is invisible. RAP(20) beats GPT-4-CoT by 33% (case-weighted) on Blocksworld and beats CoT/self-consistency on GSM8K (51.6% vs 46.8%) and PrOntoQA (78.8% vs 64.8% proof accuracy) — but the extra LLM calls MCTS requires are never quantified, and each task's reward mix was hand-picked, not derived.
