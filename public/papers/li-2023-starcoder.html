StarCoder: may the source be with you!

A 600-member open collaboration trains a 15.5B-parameter code LLM on a trillion tokens of permissively-licensed GitHub code — then ships the dataset, an opt-out governance tool, a trained PII-redaction pipeline, and code-attribution tooling alongside the weights. The most comprehensive Code-LLM evaluation of its day, wrapped in the most transparent data story.

// the gap

What problem is this solving?

By early 2023, code LLMs had become a product category — GitHub Copilot reported over a million professional users — but the strongest models were closed in two directions at once. Their weights were behind paid APIs (OpenAI's Codex, code-cushman-001), and the data they trained on was undisclosed. That opacity is not just an inconvenience for researchers: code models raise sharp copyright, privacy, and consent questions. Lawsuits had already been filed against Copilot; Italy had temporarily banned ChatGPT over GDPR concerns; and the people whose public repositories feed these models had no way to know whether their code — or the email addresses and API keys embedded in it — was in the training set.

The few open code models that existed didn't close the gap. PolyCoder (2.7B) and the authors' own SantaCoder (1.1B) released both weights and data but were small and trained on under 300 GB. Larger open models came with strings attached: CodeGen-16B-Mono withheld its Python data, CodeGeeX and GLM didn't release training sets, and LLaMA's license forbade commercial use. So the practical question StarCoder targets is concrete: can a fully open, community-governed project build a code model that is competitive with the closed frontier — and do it without quietly inheriting the data-governance problems that make the closed models legally and ethically fraught?

The paper's answer is a model and a release apparatus. The model is StarCoderBase (and a Python-specialised StarCoder). The apparatus — The Stack as a transparent, opt-out dataset; a PII-redaction pipeline; and attribution tools that let a user check whether a generation was copied from training — is treated as a first-class contribution, not an afterthought.

// how it works

The method

The data. StarCoderBase trains on 1 trillion tokens drawn from The Stack v1.2 — 6.4 TB of permissively-licensed source across 384 languages, of which the team selected 86 (every language with >500 MB, plus top-50 popularity languages and dialects). On top of code, the mix includes GitHub issues, Git commits, and Jupyter notebooks, so roughly 20% of tokens are natural language (≈7% issues, 10% Markdown, 4% HTML, 2% notebooks). Rather than re-balance languages, the team sampled proportional to volume — they found the high-resource languages (C, C++, C#, Java, JavaScript, Python, PHP) already sat in a similar 44–87 GB band after deduplication — with the only exceptions being data-heavy formats down-weighted to teach syntax, not memorise content (JSON/YAML capped to 1 GB, CSS to 3 GB). Cleaning combined heuristic filters (XML, alpha-ratio, HTML-boilerplate, long-line) with 18 community annotators inspecting 300 language extensions by hand, plus near-deduplication.

Governance and PII as part of the build

The Stack ships with "Am I in The Stack", a tool to check whether your repository is included, and an opt-out process (44 developers had opted out at processing time). To attack secrets leaking into weights, the team built a real PII pipeline: they paid 1,399 Toloka crowd-workers across 35 countries (at a PPP-adjusted $7.30/hr floor) to annotate 22,950 PII entities in 12,000 files across 31 languages (Figure 1). They then trained StarEncoder — a 125M BERT-style bidirectional encoder (MLM + next-sentence, ≈400B tokens) — and fine-tuned it as a named-entity recogniser for names, emails, keys, passwords, and IP addresses, boosting weak categories with pseudo-labelling on 18,000 extra files. Detected entities are masked with <NAME>, <EMAIL>, <KEY>, <PASSWORD> tokens before training.

The model

StarCoderBase is a 15.5B-parameter decoder-only Transformer with the SantaCoder recipe scaled up: 40 layers, hidden size 6144, 48 heads, a 49,152-token byte-level BPE vocabulary, and learned absolute positional embeddings. Three architectural choices matter. Multi-Query Attention (MQA) shares one key/value head across all query heads, slashing the inference-time KV cache so large-batch serving is fast. FlashAttention makes attention IO-aware, which let them stretch the context to an 8K-token window — long enough to condition on whole files. And Fill-in-the-Middle (FIM) transforms reorder a fraction of training documents into prefix · suffix · middle so the model can infill, not just complete left-to-right. Training ran 250k steps at a 4M-token batch (1T tokens total) on 512 A100-80GB GPUs over 64 nodes — about 320k GPU-hours and an estimated 16.7 tonnes of CO₂eq. StarCoder is then StarCoderBase fine-tuned for 2 epochs on a 35B-token Python subset.

Everything is evaluated with pass@k — the fraction of problems for which at least one of k sampled programs passes every hidden test. With n samples generated and c correct, the unbiased estimator (Chen et al., 2021) is:

pass@k = 𝔼problems [ 1 − C(n−c, k)C(n, k) ] // the headline metric is pass@1: solve it in a single attempt. Open models use n=200 samples; API models n=20.

To support the privacy/copyright story at inference time, the release also includes attribution tools: a Bloom-filter membership check (a 26 GB "Data Portrait", ≈3% of the corpus) for a fast first pass, backed by a full BM25 / Elasticsearch search index over the training files so a user can locate the source repository of a copied snippet — all wired into a VS Code demo, and shipped under a commercially-viable OpenRAIL-M license with embedded use restrictions.

// what they found

The results

On Python, StarCoder is the strongest open model on both HumanEval (33.6 pass@1) and MBPP (52.7), and it matches or beats OpenAI's code-cushman-001 (33.5 / 45.9) — the 12B model that powered the first Copilot — despite being open and trained only on permissive data. StarCoderBase (no Python fine-tuning) already reaches 30.4 / 49.0, edging out the similarly-sized, Python-tuned CodeGen-16B-Mono (29.3 / 35.3). It does not reach the much larger 175B code-davinci-002 (45.9 / 60.3) — the paper is candid that the open frontier still trailed the closed one in raw capability.

The more interesting result is that rankings reshuffle on realistic code. On DS-1000 — 1,000 real data-science problems across seven libraries — StarCoder scores 26.0 overall to StarCoderBase's 23.8 and code-cushman-001's 18.1, while CodeGen-16B-Mono collapses to 11.7 even though it was competitive on HumanEval/MBPP. The puzzle benchmarks and the practical benchmark disagree — a theme the critique returns to. StarCoder is also competitive across 19 languages via MultiPL-E (often >2× the open baselines), leads single-line fill-in-the-middle (Python 0.62 vs SantaCoder 0.44, InCoder 0.31), and demonstrably benefits from its 8K window: perplexity on held-out repos drops across all ten languages tested when the context grows from 2K to 8K tokens.

// does it hold up?

The eval read

This is one of the most honestly-evaluated code models of its era, and the headline claims are stated carefully enough to survive scrutiny: "best open multilingual code model" and "matches or outperforms code-cushman-001" are both true at the table level, and the authors take pains not to claim parity with the 175B closed frontier. The breadth is real — HumanEval, MBPP, DS-1000, ODEX, MultiPL-E over 19 languages, FIM, type prediction, a security benchmark, GSM8K, and HELM reasoning. But three things deserve a harder look.

First, the baseline set is uneven by construction. The fairest comparison — open, similar size — is CodeGen-16B-Mono, and StarCoder wins it cleanly. But several headline contrasts are softer than they read: LLaMA/PaLM/LaMDA aren't code-specialised (beating them is expected), code-cushman-001 was already deprecated by OpenAI at writing, and the obvious 2023 reference points a reader would want — ChatGPT/GPT-3.5 and GPT-4, and instruction-tuned code assistants generally — are absent from the Python headline table. StarCoder is a base completion model; its "technical assistant" behaviour is elicited by prompt-hacking, not instruction tuning, so the comparison is apples-to-oranges against the chat models people were actually using.

Second, contamination is a live risk the static benchmarks can't fully exclude. The Stack is GitHub-scraped; HumanEval is hand-authored (relatively safe), but MBPP and especially DS-1000 (built from StackOverflow) plausibly overlap the training corpus, and the team explicitly flags likely train/test overlap for the return-type and CodeXGLUE docstring evals. To their credit, they include a decontamination pass and prefer leakage-resistant designs where they can (the TypeScript type-prediction setup deliberately uses never-translated JavaScript). The tell is internal: that HumanEval rank does not predict DS-1000 rank is partly a representativeness gap and partly what contamination noise looks like. Later contamination-controlled benchmarks that cite StarCoder (e.g. LiveCodeBench) found static HumanEval numbers flatter models than held-out problems do.

Third, the headline averages hide severe variance. "StarCoder retains multilingual ability after Python fine-tuning" is true on average but masks catastrophic per-language regressions: StarCoder's MultiPL-E Ruby drops to 1.24 and Racket to 0.07 (vs StarCoderBase's 17.25 and 11.77) — Python specialisation quietly lobotomised whole languages. And the data-scaling story (Figure 2) shows low-resource languages (≲1 GB) stall or regress with more training; R's pass@1 falls between the 800B and 1000B checkpoints, with per-problem success varying almost randomly across checkpoints. The "safe release" contributions are genuine and field-leading, but bounded: PII detection F1 is still weak on keys (≈70%) and usernames were dropped from redaction entirely, so secrets can still survive into the weights; and the attribution tools index only the training subset, can't separate boilerplate from protected content, and the Bloom-filter check has false positives.

baselines

Strongest where it matters (open, same-size CodeGen-Mono — clean win; matches deprecated code-cushman-001). Softer elsewhere: non-code LLMs, and no ChatGPT/GPT-4 or instruction-tuned code assistants in the headline table. StarCoder is a base model, not a chat model.

contamination

Real and acknowledged: GitHub-scraped corpus vs MBPP/DS-1000/StackOverflow-derived tests; train/test overlap flagged for type-prediction & docstring evals. Mitigated by a decontamination pass and leakage-resistant TS type-prediction, but not eliminated.

what's missed

pass@1 on toy puzzles tracks poorly with realistic code (DS-1000 reshuffles the ranking). Per-language averages hide collapses (StarCoder Ruby 1.24, Racket 0.07) and a data-starvation cliff below ~1 GB. No instruction-following / multi-turn eval.

would raise confidence

Head-to-head with GPT-3.5/4 and instruction-tuned peers; a post-cutoff, contamination-free benchmark; an explicit decontamination accounting; and downstream evidence that the PII/attribution tooling actually prevents secret-leakage and copying in deployment.

// bottom line

The capability claims hold up as stated — careful, hedged, and broad — and StarCoder genuinely closed much of the open-vs-closed gap for code in 2023. Its more durable contribution is the one that's hardest to score on a leaderboard: proving that a fully open, opt-out-governed, PII-scrubbed, attribution-instrumented code model could be built at frontier-adjacent scale at all. Read the benchmark tables as evidence of a strong base model, not a finished assistant — and read the governance apparatus as the real template the field borrowed.

// takeaways

What to remember

  1. Open caught up to closed-ish. A 15.5B model on 1T tokens of permissively-licensed code (The Stack) matched OpenAI's code-cushman-001 and beat every open multilingual code model — without inheriting the closed models' opaque-data problem.
  2. The release is the contribution. Opt-out governance ("Am I in The Stack"), a crowd-annotated PII pipeline (StarEncoder + NER over 22,950 labelled entities), and Bloom-filter + BM25 attribution tools turned "responsible release" from a paragraph into shipped infrastructure.
  3. Architecture for serving and infilling: Multi-Query Attention for cheap large-batch inference, FlashAttention to reach an 8K window (which measurably lowers perplexity), and Fill-in-the-Middle so the model can complete code from both sides.
  4. Benchmarks that look alike aren't. HumanEval/MBPP rankings don't survive contact with realistic data-science code (DS-1000), and language-level averages hide both a sub-1 GB data cliff and Python-fine-tuning regressions that gut low-resource languages.
