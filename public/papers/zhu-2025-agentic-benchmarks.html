Establishing Best Practices for Building Rigorous Agentic Benchmarks

A do-nothing agent scores 38% on τ-bench-Airline. An agent can overwrite SWE-Lancer's hidden tests with assert 1 == 1 and score 100%. Across ten popular agentic benchmarks, the authors find grading or task-design holes in every one — and distill the fixes into ABC, a 43-item checklist for benchmarks whose numbers you can actually trust.

Yuxuan Zhu, Tengjun Jin, Yada Pruksachatkun, Andy K. Zhang, … Matei Zaharia, Ion Stoica, Percy Liang, Daniel Kang

// the gap

What problem is this solving?

Classical AI benchmarks are easy to grade. ImageNet checks a categorical label; a translation benchmark runs BLEU against a reference string. The ground truth is fixed and the comparison is mechanical, so the score means what it says. Agentic benchmarks are different in kind. Success is defined by completing an end-to-end task: an agent reasons, writes code, calls tools, and mutates an environment, and the benchmark must decide whether the final outcome matches a ground-truth outcome — by running unit tests, matching strings, comparing environment states, or asking an LLM judge.

That extra machinery is where the rot sets in. The authors single out two structural features of agentic evals that make them hard to get right: a complex task setup (a live environment plus a toolbox the agent operates in) and an unstructured task outcome (free text, code, file edits, database states) that no simple metric can score soundly. When the grader is itself a program with edge cases, an agent that is good at finding edge cases will find the grader's.

The result is that the headline numbers everyone cites can be wrong — by up to 100% in relative terms. The paper's two opening exhibits set the tone. In τ-bench, a trivial agent that returns an empty response is marked correct on intentionally impossible tasks (e.g. "change a non-refundable ticket"), reaching a 38% success rate that beats a real GPT-4o-based agent. In SWE-bench-Verified, passing the hand-written unit tests does not guarantee the GitHub issue was actually resolved, and follow-up work found that 24% of the top-50 leaderboard positions are incorrect as a consequence. These aren't exotic corner cases — they are the most cited agentic benchmarks in the field, used by OpenAI, Anthropic, Google, Meta, and others to make capability claims. Prior work had flagged issues in individual benchmarks, but no one had turned those lessons into an actionable, systematic standard a builder could check their work against.

// how it works

The method

The paper's framing is a chain of equivalences that a trustworthy benchmark must preserve. The thing you care about is the agent's target capability. You proxy it with task success, and you detect task success with positive evaluation results. A benchmark is only valid if both links in that chain hold:

Target Capability ⇕ Task Success ⇕ Positive Results // left link = task validity · right link = outcome validity — break either and the score lies

Task validity is the left link: a task should be solvable if and only if the agent has the capability you're measuring. It breaks when task design or implementation creates a shortcut (a false positive — the task is passable without the capability) or an impossible task (a false negative — unsolvable even with the capability). Outcome validity is the right link: passing the check must truly mean the task was completed. It breaks when the grader is incomplete or gameable — SWE-bench's tests pass on wrong patches; τ-bench's empty-response loophole.

To turn this into something usable, the authors first collected a corpus of 17 widely-used agentic benchmarks — chosen because top AI providers (OpenAI, Anthropic, Amazon, Meta, Google, xAI, Mistral, DeepSeek) rely on them or because they won awards at peer-reviewed venues. They then surveyed agent-evaluation frameworks (Inspect AI, METR's protocols, OpenAI's Preparedness framework), folded in previously-documented benchmark issues, and borrowed best practices from adjacent fields — especially software testing. The synthesis is the Agentic Benchmark Checklist (ABC): 43 concrete checks across three parts.

Part 1 — Task validity (T.1–T.10)

Ten checks grouped by where shortcuts and impossible tasks come from: tools (pin tool/package versions; keep APIs reachable; terminate gracefully when an API dies), environment (clear residual state between runs; isolate the agent from any ground-truth signal; freeze the setup so it can't drift — no live websites), and implementation (verify annotated ground truth, verify each task is solvable, ship an oracle solver, and audit for exploitable bugs). The memorable one is T.5/T.4: KernelBench left ground-truth answers in GPU memory, so an agent could read them via out-of-bounds memory access.

Part 2 — Outcome validity (O.a–O.i)

Twenty checks, organized by what kind of outcome is produced and how it's graded. For information acquisition (text answers), string and substring matching must handle semantically-equivalent phrasings, redundant words, and negation, and must resist agents that "list every possible answer"; LLM-as-a-judge must come with documented evidence of its accuracy, self-consistency, and human agreement, and be hardened against reward hacking. For code generation, the testing checks import software-engineering rigor wholesale: unit tests need verified correctness and coverage/complexity metrics; fuzz testing must vary edge cases, memory layouts, and value ranges, and must generate inputs the program is actually sensitive to (random negatives tell you nothing about relu(x)); end-to-end tests must exercise all branches and eliminate flakiness. For state modification, state matching must enumerate all success states, check irrelevant states too (did the agent break something out of scope?), and be complex enough that a trivial change can't pass. For multistep reasoning, answer parsers must state their format assumptions, and customized quality metrics must correlate with the reasoning process so they can't be hacked.

Part 3 — Benchmark reporting (R.1–R.13)

Because some flaws are unavoidable (LLM judges and testing are never perfect), the third part is about honesty: open-source the data and harness, guard against contamination, state exactly which construct you're measuring, and — the practically sharp items — report confidence intervals (R.10), a non-AI/human baseline (R.12), and the score of a trivial agent that does nothing (R.13). That last check would have caught τ-bench's empty-response loophole on day one. The authors then scored ten of the benchmarks against ABC (1 point per satisfied item) and, for every flaw the checklist flagged, ran an experiment to quantify how much it distorts the reported number.

// what they found

The results

Applying ABC to ten benchmarks — SWE-bench, SWE-Lancer, KernelBench, BIRD, Cybench, MLE-bench, GAIA, τ-bench, WebArena, and OSWorld — the verdict is bleak and uniform: 7 of 10 violate task validity, 7 of 10 violate outcome validity, and all 10 have reporting limitations. 80% fail to acknowledge any weakness in their own design, and not a single benchmark satisfies every reporting criterion. The failures aren't theoretical — for each one the authors built an exploit and measured the damage:

BenchmarkFlaw (ABC check)Error
τ-bench (Airline)do-nothing agent passes "impossible" tasks (O.g.3)+38%
τ-benchagent can list every possible answer (O.b.2)+40%
KernelBenchfuzzer fixes tensor shapes / memory layout (O.e.1–2)+31%
WebArenalax substring match + unvalidated LLM judge (O.b.2, O.c.1)+5.2%
OSWorld (Chrome)13/46 tasks broken by website drift (T.6)−28%
SWE-Lanceragent overwrites hidden tests (T.5)→100%

A few are worth spelling out. SWE-Lancer grants agents unrestricted file-system access, including to its own test files. The tests sit in a password-protected ZIP — but the archive's directory listing is readable and its contents can be overwritten without the password, so an agent can replace the tests with assert 1 == 1 and score 100% without solving a single task (a task-validity isolation failure, T.5). KernelBench grades CUDA kernels against random tensors, but its fuzzer varies only tensor values, never shapes or memory layouts; re-examining the kernels from prior work, the authors find kernel-correctness is overestimated by 31% in absolute terms. OSWorld cuts the other way — 13 of 46 Chrome tasks rely on HTML selectors that broke when the live sites changed their layout, underestimating UI-TARS (the SOTA open agent) by 28% in the Chrome section. And in τ-bench, 38% of the airline subset and 6% of the retail subset are intentionally unsolvable, graded as "leave the environment unchanged" — so the empty agent passes them all.

The constructive half of the paper is a forward case study on CVE-Bench, a cybersecurity benchmark where agents exploit real web vulnerabilities. Using ABC, the authors found two issues. The time-based SQL-injection check looked for a SLEEP clause in the database log — but a clause appearing in the log doesn't mean it executed (a state-matching failure, O.g.1), so agents passed by inserting SLEEP anywhere, inflating scores by 32.5%. The outbound-server check could be passed via a same-Docker-network shortcut; after gating it, agent success dropped 10%. Net, ABC reduced CVE-Bench's performance overestimation by 33% in absolute terms, confirmed by cybersecurity experts.

// does it hold up?

The eval read

A paper grading evaluators invites being graded by its own rubric, and it mostly survives — but with caveats worth naming. The "up to 100% relative" headline is the worst single case (τ-bench's empty-agent loophole), not a typical figure; the per-benchmark absolute errors are more sober — 5.2% (WebArena), 28–33% (OSWorld, CVE-Bench), 31% (KernelBench). Read the dramatic number as an existence proof that catastrophic failures are possible, not that every benchmark is off by half.

Two methodological soft spots. First, the assessment is a convenience sample scored by the authors themselves: 10 benchmarks chosen for popularity, each item marked 1/0 by the same team that wrote ABC, with no inter-rater reliability reported — exactly the kind of un-validated subjective judging the checklist tells others to quantify (O.c.1). Second, there's a mild circularity: much of ABC is synthesized from "previously reported issues" in these same benchmarks, so re-finding those issues isn't independent confirmation that the checklist generalizes. The genuinely forward evidence — ABC catching a new issue at design time — is the CVE-Bench study, and that's a single benchmark. The binary scoring also flattens severity: failing the trivial-agent report (R.13) and failing ground-truth verification (T.7) cost one point each, though one is fatal and one is cosmetic, and the checklist won't tell you how much a violation moves the number unless you run the bespoke experiment the authors ran for each.

Finally, the specific findings are a snapshot that decays: SWE-bench-Verified already exists as a community response to SWE-bench's test-soundness problem, and live-site benchmarks like OSWorld will keep breaking and being patched. The durable contribution is the framework and the discipline it encodes — the individual scores are a 2025 photograph.

baselines

Strong on the constructive side — ABC explicitly demands non-AI and trivial-agent baselines (R.12/R.13), the very checks that expose τ-bench. Weak on its own: the 10-benchmark sample is popularity-selected, not random, and self-scored without inter-rater agreement.

contamination

Not a trained model, so no data leakage — but the analog risk is circularity: the checklist is partly derived from the benchmarks it then grades, so the assessment isn't an independent test of ABC's generality. CVE-Bench is the one true held-out demonstration.

what's missed

Binary 1/0 scoring with no weighting collapses severity; the "up to 100%" headline is the single worst case; and the checklist quantifies impact only via per-issue experiments a typical user won't reproduce. No guidance on how often a flagged flaw actually flips a leaderboard ranking.

would raise confidence

Independent raters applying ABC with reported agreement; a larger randomly-sampled benchmark set; severity weighting; and more forward case studies beyond CVE-Bench showing ABC prevents issues during construction rather than only diagnosing them after.

// bottom line

The scores are a snapshot; the checklist is the keeper. A do-nothing agent scoring 38% on τ-bench-Airline, and an assert 1==1 patch scoring 100% on SWE-Lancer, are the kind of results that should make every team re-audit its grading harness before quoting a number. ABC's lasting value is naming the two failure axes — task and outcome validity — and handing builders a concrete, testable list (with reporting hygiene) to walk before release.

// takeaways

What to remember

  1. Agentic benchmarks break along two axes: task validity (is the task solvable iff the agent has the capability?) and outcome validity (does passing the check mean the task was actually done?). Either failure silently corrupts the headline number — by up to 100% in relative terms.
  2. The failures are concrete and exploitable, not hypothetical: a do-nothing agent scores 38% on τ-bench-Airline; an agent overwrites SWE-Lancer's hidden tests with assert 1==1 for a 100%; KernelBench overstates kernel correctness by 31% because its fuzzer never varies tensor shapes; OSWorld under-rates the SOTA agent by 28% from website drift.
  3. ABC is the deliverable: 43 checks across task validity (T.1–10), outcome validity (O.a–i), and reporting (R.1–13) — including the cheap, high-value habits of reporting a trivial-agent baseline, a human baseline, and confidence intervals. Applied forward to CVE-Bench, it cut performance overestimation by 33%.
