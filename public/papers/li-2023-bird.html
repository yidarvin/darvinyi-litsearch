Can LLM Already Serve as A Database Interface? A BIg Bench for Large-Scale Database Grounded Text-to-SQLs

BIRD swapped the clean toy schemas of Spider and WikiSQL for 33 GB of dirty, real-world databases — and found that even GPT-4 lands at 54.89% execution accuracy where human experts hit 92.96%.

Jinyang Li, Binyuan Hui, Ge Qu, Binhua Li, Jiaxi Yang, … Reynold Cheng, Yongbin Li

// the gap

What problem is this solving?

Text-to-SQL had a benchmark problem. The two datasets everyone reported on — Spider and WikiSQL — test parsing against tidy academic schemas with only a handful of example rows per table. They answer one question: can a model map a sentence onto a schema? But they strip away almost everything that makes querying a production database hard.

Three things go missing. First, scale and dirt: real tables hold hundreds of thousands of messy rows, with values stored in inconsistent, human-entered formats. Second, domain knowledge: an analyst knows that "eligible for a loan" maps to a specific column value, or how to compute a financial ratio — the schema alone never says so. Third, efficiency: a query can be perfectly correct and still take seconds on a large table, which on a benchmark of toy tables is invisible.

The cost of those omissions is a gap between leaderboard scores and deployment reality. By the numbers BIRD makes the gap concrete: its databases average 549K rows each, against Spider's ~2K and WikiSQL's 17. Its gold queries are correspondingly heavier — 30.9 tokens and 51.0 keywords per SQL on average, versus 18.5 / 36.0 for Spider (Figure 4). BIRD's premise: a benchmark that looks like the real world will tell a very different story about how good models actually are.

// how it works

The method

BIRD is 12,751 question–SQL pairs over 95 databases totaling 33.4 GB across 37 professional domains (finance, healthcare, sports, education, and more). The recommended split is 9,428 train / 1,534 dev / 1,789 hidden test. Eighty databases come from real analysis platforms (Kaggle, Relation.vit) and feed the train and dev sets; a further 15 freshly curated databases are reserved for a concealed test set to limit leakage. SQL queries are stratified by annotation difficulty into simple (30%) / moderate (60%) / challenging (10%).

Three design pillars define the benchmark (Figure 1):

  • Realistic, dirty database values. Models must reason over actual cell contents, not just schema names — e.g. averaging a salary stored as the string "US$57,500.00", which has to be cleaned before AVG can run.
  • External-knowledge grounding. Each question ships with hand-annotated evidence in four categories — Numeric Reasoning, Domain Knowledge, Synonym Knowledge, and Value Illustration. Without it, a model has no way to know that "eligible for loans" means account.type = 'OWNER'.
  • Efficiency. BIRD is the first text-to-SQL benchmark to score how fast a correct query runs, via the new Valid Efficiency Score (VES).

Annotation runs as an expert-plus-crowd pipeline (Figure 2). Experts first build database description files (column and value descriptions) and question blueprints. A vetted question crowd (passing an 8-of-10 enrollment exam) writes the natural-language questions; a separate SQL crowd (9 of 10) writes the queries. Quality control uses double-blind annotation: two independent annotators write SQL for the same question without discussion, and a pair is only accepted when the two queries return matching execution results, iterating until they agree.

The efficiency metric ties correctness and speed together. VES only credits a query's speed when its result set is correct:

VES = Σn=1N 𝉽(Vn, V̂n) · R(Yn, Ŷn)N,   R(Yn, Ŷn) = √E(Yn) / E(Ŷn) // indicator 𝉽(·) = 1 only when the predicted SQL's results match gold; R is the square-rooted ratio of ground-truth to predicted execution time E(·). Efficiency is scored only on correct queries.
// what they found

The results

The headline is a chasm. On the hidden test set, the strongest model — GPT-4 — reaches 54.89% execution accuracy (with the oracle external knowledge), against human experts at 92.96%. Bolting on the DIN-SQL prompting pipeline nudges GPT-4 to 55.90%. Fine-tuned T5 models trail far behind (T5-3B around 24%). The same gap appears on efficiency: GPT-4 scores 60.77 VES on the test set versus human 90.27.

The single most telling result is the value of the hand-annotated evidence. Strip the external knowledge away and GPT-4's test execution accuracy falls from 54.89% to 34.88% — a +20.01-point swing from knowledge alone. The lift is roughly +20 points for humans too (72.37% → 92.96%). That confirms BIRD's central thesis: for real databases, parsing was never the bottleneck — grounding is.

And the contrast with Spider is stark: the same baseline models that reach the 70s–80s in execution accuracy on Spider (DIN-SQL tops out at 82.8) fall to the 20s–50s on BIRD, quantifying how much harder real databases are.

// does it hold up?

The eval read

BIRD's best design choice — shipping gold evidence with every question — is also its biggest evaluation caveat. The famous 54.89% is an oracle-knowledge number: a human annotator hand-wrote exactly the domain fact each query needs. Take that away and GPT-4 sits at 34.88%. Real deployments have no oracle; they must retrieve or infer that knowledge themselves, so the honest answer to the title's question — "can an LLM serve as a database interface?" — lives closer to the no-knowledge column. The benchmark measures grounded parsing given the right hint, not end-to-end knowledge discovery.

The human ceiling deserves the same scrutiny. The 92.96% "human performance" comes from expert SQL annotators — essentially the same population that authored the gold queries, working with the database description files in hand. That is a best-case expert-with-documentation number, not a typical-analyst baseline, which likely inflates the human–model gap relative to a fair head-to-head.

VES is a genuinely useful idea but a fragile metric. It is wall-clock execution time — which the authors themselves flag carries "machine status-related uncertainty," then square-root to damp the variance. Numbers aren't portable across hardware, and because efficiency is only scored on already-correct queries, at today's accuracy VES mostly tracks execution accuracy rather than adding an independent axis.

Correctness is judged by execution-result match against a single gold SQL. Double-blind annotation guarantees that two annotators agree, not that they're right — two people can converge on the same misreading of an ambiguous question — and result-set matching can credit a wrong query that happens to return the same rows, or penalize a valid alternative interpretation.

baselines

GPT-4, Claude-2, ChatGPT, Codex, PaLM-2 plus fine-tuned T5 — all 2023-era; DIN-SQL is the only agentic pipeline tested. Strong for the moment of publication, but the field's best BIRD systems now far exceed these.

contamination

The 15-database hidden test set limits leakage by design — the right call. But nearly all published BIRD results (this paper included) report the public dev set, on the web since May 2023, so any model trained afterward may have seen it.

what's missed

The 54.89% headline assumes oracle hand-written knowledge worth ~+20 points; the realistic no-knowledge number is ~35%. The 92.96% human ceiling is expert-annotators-with-docs — an optimistic upper bound.

would raise confidence

End-to-end results where the model must retrieve its own evidence; a non-expert human baseline; hardware-normalized or query-plan-based efficiency rather than wall-clock; multiple accepted gold SQLs per question.

// bottom line

BIRD is the benchmark that dragged text-to-SQL out of clean toy schemas and into 33 GB of messy, knowledge-laden, real-world databases — and that contribution is durable. Just read the famous "GPT-4 only 54.89%" headline with two asterisks: it is with oracle knowledge, measured against an expert-with-documentation human ceiling. The realistic number was lower, and three years of agentic pipelines have since closed much of the gap — but the benchmark's framing of dirty data + knowledge + efficiency has aged far better than its leaderboard numbers.

// takeaways

What to remember

  1. BIRD reframed text-to-SQL as a real-world problem: 12,751 questions over 95 dirty, cross-domain databases (33.4 GB, 549K average rows) — orders of magnitude bigger and messier than Spider or WikiSQL.
  2. External knowledge is the bottleneck: hand-annotated evidence is worth roughly +20 execution-accuracy points to every model, showing that parsing was never the hard part — grounding is.
  3. It introduced efficiency to text-to-SQL (the VES metric) and a striking gap — GPT-4 54.89% vs human 92.96% on the hidden test set — though both numbers should be read as oracle-knowledge / expert-ceiling best cases.
