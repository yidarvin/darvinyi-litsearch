WebVoyager: Building an End-to-End Web Agent with Large Multimodal Models

The first web agent to drive live websites end-to-end from rendered screenshots: it overlays numbered boxes on every clickable element, lets GPT-4V look, reason, and act, and reaches 59.1% task success across 15 real sites — paired with a GPT-4V auto-judge that matches human verdicts 85.3% of the time.

He, Yao, Ma, Yu, Dai, Zhang, Lan, Yu · Zhejiang University · Tencent AI Lab · Westlake University

// the gap

What problem is this solving?

By early 2024, "LLM web agent" research carried two simplifications that kept it away from the real web. The first was modality. Agents were fed the page as text — raw HTML, a cleaned DOM, or an accessibility tree — and never saw what a person sees. But a webpage is designed to be looked at: layout, icons, spacing, and the visual hierarchy of a calendar or a flight picker carry meaning that a flattened HTML string buries under thousands of tokens. WebGPT fine-tuned GPT-3 inside a text-only browser; WebAgent pre-trained a T5 to extract HTML snippets. Both spent most of their engineering on taming verbose HTML rather than reading the rendered result.

The second was the environment. Agents were measured in simplified simulators (MiniWoB, WebShop) or on static snapshots with one annotated "golden" trajectory, as in Mind2Web. Offline, step-by-step matching against a single gold path penalizes every equally-valid alternative route and tells you nothing about whether the agent survives the live web — floating ads, pop-ups, latency, and content that changes between runs. So you could report a number on a frozen dataset and still have no idea whether the agent could book a real flight on a real site.

WebVoyager attacks exactly this pair. It builds an agent that perceives the rendered screenshot and runs end-to-end on the open web with no human in the loop — and, because open-ended online tasks have no golden path to match against, it also needs a new way to score success. Both halves — the multimodal agent and an automatic trajectory judge — are the contribution.

// how it works

The method

WebVoyager is a perception–action loop wrapped around a frozen large multimodal model — primarily GPT-4 Turbo with vision (gpt-4-vision-preview), with Claude 3 Opus and GPT-4o tested as alternates. At each step the agent observes a screenshot of the current viewport plus a little auxiliary text, writes a natural-language thought, then emits exactly one action, which a Selenium-driven browser executes; the resulting page becomes the next observation. The loop runs until the model emits Answer or hits a 15-step budget.

Making a screenshot clickable: Set-of-Mark

The pivotal trick is Set-of-Mark prompting. A rule-based JavaScript tool (GPT-4V-Act) walks the DOM, finds the interactive elements, and overlays a black-bordered box with a numeric label on each one. The model never has to output pixel coordinates — it just names a label ("Click [17]"), which is mapped back to the real element. Unlike the original Set-of-Mark, no object detector is needed: the labels come straight from the DOM, so the marking is cheap and exact. The authors found a single black color for borders and labels beat multi-color schemes. Each observation also carries the element's type and inner text (and any aria-label) as auxiliary text, which rescues dense or low-resolution regions where the screenshot alone is unreadable.

The interaction loop, formally

The model receives a context of past observations and actions and produces the next action, which is split into a thought and the executable action code; the environment returns the next observation. To keep the prompt affordable, context is clipped to the three most recent screenshots (older images drop out) while the full textual history of thoughts and actions is retained.

ct = (o1, a1, …, ot-1, at-1, ot, I)   →   at = M(ct) = (st, ât)   →   ot+1 = E(ot, at) // M = multimodal model, E = browser environment, I = instruction; s = thought, â = action code (ReAct-style: think, then act)

The action space mirrors a human's mouse and keyboard: Click, Input (select a box, clear it, type), Scroll, Wait (for loads), Back, Jump to Search Engine (a reset valve when the agent is stuck on a dead end), and Answer. Crucially, the sites are not hosted locally — unlike WebArena, the agent explores the real open web and must cope with ads, pop-ups, and live updates as part of the task. Login- and CAPTCHA-gated content is excluded; multiple tabs are disabled so all interaction stays in one tab.

The benchmark: 643 tasks on 15 real sites

The evaluation set spans 15 everyday sites — Allrecipes, Amazon, Apple, ArXiv, BBC News, Booking, Cambridge Dictionary, Coursera, ESPN, GitHub, Google Flights, Google Map, Google Search, Huggingface, and Wolfram Alpha — with 40+ tasks each, 643 total. Tasks are built by self-instruct: humans seed a few tasks per site, GPT-4 Turbo generates ~100 more from in-context examples, and humans verify and rewrite each, confirming the answer can actually be found online. Diversity is checked by embedding similarity (all-mpnet-base-v2): of 206,403 task pairs, only 49 exceed 0.8 cosine similarity and 99.68% fall below 0.6. Each task is annotated with a "Golden" answer (22.3% — exhaustive, stable lists) or a "Possible" answer (open-ended, multi-answer, or real-time like flight prices).

An LMM that grades the agent

Because most answers are open-ended, there is no exact-match key to score against. So the second contribution is an automatic evaluation protocol: feed GPT-4V the task, the agent's textual responses, and the last k screenshots of the trajectory, and ask for a binary "did it succeed?" verdict — an LMM emulating a human grader at scale.

// what they found

The results

With the GPT-4 Turbo (vision) backbone, WebVoyager reaches 59.1% overall task success under human evaluation — far ahead of the text-only accessibility-tree variant at 40.1% and of GPT-4 (All Tools), OpenAI's own integrated browsing/tools agent, at 30.8%. The discussion pins GPT-4 (All Tools)' weakness on its reliance on Bing-fetched pages: it can't directly drive sites like Apple, Amazon, or BBC to search, click, or sort.

The advantage is uneven, and the unevenness is the story. WebVoyager wins big on visually structured sites — Google Map 76.7%, Coursera 73.8%, Google Flights 70.7%, Apple 65.1%, Cambridge Dictionary 65.1% — where screenshots make calendars and pickers tractable and the text accessibility tree becomes hopelessly verbose. But it is flat or behind on text-heavy sites: 53.3% on Allrecipes (vs text-only's 55.6%), and roughly tied on GitHub, ESPN, and Wolfram Alpha — precisely where small-font OCR from a screenshot fails and reading the HTML wins. On the SeeAct online test set WebVoyager scores 30% vs SeeAct's best autonomous agent at 26%.

On 90 web-browsing tasks drawn from GAIA (Level 1 and 2, starting from Google Search), WebVoyager again leads both baselines: 38.5% at Level 1 (vs 23.1% / 19.2%) and 15.6% at Level 2 (vs 12.5% / 12.5%) — the gap shrinks on the harder, longer L2 tasks.

The auto-judge tracks humans

On a 300-task subset with three annotators per trajectory, GPT-4V's verdicts agree with the consolidated human labels 85.3% of the time at κ = 0.70 when given the full trajectory — matching the humans' own inter-annotator Fleiss κ of 0.7. Agreement rises monotonically with more screenshots, and a GPT-4o judge edges slightly higher (κ 0.72) while Claude-3-Opus is lower (κ 0.6).

Where it fails, and why

Hand-labeling 300 failures gives a clear hierarchy. The dominant problem is Navigation Stuck (44.4%) — the agent burns its 15 steps without finishing: vague search queries drown it in irrelevant results, it can't locate a small scroll area, or it oscillates between scrolling up and down (and tends to repeat mistakes because the screenshot context was clipped). Next is Visual Grounding (24.8%) — misreading small text, pronunciation marks, or math; treating two near-identical pages as a failed action; or clicking an adjacent element (e.g. confusing calendar digits with Set-of-Mark labels). Then Hallucination (21.8%) — plausible-but-wrong answers, such as naming a cheap product without first sorting by price. Finally Prompt Misalignment (9.0%) — unparseable output or a premature Answer.

// does it hold up?

The eval read

The headline gain is real and the auto-judge is a genuinely useful contribution — both became standard practice. But the rigor caveats here are structural, baked into the design choices that make the paper interesting, and the authors are admirably candid about several.

1. The benchmark is a moving target. By design WebVoyager runs on the live web, and 77.7% of tasks carry only "Possible" answers (open-ended or real-time, e.g. ticket prices). The identical task can succeed today and fail tomorrow as listings, prices, and layouts shift — so 59.1% is a snapshot, not a reproducible number, and re-running the same agent later is not directly comparable to the paper's figure. This is the realism/reproducibility trade-off taken to the realism extreme; later work pushes back by archiving pages or freezing environments.

2. Judge and subject share a brain. GPT-4V is simultaneously the strongest agent backbone and the evaluator. The paper itself shows evaluators favor their own family — Table 3: GPT-4o grades leniently, GPT-4V strictly, and Claude-3-Opus rates its own runs best — so scoring a GPT-4V agent with a GPT-4V judge is not an independent measurement. The 85.3% agreement is reassuring but still leaves ~1-in-7 disagreements, and the "ground truth" it is measured against is itself only κ 0.70 among humans. The judge was validated on 300 tasks, then applied at scale.

3. The task distribution is the easy slice of the web. Login and CAPTCHA sites are excluded, multi-tab is disabled, and there are no transactions, form submissions, or authenticated state — i.e. exactly the consequential, state-changing actions a real assistant needs. Success here means "can it find and read things on public pages," not "can it run my errands."

4. Tasks are GPT-written, then GPT-graded. Self-instruct seeds the set with GPT-4 and GPT-4V grades it. Even with human verification, the task style can align with model strengths, and difficulty is never calibrated against an external standard. Diversity is measured (embedding similarity), but diversity is not difficulty.

5. Closed-model dependence. The agent is a prompt scaffold around frontier proprietary LMMs. The authors note open models can't even ingest the inputs — LLaVA downsizes images to 224/336px (small fonts vanish) and caps context at 4k, while a 15-step trajectory needs ~7k tokens — so there is no open, reproducible path, and every number is hostage to silent API/model updates.

6. Some failures are the harness, not the model. Navigation Stuck — 44% of all errors — is partly the 15-step cap and the 3-screenshot context clip biting: the agent "repeat[s] its previous mistakes due to the input clipping," by the authors' own account. So the score conflates model capability with scaffold design choices.

baselines

Fair and current for early 2024: GPT-4 (All Tools), a same-backbone text-only ablation, plus SeeAct and GAIA cross-checks, and three backbones (GPT-4V, Claude-3-Opus, GPT-4o). All closed models, though — no open-LMM point of comparison.

contamination

Not a leakage benchmark per se, but the live web means content drifts run-to-run, and tasks are GPT-generated and GPT-graded — the risk is entanglement and style-fit, not memorized answers.

what's missed

No login / transaction / multi-tab tasks; the score is a non-reproducible live-web snapshot; the GPT-4V judge isn't independent of the GPT-4V agent; cost and latency go unreported.

would raise confidence

Frozen or archived site snapshots for replayability; a judge from a different model family (or human spot-audit at scale); authenticated, state-changing tasks; an open-model baseline; per-task tokens / $ / steps.

// bottom line

WebVoyager proved two things that stuck: a multimodal agent that looks at the rendered page (Set-of-Mark + screenshots) beats one that reads its HTML, and an LMM can grade open-ended web trajectories about as well as humans agree with each other. Both became templates for the field. But the 59.1% is best read as a 2024 live-web snapshot rather than a leaderboard number — the live environment that makes it realistic also makes it unrepeatable, and the GPT-4V-judges-GPT-4V loop means the headline is part self-assessment, part measurement.

// takeaways

What to remember

  1. See the page, don't parse it. Overlaying numbered boxes on interactive elements (Set-of-Mark) turns a screenshot into a discrete, clickable action space — the model says "Click [17]" instead of emitting coordinates or wading through a DOM. This screenshot-first recipe became the template for the GUI / computer-use agent wave that followed (OSWorld and successors cite it directly).
  2. LMM-as-judge for open-ended agents works — within limits. GPT-4V scoring full trajectories reaches 85.3% agreement / κ 0.70 with humans, enough to scale evaluation past hand-grading. The catch is independence: if the judge shares a model family with the agent, the score drifts toward self-assessment.
  3. Live web = realistic but unrepeatable. Running on real sites — ads, pop-ups, changing prices — is the paper's headline strength and its deepest measurement weakness; 59.1% is a moving target. And the dominant failure (Navigation Stuck, 44%) is a long-horizon control problem more than a perception one.
