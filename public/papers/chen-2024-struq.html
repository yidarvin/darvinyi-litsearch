StruQ: Defending Against Prompt Injection with Structured Queries

Prompt injection is the classic injection vulnerability — mixing control and data in one channel — replayed for LLMs. StruQ's fix is the classic fix: separate them. A filtering front-end plus a purpose-built fine-tuning recipe drive every manual attack the paper tests to under 2% success, with almost no utility cost.

// the gap

What problem is this solving?

Every LLM-integrated application concatenates a developer-written prompt with untrusted user data into one string and hands it to the model. That single-string API is, in the paper's framing, "unsafe-by-design" — and it is the exact same design mistake behind fifty years of injection vulnerabilities in classical software security. SQL injection happens because a database API accepts one string mixing the query template (control) with user-supplied values (data); the fix that actually worked was prepared statements, which pass control and data to the database as two separate arguments instead of one interpolated string. Command injection, cross-site scripting, and even a phone-phreaking exploit from the 1970s (a 2600 Hz tone smuggled over the same voice channel used for call-control signaling) are all the same pattern. LLMs never got the equivalent fix — there was no API that let a developer mark which part of the input is instructions and which is data, and have the model actually honor that distinction.

Prior defenses patched around this gap rather than closing it: adding a warning sentence to the prompt, wrapping the data in delimiters, or fine-tuning on injected examples drawn from a different distribution than the real instructions (as in the concurrent BIPIA scheme). None of them changed the fundamental API. StruQ's premise is that the fix has to be structural: build an actual two-channel interface — a structured query — and train the model to only ever treat the prompt channel as a source of instructions.

// how it works

The method

StruQ has two components that have to work together: a secure front-end that encodes a structured query, and a model trained to respect that encoding.

The secure front-end. StruQ adapts the Alpaca prompt format but replaces its plain-text delimiters ("### instruction:", "### input:", "### response:") with reserved special tokens — [MARK], [INST], [INPT], [RESP], [COLN] — that a user's data cannot contain, because the front-end recursively filters those exact strings (and a bare ##) out of the data before assembly. This one design choice is what stops Completion attacks, where an attacker fakes a "### response:" block to trick the model into thinking the target task is already finished and it's now reading a new instruction: if the real delimiter can't appear in the data at all, that class of attack cannot be constructed.

Structured instruction tuning. A filtered delimiter is not enough on its own — the model itself has to learn to ignore instructions wherever they land outside the prompt channel. StruQ fine-tunes a base (non-instruction-tuned) Llama-7B or Mistral-7B on an augmented instruction-tuning set: 50% of examples are left clean, 25% get a second, unrelated instruction concatenated directly into the data portion (a Naive-attack-shaped augmentation), and 25% get a fake response plus a second instruction appended after fake delimiters (a Completion-attack-shaped augmentation). In every augmented example, the target label is still the response to the correctly-positioned prompt-channel instruction — the model is explicitly taught to answer the real instruction and never the one smuggled into the data:

dj′ = dj ∥ dresp ∥ r′ ∥ dinst ∥ pi ∥ di // Completion-Other training augmentation — sample j's clean data dⱼ, then a fake response delimiter + fake response r′ + fake instruction delimiter + an unrelated sample's instruction/data (pᵢ, dᵢ), all appended into the data channel; the training label stays rⱼ, the response to prompt pⱼ

The special tokens are seeded with sensible initial embeddings ([MARK] starts from the embedding of "###", [INST] from "instruction", and so on) because the paper found training a brand-new token embedding from scratch under standard instruction tuning doesn't converge to something useful — initialization from the nearest existing token turned out to matter a lot for utility.

// what they found

The results

Against the paper's suite of manual, hand-crafted attacks, StruQ is close to airtight. On Llama, undefended attack success rates of 6% (Naive), 12% (Ignore), 29–41% (Completion-Other and its combo variant), and a striking 96% (Completion-Real, the attack that spoofs the model's own exact delimiters) all fall to 0% under StruQ. Mistral shows the same pattern — 96% and 71–83% undefended success rates on the strongest Completion attacks collapse to 0% (2% for one combo variant). Nine "near-miss" adaptive delimiter variants — different capitalization, missing colons, single-character typos, semantically similar words — that remain highly effective against an undefended model (61–93% success) are all reduced to ≤1% by StruQ, because the correct delimiters map to reserved tokens with very different embeddings than any near-miss text, and the model has explicitly learned to only trust those reserved tokens.

Utility barely moves: AlpacaEval win rate goes from 67.2% to 67.6% on Llama (StruQ is, if anything, marginally higher — noise) and from 80.0% to 78.7% on Mistral, a roughly one-point drop the authors note is borderline statistically significant against AlpacaEval's 0.7% standard error. Against the concurrent BIPIA defense — the closest prior work, which also uses special delimiters and injection-augmented fine-tuning — StruQ wins on both axes at once: 0% attack success on both BIPIA's own test set and StruQ's harder one, versus BIPIA's 54% Ignore-attack success rate on StruQ's test set, while BIPIA's AlpacaEval win rate drops from 53.9% to 26.0% (StruQ: 67.2% → 67.7%, no comparable utility hit).

The one place StruQ clearly does not hold is optimization-based, white-box attacks. TAP (a jailbreak search method the authors adapt for prompt injection) drops from 97% success to 9% on Llama but only to 36% on Mistral; GCG — a gradient-guided suffix search — drops from 97% to a still-substantial 58% on Llama and 99% to 56% on Mistral. The authors are explicit that StruQ was trained only on task-agnostic, manually-crafted injections and "does not fully generalize to task-specific injections that are carefully optimized with much more resources."

// does it hold up?

The eval read

StruQ's own headline claim survives scrutiny only because the authors state its boundary clearly: the paper is honest that "StruQ is not yet fully secure," and the two numbers that matter most for judging real-world deployment risk — 58% and 56% residual GCG success on Llama and Mistral — sit right there in the paper's own Table 2, not buried. That transparency is a real strength. The evaluation rigor around what StruQ tests against 15+ manual/near-miss attack variants is also genuinely thorough, including a dedicated adaptive-attack section (Section 5.2) where the authors specifically try to break their own filter rather than only reporting attacks designed before the defense existed.

Where it's thinner: utility is measured with a single instrument, AlpacaEval 1.0 against a single reference model (text-davinci-003) using GPT-4 as judge — a metric known to be sensitive to response length and stylistic preferences of the judge model, and the paper reports only the aggregate win rate with no breakdown by task type or qualitative failure analysis of what utility loss actually looks like. Security is measured almost entirely with a single injected string, "Print exactly Hacked!" (plus translated/base64 variants), scored by checking whether the output contains "Hacked!"/"hacked!" and then manually inspecting each hit to exclude false positives — a clean, unambiguous canary, but not necessarily representative of subtler injected instructions (a fake tool call, a change to a numeric answer, an instruction with no fixed lexical signature) that a substring-plus-manual-check protocol like this one isn't built to catch. GCG and TAP are each evaluated on a limited attack budget (500 GCG iterations, TAP's standard search depth) against only two 7B-parameter open models; whether residual success climbs further with more optimization budget, larger models, or black-box adaptation to a deployed StruQ system is untested here.

baselines

Strong and fair — StruQ is the only paper in this comparison set to reproduce a concurrent defense (BIPIA) end-to-end on both test sets rather than citing its reported numbers, and it credits BIPIA's shared ideas explicitly.

contamination

Not classic leakage, but AlpacaFarm/Alpaca instruction data is a widely-used, well-known open dataset; utility numbers reflect performance on a familiar distribution, and the "208 AlpacaFarm samples" security test set is small enough that individual example idiosyncrasies could move percentages by a few points.

what's missed

Security is measured almost entirely via one lexical canary string; no larger-budget or black-box adaptive attacks against the deployed defense; utility measured on one benchmark (AlpacaEval 1.0) with one judge model; no evaluation beyond 7B-scale open models.

would raise confidence

Test against injected instructions with no fixed string signature (e.g. numeric or tool-call manipulation scored semantically, not by substring match); scale GCG/TAP attack budgets to see where residual success saturates; add a second utility benchmark and model scale to rule out AlpacaEval- or size-specific effects.

// bottom line

StruQ earns its result: near-total elimination of every manual and near-miss attack it tests, at essentially no utility cost, by genuinely closing the spoofable-delimiter hole rather than papering over it. But its own numbers are also the clearest evidence in the prompt-injection literature that structural, non-adaptive defenses and optimization-based attackers are not yet playing the same game — a 56–58% residual GCG success rate is a long way from secure, and the paper's honesty about that gap is exactly what later work (SecAlign, from the same authors, and Meta SecAlign) set out to close.

// takeaways

What to remember

  1. Prompt injection is the LLM-era instance of a 50-year-old injection pattern — mixing control and data in one channel — and StruQ applies the classical fix: a structured query that separates them into two channels, enforced by a front-end filter plus a model trained to only trust the prompt channel.
  2. The combination works extremely well against hand-crafted attacks: Completion-Real, the strongest manual attack, falls from 96% to 0% success under StruQ on both Llama and Mistral, and all nine near-miss adaptive-delimiter variants (61–93% undefended) drop to ≤1% — at roughly a one-point AlpacaEval utility cost.
  3. It does not close the gap against optimization-based, white-box attackers — GCG still succeeds 56–58% of the time even after structured instruction tuning — which the paper states plainly and which motivated its own follow-up work (SecAlign).
