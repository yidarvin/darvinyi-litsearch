Agentic Rubrics as Contextual Verifiers for SWE Agents

Verifying a coding agent's patch usually means spinning up a sandbox and running tests. This paper asks an expert agent to crawl the repository first and write a context-grounded rubric checklist — then grades candidate patches against it with no code execution at all, beating patch classifiers and test-based selection on SWE-Bench Verified.

Mohit Raghavendra*, Anisha Gunjal*, Bing Liu, Yunzhong He · Scale AI

// the gap

What problem is this solving?

For software-engineering agents, verification is the bottleneck. It plays a dual role: it is the reward signal that supervises post-training with verifiable rewards, and it is the selector that turns extra inference compute into accuracy through test-time scaling — sample many candidate patches, then pick the best one. Verifier's Law makes the dependence explicit: how easily you can train an agent on a task is bounded by how efficiently and reliably you can verify a candidate solution.

Today's verifiers split into two families, and both have a flaw. Execution-based methods run the repository's unit tests (human-written or LLM-generated). They are environment-aware and grounded in real behavior, but every instance needs its own sandbox setup, the signal is sparse and brittle (limited distinguishability between near-correct candidates, "test toxicity"), and standing up an executable environment for a long-tail repo is exactly the expensive part. Execution-free methods — learned patch classifiers, diff-similarity, LLM judges — are operationally lightweight but ungrounded: less reliable, harder to interpret, and prone to latching onto surface cues (stylistic patterns, non-semantic artifacts) instead of functional correctness.

As agents move toward more open-ended, goal-driven tasks and unfamiliar repositories, a verifier needs to be both scalable and codebase-specific — and neither family is. Rubric-based scoring is a natural middle ground, but a rubric written from the issue text alone is under-specified: it doesn't know the repo's interfaces, call sites, or conventions, so its criteria are vague and inconsistently gradable. The gap this paper targets is making a rubric contextual — grounded in the actual codebase — without paying the per-instance cost of execution.

// how it works

The method

Agentic Rubrics splits verification into two phases. In the rubric-generation phase, an expert rubric agent — built on the SWE-Agent scaffold, so it has repository navigation, file inspection/editing, and shell tools — explores a sandboxed copy of the repo alongside the PR description, then submits a structured rubrics.yaml. This mirrors how a developer validates a fix when tests are missing: read the surrounding code and contracts, trace call sites, reason about edge cases. In the inference phase, a candidate patch is scored against that rubric with no code execution.

Each rubric item is a pair (ti, wi) — a short natural-language criterion and an importance weight wi ∈ {1, 2, 3} (nice-to-have / important / must-have) — and is filed under one of four axes:

  • File Change (4–8 items): edits are minimal, local, and sufficient for the fix — no over-scoping.
  • Spec Alignment (3–6): the patch satisfies the requirements stated in the issue.
  • Integrity (3–6): "no-cheating" and hygiene constraints — no test weakening, broad refactors, mass renames, or dependency churn.
  • Runtime (3–6): the change implies the intended runtime behavior and avoids obvious execution-time issues.

At grading time an LLM judge assigns each item a binary score si ∈ {0, 1}, and the verifier score is the weight-normalized sum — a graded value in [0, 1] used to rerank candidates:

S = Σi wi · siΣi wi // weighted rubric score, S ∈ [0,1]; a failed-to-parse rubric or missing artifact scores 0

One design choice does a lot of quiet work: grading sees only the problem statement, the rubric, and the final submitted patch — never the agent's rollout trajectory or tool traces. Prior work (R2E-Gym) found verifier decisions are unduly swayed by an agent's thinking trace, so withholding it keeps scoring uniform across methods and curbs verifier hacking.

The test-time-scaling protocol

Everything is evaluated as BEST@K selection on the 500 problems of SWE-Bench Verified. For each problem a fixed generator samples K = 16 independent rollouts at temperature 1.0 — the paper uses two generators, Qwen3-32B (max 30 turns) and Qwen3-Coder-30B-A3B (max 50 turns). The verifier scores all 16 candidates; the highest-scoring patch is submitted and counted as resolved only if it passes the ground-truth Fail-to-Pass and Pass-to-Pass tests. Two reference lines bracket the result: Oracle Pass@K (select using the hidden GT tests — an upper bound) and Random@K (uniform pick). All artifact-based verifiers use Claude Sonnet-4.5 (30-turn budget) as the expert agent and GPT-5 (low reasoning) as the judge; the repo is reset to a pre-PR snapshot, and agents cannot touch the hidden tests, reference patch, or git history.

// what they found

The results

Across both generators, Agentic Rubrics is the top verifier at K=16. On Qwen3-32B rollouts it reaches 40.6% BEST@16 — +3.5 points over the best non-agentic baseline (Patch Classifier, 37.1%) and +4.6 over the best artifact-based alternative (Agentic Patch Similarity, 35.0%). On the stronger Qwen3-Coder-30B-A3B rollouts it reaches 54.2% — +4.0 over the best non-agentic baseline (50.2%) and +4.6 over the best agentic one (49.6%). The scaling curve shows the lead holds as K grows, so this isn't a single lucky operating point. For reference the Oracle ceilings are 51.4% and 65.6%, and Random is 22.6% / 39.6%.

Rubric scores track ground truth — and add detail

Beyond aggregate selection, the scores are well-calibrated against the test suite. Split Qwen3-32B rollouts by whether they pass the GT tests and the rubric distributions separate cleanly: passing patches concentrate at 0.85–1.0, failing ones spread lower (often ~0.4–0.5), giving an ROC-AUC of 0.886 and PR-AUC of 0.722 for predicting test pass/fail. Critically, the signal is denser than a binary pass/fail — it assigns partial credit, so it can rank two failing candidates by how close each is. Broken out by axis, failing patches lose points on File Change (over-scoped edits), Spec Alignment (missed requirements), and Runtime, while staying strong on Integrity. Scored against the human reference patches, those GT fixes earn a mean rubric score > 0.8 across axes — the one recurring exception being File Change, where rubrics are sometimes more prescriptive about exact edit location than the real implementation.

When rubrics disagree with tests, half the time they're right

Because the rubrics are generated synthetically, the authors audit their utility: on 100 instances, GPT-5 (medium reasoning) tags each judgment high- or low-utility using the GT tests and patch as the reference. When rubric and test outcomes agree, 78% of rubric judgments are high-utility — dominated by Core Semantics (45.0%), API/Compatibility (12.8%), Structure/Scope (10.1%), and Edge Coverage (8.2%). The more interesting case is when a patch passes the tests but the rubric rejects it (score < 0.7): even there, 54% of the rubric's complaints are high-utility, most often flagging a missed root cause (17.5%) or missing edge cases (15.1%) — real issues the available tests simply don't cover. The remaining 46% are genuine low-utility noise (over-specification, redundancy, rubric–test mismatches), which the authors point to human-in-the-loop refinement to clean up.

Ablations: the agent, the grounding, the judge

The rubric-agent's capability shows up directly in the score. Holding the generator and judge fixed and swapping who writes the rubric, frontier coding models — Claude Opus-4.5, Claude Sonnet-4.5, Gemini-3-Pro — top out around 54% BEST@16; open-weight coders (Qwen3-Coder-30B-A3B, Code World Model) land near 45%, and a non-coding agent (Qwen3-32B) around 43%. The mechanism is granularity: stronger models write more criteria per instance (Sonnet-4.5 averages >20, roughly double Qwen3-32B and CWM), giving finer-grained discrimination between candidates.

The capability distills. Fine-tuning Qwen3-32B as an agentic rubric generator (2,000 Sonnet-4.5 rubric-generation trajectories) substantially outperforms fine-tuning the same model as a patch classifier (the R2E-Gym recipe, 4,696 YES/NO-labeled examples) — and both base models — showing structured rubric generation is a trainable, stronger objective than binary classification for execution-free verification.

Two more knobs. Repository grounding matters: generate the same rubrics from the issue text with no codebase access and BEST@16 drops 4.0 points on Qwen3-32B rollouts (1.4 on Qwen3-Coder) — agentic rubrics name concrete files, classes, and methods via find/grep/view, where non-agentic ones stay vague ("touches the right code path"). Judge capability matters only a little: scoring fixed Sonnet-4.5 rubrics, GPT-5-mini gets 52.6, and Low / Medium / High reasoning give 54.2 / 54.3 / 55.0 — because the rubrics are designed to be self-contained and atomic, grading doesn't need a heavy reasoner.

// does it hold up?

The eval read

The core claim — that an agent-written, repo-grounded rubric is a better execution-free selector than a patch classifier, and that the skill distills into a 32B model — is supported by a clean, well-instrumented study. But the evidence sits on a narrow base, and the framing oversells in two places.

The benchmark is the opposite of the motivation. The paper argues rubrics are needed for "long-tail repositories" and "open-ended, goal-driven tasks," yet every number is on SWE-Bench Verified — 500 instances from a dozen popular, heavily-maintained Python libraries that are prime pretraining material and the easiest possible codebases for a model to "understand." An execution-free verifier grounded in codebase context is most flattered exactly where the model already half-knows the code; the hard case it's pitched for (an obscure repo the model has never seen) is never tested.

The whole loop is one frontier model judging another. Sonnet-4.5 writes the rubric, GPT-5 grades it, and the most striking result — that 54% of cases where the rubric rejects a test-passing patch are catching real under-tested bugs — is itself adjudicated by GPT-5, with the GT tests/patch as reference. No human ever confirms a single "missed root cause" or "missing edge" label. So "rubrics catch what tests miss" currently rests on one LLM's opinion about another LLM's rubric.

The comparison set omits the actual SOTA. The baselines are non-agentic (self-consistency, patch classifier) and single-signal agentic (tests, patch-similarity, rubric). But R2E-Gym — which this paper cites and borrows its classifier recipe from — reports that hybrid execution + execution-free verification reaches ~51% where either signal alone saturates ~42–43%. Agentic Rubrics is purely execution-free and never combined with execution, so "+3.5 over the strongest baseline in our comparison set" is a comparison the authors scoped; the most competitive published idea isn't on the table.

baselines

Reasonable execution-free and single-signal agentic verifiers — but the comparison excludes the hybrid execution + execution-free verifier (R2E-Gym, ~51%) that is the real SOTA, and only mid-capability Qwen3 generators are reranked, never frontier rollouts where patches are harder to tell apart.

contamination

SWE-Bench Verified's dozen popular Python repos are heavily represented in pretraining; a "codebase-grounded" verifier is most advantaged precisely where the model already knows the repo. No long-tail / held-out-repo split, despite that being the paper's stated motivation.

what's missed

Cost. Each rubric is a 30-turn Sonnet-4.5 agentic rollout, and grading is GPT-5 over ~K×20 (candidate × criterion) calls per problem. "Execution-free" trades sandbox setup for hundreds of frontier-API calls and is never cost-accounted against simply running the tests.

would raise confidence

A long-tail / unseen-repo split; human verification of the high-utility-disagreement labels; a hybrid rubric+execution verifier compared to R2E-Gym's SOTA; cost-normalized resolve-per-token curves; reranking frontier-model rollouts; and the deferred RLVR run, where reward hacking and non-stationarity are the real stress test.

// bottom line

A genuinely useful result — execution-free verification can be made repo-grounded, can beat patch classifiers for best-of-K, and distills into a small open model. But every headline number is frontier-LLM-grading-frontier-LLM on the field's most-pretrained benchmark, the baseline table is missing the hybrid SOTA it cites, and the "scalable, lightweight" pitch is undercut by a frontier-agent-plus-many-call grading cost the paper never measures. A promising verifier signal, not yet the scalable-verification story the title gestures at — and the RLVR payoff that would prove it out is left for future work.

// takeaways

What to remember

  1. Verification as a repo-grounded checklist. An expert agent explores the codebase to write weighted criteria across four axes (File Change, Spec Alignment, Integrity, Runtime); patches are then graded execution-free into a [0,1] score. On SWE-Bench Verified that hits 40.6% (Qwen3-32B) and 54.2% (Qwen3-Coder) BEST@16 — +3.5 to +4.6 over the best baseline.
  2. Grounding and granularity are the levers. Strip repository access and BEST@16 drops up to 4 points; stronger rubric-agents win by writing more, finer criteria (Sonnet-4.5 averages >20 per instance, ~2× a non-coding model). And the skill is trainable — a fine-tuned 32B rubric generator beats a same-size patch classifier.
  3. A denser signal than pass/fail, but lightly validated. Rubric scores predict test outcomes at ROC-AUC 0.886 and sometimes flag real bugs tests miss — yet the evidence is LLM-graded on the most-pretrained benchmark in the field, with no hybrid-verifier comparison and no cost accounting. The RLVR use-case the framing points at is deferred.
