DAPO: An Open-Source LLM Reinforcement Learning System at Scale

The reasoning-model wave runs on large-scale RL, but the actual recipe stayed hidden. DAPO opens it: four surgical fixes to GRPO lift a Qwen2.5-32B base model from near-0% to 50 on AIME 2024 — beating DeepSeek-R1-Zero-Qwen-32B (47) in half the training steps — and the algorithm, the verl training code, and the dataset all ship.

// the gap

What problem is this solving?

Test-time scaling — OpenAI's o1, DeepSeek's R1 — turned reinforcement learning into the central engine of frontier reasoning. The models learn to think in long chains, self-verify, and backtrack, and they do it through RL on verifiable answers. But the recipe is a black box. The o1 blog and the R1 technical report publish results, not the knobs: the exact objective, the clipping ranges, the sampling discipline, the reward shaping. The community has been left trying to reverse-engineer industry-grade RL from headline numbers.

And the reproduction wall is real. The authors' own naive GRPO run on Qwen2.5-32B tops out at 30 points on AIME 2024 — far below DeepSeek's reported 47 on the same base model. Worse, the run is sick in characteristic ways: entropy collapses early (the policy goes deterministic and stops exploring), response length and entropy blow up unhealthily, and truncated generations inject reward noise that destabilizes training. Multiple independent teams hit the same symptoms, which strongly suggests the missing training details aren't incidental — they're load-bearing.

DAPO's contribution is to close that gap as an open system: name the specific failure modes of large-scale long-CoT RL, give each a concrete fix, and release the whole stack — the algorithm, the training code (built on verl), and a curated dataset (DAPO-Math-17K) — so the 50-on-AIME result is something anyone can rerun rather than admire from a distance.

// how it works

The method

DAPO starts from GRPO (Group Relative Policy Optimization), the critic-free workhorse from DeepSeekMath: for each prompt, sample a group of G responses, score each with a reward, and use the group-normalized reward as the advantage — no value network needed. On top of that, DAPO makes two structural simplifications and adds four targeted techniques. The simplifications: it drops the KL-divergence penalty entirely (a long-CoT reasoning policy is supposed to drift far from the base model, so anchoring it to the reference is counterproductive), and it uses a rule-based ±1 reward — +1 if the parsed answer is equivalent to the ground truth, −1 otherwise — so there is no learned reward model to hack.

1 · Clip-Higher — stop entropy collapse

PPO's clip is symmetric: 1±ε. With the default ε=0.2 and a positive advantage, a high-probability "exploitation" token at 0.9 may rise toward 1.08 (effectively uncapped), while a low-probability "exploration" token at 0.01 is bounded to 0.012 — a trivial increase. The upper clip therefore throttles exactly the tokens you want to encourage, the policy sharpens, and entropy collapses (Figure 1, right). DAPO decouples the clip into εlow=0.2 and εhigh=0.28, raising only the ceiling so low-probability tokens get room to grow. The lower bound is left alone — pushing it up would crush tokens toward zero and collapse the sampling space the other way.

2 · Dynamic Sampling — keep gradients non-zero

If every one of a prompt's G samples is correct (reward all +1), the group-normalized advantage is exactly zero — and a zero advantage means zero gradient for that prompt. As training improves, more and more prompts hit accuracy = 1, so the effective batch silently shrinks, gradient variance grows, and sample efficiency degrades. DAPO over-samples and filters: it keeps drawing until the batch is full of prompts with 0 < correct < G (the constraint in the objective above), discarding all-correct and all-wrong groups. Counterintuitively this speeds up wall-clock convergence — the extra sampling is cheap relative to the steps saved (Figure 3).

3 · Token-Level Policy Gradient Loss

Vanilla GRPO reduces the loss per sample: it averages token losses within a response, then averages across responses — so every response carries equal weight regardless of length. In long-CoT training that backfires. Tokens inside long responses are systematically underweighted, so the model can't properly learn from high-quality long reasoning; and because long, low-quality samples (gibberish, repetition) aren't penalized in proportion to their length, entropy and response length inflate unhealthily. DAPO normalizes by the total token count in the batch (the 1/Σ|oi| term), so every token counts equally and a bad pattern is suppressed wherever it appears.

4 · Overlong Reward Shaping

Generations are capped at a maximum length and truncated past it. Naïvely assigning the truncated sample a punitive −1 is reward noise: a perfectly sound chain of reasoning gets penalized purely for being long, confusing the model about whether its reasoning was valid. DAPO first tries Overlong Filtering — simply mask the loss of truncated samples — which already stabilizes training markedly. It then adds Soft Overlong Punishment: a graded length penalty that kicks in only inside a cache zone before the hard cap, so longer-but-not-yet-truncated responses are gently discouraged rather than cliff-edged.

Rlength(y) = 0,  if |y| ≤ Lmax − Lcache
(Lmax − Lcache) − |y|Lcache,  if Lmax − Lcache < |y| ≤ Lmax
−1,  if Lmax < |y| // Lmax = 16,384 tokens, Lcache = 4,096 (max generation 20,480); added on top of the ±1 correctness reward

The whole thing trains with verl, AdamW at a constant 1×10⁻⁶ learning rate, a prompt batch of 512 with 16 samples each, and a curated DAPO-Math-17K dataset whose answers are all transformed into integers so a rule-based checker can grade them without a brittle formula parser.

// what they found

The results

On AIME 2024, DAPO takes a Qwen2.5-32B base model from near 0% to 50 avg@32, surpassing DeepSeek-R1-Zero-Qwen-32B's 47 — and reaching it in 50% of the training steps. The progressive ablation (Table 1) is the paper's core evidence: each technique is layered onto naive GRPO and contributes a few points, with Clip-Higher and Dynamic Sampling doing the heavy lifting at the two ends.

Beyond the headline, the paper leans hard on training-dynamics monitoring — response length, reward score, generation entropy, and mean token probability — arguing that in a system this coupled, watching these curves is how you catch a run going bad before it wastes a week of compute. They also report a qualitative finding: reflective behaviors emerge during training — checking and backtracking ("however, wait a moment, let's rethink…") that were essentially absent early on appear on their own as RL progresses.

// does it hold up?

The eval read

The artifact is excellent and the four fixes are mechanistically well-motivated — but as an empirical paper, the evidence base is surprisingly thin for the strength of the claims. Everything rests on one benchmark (AIME 2024, just 30 problems), one base model (Qwen2.5-32B), and one domain (math). The "readily transferred to other tasks" line is asserted, not demonstrated: there's no coding eval, no MATH/AMC/OlympiadBench, no out-of-domain check.

The two flagship comparisons both lean on a quoted number. The "47, in 2× the steps" for DeepSeek-R1-Zero-Qwen-32B comes from R1's own setup, not a controlled re-run on matched infrastructure, data, and hyperparameters — so "50% of the training steps" is a headline framing more than an apples-to-apples measurement. And Table 1 is a single progressive chain with no seeds and no error bars. On a 30-problem set scored avg@32, several of the per-technique deltas (token-level loss +1, Clip-Higher +2) sit comfortably inside sampling noise; the layering order is plausible and useful, but the exact attributions shouldn't be read as cleanly isolated effects.

baselines

Only DeepSeek-R1-Zero-Qwen-32B (47), and that figure is quoted from R1's paper rather than re-run here. No head-to-head with QwQ-32B, o1, or full R1; the internal baseline is the authors' own naive GRPO (30).

contamination

AIME 2024 (Feb 2024) predates Qwen2.5's release, and DAPO-Math-17K is web-scraped from competition homepages. No decontamination of the training set against AIME is reported — a real leakage risk for a math-competition benchmark.

what's missed

Single benchmark / single model / math-only, with no variance reporting. avg@32 controls scoring noise but not the underlying 30-item, single-seed fragility, so the small Table-1 increments are suggestive rather than established.

would raise confidence

Multiple seeds with error bars; more base models and sizes; MATH/AMC/OlympiadBench plus a coding eval; a same-infrastructure, step-matched DAPO-vs-GRPO comparison; and a contamination audit of DAPO-Math-17K against the test set.

// bottom line

As an algorithm and an open system, DAPO is the most useful artifact to come out of the post-R1 reproduction scramble: the four techniques are sound engineering defaults and the code + data are genuinely public. As an empirical result it overclaims from a single math benchmark on a single model — treat the four fixes as well-justified defaults to adopt, and the precise "+N per technique" and "half the steps" as headline-grade, not rigorously isolated.

// takeaways

What to remember

  1. The real contribution is reproducibility, not a new objective: DAPO is GRPO with the KL penalty dropped plus four surgical fixes — decoupled clip, dynamic sampling, token-level loss, overlong shaping — shipped with the verl training code and the DAPO-Math-17K dataset so the 50-on-AIME result is rerunnable.
  2. Entropy collapse is the silent killer of LLM RL, and the cheapest high-leverage fix is Clip-Higher: just split PPO's clip into εlow=0.2 / εhigh=0.28 to give low-probability "exploration" tokens room to grow.
  3. In long-CoT RL, how you reduce the loss matters: averaging per-sample (GRPO) underweights long responses and lets gibberish inflate length and entropy; normalizing per-token is what keeps length growth healthy — a subtle implementation detail with outsized effects.
