ToolLLM: Facilitating Large Language Models to Master 16000+ Real-world APIs

Open-source LLMs can chat but can't reliably use tools — ToolLLM fixes this the ChatGPT-bootstrapped way: crawl 16,464 real, executable REST APIs from RapidAPI, get ChatGPT to write instructions and search out multi-step solution paths for them with a new decision-tree strategy (DFSDT), then fine-tune LLaMA-2 on the result. The resulting ToolLLaMA lands within a few points of ChatGPT on the paper's own ToolEval suite and generalizes to APIs it never trained on.

Yujia Qin, Shihao Liang, Yining Ye, Kunlun Zhu, Lan Yan, … Dahai Li, Zhiyuan Liu, Maosong Sun · Tsinghua University · ModelBest Inc.

// the gap

What problem is this solving?

By mid-2023, ChatGPT and GPT-4 had clearly demonstrated that large language models could call external tools — search engines, calculators, code interpreters, arbitrary REST APIs — to extend what they could do beyond next-token prediction. But that competence lived entirely behind closed weights. Open LLaMA-family models, even after heavy instruction tuning (Alpaca, Vicuna), showed essentially no tool-use ability, because the instruction-tuning corpora that produced them were built for general dialogue — summarizing, chatting, following formatting instructions — and simply never included examples of calling an API, reading back its response, and using that to complete a task. The authors frame this bluntly: instruction tuning has focused on basic language skills while "the tool-use domain" was left untouched, and how SOTA models like ChatGPT actually acquired their tool competence remains opaque because the models are closed.

A few groups had already tried to close this gap with synthetic tool-use datasets, and the paper's own Table 1 lays out exactly why each one falls short along three axes. Limited APIs: Gorilla's APIBench covers only 3 tools (1,645 APIs, all drawn from three ML hubs — TorchHub, TensorHub, HuggingFace); API-Bank covers 53 tools; ToolAlpaca covers 400 tools but the API responses are simulated by an LLM rather than real HTTP calls. Constrained scenario: every one of these datasets confines itself to instructions that invoke a single tool at a time, which is a poor model of realistic tasks — booking a hotel while cross-referencing a movie API and a facts API is a two- or three-tool problem, not a one-tool problem, and none of these works assume the model has to select the right tool itself from a large pool rather than being handed a curated, tiny candidate set. Inferior planning and reasoning: the standard toolkit for eliciting multi-step tool behavior was chain-of-thought or ReAct-style prompting, and both are strictly linear — one wrong turn early in a reasoning chain propagates forward with no way to back out, and even GPT-4 running ReAct-style search frequently fails to land on a valid multi-step solution for anything but the simplest instructions. Some of the same datasets (APIBench, ToolAlpaca) don't even execute real API calls during their own construction, which means the "ground truth" solution paths they supply were never actually verified to work.

ToolLLM's premise is that none of these are separately fatal, but together they explain why no open model could really use tools: you need real, executable, large-scale APIs; you need instructions that plausibly chain several of them together; and you need a search procedure patient enough to find a working solution path even when the first attempt fails. Confusingly, one of the baselines in that same Table 1 — Xu et al. (2023b), "On the Tool Manipulation Capability of Open-source Large Language Models" — is also called "ToolBench," a naming collision with this paper's own dataset that is worth flagging so the two aren't conflated when reading citations elsewhere.

// how it works

The method

ToolLLM is really three things bundled together: an instruction-tuning dataset called ToolBench, built almost entirely by ChatGPT in three stages; a new search strategy for annotating multi-step solutions called DFSDT (depth-first search-based decision tree); and the resulting fine-tuned model, ToolLLaMA, paired with a lightweight neural API retriever so a user never has to hand-pick which of 16,000+ APIs to expose to the model.

Stage 1 — API collection

The APIs come from RapidAPI Hub, a marketplace of real, live, developer-published REST APIs organized into 49 coarse-grained categories (finance, sports, weather, movies, …) and 500+ finer-grained "collections" (e.g., all Chinese-language APIs). For each tool the authors crawl its name, description, host URL, and the full documentation of every API it exposes: HTTP method, required and optional parameters, executable code snippets, and an example response — everything an LLM would need to call the API zero-shot from documentation alone, without ever having seen it during training. They start from 10,853 tools (53,190 APIs) and run them through a filtering pass — dropping anything that returns errors, times out, or otherwise looks unreliable — leaving 3,451 high-quality tools spanning 16,464 APIs across the 49 categories.

Stage 2 — instruction generation

Rather than brainstorming tasks first and searching for APIs to match, the pipeline samples a set of APIs and prompts ChatGPT (gpt-3.5-turbo-16k, using its function-calling interface) to invent instructions that plausibly require exactly those APIs, conditioned on 3 in-context seed examples pulled from a human-written pool of 12 single-tool and 36 multi-tool seeds. To get real multi-tool coverage without random tool combinations producing nonsensical instructions (most tools on RapidAPI have nothing to do with each other), the authors exploit the RapidAPI hierarchy directly: single-tool instructions (I1) iterate over each tool's own APIs; intra-category multi-tool instructions (I2) sample 2–5 tools from the same coarse category; intra-collection multi-tool instructions (I3) sample from the same fine-grained collection, where tools are known to share functionality or goals. After filtering out instructions whose "relevant APIs" turned out to be hallucinated (i.e., not actually present in the sampled set), this stage yields 87,413 / 84,815 / 25,251 qualified (instruction, relevant-API) pairs for I1 / I2 / I3 respectively — nearly 200k in total.

Stage 3 — solution-path annotation with DFSDT

This is the paper's central methodological contribution. Given an instruction, the model needs to produce a full action sequence — a chain of "thought → API call → observation" steps culminating in either a final answer or a documented give-up — and the authors cast this as a multi-round conversation:

at ~ ChatGPT( at | {a1, r1, …, at-1, rt-1}, Inst* ) // the next action is sampled conditioned on every prior action a and every real API response r so far, plus the original instruction

Conventional CoT or ReAct reasoning walks this chain forward with no way back: if step 3 calls the wrong API or hallucinates a bad argument, every subsequent step inherits that mistake (error propagation), and because only one path through the decision space is ever explored, there's no recovery even in principle (limited exploration). In the authors' own pilot studies, even GPT-4 running ReAct often can't find a single valid path for anything beyond simple single-tool instructions. DFSDT instead expands a decision tree: at each node the model can either extend the most promising path, or call a "Finish by Giving Up" action that prunes the current branch and backtracks to try a diversified alternative — the prompt explicitly shows the model its previously-tried and failed actions so the next attempt is forced to differ. Two special terminal actions close out a trace: Finish with Final Answer and Finish by Giving Up. Naively this would cost O(n log n) ChatGPT calls per node to rank n candidate children, but the authors find empirically that the first-generated child is usually the best-ranked one anyway, so they skip sorting and use a cheap pre-order DFS traversal instead — a design that collapses to plain ReAct when no retraction is ever needed (so it's no more expensive in the easy case) while still reaching almost the same solutions a full DFS would find in the hard case.

Only passed solution paths are retained as training data, which converts DFSDT's search-space advantage directly into annotation efficiency: at matched ChatGPT API-call budget (a "ReACT@N" baseline that reruns ReAct repeatedly until it burns the same cost as one DFSDT search), DFSDT still finds a valid path far more often — see Results below — so the same annotation budget buys more usable, and harder, training instructions. Running DFSDT over the full instruction set yields the final ToolBench: 126,486 (instruction, solution-path) pairs, built from 469,585 real API calls across the 3,451 tools, averaging 4.0 reasoning traces (API calls) per instance. Because raw API responses can be long JSON blobs that eat into the context budget, a separate ChatGPT-driven compression step strips response fields judged unimportant relative to a documented schema, truncating anything still over 1,024 tokens.

Training ToolLLaMA and the neural API retriever

ToolLLaMA is a LLaMA-2 7B model fine-tuned directly on these instruction–solution-path conversations, in the same multi-round format ChatGPT produced them in. Because API responses can be verbose, the authors extend LLaMA-2's native 4,096-token context to 8,192 via positional interpolation before training for 2 epochs (learning rate 5×10⁻⁵, batch size 64). Since no one wants to hand-pick from a pool of 16,000+ APIs at inference time, the paper also trains a small dense retriever — a Sentence-BERT model built on BERT-BASE — with contrastive learning: an instruction's ground-truth relevant APIs are positives, a sample of other APIs are negatives, and at inference the retriever ranks the full 16,464-API pool by embedding similarity to recommend a shortlist. Finally, to actually score all of this, the authors build ToolEval, an automatic ChatGPT-backed evaluator with two metrics: pass rate (did the model complete the instruction within a call budget?) and win rate (given two solution paths for the same instruction, which is better, judged on information richness, factual accuracy, quality of failure explanation, milestones reached, breadth of API exploration, and redundant-call cost). Validated against 300 human-annotated instructions per method across four method pairs, ToolEval agrees with human judgment 87.1% of the time on pass rate and 80.3% on win rate.

// what they found

The results

The clearest baseline result is also the starkest: Vicuna and Alpaca — general-purpose instruction-tuned LLaMA variants — score 0.0 pass rate and 0.0 win rate on every single ToolBench scenario, even after extensive prompt engineering. Whatever instruction-following ability those models have simply does not transfer to calling an API correctly. That's the paper's headline evidence that tool use is a data problem, not a scale or prompting problem: the fix isn't a better prompt, it's training data that actually contains tool-call traces.

Before even training ToolLLaMA, the authors validate their two building blocks in isolation. The neural API retriever beats both BM25 and OpenAI's text-embedding-ada-002 by a wide margin — averaged NDCG@1/@5 across the three instruction types (I1/I2/I3) is 78.0 / 84.9 for their retriever versus 18.5 / 17.0 for BM25 and 49.6 / 45.4 for ada-002. And DFSDT beats both plain ReAct and the compute-matched "ReACT@N" control on ChatGPT: average pass rate 63.8 for DFSDT versus 35.3 for ReAct and 44.5 for ReACT@N, with the DFSDT advantage growing on the harder multi-tool instructions — 70.6 vs. 40.6/49.4 on I2 and 62.8 vs. 27.6/34.6 on I3, compared to a smaller 58.0 vs. 37.8/49.4 gap on the easier single-tool I1. In other words, DFSDT's backtracking pays off exactly where a linear chain is most likely to get stuck.

Two things stand out. First, ToolLLaMA+DFSDT (66.7 pass / 60.0 win) lands just under ChatGPT+DFSDT (64.8 / 64.3 — essentially tied) and only behind GPT4+DFSDT (71.1 / 70.4), while comfortably beating Claude-2 and Text-Davinci-003 running the same DFSDT strategy — a 7B open model, fine-tuned on 126k synthetic traces, closing most of the gap to its own teacher. Second, swapping the oracle API set for the trained retriever's top-5 suggestions improves ToolLLaMA's score slightly (67.3 / 63.1 vs. 66.7 / 60.0) — the authors' explanation is that the retriever sometimes substitutes a functionally-better API for the "ground truth" one, evidence the retriever generalizes usefully across the 16,000+ API pool rather than just memorizing.

The last experiment tests true out-of-distribution transfer: without any additional training, ToolLLaMA is pointed at Gorilla's APIBench (HuggingFace, TorchHub, TensorHub — domains ToolBench never covered). Using their own retriever, ToolLLaMA reaches AST accuracy of 16.77 / 51.16 / 40.59 across the three domains with hallucination rates of 10.60 / 15.70 / 6.48 — beating Gorilla's zero-shot variant (Gorilla-ZS+BM25: AST 10.51 / 44.62 / 34.31, hallucination 46.90 / 17.20 / 20.58) on every domain. Gorilla's retrieval-aware variant trained specifically on APIBench (Gorilla-RS) still edges out ToolLLaMA on raw AST accuracy under an oracle retriever (89.27 / 93.01 / 94.16 vs. ToolLLaMA's 88.80 / 85.88 / 88.62) — unsurprising, since Gorilla-RS is trained in-domain on exactly these three API families, while ToolLLaMA has never seen a single HuggingFace, TorchHub, or TensorHub API.

// does it hold up?

The eval read

ToolLLM's entire pipeline — instruction generation, solution-path search, response compression, and final scoring — runs on ChatGPT. That's an efficient way to bootstrap a dataset with "minimal human supervision," and the authors do the right validating move: they hand-annotate 300 test instructions per method across four method pairs (1,200 human judgments total) and report 87.1% pass-rate agreement and 80.3% win-rate agreement between ToolEval and humans. But those numbers deserve a second look rather than a nod. An 80% pairwise-preference agreement rate is respectable for LLM-as-judge work, yet it means roughly one in five win/lose calls in Table 4 and Table 6 disagrees with what a human annotator would say — on a benchmark whose entire headline claim ("ToolLLaMA ≈ ChatGPT, second only to GPT-4") is a difference of a few win-rate points. The authors themselves go further and undercut their own metric's authority in the appendix: they report that even human experts frequently disagree with each other about which of two tool-use solution paths is "better," because there's no single correct way to solve an open-ended tool task — one annotator may prefer a path that finishes fast with a few calls, another may prefer one that cross-validates across more APIs. If human-vs-human agreement is already shaky, ChatGPT-vs-human agreement of 80% is arguably close to the ceiling the setup can achieve, not a strong endorsement of ToolEval's reliability in an absolute sense.

There is also a closed-loop concern the paper doesn't address directly: ChatGPT is simultaneously the instruction author, the DFSDT solution-path annotator (whose passed traces become ToolLLaMA's training data), and the ToolEval judge that later scores ToolLLaMA and its ChatGPT-family competitors. Any systematic bias in what ChatGPT considers a "complete" or "well-reasoned" answer gets baked into the training distribution and then rewarded again at evaluation time — a model trained to imitate ChatGPT's tool-use style is being judged, in part, by ChatGPT's own taste for that style. This doesn't invalidate the pass-rate numbers (executing a working solution within budget is a fairly objective bar), but it does mean the win-rate comparisons, especially "ToolLLaMA performs comparably to ChatGPT," should be read as "ToolLLaMA imitates ChatGPT's manner of tool use convincingly enough to fool a ChatGPT judge," which is a narrower and more circular claim than it first appears.

On the data side: the 16,464 APIs are genuinely real and executable — that part isn't synthetic. But every instruction in ToolBench is invented by ChatGPT from API documentation and a handful of seed examples, not sourced from logs of real users trying to get something done. There's no evidence in the paper that these are the kinds of tool-use requests people actually make; "diversity" is measured by how varied ChatGPT's own inventions look (validated only by "rigorous human evaluation" and an Atlas visualization, both under-specified), not by comparison to any external distribution of real tasks. The paper also candidly flags, in its own appendix, that RapidAPI has "temporal variability" — APIs go down, change schemas, or get deprecated over time — which is precisely why they say a fixed ground-truth solution path can't be annotated once and reused forever. That's an honest limitation, but it also means ToolBench's live-API evaluation numbers are not fully reproducible months or years later without re-running against whatever the RapidAPI endpoints currently return (a problem serious enough that later work built a "stable" frozen-response variant of this exact benchmark specifically to fix it).

Finally, on what pass rate misses: it only asks whether an instruction was completed within a call budget, using ChatGPT's own judgment of whether "the final answer completely resolves the original instruction" — but ChatGPT-as-judge cannot independently re-execute the APIs to verify the final answer's factual content is accurate, only that it reads as though it resolves the request. A model could hallucinate a plausible-sounding final answer that happens to satisfy the judge without ever correctly parsing the underlying API response. Pass rate also doesn't penalize inefficiency (win rate has a "cost" criterion, pass rate doesn't) or reward finding the objectively best API when multiple would technically "pass" — so two very different quality solutions can register as identical successes.

baselines

Vicuna/Alpaca are appropriately weak strawmen (0.0 everywhere); the real comparison set — ChatGPT, GPT-4, Claude-2, Text-Davinci-003, all run with both ReACT and the paper's own DFSDT — is reasonably strong and current for 2023, though it excludes any other tool-tuned open model as a baseline for ToolLLaMA specifically.

contamination

Within-dataset generalization is controlled via held-out instructions/tools/categories (Inst./Tool/Cat. splits), which is sound. The only genuine external test is APIBench, and there Gorilla-RS (trained in-domain on APIBench) still edges out ToolLLaMA on raw AST accuracy under oracle retrieval — a nuance the "remarkable OOD generalization" framing slightly undersells.

what's missed

Pass rate can't verify factual correctness of the final answer against the real API data, doesn't penalize call-count inefficiency, and the whole scoring pipeline shares a model family with the data-generation and annotation pipeline.

would raise confidence

A judge model from a different lineage than the data-generating model; a frozen/replayable snapshot of the RapidAPI responses so scores don't drift as live endpoints change; a larger human-annotated sample given the acknowledged low human-human agreement; and evaluation against instructions sourced from real user logs rather than only LLM-invented ones.

// bottom line

The scale and realism of the API side of ToolBench is a genuine, verifiable advance — 16,464 live, executable REST APIs is an order of magnitude beyond anything that came before it, and DFSDT is a real, measurable improvement over ReAct at matched cost. But the evaluation loop is entirely self-referential (ChatGPT writes the tasks, ChatGPT searches the solutions, ChatGPT grades the outcome), validated against a human sample whose own inter-annotator agreement the authors admit is shaky. Treat "ToolLLaMA ≈ ChatGPT" as "convincing to a ChatGPT-family judge on ChatGPT-style tool-use instructions," not as a claim independently verified against ground-truth task success.

// takeaways

What to remember

  1. Open-model tool use in 2023 was a data problem, not a scale or prompting problem: Vicuna and Alpaca score exactly 0.0 pass rate on every ToolBench scenario, while a 7B LLaMA-2 fine-tuned on 126,486 real API-call traces lands within a few points of ChatGPT — the fix was showing the model 16,464 real, documented APIs and their responses, not a bigger base model.
  2. DFSDT beats ReAct precisely where linear reasoning chains are most fragile: the gap over a compute-matched ReAct baseline is small on easy single-tool instructions (58.0 vs. 49.4 pass rate) and grows sharply on hard multi-tool ones (70.6 vs. 49.4 on I2, 62.8 vs. 34.6 on I3) — because a decision tree with a "give up and retract" action recovers from an early wrong turn instead of propagating it.
  3. The whole system — instruction generation, DFSDT solution search, and the ToolEval judge — is ChatGPT evaluating ChatGPT-descended behavior, validated on a human sample the authors themselves note has low inter-annotator agreement. That's a reasonable way to bootstrap a benchmark cheaply, but it's worth remembering every time "ToolLLaMA ≈ ChatGPT" gets cited as a standalone fact.
