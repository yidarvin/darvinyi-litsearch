From Local to Global: A Graph RAG Approach to Query-Focused Summarization

Vector RAG retrieves the handful of chunks nearest your query, so it falls apart on global questions like “what are the main themes?”. GraphRAG indexes a corpus as an LLM-built knowledge graph, pre-summarizes its detected communities, and map-reduces those summaries into a global answer — winning 72–83% of head-to-head judgments over vector RAG on comprehensiveness and diversity.

Edge, Trinh, Cheng, Bradley, Chao, Mody, Truitt, Metropolitansky, Ness, Larson · Microsoft Research

// the gap

What problem is this solving?

Retrieval-augmented generation (RAG) lets an LLM answer questions over a corpus too large for its context window: embed the query, retrieve the top-k chunks that are individually most similar, stuff them into the prompt, and generate. This works beautifully for local questions — ones whose answer lives in a small, findable set of records (“what was NeoChip’s acquisition price?”).

It breaks on global questions: “what are the main themes in this dataset?”, “what are the key trends across these reports?”. The authors call these sensemaking queries — they require reasoning over connections spread across the entire corpus, so the answer is in no top-k neighbourhood to retrieve. This is fundamentally a query-focused summarization (QFS) problem, not a retrieval problem.

The obvious fix — classic abstractive summarization — has the opposite failure mode: QFS methods produce good global summaries but don’t scale to the millions of tokens a RAG system routinely indexes. You can’t fit a 1M-token corpus in a context window, and naively map-reducing a summary over all the raw source text is enormously expensive. So the field had two halves of a solution that didn’t compose: retrieval that scales but can’t globalize, and summarization that globalizes but can’t scale. GraphRAG is an attempt to get both at once.

// how it works

The method

GraphRAG front-loads the work into an index-time graph build, then answers each query with a cheap map-reduce over precomputed summaries. The build has five stages:

1 · Chunk. Split the documents into text chunks (the paper uses 600-token chunks with 100-token overlap). Chunk size is a real lever: longer chunks mean fewer LLM extraction calls (cheaper) but worse recall of entities near the start of the chunk.

2 · Extract entities & relationships. An LLM reads each chunk and emits the important entities (with descriptions), the relationships between them, and optionally claims — verifiable factual statements like dates and events. Prompts carry domain-specific few-shot exemplars. A self-reflection (“gleaning”) loop then asks the model whether any entities were missed and re-extracts, which lets you use larger, cheaper chunks without losing recall.

3 · Build the knowledge graph. The same entity is detected many times across chunks; those instances are merged into single graph nodes (entities) and edges (relationships), with the duplicate count becoming the edge weight. Entity matching here is plain exact-string matching — resilient enough because duplicates get clustered together downstream anyway.

4 · Detect communities. Run hierarchical Leiden community detection to partition the graph into nested communities of closely related entities. Each level is a mutually-exclusive, collectively-exhaustive cover of the nodes — root communities (C0, few and broad) recursively split into sub-communities (down to C3, many and fine). This modular structure is the part GraphRAG exploits that prior graph-RAG work did not.

5 · Summarize communities, bottom-up. The LLM writes a report-like summary for every community. Leaf communities prioritize their elements by combined source+target node degree (most prominent relationships first) until the token budget fills; higher levels substitute shorter sub-community summaries for their longer element summaries when needed. These summaries are independently useful — you can browse them to understand a corpus with no query at all.

Query time: map-reduce over community summaries

Given a question, GraphRAG picks a community level and answers in two passes. In the map step, the community summaries are shuffled, chunked, and each chunk independently produces a partial answer plus a self-assigned helpfulness score (0–100); answers scored 0 are dropped. In the reduce step, the surviving partial answers are sorted by helpfulness, packed into a fresh context window, and summarized once more into the final global answer.

global_answer = LLMreduce( sort↓score { (ai, si) = LLMmap(summaryi, q) : si > 0 } ) // each community summary independently answers the query q and rates its own helpfulness s; the reduce step folds the high-scoring partials into one answer (schematic)

Crucially, the shuffling spreads relevant information across chunks so it isn’t concentrated — and potentially lost — in a single context window, a nod to the “lost in the middle” long-context failure mode. Because the expensive part (the graph and its summaries) is computed once, query cost depends only on how many community summaries you feed in, which is exactly what the community level controls.

// what they found

The results

The evaluation covers two corpora in the ~1M-token range: podcast transcripts (Behind the Tech, ~1M tokens → an 8,564-node / 20,691-edge graph) and news articles (~1.7M tokens → 15,754 nodes / 19,520 edges). Because global sensemaking questions have no gold answers, the authors generate 125 questions per dataset from LLM-invented user personas, then score answers with an LLM-as-judge on four criteria: comprehensiveness, diversity, empowerment, and a control criterion, directness. The model throughout is gpt-4-turbo.

Against conventional vector RAG (the “semantic search” baseline, SS), the global graph methods win decisively on the two headline criteria: comprehensiveness win rates of 72–83% (Podcast) and 72–80% (News), and diversity win rates of 75–82% (Podcast) and 62–71% (News), all significant (p<.001, diversity-News p<.01). As a sanity check, vector RAG wins on directness — confirming the criteria measure a real breadth-vs-concision trade-off rather than a uniform “GraphRAG is just better” artifact. Empowerment was mixed.

The most practically important result is about cost. Answering from root-level summaries (C0) costs only 26,657 tokens/query (2.6% of the maximum) on Podcast and 39,770 (2.3%) on News — roughly 9×–43× fewer tokens than higher GraphRAG levels and >97% fewer than map-reducing over raw source text (TS) — while still beating vector RAG by 72% on comprehensiveness and 62% on diversity. In other words, the cheapest configuration keeps most of the quality benefit.

A second experiment validates the LLM-judge scores with a model-free metric. Using Claimify to extract verifiable claims (47,075 unique claims, ~31 per answer), the authors measure comprehensiveness as average claim count and diversity as average claim-cluster count. All global conditions (C0–C3, TS) produced significantly more claims than vector RAG (C0: 34.18 News / 32.21 Podcast vs SS’s 25.23 / 26.50, p<.05). Where the LLM judge returned a non-tie majority, it agreed with the claim-based winner 78% of the time on comprehensiveness and 69–70% on diversity — moderate corroboration. Notably, the graph-free TS baseline matches GraphRAG on quality at every level; GraphRAG’s separation from it is in tokens, not answers.

// does it hold up?

The eval read

GraphRAG is a clean systems idea with a soft evaluation underneath it. The headline “substantial improvements over a conventional RAG baseline” is true — but the baseline is naive top-k vector RAG, and the comparison that would actually isolate the contribution is missing. GraphRAG’s related work cites RAPTOR (recursive hierarchical-tree summarization) and other advanced/graph RAG methods, yet benchmarks against none of them. RAPTOR is the obvious control: it also builds a hierarchy of summaries, so beating it would show the graph + community detection specifically helps — beating only vector RAG and raw-text summarization does not.

And the paper’s own numbers undercut the quality story. The graph-free TS condition (map-reduce over raw text, no graph at all) ties or edges out GraphRAG on comprehensiveness and diversity at every level; in fact TS posts the single highest comprehensiveness win over vector RAG (83%). So the graph’s demonstrated payoff is efficiency — root summaries deliver ~72% of the win at 2.6% of the tokens — not better answers than plain global summarization. That’s a real and valuable result, but it’s a different claim than “GraphRAG produces better answers,” which the framing invites.

Then there’s the judge. Comprehensiveness and diversity both reward longer, more varied answers — exactly the qualities LLM judges are known to over-prefer (verbosity and self-enhancement bias, the latter sharpened by judging same-family gpt-4 outputs). The directness control confirms vector RAG’s answers are more concise; answer length is never reported or controlled, so an unknown share of the win could be verbosity rather than substance.

baselines

Only naive top-k vector RAG (SS) and graph-free map-reduce (TS) — no RAPTOR or other advanced/hierarchical RAG it cites. TS matches GraphRAG on answer quality, so the graph’s edge is cost, not accuracy.

contamination

No train/test leakage (private corpora), but the eval is GPT-4 judging GPT-4 with no gold answer, on criteria that favor long answers, with answer length uncontrolled — a self-preference/length-bias confound.

what's missed

Only two English corpora (one podcast, one news set); questions are LLM-synthesized via personas, not real users; no hallucination/fabrication rate (flagged as future work); the ~281-min index build is acknowledged but never amortized into a cost-benefit; exact-string entity matching ignores coreference/aliasing.

would raise confidence

Head-to-head vs RAPTOR and graph-free hierarchical summarization; length-controlled and/or human evaluation; a measured fabrication rate; an ablation (does Leiden community detection beat random clustering of the same entities?); diverse and multilingual corpora.

// bottom line

GraphRAG turns a corpus into an LLM-built, community-summarized knowledge graph and makes global “sensemaking” queries tractable — with the elegant practical finding that root-level summaries give most of the benefit at ~2.6% of the token cost. But the quality claims rest on a single GPT-4-as-judge setup over two narrow corpora, against weak baselines, on criteria that reward verbosity; the paper’s own results suggest the graph’s defensible advantage is scalability and cost, not proven answer-quality superiority over plain global summarization.

// takeaways

What to remember

  1. RAG ≠ summarization. Top-k retrieval answers local fact questions; “what are the main themes?” is query-focused summarization over the whole corpus, which retrieval structurally cannot do. GraphRAG reframes global QA as map-reduce over precomputed community summaries.
  2. The graph is an index, not the answer. Value comes from doing the expensive work once: LLM-extracted entities → hierarchical Leiden communities → bottom-up community summaries. Queries just map-reduce those summaries, and the community level is a cost/quality dial — root-level C0 wins 72% comprehensiveness / 62% diversity over vector RAG at 2.6% of the tokens.
  3. Read the eval as efficiency, not superiority. Gains are GPT-4 judging GPT-4 on length-favoring criteria, over two corpora, against naive vector RAG — and graph-free global summarization nearly matches GraphRAG on quality. The robust, replicated win is scalability; the answer-quality edge over strong baselines is unproven.
