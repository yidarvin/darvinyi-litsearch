VeRO: A Harness for Agents to Optimize Agents

A reproducible harness and benchmark for the "agents optimizing agents" task — a coding agent that edits, evaluates, and versions a target agent's code under a fixed evaluation budget. Across five tasks, the full harness lifts tool-use agents ~8% over baseline while reasoning tasks barely move.

// the gap

What problem is this solving?

There are two ways to make an LLM agent better: optimize the model's weights (RL, fine-tuning), or optimize the agent program itself — its prompts, tools, control flow, and memory. The program-optimization story is dominated by frameworks that only touch textual context: DSPy and TextGrad mutate prompts and tool descriptions, GEPA evolves them reflectively. The broader program structure — the agent harness — stays off-limits, even though in practice it's where most of an agent's behavior lives.

And in production, optimizing that harness is a slow, manual loop: stand up an agent, read failure modes out of its traces, hand-edit a prompt or add a tool, re-evaluate, repeat. As demand for specialized agents grows, that loop doesn't scale — which is why "let a coding agent do the optimizing" has become an obvious idea. The trouble is the community has no systematic way to measure how good coding agents actually are at this, for two reasons:

  • The task isn't ordinary software engineering. An agent harness interleaves deterministic code with stochastic LLM completions, so improvement signal has to come from structured capture of both the intermediate reasoning traces and the downstream task outcomes — not just "do the tests pass."
  • Fair comparison is nearly impossible without infrastructure. Different optimizers are different coding agents (Claude Code, the OpenAI Agents SDK, bespoke loops) with different tools and interfaces. Without standardization you can't tell whether one beat another because it searched smarter — or because it spent more compute, or peeked at held-out test data, or edited the evaluation harness itself.

VeRO is the missing measurement layer: a protocol and harness that makes "an agent optimizing an agent" a controlled, reproducible, comparable task, plus VeRO-Bench, a suite of target agents and tasks to run it on.

// how it works

The method

VeRO (Versioning, Rewards, and Observations) is an outer harness that wraps a target agent and exposes it to an optimizer through one standardized interface. The optimizer S — itself a coding agent — produces a sequence of target-agent versions A0, A1, …, AT, each conditioned on the full history of prior versions and their evaluation traces, under a fixed budget B of evaluation calls and a restricted search space:

At+1  =  S( f({Ai, τi}ti=0),  C )    s.t.   nE ≤ B // optimizer S proposes the next agent version from observed history (traces τ) and context C; the harness blocks any request past the evaluation budget B

Those constraints translate into six concrete requirements — versioning (every edit is a discrete snapshot, e.g. a git commit), budget enforcement (block any evaluation past B), permission control (no access to held-out test data, no swapping the underlying model, restricted file writes), reproducible execution (dependency lockfiles + isolation), structured tracing (inputs, outputs, intermediate steps, scores), and a standardized observation interface so no optimizer gets privileged information. VeRO implements them as five core abstractions, each exposed through explicit tools and transparent hooks:

  • Git Worktree — the target codebase is a git repo; an auto-commit hook records every change as an immutable diff, so the optimization trajectory is fully inspectable and rollback-able.
  • Dataset — manages train/val/test splits; the DatasetViewer enforces access control so the optimizer literally cannot view the held-out test set.
  • Filesystem — pattern-based access control that walls off tests, ground-truth implementations, and evaluation infrastructure from edits.
  • Experiment Database — stores every trace: per-sample scores, errors, agent rollouts, aggregate stats; the ExperimentViewer is how the optimizer "sees" what went wrong.
  • Evaluator — the ExperimentRunner is a gated interface: each request names a commit and samples, the engine checks out that version, runs it, stores results, and decrements the budget. This is what stops an optimizer from buying its way to a better score with extra compute.

Crucially, VeRO is coding-agent-agnostic: any coding agent that consumes its interfaces and preserves commit-level traceability can be benchmarked. The authors ship a minimal reference optimizer, the VeRO-Agent, and also wire Anthropic's Claude Code into the harness via MCP tools and bash hooks. Because hooks mediate the tool layer — file edits trigger auto-commits, evaluations route through the gated Evaluator, data access goes through viewers — optimizers with totally different native toolsets can still be compared on equal footing. The whole reproducibility story (uv-pinned packages + git versioning) deliberately mirrors how SWE-bench uses Docker to make patch evaluation deterministic.

// what they found

The results

The benchmark study compares 8 optimizer configurations across 5 tasks (MATH, TAU-Bench Retail, GAIA, SimpleQA, GPQA), averaging over N = 3 iterations — 120 experiments in all. The target agent is always GPT-4.1 mini, the budget is fixed at B = 8, and the best commit is selected on validation. The headline number: holding the optimizer model constant at Claude Sonnet 4.5, the full VeRO harness lifts average score to 0.61, +8 points over the 0.53 no-VeRO Claude Code Pure baseline (+11 over the raw 0.50 initial-commit score) — versus only +2 for Claude Code given just VeRO's tools, and ~0 for prompt-only optimization.

The harness — not the model — is what drives gains

The external baseline GEPA reaches 0.54, essentially tied with VeRO's prompt-restricted Resources Only variant (also 0.54) and well below Default (0.61). Since GEPA and Resources Only mutate the same components (prompts, tool descriptions, parameters), their convergence pins the +7-point gap to Default on the broader harness changes — adding tools, restructuring control flow, delegating to sub-agents — that only the full harness permits.

Gains are tool-use-shaped, not reasoning-shaped

The improvement is sharply task-dependent. Tool-use tasks see consistent lift — mean lift of +0.09 on GAIA, +0.09 on TAU-Bench Retail, and +0.08 on SimpleQA — while reasoning-heavy tasks barely move: +0.03 on GPQA and +0.00 on MATH. A budget ablation across B ∈ {2, 4, 8, 16, 32} shows GPQA and MATH stay flat throughout, ruling out "not enough budget" as the explanation. The authors read this as a feature, not a bug: agent-driven optimization pays off exactly on the tasks that demand richer agentic behavior.

Transfer, complexity, and what the optimizer actually does

  • Transfer is family-bound (robustness study). Taking best commits optimized against GPT-4.1 mini and swapping in new target models, gains hold within-family (a Sonnet-optimized TAU-Bench Retail agent transfers to +0.22 on Sonnet-4.5 and +0.24 on GPT-4.1) but degrade out-of-family — GPT-5.2-optimized GAIA commits regress on Gemini-Flash and Qwen3.
  • Headroom shrinks with sophistication (case study). On GAIA, optimizing a minimal "Pawn" agent (4 tools, 25-line prompt) yields +11.5% / +10.5% / +13.3% on GAIA / FACTS / SimpleQA, but a sophisticated "Knight" (6 tools, 140-line ReAct prompt) gains only +6.9% / +5.6% / +4.5%. And prescriptive guidance helps the Pawn but hurts the Knight, which does best under a Minimal template that cuts its prompt 65% — sophistication wants creative freedom, not a cookbook. Optimizing for one task doesn't transfer for free: one Pawn iteration gained +5.75% on GAIA while regressing −17.8% on SimpleQA, by adding a verification tool that helped multi-hop reasoning but bloated simple lookups.
  • It scales to long-horizon coding. On TerminalBench-2 (89 terminal tasks, Terminus-KIRA + Claude Haiku 4.5), VeRO lifts pass rate from a 30.3% baseline to 37.1% (+6.8pp at B=178), and one run independently re-discovered a structural type-checking bug in the agent's tool-call parser responsible for 11 of 41 baseline crashes.
  • Prompts dominate the edit distribution (interpretability). Tagging every change, prompt edits make up >50% of modifications in all phases beyond the first. The Orchestrator variant keeps the most diverse edit mix; across all variants, change-type entropy collapses after the first phase — agents fall back to prompt tweaks once their more ambitious structural edits fail to pay off.
// does it hold up?

The eval read

VeRO's strongest move is methodological: including GEPA as an external baseline that mutates the same components as the Resources-Only variant cleanly isolates "better search algorithm" from "bigger search space," and the convergence of the two (both 0.54) is the paper's most convincing single result. But the quantitative rankings rest on a softer foundation than the table suggests.

The fairness claim at the center of the design — "no optimizer gains advantage through extra compute" — is undercut by the authors' own budget definition: B counts only ExperimentRunner invocations, not the optimizer's tokens, tool calls, or API spend. Two optimizers at the same nE=8 can burn wildly different compute, so the +8-point "harness lift" partly reflects that VeRO-Agent Default simply has more scaffolding (sub-agent delegation, a Cookbook of design patterns) than Claude Code Pure — more machinery, not provably better search at equal cost.

Statistical strength is thin: the benchmark runs N=3 and captures optimizer stochasticity but not evaluation stochasticity (each commit is scored once, on a temperature-1.0, GPT-4.1-mini target). The case studies are N=1 (TerminalBench-2) and N=4 (GAIA). At that resolution, a "+2% vs 0% best" difference for VeRO Tools sits comfortably inside the noise — the authors themselves recommend multiple evaluations per commit as future work.

baselines

GEPA is a genuinely fair external baseline (same component search space as Resources Only). But the optimizer-model comparison is lopsided: Sonnet 4.5 runs 75 of 120 experiments; Opus and GPT-5.2-Codex appear only on the Orchestrator variant (15 runs each), so "no single model dominates" rests on a thin slice.

contamination

GPQA, MATH, and SimpleQA are public. Permission control hides the test split from the optimizer but not the target model's pretrained knowledge or web-search tools fetching leaked answers. SimpleQA is hardened (filtered to queries all three frontier models fail zero-shot); GPQA and MATH are not, and GPQA has no validation split, so its best commits are train-selected — an overfitting risk the authors flag.

what's missed

No human baseline: there's no measure of how a human engineer would optimize the same agent under the same budget, so an "+8% lift" can't be calibrated as good or trivial. And because B ignores tokens/cost, the harness-vs-no-harness comparison conflates capability with compute. External APIs (Wikipedia, web search) inject uncontrolled temporal noise into scores.

would raise confidence

Multiple evaluations per commit with confidence intervals; multi-dimensional budgets (tokens, wall-clock, API cost); human optimizer baselines; and contamination guards across all tasks (cached API snapshots, the SimpleQA-style frontier-failure filter applied uniformly, explicit memorization detection).

// bottom line

The lasting contribution is infrastructure: a reproducible, permission-controlled, budget-gated harness that turns "agents optimizing agents" into something you can actually measure and compare — plus a clean, believable finding that today's gains live in tool-use scaffolding, not reasoning. Treat the precise lifts (+2% vs +8%) as directional: single-eval scoring, a budget that ignores tokens, and uneven contamination control mean the rankings point the right way without being definitive.

// takeaways

What to remember

  1. VeRO turns "agents optimizing agents" into a controlled task — versioned git snapshots, a gated evaluation budget B, permission control over test data and files, and standardized trace observations — so different coding agents can be compared without one cheating on compute or peeking at test data.
  2. The harness, not the model, drives the gains: holding Sonnet 4.5 fixed, the full VeRO-Agent reaches 0.61 — +8 over the 0.53 no-VeRO Claude Code Pure baseline (+11 over the raw 0.50 initial commit) — versus +2 for Claude Code with VeRO tools alone and ~0 for prompt-only optimizers (GEPA and Resources Only both land at 0.54).
  3. Optimization is tool-use-shaped: ~+0.08–0.09 mean lift on GAIA / TAU-Bench Retail / SimpleQA but ~0 on GPQA / MATH (flat across budgets 2–32), and headroom shrinks as the base agent gets more sophisticated — a minimal Pawn gains +11–13% where a polished Knight gains only +4–7%.
