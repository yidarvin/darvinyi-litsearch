Planning In Natural Language Improves LLM Search For Code Generation

Scaling training compute made LLMs far better; scaling inference compute mostly didn't — because models re-sample near-identical wrong solutions. PLANSEARCH searches over diverse natural-language plans before writing any code, lifting Claude 3.5 Sonnet from a 41.4% best no-search pass@1 to 77.0% pass@200 on contamination-controlled LiveCodeBench.

Wang, Cassano, Wu, Bai, Song, Nath, Han, Hendryx, Yue, Zhang — Scale AI

// the gap

What problem is this solving?

The "bitter lesson" says two things scale better than everything else: learning and search. Modern LLMs have ridden the learning curve hard — but search, the workhorse behind superhuman Go, poker, and chess, has stubbornly refused to pay off for language models. Drawing more samples from a model at higher temperature (repeated sampling) buys only a slow, roughly logarithmic rise in pass@k. The paper's diagnosis: the bottleneck isn't the search procedure, it's a lack of high-level diversity in what the model produces. Sampled 200 times, an LLM tends to emit 200 minor variations of the same idea — so the search keeps re-submitting the same wrong solution with cosmetic edits.

Why so little diversity? The authors point at post-training. Instruction tuning and preference optimization (RLHF, DPO) reward producing the single most-likely correct answer, which collapses the model's output distribution around one mode. The tell-tale sign: base models frequently overtake their instruct counterparts at large k even though they lose at pass@1 — e.g. DeepSeek-Coder-V2-Lite-Base starts below its instruct version but crosses above it as k grows, because the base model still carries more idea diversity (Figure 3).

Code generation is the ideal place to study this, because it sidesteps the other hard half of search — selection. For open-ended tasks you must both generate good candidates and somehow pick the winner; for code, any candidate can simply be run against test cases. So if search underperforms here, the culprit is generation diversity, not verification. Two motivating experiments pin down where the diversity should live:

  • Backtranslation. Take a correct solution, compress it back into a short natural-language sketch, and hand that sketch to the model. Even a 10-token sketch sharply increases solve rate, and accuracy keeps climbing with sketch length (Figure 4a) — so possessing the right idea is most of the battle.
  • Conditioning on an idea. When you fix a sketch and sample many implementations, solve rates polarize toward 0% or 100% (Figure 4b): whether a problem gets solved is largely determined by whether the model lands on a correct idea, not by implementation noise.

The conclusion that organizes the whole paper: don't search over tokens or over finished programs — search over ideas, expressed in natural language, the medium LLMs were trained to reason in.

// how it works

The method

PLANSEARCH is the top rung of a three-method ladder, each one searching in a more abstract space than the last:

  • Repeated Sampling — the baseline. Few-shot prompt the model and sample n code solutions directly; pass@k measures whether any of the first k pass.
  • IdeaSearch — don't ask for code first. Ask the model for a natural-language solution S to problem P; then a separate instance is given P and S and told to implement that plan. Splitting the idea and the code into two responses isolates the value of a good idea — and empirically beats plain chain-of-thought, which folds reasoning and code into one response.
  • PLANSEARCH — search explicitly over a combinatorial tree of plans.

Stage 1 — generate diverse observations

Starting from the problem statement P, prompt the model for a set of "observations" or hints — typically 3 to 6 first-order observations O¹. Then form every subset of size at most S = 2 of those observations; each subset is a combination C¹. This defines a depth-1 tree rooted at P. The procedure is repeated on each leaf: feed the model P plus the observations in a combination, framed as primitives needed to solve P, and ask it to merge them into new, second-order observations. The tree is truncated at depth L = 2 for cost. Crucially, the observations need not be correct — many are wrong on purpose. Their only job is to push the model toward a more diverse set of ideas.

Stage 2 — turn observations into code

For each leaf, the model is prompted with all the accumulated observations and the original problem to write a natural-language solution. To squeeze out more diversity, every idea is doubled: the model is told to assume its own idea is wrong and to produce a critique, yielding an adversarial second idea. Each idea is then translated idea → pseudocode → Python in deliberate, granular steps, so the model doesn't quietly revert to its default solution and discard the reasoning. A single run of PLANSEARCH ends up generating a variable pool of roughly 300–400 candidate programs.

Measuring idea diversity

The paper's analytical centerpiece is a way to quantify idea diversity that plain entropy misses (two models can share an entropy while one varies the idea and the other only the surface form). Given a pool of n generated programs, construct all C(n,2) pairs and ask an LLM judge for a binary similarity verdict S(cᵢ,cⱼ) ∈ {0,1} — judged by backtranslating each program to natural language and comparing the ideas. The diversity score is one minus the fraction of similar pairs:

D = 1 − Σi<j S(cᵢ, cⱼ)C(n, 2) // D = 0 if every program implements the same idea; D = 1 if all ideas are distinct. For large n, 40 codes are sampled and all pairs compared; GPT-4o-mini is the judge.

Equivalently, D is the probability that two randomly chosen programs from the method are judged to have different ideas — a clean scalar for how much room a search method has to find a winner.

// what they found

The results

Across four models (GPT-4o-mini, GPT-4o, DeepSeek-Coder-V2, Claude 3.5 Sonnet) and three benchmarks (LiveCodeBench, HumanEval+, MBPP+), PLANSEARCH beats both repeated sampling and IdeaSearch at pass@200, everywhere. The headline lives on LiveCodeBench, the contamination-controlled competitive-programming set: PLANSEARCH on Claude 3.5 Sonnet hits 77.0% pass@200 — nearly double the best pass@1 achievable by any model without search (41.4%, DeepSeek-Coder-V2), and well above the best non-search pass@200 (60.6%, GPT-4o repeated sampling).

A few details that matter:

  • Already-saturated benchmarks still move. On HumanEval+, PLANSEARCH takes GPT-4o from 86.4 → 99.5 and Sonnet from 81.6 → 98.5 pass@200; on MBPP+, Sonnet goes 77.1 → 93.7.
  • Against a search-native model the edge shrinks. o1-mini already spends inference compute, so PLANSEARCH@200 (91.3) only barely tops its repeated-sampling@200 (90.8) — evidence that PLANSEARCH is buying the same thing o1's training already bought.
  • Filtering makes it deployable. Public-test filtering shifts the pass@k curve ~10× leftward: PLANSEARCH reaches 77.1% at pass@20, matching its 77.0% pass@200 without filtering, and beats filtered repeated sampling by a flat 30–40% even at 10 completions (Figure 5).
  • It can hurt pass@1. Diversifying ideas dilutes the probability of any single idea, so PLANSEARCH often lowers pass@1 (most notably Sonnet on LiveCodeBench) while raising the chance that some idea in the pool is correct.

The deepest result isn't a leaderboard number — it's a law. Plotting each (model, method, benchmark)'s measured idea-diversity against its relative pass@1→pass@200 gain produces a tight positive correlation (Figure 6). Diversity predicts how much search will help, full stop. PLANSEARCH sits at the high-diversity, high-gain corner (Sonnet at D ≈ 0.59, relative gain ≈ 1.78), and the search-native o1-mini falls on the same line.

// does it hold up?

The eval read

The contamination story is the paper's strongest methodological move, and it's done right: LiveCodeBench is restricted to problems from May–September 2024, after Claude 3.5 Sonnet's April 2024 knowledge cutoff, with the same window applied to every model. That's exactly why the LiveCodeBench result carries the argument while HumanEval+/MBPP+ (which the authors openly treat as saturated) are along for the ride. Including o1-mini as a search-native baseline is also commendably honest — it's the data point that most undercuts the method's apparent magic, and they show it anyway.

But the headline number is an oracle metric. pass@200 = 77.0% means "a correct solution exists among 200 samples" — recovering it requires a verifier (the hidden test suite) to do the selection. Without test-based selection that 77.0 is an upper bound on recoverable accuracy, not delivered accuracy, and PLANSEARCH frequently lowers pass@1. The paper's answer — public-test filtering brings pass@20 up to 77.1% — is the right instinct, but it leans on each problem shipping usable public tests, and the main tables are normalized per attempt, not per token. Since PLANSEARCH spends a whole observation tree before each solution, the attempt-matched axis flatters it; the compute-matched comparison (Appendix D) is the fairer one, and the authors concede PLANSEARCH only wins there "if sufficient compute is expended."

The diversity law deserves a skeptical read too. The diversity score is itself LLM-judged (GPT-4o-mini deciding whether two ideas are "the same"), and it's measured on the same problems whose gains it predicts — so "diversity predicts search gains" is partly a statement about one model family's internal notion of sameness, not a model-independent quantity. And the pass@k unbiased estimator assumes successes are independent; the authors note (Appendix O) that PLANSEARCH's generations are explicitly not independent, so its pass@k figures rest on a knowingly violated assumption.

baselines

Fair in spirit (equal-k across methods) and unusually honest in including search-native o1-mini, where the gain nearly vanishes (91.3 vs 90.8). Weak spot: headline numbers are attempt-matched, not compute-matched, which favors a method that plans before every solution.

contamination

Handled well — the load-bearing benchmark (LiveCodeBench, May–Sept 2024) post-dates the strongest model's cutoff, same window for all. HumanEval+/MBPP+ are acknowledged as saturated.

what's missed

pass@200 is an oracle metric needing a verifier to extract the right sample; it overstates deployable accuracy and PLANSEARCH often hurts pass@1. The diversity metric is LLM-judged and self-referential, and pass@k assumes an independence PLANSEARCH violates.

would raise confidence

Report verifier-selected / filtered pass@1 head-to-head as the deployable number; use a diversity judge outside the generation family; add frontier reasoning models (o1/o3) where training may already fix diversity; and present a cost-matched (tokens or $) frontier rather than attempt-matched.

// bottom line

A clean, well-motivated demonstration that the bottleneck to inference-time search for code is idea diversity, not verification — and that searching in natural-language plan space is a cheap, model-agnostic way to restore it. The 77.0% pass@200 is real and contamination-guarded, but it's an oracle figure; the durable contributions are the diversity-predicts-search-gains relationship and the plan-space recipe, both of which point at post-training as the place to fix diversity for good.

// takeaways

What to remember

  1. Repeated sampling stalls because instruct/RLHF training collapses output diversity — the model re-samples the same idea. The fix is to search where the diversity actually lives: natural-language plans, not tokens or finished code.
  2. PLANSEARCH builds a depth-2 tree of combinatorial "observations," expands each leaf into a plan (plus an adversarial "assume it's wrong" variant), then translates plan → pseudocode → code — lifting Claude 3.5 Sonnet from a 41.4% best no-search pass@1 to 77.0% pass@200 on contamination-controlled LiveCodeBench.
  3. A single LLM-judged idea-diversity score predicts how much any search method will gain from scaling samples, turning diversity into a measurable design target — and suggesting the real win is re-architecting post-training around search rather than patching it at inference time.
