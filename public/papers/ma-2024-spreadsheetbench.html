SpreadsheetBench: Towards Challenging Real World Spreadsheet Manipulation

Most spreadsheet benchmarks test toy tables with made-up instructions. SpreadsheetBench instead scrapes 912 real questions off Excel forums, keeps the spreadsheets as messy as users left them, and grades each model's code online-judge style — one solution must pass several input–output spreadsheets. The result: a 2024 frontier stuck near 18% while Excel experts hit 71%.

What problem is this solving?

Spreadsheet agents like SheetCopilot and SheetAgent promise to turn a plain-English request into the formulas or VBA that does the work. But the benchmarks used to judge them flatter the models, and the paper pins this on three concrete failures.

The instructions are fake or trivial. Prior benchmarks either synthesize queries with self-instruct (SheetCopilotBench, SheetRM) or have crowdworkers write short prompts for a given sheet (InstructExcel). The tell is length: average instruction words run 27.9 (SheetCopilotBench) and 9.8 (InstructExcel) versus 85.7 here. Real forum questions carry context a synthetic prompt never does — the user's previous attempts, the wrong answers they got, the functions they already tried (=VALUE, =SUM, =SUMPRODUCT), and a worked example of the output they want.

The spreadsheets are too clean. Existing benchmarks use a single regular relational table, much like the inputs to TableQA or Text2SQL. Real spreadsheets have no fixed boundaries: multiple tables in one sheet, non-standard tables with nested / incomplete / missing headers, free-form text inside a cell, and non-textual signal like color and bold that the task may depend on.

One test per instruction is a leaky ruler. Grading a single (spreadsheet → answer) pair rewards a solution that happens to be right for that one sheet but breaks the moment the data changes — a false positive. A robust solution should keep working when values shift, including on corner cases.

// how it works

The method

The task is deliberately framed around code, not answers. The dataset is D = {(qᵢ, cᵢ, aᵢ)}: an instruction qᵢ, a spreadsheet cᵢ, and the ground-truth modified spreadsheet aᵢ. A model must emit a code solution sᵢ that, applied to cᵢ, produces âᵢ — graded against aᵢ. Instructions are either cell-level (touch specific cells, e.g. D2:D6) or sheet-level (rewrite a whole range, e.g. A2:D6); the target range is annotated into the instruction so answers land where the grader looks.

The corpus comes from four high-ranked Excel forums and blogs — ExcelForum, Chandoo, MrExcel, ExcelGuru — through a four-stage pipeline. (1) Sourcing pulls posts tagged "solved" (GPT-4 infers the tag where it's missing, from cues like closing thank-you notes), across categories like Formula, VBA & Macros, Pivot Table, and Power Query. (2) Filtering keeps only questions that are solved, purely about spreadsheet manipulation (software-specific UI questions about input boxes / buttons / forms are dropped), feasible & testable, and representative. (3) Formatting has GPT-4 condense the answer — often scattered across several replies — into one coherent self-contained instruction, which annotators then verify and tag with the answer position. (4) Test-case construction is the heart of it: annotators take the original sheet, then hand-edit it twice into two more sheets — deliberately seeding real-world corner cases (e.g. a cell that should hold a zero but is left blank) — and apply the same human solution to each, yielding three (input, output) test cases per instruction.

Three perturbations make the set leakage-resistant by construction: instructions are GPT-4-rewritten (so a model can't recognize the original post), spreadsheets are modified (so memorized files don't help), and answer positions are moved (so a copied forum solution writes to the wrong place). All files are kept in software-independent XLSX and models manipulate them with plain Python rather than the Excel API. A team of 20 Excel specialists annotated it, with two validators and two authors doing two rounds of QA.

Grading borrows the online-judge (OJ) idea from competitive programming: one solution is run against all of an instruction's test cases. Two scores bracket the result — a lenient IOI-style soft restriction that gives partial credit per passed test case, and a strict ICPC-style hard restriction that pays only if every test case passes.

Ssoft = 1|D| Σi=1|D| 1|Ti| Σj=1|Ti| 𝟙rij=ACC      Shard = 1|D| Σi=1|D| 𝟙rij=ACC, ∀j // soft = fraction of test cases passed (partial credit); hard = all-or-nothing per instruction

Crucially this needs one model inference per instruction — the generated code is reused across all three test cases — unlike data-perturbation benchmarks that re-run the model on every variant.

What the 912 instructions actually look like

The final set is 912 instructions and 2,729 test cases (avg 3 per instruction), spanning ten manipulation verbs — find, extract, sum, highlight, remove, modify, count, delete, calculate, display. The sheets are large and irregular: 80% sit within 49 rows and 13 columns, but the long tail runs past 100 columns and 20,000 rows; 35.7% contain multiple tables and 42.7% use non-standard (nested / incomplete / missing-header) tables. Against prior work the contrast is stark — SpreadsheetBench's 2,729 spreadsheet files and 85.7-word instructions dwarf SheetCopilotBench (221 instructions / 29 files), InstructExcel (4,850 / 940 but 9.8 words each), and SheetRM (201 / 25).

The results

Across every model class — fine-tuned TableQA models, open code and general LLMs, closed frontier models, and shipping Excel products — performance is poor and spread wide, from 0.05% to 23.65% soft-restriction accuracy. The strongest LLM, GPT-4o, scores 18.35% overall (15.03% cell-level, 23.65% sheet-level) on soft restriction and 15.02% on hard. The best product, Copilot in Excel, reaches ~20%, with SheetCopilot (GPT-4) at 14% — though both, along with the human baseline, were measured on small subsets, not the full 912. Excel experts score 71.33% soft (62.00% hard). The TableQA models TaPEx and TaPas score zero and are omitted, underscoring that spreadsheet manipulation is a different beast from table QA.

Three patterns emerge. Coding ability beats general ability: DeepseekCoder-33B (2.60) tops the larger general Llama-3-70B (1.32), because solutions are fundamentally code. Multi-round helps almost everyone — except GPT-4o: giving models ReAct-style turns to read the sheet and see code-execution errors lifts most by up to ~10× (GPT-3.5 2.34 → 7.09), but GPT-4o actually drops (18.35 → 16.96). The paper's explanation: GPT-4o dutifully follows the "retrieve the spreadsheet" instruction and re-reads rows already in its prompt, duplicating context, whereas weaker models ignore the retrieval step and just use the initial rows. Hard barely trails soft (GPT-4o 18.35 → 15.02), meaning models rarely produce a half-robust solution — they tend to pass all three test cases or none.

Difficulty also scales cleanly with spreadsheet complexity. On GPT-4o, accuracy falls from 20.63% on sheets ≤50 rows to 11.50% above, from 22.50% on ≤10 columns to 10.51% above, from 21.12% on single-table to 13.71% on multi-table, and from 19.50% on relational to 16.95% on non-standard tables. And simply feeding more rows into the prompt doesn't fix it — going from 5 to 10 input rows barely moves GPT-4o, likely a context-length effect.

// does it hold up?

The eval read

SpreadsheetBench's design is genuinely good — the weak points are about the numbers, not the construction. The single biggest caveat is that the leaderboard is a mid-2024 snapshot: the model roster is GPT-4o, GPT-3.5, Llama-3-70B, Mixtral-8×7B, DeepseekCoder-33B, CodeQwen-7B — no reasoning models (o1/o3), no Claude, no Gemini, nothing from the 2025–26 frontier. The "18% ceiling" headline says more about 2024 than about the task's intrinsic difficulty, which is exactly why later work reuses the benchmark while re-measuring it with newer agents.

The other soft spot is the baseline apples-to-oranges. GPT-4o's 18.35% is on the full 912 instructions, but the two product baselines (Copilot in Excel ~20%, SheetCopilot 14%) and the human 71.33% were measured on small subsets — a 5% sample and 50 instructions respectively — and the products were graded by hand inside their own UIs rather than through the reproducible Python pipeline. The paper is upfront about the cost reasons, but it means the marquee "20% product vs 71% human vs 18% GPT-4o" comparison isn't strictly like-for-like.

baselines

Strong on breadth (TableQA, open code/general, closed, products) but frozen at mid-2024 — no reasoning/Claude/Gemini models; products and humans scored on subsets and via different harnesses, so cross-baseline gaps are directional.

contamination

A real strength and the design's whole point: GPT-4-rewritten instructions, twice-modified spreadsheets, and moved answer positions defeat memorized forum solutions, and the multi-test-case grader catches single-sheet overfitting. Caveat: GPT-4 rewriting may smooth real-user phrasing toward model-friendly text.

what's missed

Grading is exact-match at annotator-placed answer positions, so a correct result written slightly off-target can read as wrong; soft and hard scores barely diverge, hinting the 3-test-case design is too coarse to grade robustness finely (the authors concede they "did not meticulously devise corner cases").

would raise confidence

Re-run current frontier + reasoning models on the full set; score every product and the human baseline on the same 912 through one harness; add more test cases per instruction toward true OJ density; report inter-annotator agreement on answer-position placement.

// bottom line

The lasting contribution is methodological: real-forum instructions plus OJ-style multi-spreadsheet grading is a harder, leakage-resistant way to test spreadsheet manipulation, and it surfaced a wide human–model gap that single-test benchmarks hid. Treat the specific 18%-vs-71% numbers as a 2024 snapshot with mismatched sample sizes, not a fixed law — the benchmark is the durable part.

// takeaways

What to remember

  1. Real forum questions average 85.7 words (vs 9.8–27.9 in prior benchmarks) and the spreadsheets are authentically messy — 35.7% multi-table, 42.7% non-standard headers, up to 100+ columns and 20,000 rows. That realism is the difficulty.
  2. The online-judge metric — one code solution must pass three input–output spreadsheet test cases — is the core idea: it kills solutions that overfit a single sheet and makes the benchmark leakage-resistant by construction.
  3. In 2024 the field topped out near 18–20% (GPT-4o 18.35%, Copilot in Excel ~20%) against 71.33% for Excel experts; coding ability mattered more than general ability, and multi-round execution feedback helped every model except GPT-4o.
