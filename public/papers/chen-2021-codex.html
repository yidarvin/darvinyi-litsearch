Evaluating Large Language Models Trained on Code

OpenAI fine-tunes GPT on public GitHub Python and introduces HumanEval — 164 hand-written problems graded not by string-matching but by running the code against hidden unit tests. Codex-12B solves 28.8% from a single sample; with 100 samples and an oracle pick, 77.5%. This is the paper that made functional correctness and pass@k the measurement substrate of the coding-agent era.

// the gap

What problem is this solving?

By 2021, generative models for code were almost always benchmarked by matching the generated text against a reference solution — exact match, or a fuzzy overlap score like BLEU. That measurement is fundamentally mismatched to the object being measured. A program can be completely correct yet share almost no tokens with the reference (there are vastly many functionally-equivalent implementations of any spec), and a program can overlap heavily with the reference yet be subtly, totally wrong. The authors point to prior work (Ren et al., 2020) finding that BLEU fails to capture code-specific semantics, and argue that match-based metrics simply cannot account for the large, complex space of programs functionally equivalent to a reference solution.

At the same time, OpenAI's early probing of GPT-3 revealed something suggestive: even a model never explicitly trained for code could draft simple programs from Python docstrings. That hinted a code-specialized GPT might be far more capable — but there was no rigorous, contamination-resistant way to measure how capable. The field needed an evaluation grounded in the only ground truth that matters for code: does it run and do what was asked?

// how it works

The method

Codex is a GPT language model fine-tuned on publicly available code from GitHub — hundreds of millions of lines of Python. A second variant, Codex-S, is further fine-tuned (supervised) on a curated set of correctly-implemented, standalone functions. A distinct production descendant of these models powers GitHub Copilot. The models range from 12M up to 12B non-embedding parameters, letting the paper trace capability against scale.

The evaluation contribution is HumanEval: a set of 164 original programming problems written by hand specifically so they would not appear in the training corpus — a deliberate contamination control. Each problem ships as a function signature, a docstring (the natural-language spec the model conditions on), a reference body, and a battery of unit tests (an average of 7.7 tests per problem). The problems probe language comprehension, algorithms, and simple mathematics — roughly the difficulty of easy software-interview questions.

Grading is functional correctness, not text similarity: the model samples one or more completions; each is executed against the problem's hidden unit tests; a problem counts as solved if a sample passes all of them. This is the move that reframed the whole field — evaluate code the way an engineer would (run it), not the way a translation system is scored (compare strings).

Measuring with pass@k — without fooling yourself

Because sampling is stochastic, the natural metric is pass@k: generate k samples and check whether any of them passes. But estimating this naïvely is treacherous. Computing pass@k by literally drawing k samples per problem has high variance, and the tempting shortcut 1 − (1 − p̂)k (built from the empirical pass@1 rate p̂) is biased. Instead, Codex draws a larger pool of n samples (here n = 200, k ≤ 100), counts the number c that pass, and uses an unbiased combinatorial estimator:

pass@k := Eproblems [ 1 − C(n − c, k)C(n, k) ] // n = 200 samples per task, c = number that pass the unit tests, k ≤ 100 — the probability that a random size-k subset contains no correct sample, subtracted from 1. Evaluated with a numerically stable product to avoid huge binomials.

One more subtlety the paper surfaces: the sampling temperature must be tuned to the value of k. Low temperature (≈0.2) concentrates probability mass and is best when you only get one shot (pass@1); higher temperature (≈0.8) injects the diversity that pays off when you can draw many samples (large k).

// what they found

The results

From a single sample, a 12B-parameter Codex solves 28.8% of HumanEval, while GPT-3 — never fine-tuned on code — solves 0% and the 6B open GPT-J solves 11.4%. Capability scales smoothly with size: a 300M Codex already reaches 13.2%. Supervised fine-tuning helps further: Codex-S-12B reaches 37.7% pass@1.

The paper's most influential empirical claim is that repeated sampling is a surprisingly strong strategy. Generating 100 samples per problem and accepting the problem if any sample passes lifts Codex to 70.2%. For Codex-S, an oracle that always picks a passing sample reaches 77.5% — meaning the model can produce a correct solution for more than three-quarters of the problems; the hard part is selecting it. A cheap heuristic — keep the sample with the highest mean log-probability — recovers 44.5%, far above pass@1 but well short of the oracle.

Two scaling regularities round out the picture. After code fine-tuning, test loss follows a clean power law, (N / 5.92×10⁷)−0.13 in non-embedding parameters N — the same Kaplan-style scaling seen in language modeling, now for code. And the temperature sweep confirms the sampling intuition: the temperature that maximizes pass@k rises monotonically with k, so the optimal way to sample depends on how many shots you are allowed.

// does it hold up?

The eval read

The durable contribution here is not the 164 problems — it is the measurement design: execution-graded functional correctness plus an unbiased pass@k estimator. That reframing ("run the code, don't match the tokens") is the foundation under nearly every serious code and agent benchmark since. Judged as a 2021 measurement instrument, though, HumanEval has real limits, several of which the paper names itself.

First, scope. HumanEval is small (164 tasks) and narrow: self-contained, single-function, short-docstring algorithmic puzzles. It measures "write a small function from a clean spec," which is a long way from real software engineering — multi-file, stateful, repository-scale change driven by a messy issue. Saturating HumanEval therefore does not mean a model is a good engineer; that exact gap is what SWE-bench later opened.

Second, the headline-number trap. pass@100 = 70.2% and the 77.5% oracle are genuinely impressive, but they describe a setting you cannot deploy: they assume a free, perfect verifier that already knows which sample passes. The honest number for "one answer, chosen automatically" is the 44.5% from mean-logp reranking — a 33-point gap the oracle hides. Reporting oracle pass@k as the headline, as later code papers often did, overstates real-world capability.

Third, test adequacy and contamination. Hand-writing the problems to avoid training overlap was forward-thinking, but the training set is "hundreds of millions of lines of public GitHub," and over the following years HumanEval's own solutions leaked into web corpora. Worse, the tests are sparse (≈7.7 per problem) — later work (EvalPlus) showed those tests under-constrain correctness and inflate pass rates, and contamination-resistant successors (LiveBench) emerged precisely because static, public sets like this decay. The model's self-reported weaknesses sharpen the picture: pass rate drops by roughly a factor of 2–3 with each additional chained operation in a synthetic 13-building-block docstring (exponential degradation no competent human would show), it makes variable-binding errors as operations multiply, and it is wildly sample-inefficient — a strong intro-CS student solves a larger fraction than Codex-12B after seeing a tiny fraction of the code.

baselines

Fair contemporaries (GPT-3 at 0%, GPT-J 6B at 11.4%), but in 2021 no other code-specialized model existed, so Codex is largely compared against itself across scale rather than against rival systems.

contamination

Problems hand-written to avoid training overlap — good hygiene for the era — but HumanEval later leaked into web/training corpora, and its sparse tests (~7.7/problem) were shown to under-constrain correctness (EvalPlus).

what's missed

Single-function algorithmic puzzles ≠ real engineering; the deployable metric is pass@1-with-reranking (44.5%), not the oracle pass@100 (77.5%) that headlines tend to quote.

would raise confidence

Denser, harder test suites; repository-scale and multi-file tasks; reporting realistic single-sample selection as the headline; continually-refreshed, held-out problem sets.

// bottom line

By today's standards HumanEval is small, partly saturated, and contamination-prone — but the paper's legacy was never the 164 problems. It is the shift to execution-graded functional correctness and the pass@k estimator: the measurement vocabulary the entire coding-agent line (SWE-bench, SWE-Bench Pro, SWE Atlas, MCP-Atlas) still speaks.

// takeaways

What to remember

  1. Codex — GPT fine-tuned on GitHub Python — solves 28.8% of HumanEval from a single sample (GPT-3: 0%, GPT-J: 11.4%); with 100 samples an oracle reaches 77.5%, establishing repeated sampling as a powerful strategy for hard coding prompts.
  2. The lasting contribution is methodological: 164 hand-written problems graded by running unit tests, plus the unbiased pass@k estimator (n=200, k≤100). Execution-based functional correctness replaced BLEU and exact-match for code.
  3. The honest capability ceiling is pass@1 with realistic reranking (44.5%), not oracle pass@100 (77.5%) — and the benchmark is single-function puzzles, not engineering. Those gaps motivated EvalPlus, LiveBench, and the repo-scale SWE-bench lineage that followed.
