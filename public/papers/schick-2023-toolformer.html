Toolformer: Language Models Can Teach Themselves to Use Tools

Teach a language model to use a calculator, a search engine, a translator and a calendar — without a single human annotation. Toolformer annotates its own pretraining text with candidate API calls, keeps only the calls that make the next tokens easier to predict, and finetunes on the survivors. A 6.7B GPT-J learns when and how to call tools, and beats GPT-3 (175B) zero-shot on factual and math tasks.

Timo Schick, Jane Dwivedi-Yu, Roberto Dessì, Roberta Raileanu, Maria Lomeli, Luke Zettlemoyer, Nicola Cancedda, Thomas Scialom · Meta AI Research

What problem is this solving?

Large language models are remarkable few-shot learners and yet, paradoxically, fail at things a pocket calculator or a database nails: exact arithmetic, up-to-date factual lookup, low-resource translation, and knowing what today's date is. These are not bugs that more scale cleanly fixes — they are structural. A frozen-weights model hallucinates facts it never memorized, cannot know events after its training cutoff, and computes 658,893 / 11.4% by guessing rather than dividing.

The obvious fix is to let the model call external tools. But in early 2023 the two known ways to do that both had a catch. One camp (WebGPT, LaMDA, internet-augmented dialogue) trained tool use from large amounts of human supervision — expensive, and biased toward what humans think is useful rather than what the model finds useful. The other camp (PAL, ReAct, few-shot retrieval) used task-specific prompting, where you already know in advance which tool solves the task and hand it a demonstration. Neither gives you a general model that decides for itself, across arbitrary text, which of many tools to invoke, when, and with what arguments.

Toolformer's target is exactly that gap: tool use that is learned self-supervised (a handful of demos per API, no human annotation of where calls belong) and general (not tied to one task, and not costing the model its core language-modeling ability).

// how it works

The method

Every API call is written as plain text the model can emit: a question-answering call looks like [QA("Who published NEJM?") → Massachusetts Medical Society]. The start marker, end marker and the → separator are implemented as ordinary token sequences ([, ], ->), so nothing about the model's vocabulary changes. The whole trick is then a way to build a training set where these calls appear in exactly the places they help.

Annotate, then filter on your own loss

Starting from a plain-text corpus (a subset of CCNet) and a pretrained GPT-J (6.7B), Toolformer converts the corpus into an augmented version C* in three steps:

  • Sample. For each tool, a short few-shot prompt P(x) shows the model how the API is called. The model proposes positions where an API call is likely (the probability assigned to the <API> token exceeds a threshold τs), keeps the top k, and samples up to m candidate calls at each.
  • Execute. Each candidate call is actually run against the real tool to get a result r — calling another network, running a Python script, or querying a retrieval index.
  • Filter. Keep a call only if it earns its place: inserting the call and its result must reduce the model's weighted cross-entropy over the following tokens.

The filtering criterion is the heart of the paper. Let Li(z) be the weighted loss over tokens xi…xn when the model is prefixed with z. A call ci with result ri survives when it beats the better of doing nothing and doing the call without seeing its answer:

Li− − Li+  ≥  τf // Li+ = Li(e(ci, ri))  ·  Li− = min( Li(ε), Li(e(ci, ε)) )  —  keep the call iff its answer cuts the loss by at least τf

The surviving calls are interleaved back into the original text, producing C* — the same text as the corpus, plus a sprinkling of useful, self-vetted API calls. The model is then finetuned on C* with a standard language-modeling objective. Because the augmented corpus differs from the original only by helpful calls, the model keeps its general ability while learning to call tools where they pay off. At inference, decoding proceeds normally until the model emits →; decoding pauses, the API is called, the response is spliced in, and generation continues.

The five tools span the model's weak spots: a question-answering system (the retrieval-augmented Atlas finetuned on Natural Questions), a Wikipedia search engine (a BM25 retriever over the KILT dump), a calculator (a Python script, four operations, rounded to two decimals), a 600M-parameter NLLB machine-translation model (any of 200 languages into English), and a calendar that returns the current date.

// what they found

The results

All comparisons are zero-shot (models are instructed in natural language but given no in-context examples). The same-size baselines are plain GPT-J, GPT-J finetuned on the corpus without calls (GPT-J + CC), and "Toolformer (disabled)" — the trained model with API calls switched off at decode time. The headline: Toolformer (6.7B) clearly beats every same-size baseline, and on factual and math tasks it beats GPT-3 (175B) and OPT (66B) — models roughly 25× and 10× larger.

On the LAMA factual-completion subsets it scores 33.8 / 11.5 / 53.5 on SQuAD / Google-RE / T-REx, improving on the best same-size baseline by +11.7, +5.2, +18.6 points and clearing GPT-3's 26.8 / 7.0 / 39.8 — by autonomously reaching for the QA tool on 98.1% of examples (the Wikipedia search tool is disabled here, since LAMA is built from Wikipedia).

The pattern repeats on math (ASDiv / SVAMP / MAWPS): Toolformer hits 40.4 / 29.4 / 44.0 versus GPT-3's 14.0 / 10.0 / 19.8 — more than double every same-size baseline — calling the calculator on 97.9% of examples. On question answering (WebQS / NQ / TriviaQA) it reaches 26.3 / 17.7 / 48.8, beating its same-size peers via the search tool (used 99.3% of the time) but still trailing GPT-3 (175B) at 29.0 / 22.6 / 65.9 — the search engine is simple and non-interactive. Temporal tasks are mixed: on the synthetic DATESET the calendar tool drives a jump to 27.3 (used 54.8% of the time), but on TEMPLAMA the calendar is barely touched (0.2%). And crucially, language-modeling ability is not sacrificed — perplexity with calls disabled is 10.3 (WikiText) / 10.5 (CCNet), matching plain finetuning.

That scaling curve is the conceptual payoff: making good use of a tool is itself an emergent capability. The smallest models cannot turn an API result into a better prediction; larger ones can, and the with-/without-tool gap keeps widening as the model grows.

// does it hold up?

The eval read

The flagship "a 6.7B model beats GPT-3 (175B)" line is true but narrower than it sounds. Toolformer wins on the tasks its five tools were built for — factual completion and arithmetic — but still loses to GPT-3 on the QA benchmarks, and on multilingual MLQA the whole approach barely helps (finetuning on CCNet even degrades some languages). The big-model baselines are also the un-instruction-tuned davinci GPT-3 and OPT, evaluated zero-shot — a setting that handicaps general models (on MLQA, OPT and GPT-3 mostly fail simply by not answering in English). So "competitive with a 25×-larger model" holds for the tool-suited slice, not in general.

The deeper subtlety is the training signal. Calls are kept when they reduce next-token perplexity, not when they are correct. A call that makes the continuation more predictable survives even if its content is wrong or irrelevant — the paper's own Table 10 shows a "Fast train success" Wikipedia call that returns nothing relevant yet still passes the filter. The model is optimized to be helped in prediction, which is correlated with, but not the same as, being right. The eval criteria are also deliberately lenient (a LAMA answer counts if the correct word is in the first five predicted; a QA answer if it's in the first twenty; math takes the first number), which inflates absolute scores and complicates cross-paper comparison.

baselines

Same-size GPT-J set is fair and well-controlled. The headline GPT-3 (175B) / OPT (66B) comparison uses un-instruction-tuned models, zero-shot, and tilts toward the five tools' home turf — Toolformer still loses on QA and on MLQA.

contamination

The QA tool (Atlas) is itself finetuned on Natural Questions, so QA is disabled on the NQ eval to avoid trivializing it — a real entanglement, handled but telling. Lenient first-5/first-20-words matching inflates the absolute numbers.

what's missed

The filter rewards perplexity reduction, not factual correctness. Calls are sampled independently, capped at one per example, with no tool chaining (calendar→QA is structurally impossible, which is exactly why TEMPLAMA fails), no interactive/reformulated search, and no model of a tool's compute cost.

would raise confidence

A correctness-aware (not just loss-based) filter; chained and interactive tool use; exact-match evaluation; and a head-to-head against an instruction-tuned model of equal size.

// bottom line

Toolformer is the proof that a language model can supervise its own tool use — fit API calls into a corpus by keeping only the ones that lower its own loss, finetune, done — and it is the conceptual seed of every modern tool-calling agent. But it is a 2023 prototype: single non-chained calls, a perplexity-not-correctness signal, and a "beats GPT-3" headline that holds only on the tasks its five tools were designed for.

// takeaways

What to remember

  1. Tool use can be learned self-supervised. Annotate a plain corpus with candidate API calls, keep only those that reduce the model's own next-token loss, and finetune on the survivors — no human labels for where or how tools should be called.
  2. Using a tool well is an emergent capability. Below ~775M parameters the model cannot turn an API result into a better prediction; the with-/without-tool gap only opens up — and keeps widening — as models scale.
  3. It buys small models big-model competence on the right tasks — a 6.7B GPT-J beating GPT-3 (175B) on LAMA and math, with no loss in plain language modeling — but the filter optimizes predictability rather than correctness, and the calls cannot yet chain or interact.
