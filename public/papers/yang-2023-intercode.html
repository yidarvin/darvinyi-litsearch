InterCode: Standardizing and Benchmarking Interactive Coding with Execution Feedback

Recasts interactive coding as a standard POMDP — code as actions, execution output as observations — and wraps NL2Bash, Spider, and MBPP in Docker-sandboxed Bash, SQL, and Python environments, so wildly different execution-feedback methods can finally be compared on one benchmark instead of each rolling its own bespoke setup.

John Yang, Akshara Prabhakar, Karthik Narasimhan, Shunyu Yao

// the gap

What problem is this solving?

Humans write code interactively — a constant "write, execute, test" loop where compiler and interpreter output resolves ambiguity, catches typos before they compound, and helps decompose a hard problem into pieces. The dominant code-generation benchmarks of the time (HumanEval, APPS, MBPP, CodeXGLUE) don't test that loop at all: they frame code generation as a static, one-shot instruction-to-code transduction — the model gets one shot, is graded once, and never sees what its own code actually does when it runs. That leaves no room for error recovery and creates a disconnect between the generated code and its downstream execution environment.

A handful of papers had started experimenting with execution feedback or interaction, but each one built its own bespoke setup — different compilers, execution environments, feedback signals, and assumptions about human-in-the-loop participation — making cross-paper comparison effectively impossible. There was no standard interactive-coding benchmark that a new method could be measured against, the way HumanEval had become the default yardstick for static code generation.

// how it works

The method

InterCode formalizes interactive coding with execution feedback as a partially observable Markov decision process — instruction space U, state space S, action space A, observation space O, transition function T: S×A→S, reward function R: S×A→[0,1]. Given a natural-language instruction, an agent issues code (or a special submit keyword) as an action; an admissible action — one that parses and executes — changes the environment's latent state and returns execution feedback as an observation. The loop repeats until submit, at which point a reward is computed, with 1 meaning task completion. Success Rate (SR) is the share of episodes ending in reward 1; Error % is the share of non-admissible (unparseable / failed) actions.

Building a new InterCode environment takes three modular pieces: a Dockerfile defining a safe, reproducible virtual-container sandbox; a dataset with at minimum a natural-language query and a gold reference; and a reward function with access to the full interaction history and the live container. The paper instantiates three environments this way. InterCode-Bash wraps a bash shell in Ubuntu, bootstrapping 200 commands from NL2Bash across 4 distinct file systems, rewarded by a lexical-similarity score on the latest stdout plus a file-system diff (list of changed paths, with md5sum checks on commonly edited files) against the gold command. InterCode-SQL wraps a MySQL database built from 1,034 Spider dev-set instances across 20 databases, rewarded by Intersection-over-Union (Jaccard index) between the agent's and gold's returned record sets, discounted by a Kendall's-τ penalty when the rows come back in the wrong order. InterCode-Python wraps a bare Python interpreter around MBPP's 117 tasks, inheriting MBPP's own unit-test pass rate directly.

On top of the three environments, the paper evaluates four prompting strategies (Figure 2): Single Turn (a zero-shot attempt, graded after the first generation only), "Try Again" (up to n turns of raw execution-feedback retry with no reasoning scaffold), ReAct (interleaved thought + action, with the agent itself deciding when to stop), and Plan & Solve (the agent first writes a plan, then executes it step by step). Models spanned three families: OpenAI (text-davinci-003, gpt-3.5-turbo, gpt-4), PaLM-2 (text-bison-001, chat-bison-001), and open source (Vicuna-13B, StarChat-16B).

// what they found

The results

Interactive, multi-turn evaluation beats single-shot evaluation for every model tested, on both environments — but the size of the win depends heavily on the task. The standout: GPT-4 on InterCode-SQL jumps from 9.1% Single Turn to 73.7% with "Try Again" (n = 10), an 8× increase. GPT-3.5-turbo shows a similar pattern (10.5% → 47.3%). InterCode-Bash gains are far more modest across the board — GPT-4 moves from 34.0% to 48.5%, GPT-3.5-turbo from 34.5% to 46.5% — and Figure 3 shows several models' Bash success rate plateauing by turn 6–8 even as SQL performance is still climbing at turn 10, consistent with the paper's own observation that as context and feedback accumulate, models struggle to pick out what's actually relevant to the next action.

Reasoning scaffolds help — sometimes. Comparing prompting strategies with gpt-3.5-turbo as the fixed backbone (Table 4 in the paper), ReAct lifts InterCode-SQL success rate from Try Again's 47.3% to 58.7%, with fewer average turns (5.30 vs 7.25) and a far lower error rate (6.94% vs 46.4%). But on InterCode-Bash, ReAct underperforms Try Again — 20.5% vs 46.5%, less than half — while Plan & Solve lands in between (28.0%). The authors attribute this split to task structure: SQL instructions are phrased as questions that reward discovering context and correcting course, which suits ReAct's flexible thought-then-act loop, while Bash instructions are more declarative and multi-step, which Plan & Solve's upfront decomposition handles better than ReAct's turn-by-turn improvisation.

Finally, the paper demonstrates InterCode's extensibility by building InterCode-CTF, wrapping picoCTF-style Capture-the-Flag challenges (forensics, binary exploitation, cryptography) as a harder, more open-ended interactive task requiring multi-language, multi-step reasoning — GPT-4's trajectory on one forensics task is walked through qualitatively (Figure 4 in the paper).

// does it hold up?

The eval read

The paper contains a genuine internal inconsistency about its own newest contribution. Section 5.3 introduces InterCode-CTF as a new dataset built from 100 picoCTF objectives. The Limitations section (Appendix, §6) says the opposite: it describes the released CTF dataset as manually curated down to just four task instances. Both statements describe InterCode-CTF in the same paper version (v3); they can't both be right, and the paper doesn't reconcile them — a reader is left guessing whether the released CTF benchmark has 4 tasks or 100.

A second gap: InterCode-Python is defined but never benchmarked. The paper's own scope claims three environments — Bash, SQL, and Python — but the main results (§5.1, §5.2) report Success Rate tables only for SQL (Table 2) and Bash (Table 3); the prompting-strategy comparison (Table 4) likewise covers only SQL and Bash. InterCode-Python / MBPP appears solely as a qualitative walkthrough (Figures 11–12 in the appendix, illustrating "how InterCode-Python performs automatic evaluation"), with no Success Rate numbers for any of the seven evaluated models. A third of the framework's headline scope is effectively unevaluated in the paper that introduces it.

The reward functions are also reasonable but imperfect proxies for correctness, and the paper doesn't stress-test them. InterCode-SQL's IoU-over-record-sets reward would score a query that returns the right rows through entirely wrong logic (e.g. a coincidental filter that happens to match, or a query missing a JOIN condition that accidentally returns the same rows on a small table) identically to a semantically correct one — there's no query-plan or logic check. InterCode-Bash's lexical-similarity-plus-file-diff reward is coarser still, and — as the authors note themselves — favors any sequence of commands that reaches the right final file-system state, not necessarily one that reflects the intended approach. Finally, the fixed n = 10 turn budget is applied uniformly across models with very different convergence rates (Figure 3 shows some models still climbing at turn 10 while others plateau by turn 4–6), so the reported ceiling may understate stronger models' true multi-turn capability while flattering models that happen to converge within the budget.

baselines

Reasonable spread of contemporary commercial and open models (OpenAI, PaLM-2, Vicuna, StarChat), evaluated consistently across strategies with fixed decoding settings (temperature 0). No contamination check, though.

contamination

Not discussed. NL2Bash, Spider, and MBPP are all older, widely circulated static datasets; the paper doesn't address whether their text (queries and gold code) could already sit in the pretraining data of GPT-4 / GPT-3.5 / PaLM-2, which would inflate multi-turn gains that look like "reasoning" but are partly recall.

what's missed

InterCode-Python/MBPP results are entirely absent from the quantitative results despite being one of three claimed environments; the CTF dataset's actual size (4 vs. 100 task instances) is internally contradictory; reward-hacking risk in the IoU/lexical-similarity proxies isn't examined.

would raise confidence

Report Success Rate for InterCode-Python alongside Bash and SQL; reconcile and state the actual InterCode-CTF task count; audit a sample of "successful" trajectories for reward-function false positives (right output, wrong reasoning); vary the turn budget per model or report turn-normalized ceilings.

// bottom line

InterCode's core contribution — a standard POMDP formulation plus a Docker-sandboxed harness that turns any {query, gold} dataset into an interactive benchmark — is a genuinely useful piece of infrastructure that later agent-benchmark papers built on. But the paper's own evaluation doesn't fully deliver on its three-environment premise (Python is unbenchmarked), contains a real self-contradiction about the size of its newest dataset (CTF), and leans on reward functions whose failure modes (reward hacking via coincidental output matches) are never examined.

// takeaways

What to remember

  1. Framing interactive coding as a POMDP with code-as-action and execution-output-as-observation gave later agent-benchmark work a shared vocabulary and a Docker-sandboxed harness to build on, instead of every paper inventing its own setup.
  2. Multi-turn execution feedback helps a lot on some tasks and barely at all on others in the same paper — GPT-4 on SQL jumps 9.1%→73.7% with 10 turns of retry, while Bash only moves 34.0%→48.5% — a reminder that "agents benefit from interaction" is task-dependent, not a universal law.
  3. Reasoning scaffolds aren't universally good: ReAct more than doubles gpt-3.5-turbo's SQL success rate over plain retry (47.3%→58.7%) but more than halves its Bash success rate (46.5%→20.5%) — the right prompting strategy depends on whether the task rewards flexible discovery (SQL) or upfront decomposition (Bash).
  4. Read past the headline environment count: InterCode advertises Bash, SQL, and Python, but only Bash and SQL get quantitative results in this paper — a useful reminder to check whether every claimed capability in a benchmark paper is actually benchmarked.
