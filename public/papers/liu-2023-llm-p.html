LLM+P: Empowering Large Language Models with Optimal Planning Proficiency

LLMs are fluent but unreliable long-horizon planners; classical planners are reliable but need a formal problem spec. LLM+P has the LLM translate a natural-language planning problem into PDDL, hands it to a classical sound-and-complete planner to actually solve, then translates the plan back into natural language.

Bo Liu, Yuqian Jiang, Xiaohan Zhang, Qiang Liu, Shiqi Zhang, Joydeep Biswas, Peter Stone · UT Austin & SUNY Binghamton

// the gap

What problem is this solving?

LLMs are, in the authors' framing, strong on linguistic competence (knowing how to say things) but weak on functional competence (knowing what to say) when the task requires genuine long-horizon reasoning about how actions change the world. The paper opens with a concrete failure: given a 5-block stacking problem, GPT-4 outputs a fluent-looking 6-step plan that is nonetheless invalid — step 4 ("Move b2 from b1 to b5") tries to stack b2 on top of b5, but two steps earlier the plan itself moved b3 onto b5, so b5 is no longer clear and the move violates a basic precondition of the domain.

Classical AI planning already solves this class of problem reliably: given a formal specification in PDDL (the Planning Domain Definition Language) — a set of world states, an initial state, goal conditions, and symbolic actions with preconditions and effects — a sound-and-complete symbolic planner is guaranteed to find a correct plan if one exists, and often the optimal (lowest-cost) one. The catch is that nobody wants to hand-write PDDL for every new task; that formality is exactly what made classical planning unwieldy outside narrow, pre-specified domains. The gap LLM+P targets: LLMs are good at exactly the translation step classical planning has always needed a human for — turning an informal natural-language task description into a formal spec — while the actual multi-step reasoning is better left to a solver that can't hallucinate an invalid move.

// how it works

The method

LLM+P is a three-stage pipeline, assuming a human or domain expert has already supplied a fixed domain PDDL file for the task family (the world's action vocabulary — e.g. pick-up, stack — with each action's preconditions and effects) plus one worked example pairing a natural-language problem with its corresponding problem PDDL:

  1. LLM as PDDL writer. Given the new natural-language problem P, the one in-context example (problem + its PDDL), and the domain file, the LLM writes the problem PDDL — the objects, initial state, and goal conditions specific to P. The paper frames this as a machine-translation task LLMs are already good at, not as reasoning.
  2. Classical planner solves it. The generated problem PDDL plus the fixed domain PDDL are fed to an off-the-shelf planner (Fast Downward), which searches for a plan — with no risk of an invalid move, because the planner only ever outputs actions whose preconditions actually hold.
  3. LLM translates the plan back. The planner's symbolic action sequence is translated back into natural language (or, for a robot, into an action-executor call) by the LLM.

The paper's own worked example shows why the in-context example matters: asked cold to write the problem PDDL for a 5-block rearrangement task, GPT-4 invents a nonexistent predicate (empty) and omits that block b1 starts on the table. Given one example pair (a different 5-block problem plus its correct PDDL) as context, GPT-4's PDDL for the same target problem becomes syntactically and semantically correct, and the planner solves it directly — the paper reports this as necessary, not incidental: "Without the context... LLMs fail to produce correct problem PDDL files."

// what they found

The results

The evaluation uses seven robot-planning domains from past International Planning Competitions (BARMAN, BLOCKSWORLD, FLOORTILE, GRIPPERS, STORAGE, TERMES, TYREWORLD), 20 auto-generated problems per domain, run through GPT-4 (temperature 0) and Fast Downward as the planner (200-second timeout, optimal alias SEQ-OPT-FDSS-1, sub-optimal LAMA in parentheses when the optimal alias times out). Baselines are LLM-As-Planner with and without a one-shot example (LLM−/LLM), and — notably — Tree of Thoughts adapted into a breadth-first-search planner (LLMToT), plus an ablation of LLM+P run without its own in-context example (LLM+P−).

The pattern is stark: LLM+P clears 85% success — counting the slower sub-optimal fallback where the optimal search timed out — in five of seven domains (BARMAN 100% sub-optimal / 20% optimal, BLOCKSWORLD 90%, GRIPPERS 95%, STORAGE 85%, TYREWORLD 90% sub-optimal / 10% optimal), drops to a modest 20% on TERMES, and is 0% on FLOORTILE. Every LLM-As-Planner variant — including the Tree-of-Thoughts-style search adaptation — scores 0% or near-0% on most domains, and 0% everywhere without an in-context example. The paper attributes LLM-As-Planner's failures to an inability to track state predicates like on and clear in BLOCKSWORLD, and to reason about spatial constraints in FLOORTILE and TERMES (e.g. proposing to paint a non-adjacent tile). Tree-of-Thoughts-as-planner (LLMToT) is reported to time out in most cases from the sheer number of LLM calls its search requires, making it impractical for these longer-horizon problems despite giving "reasonable rankings" on individual partial plans.

Both LLM+P and every baseline score 0% on FLOORTILE — the paper attributes LLM+P's failures there to mis-specified problem PDDL, such as omitted initial conditions that leave tiles disconnected and make the derived planning problem literally unsolvable, a translation failure rather than a search failure.

Beyond the benchmark suite, the authors deploy LLM+P on a real service robot for a natural-language tidy-up task and report that its plan (cost 22, re-grasping the bottle mid-route) is cheaper than the sub-optimal plan LLM-As-Planner produces for the same request (cost 31, an unnecessary round trip) — a single qualitative case, not a benchmarked comparison, but concrete evidence the approach ports off paper-only planning domains onto a physical system.

// does it hold up?

The eval read

The LLM-As-Planner and Tree-of-Thoughts-as-planner baselines are a fair and, for a paper posted just after ToT's own release, genuinely current comparison — the paper's later revision (arXiv v3, dated 27 Sep 2023) even explicitly adapts ToT's own breadth-first-search implementation for the planning setting rather than strawmanning it. But the comparison is structurally lopsided in one respect worth naming plainly: LLM+P gets to call a sound-and-complete symbolic solver that is unbeatable by construction on problems within its expressiveness, while the baselines ask an LLM to do the entire search itself. That LLM+P wins is close to true by design once the LLM can write correct-enough PDDL — in effect, the paper mostly measures whether GPT-4 can translate to PDDL reliably, and the FLOORTILE result (0% for LLM+P too) shows that when translation fails, the whole pipeline fails with it, planner or no planner.

Every quantitative result is on 20 auto-generated problems per domain (140 total) with a single GPT-4 run at temperature 0 — no repeated sampling, no variance reported, and no second LLM tested to see whether the PDDL-translation step depends on GPT-4 specifically versus generalizing to other models. The domains themselves are drawn from published International Planning Competition benchmarks that plausibly have public solutions and worked PDDL examples circulating on the web — the paper doesn't discuss whether this could let GPT-4 pattern-match toward already-seen problem/PDDL pairs rather than genuinely translating each new instance, though the one-shot in-context format at least requires per-problem generation rather than verbatim recall.

baselines

Fair and current — LLM-As-Planner (zero- and one-shot) plus an adapted Tree-of-Thoughts search baseline — but structurally asymmetric: LLM+P's planner component is correct by construction, so the comparison mostly isolates the PDDL-translation step rather than testing planning ability head-to-head.

contamination

Not discussed. The seven domains are established IPC benchmarks; the paper doesn't check whether GPT-4 has seen matching PDDL solutions during training, which would inflate the translation-quality numbers specifically.

what's missed

Single GPT-4 run per problem, no variance; only 20 generated instances per domain; and the FLOORTILE 0%-across-the-board result is a reminder that LLM+P's guarantee is conditional on correct translation, which the paper shows can fail silently (a satisfiable-looking but actually unsolvable PDDL problem).

would raise confidence

Multiple runs/seeds per problem to separate translation noise from genuine failure; testing the translation step on a second LLM; and a contamination check given these are long-published competition benchmarks.

// bottom line

The neuro-symbolic split — LLM for translation, solver for search — is a clean, well-motivated idea and the gap between LLM+P (85%+ success, counting the sub-optimal fallback, in five of seven domains) and every direct-LLM baseline (mostly 0%) is large enough that no error bars would erase it. What the paper doesn't fully separate is how much of that gap reflects planning being genuinely hard for LLMs versus the baselines simply not being given the planner's advantages — a comparison against, say, an LLM allowed iterative self-correction against the domain's precondition checker (rather than one-shot, no-feedback generation) would sharpen the claim considerably.

// takeaways

What to remember

  1. LLM+P routes planning through a three-stage pipeline — LLM writes problem PDDL, a classical sound-and-complete planner (Fast Downward) solves it, LLM translates the plan back — so the actual multi-step search is never done by the language model itself.
  2. Across seven IPC-derived robot-planning domains, LLM+P clears 85%+ success (counting the sub-optimal fallback) in five domains, falls to 20% on TERMES, and is 0% on FLOORTILE for everyone, while LLM-As-Planner and an adapted Tree-of-Thoughts search baseline score near 0% almost everywhere — and the one-shot in-context example is reported as necessary, not optional (0% without it).
  3. The comparison is structurally asymmetric — LLM+P's planner is correct by construction — so the real quantity being measured is how reliably GPT-4 can translate a natural-language problem into valid PDDL, not planning ability per se; FLOORTILE's 0%-for-everyone result shows what happens when that translation step itself fails.
