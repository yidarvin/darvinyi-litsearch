AutoGen: Enabling Next-Gen LLM Applications via Multi-Agent Conversation

Microsoft Research's framework for building LLM applications out of customizable, "conversable" agents wired together by one generic send/receive/generate_reply interface — replacing hand-rolled agent-orchestration code with a small set of reusable primitives, then using them to beat GPT-4, ChatGPT+Code Interpreter, ReAct, and single-agent baselines across six very different applications.

// the gap

What problem is this solving?

By mid-2023, "LLM agent" applications were exploding in variety — ReAct-style single-agent loops, LangChain chains, AutoGPT/BabyAGI task managers, CAMEL's role-playing pairs, ad hoc multi-agent debate scripts — but each was built as its own bespoke stack. Every new application (solving a math problem, answering a question with retrieval, executing actions in a text-adventure environment, writing and safety-checking code, running a group brainstorm) meant re-inventing the plumbing that lets an LLM, a human, and a tool trade messages and take turns, usually hard-coded for that one use case.

The authors frame the underlying design problem as two open questions: (1) how do you build individual agents that are capable, reusable, and customizable enough to cooperate in different combinations, and (2) how do you give developers a single, unified interface that can express both simple two-turn exchanges and complex, dynamic, many-agent conversations — in natural language, code, or a mix of both — without hand-rolling a new control flow for every application? Existing single-agent frameworks answer neither question by design, and the handful of multi-agent systems that existed (CAMEL, Multi-Agent Debate scripts, MetaGPT) each hard-code a fixed conversation topology for one scenario rather than offering a general, reusable substrate.

// how it works

The method

AutoGen's unit of composition is the conversable agent: an entity with a unified interface — send, receive, generate_reply — that can be backed by an LLM, a human, tool/code execution, or any combination. Two built-in specializations cover most cases: AssistantAgent is LLM-backed and defaults to suggesting Python code; UserProxyAgent defaults to soliciting human input and/or executing code or function calls the assistant proposes, with a configurable human_input_mode (ALWAYS / NEVER / conditional) controlling how much a human is actually in the loop. A GroupChatManager extends the same interface to coordinate more than two agents in a shared, dynamically-ordered chat.

The second half of the idea is conversation programming: once agents are defined, a developer wires their interaction by registering custom reply functions and calling A.initiate_chat(message, B). From there, an auto-reply mechanism takes over — whenever an agent receives a message, it automatically invokes generate_reply and sends the result back unless a termination condition is met — so the conversation drives itself with no separate control-flow module once it starts. Developers can steer this either in natural language (the assistant's system prompt) or in code (a registered reply function), and mix both freely.

The paper positions AutoGen directly against the multi-agent systems that existed at the time (its own Table 1): unlike Multi-Agent Debate scripts and BabyAGI, AutoGen supports both static and dynamic conversation topologies; unlike CAMEL, it natively executes code and tools rather than only role-playing dialogue; and unlike all four comparison systems (Multi-Agent Debate, CAMEL, BabyAGI, MetaGPT), it supports flexible human involvement rather than none. To demonstrate the abstraction is actually general, the paper builds six structurally different applications on top of the same primitives — autonomous and human-in-the-loop math solving, retrieval-augmented question answering and code generation, ALFWorld decision-making with an added "grounding" agent, a multi-agent coding pipeline with a dedicated safety-checking agent, dynamic group chat, and two-player conversational chess.

// what they found

The results

Each application is evaluated against its own natural point of comparison rather than one shared benchmark. On 120 Level-5 (hardest) MATH problems with a GPT-4 backend, AutoGen's default two-agent setup solves 52.5%, ahead of ChatGPT+Code Interpreter (48.33%), ChatGPT+Plugin (45.0%), vanilla GPT-4 alone (30.0%), Multi-Agent Debate (26.67%), and LangChain ReAct (23.33%). On the full MATH dataset — where only AutoGen and vanilla GPT-4 were run, since the commercial UIs and weaker baselines were dropped for cost and rate-limit reasons — AutoGen reaches 69.48% against GPT-4's 55.18%.

On retrieval-augmented Q&A over Natural Questions (GPT-3.5 backend), AutoGen's Retrieval-augmented Chat scores F1 25.88% / recall 66.65%, versus 22.79% / 62.59% with its own "interactive retrieval" feature disabled (the assistant is instructed to say "I don't know" instead of triggering another retrieval round), and 15.12% / 58.56% for a DPR baseline — isolating the interactive-retry mechanism as the actual source of the gain over plain retrieval-augmented generation.

On ALFWorld's 134 unseen household tasks, adding a third "grounding" agent that injects commonsense knowledge when the two-agent system starts looping on a flawed plan raises success from 54% average / 63% best-of-3 (two-agent) to 69% average / 77% best-of-3 (three-agent) — both ahead of the ReAct baseline's 54% average / 66% best-of-3. And on the OptiGuide multi-agent coding rebuild, splitting code-writing and safety-checking into separate Writer and Safeguard agents (rather than one agent doing both) raises unsafe-code-detection F1 from 88% to 96% with GPT-4, and from 48% to 83% with GPT-3.5-turbo — an 8-point and 35-point gain respectively — while cutting the core workflow's implementation from over 430 lines of code to about 100.

// does it hold up?

The eval read

This is fundamentally a systems paper wearing a benchmarks paper's clothing: six structurally different applications, six different metrics (success ratio, F1/recall, lines of code, LLM-call counts), each run once. That's a defensible way to demonstrate a general-purpose abstraction actually generalizes, but it means none of the individual results carry the statistical weight the abstract's "empirical studies demonstrate the effectiveness of the framework" implies — there is no shared benchmark suite, and every accuracy/success-ratio comparison in A1–A4 is a single run per system with no seed-level variance reported (the one place standard deviations do appear, Appendix D's Table 4, measures manual-effort savings and user-study timing, not task success).

The MATH comparison is the clearest case of a baseline set shaped by convenience rather than symmetry. The paper explicitly says it "did not evaluate ChatGPT on the whole dataset since it requires substantial manual effort and is restricted by its hourly message-number limitation," and separately drops Multi-Agent Debate and LangChain ReAct from the whole-dataset run because "they underperformed vanilla GPT-4 on the smaller test set." The result is that AutoGen's most favorable number — 69.48% vs. GPT-4's 55.18% on the full dataset — is a two-system comparison with the weaker baselines pre-filtered out, while the six-way comparison against ChatGPT+Code Interpreter, ChatGPT+Plugin, Multi-Agent Debate, and ReAct only exists on the 120-problem hard subset.

There's an analog of contamination worth naming even though AutoGen trains nothing itself: every reported number is really "AutoGen + GPT-4" or "AutoGen + GPT-3.5," and MATH/GSM8K-style problems are widely suspected to already sit inside GPT-4's pretraining data. If so, that inflates AutoGen's number and its baselines' numbers roughly in step — but the paper never checks, so it's impossible to tell how much of the reported gap is "better orchestration" versus "the backend already partially knows this problem." Sample sizes elsewhere are also small and explicitly exploratory: OptiGuide's safety ablation uses 100 hand-crafted tasks (50 safe / 50 unsafe), and the dynamic-group-chat result (A5, Tables 5–6) is run on "12 manually crafted complex tasks" that the paper itself calls a "pilot study."

Most notably, the paper has the right instrument for the multi-agent-systems tradeoff that matters most — extra agents mean extra LLM calls — but only points it at itself. Table 6 reports average LLM-call counts for A5's internal comparison (two-agent vs. group-chat vs. task-prompted group-chat, all AutoGen configurations), but no LLM-call or cost accounting appears anywhere in the A1–A4 headline comparisons against ChatGPT+Code Interpreter, GPT-4, ReAct, DPR, or Multi-Agent Debate — the comparisons that most need it, since a multi-agent system that wins by spending more inference budget per task is a different claim than one that wins by being a better method.

baselines

Reasonable choices (GPT-4, ChatGPT+Code Interpreter, ReAct, DPR, Multi-Agent Debate), but inconsistently applied: the strongest MATH result (69.48% vs. 55.18%) drops every baseline except vanilla GPT-4, while the full six-way comparison only covers a 120-problem subset.

contamination

Not directly applicable (no training happens), but every result is "AutoGen + a frontier LLM backend," and the paper never checks whether GPT-4/GPT-3.5 had prior exposure to the MATH/GSM8K-style problems being solved — a confound that would inflate AutoGen and its baselines together.

what's missed

No seed-level variance on any accuracy/success-ratio result (A1–A4); small hand-crafted eval sets in several places (100 OptiGuide tasks, 12 group-chat tasks); LLM-call/cost accounting exists only for AutoGen-vs-AutoGen comparisons (Table 6), never for the external baselines it claims to beat.

would raise confidence

Multi-seed results with variance bars; a symmetric baseline set applied at the same problem-set scale throughout; a documented contamination check on the backend LLMs; compute/cost-normalized comparisons against ChatGPT+Code Interpreter, ReAct, and Multi-Agent Debate specifically.

// bottom line

The core abstraction — a unified conversable-agent interface plus an auto-reply mechanism that lets a conversation drive its own control flow — is a genuinely useful, well-motivated systems contribution, and the six demonstrations make a real case that splitting responsibilities across agents (a grounding agent, a safeguard agent, an interactive-retrieval retry) helps on tasks where a single agent gets stuck. But the "AutoGen wins" numbers should be read as six single-run case studies with convenience-shaped baseline sets, not as a controlled benchmark result — and the paper's own LLM-call instrumentation, which could settle whether the gains survive a compute-normalized comparison, is applied to the wrong half of the experiments.

// takeaways

What to remember

  1. AutoGen's core abstraction is the conversable agent — a unified send/receive/generate_reply interface backed by any mix of LLM, human, and tools — plus an auto-reply mechanism, so a single initiate_chat() call sets a conversation running with no separate control-flow module.
  2. Splitting responsibilities across agents measurably helps across very different tasks: a third grounding agent lifts ALFWorld success from 54%/63% to 69%/77% (average/best-of-3); a dedicated Safeguard agent raises unsafe-code F1 by 8 points (GPT-4) and 35 points (GPT-3.5) over one agent doing both jobs; on 120 hard MATH problems, AutoGen's default setup beats ChatGPT+Code Interpreter (52.5% vs. 48.33%) and vanilla GPT-4 (30.0%).
  3. The evidence is six single-run, differently-scoped demonstrations rather than one controlled benchmark: the strongest MATH comparison drops every baseline except GPT-4, no accuracy/success-ratio result carries seed-level variance, and the paper's own LLM-call accounting (Table 6) is applied only to AutoGen-vs-AutoGen comparisons, never to the external baselines the headline numbers claim to beat.
