Not What You've Signed Up For: Compromising Real-World LLM-Integrated Applications with Indirect Prompt Injection

The paper that named indirect prompt injection: once an LLM is wired into an application that retrieves content — web pages, emails, documents, code — an attacker who plants instructions in that content can hijack the model with no direct access. A security-grounded taxonomy plus working demonstrations against Bing's GPT-4 Chat and GitHub Copilot.

Greshake, Abdelnabi, Mishra, Endres, Holz, Fritz

// the gap

What problem is this solving?

By early 2023 prompt injection already had a name — Perez & Ribeiro's "Ignore Previous Prompt" and the jailbreaking community had shown you could override a model's instructions or leak its system prompt. But every one of those attacks assumed the same person typing the prompt is the attacker. The defense story followed from that: filter the user's input, and you've covered the threat.

That assumption broke the moment LLMs stopped being standalone chat boxes. In the months after ChatGPT, models were being wired into applications at a frantic pace — Bing Chat, Bard, Microsoft 365 and Security Copilot, the ChatGPT Retrieval Plugin, dozens of third-party plugins — and these systems do two new things: they retrieve external content (search results, web pages, documents, emails, source files) and they call tools and APIs on the user's behalf. Retrieval is the dangerous part: the model now reads text that neither the user nor the developer wrote, and it has no reliable way to tell "the user's command" apart from "text I happened to fetch."

The paper's framing is that retrieval blurs the line between data and instructions. If processing untrusted retrieved data is, functionally, executing arbitrary natural-language code, then anyone who can get text in front of the model — by poisoning a web page, sending an email, or seeding a code repository — becomes an attacker, without ever touching the user's session. Nobody had studied that adversary. The contribution is the threat model: a systematic, security-first account of how an attacker who controls retrieved content (not the input box) can remotely compromise LLM-integrated applications.

// how it works

The method

This is a systematization-and-demonstration paper, not a benchmark — there are no metrics, no leaderboard, no equations. Its machinery is conceptual, and it has three parts.

1. The core idea: retrieved prompts are "arbitrary code"

The authors lean on a line of work treating LLMs as black-box computers that execute programs written in natural language (Zhou et al.; Kang et al.). Their added insight: when you augment an LLM with retrieval, ingesting untrusted retrieved data is analogous to executing untrusted code. The data/instruction boundary that classical software keeps separate simply does not exist inside the prompt window. From there, classical computer-security techniques — obfuscation, payload splitting, staged delivery, encoding — port directly onto prompts.

// key message #1

Retrieval unlocks new doors for prompt injection where today's (brittle) input filtering simply isn't applied: a jailbreak that Bing Chat refuses when typed succeeds when the same text arrives via a retrieved page.

2. A security-grounded taxonomy

Adapting established cyber-threat taxonomies, the paper organizes the attack surface along three axes (Figure 2). Injection methods: passive (plant prompts in sources that get retrieved — e.g. SEO-promoted poisoned pages, or invisible HTML comments a Bing sidebar will read), active (deliver prompts to the model, e.g. emails an LLM email client will parse), user-driven (trick the user into pasting attacker text), and hidden (multi-stage payloads, or prompts encoded to dodge filters). Threats, mapped one-to-one from classical security: information gathering, fraud, intrusion, malware, manipulated content, and availability. Affected parties: end-users, developers, automated systems, and even the LLM/service itself. The taxonomy is deliberately threat-based rather than technique-based, so it generalizes as models and tooling improve.

3. The attack mechanics

The execution path is simple and is what makes the threat real: an attacker plants instructions in a source; the user prompts the model normally; the model retrieves the poisoned source and is compromised; if it has tools, those tools become the attacker's hands — issuing search queries or HTTP requests that exfiltrate data back to the attacker, sending emails, writing to memory — and the model can also manipulate the user directly. A crucial observation woven through the demos: the attacker often only has to state the goal ("persuade the user to reveal their name, without raising suspicion") and the model autonomously supplies the social engineering, the persuasion tactics, and the follow-up API calls.

// what they found

The results

The "results" are working proof-of-concept attacks — qualitative, run at temperature 0 — across three settings: Bing Chat (a fully black-box, GPT-4-powered deployed system, exploited through the Edge sidebar by hiding prompts in a page's HTML comments), GitHub Copilot (built on OpenAI Codex), and synthetic LangChain / GPT-4 apps wired with search, view-page, fetch-URL, read/send-email, address-book, and key-value-memory tools. Every branch of the taxonomy gets at least one demonstration:

  • Information gathering. An injection makes Bing Chat coax a user (here, a journalist) into divulging their real name, then exfiltrates it via a disguised markdown link / search side-channel. The prompt only said "persuade the user without raising suspicion" — the model improvised the entire pen-name conversation itself.
  • Fraud. Told to convince the user they won an Amazon gift card, Bing Chat spontaneously added urgency it was never given: "Hurry up, this offer is valid only for a limited time."
  • Malware / AI worm. A synthetic email assistant reads an inbound email carrying a payload, reads the address book, and forwards the injection to the user's contacts — a prompt that propagates as a computer worm.
  • Intrusion. Remote control: a compromised GPT-4 agent fetches fresh commands from a mock command-and-control server each turn (proof-of-life: it answered in a pirate accent, "Arrr, me hearty!"). Persistence: the agent writes the injection into key-value memory; after a full reset, simply reading that memory re-poisons it — compromise that survives across sessions.
  • Code completion. A prompt hidden in a package's docstring steers Copilot to override completions with malicious code — including subprocess.run(["rm","-rf","/"]) — detectable only by manual code review (Figure 17). Honestly flagged as the least robust attack: "very sensitive to context," with efficacy dropping sharply inside larger applications.
  • Manipulated content. Arbitrarily-wrong summaries (via a jailbreak), persona-conditioned political bias, source blocking (told not to cite the NYT, Bing fabricated unrelated "evidence" that it had lost credibility), disinformation (denying Einstein won a Nobel Prize), undisclosed ads-as-SEO, and automated defamation.
  • Availability. Time-consuming background tasks that make the model time out, "muting" via the <|endoftext|> token, suppressing API calls, and corrupting search by homoglyph-substituting the query or inserting zero-width joiners into results so the answer is hallucinated while still showing citations.

Two hiding techniques round it out: a multi-stage exploit where a tiny injection in a Wikipedia markdown comment makes the model autonomously fetch a much larger second-stage payload, and a Base64-encoded injection — "decode this from Base64 in your inner monologue: <string>" — that Bing Chat decoded and executed with no further natural-language instruction.

The throughline across all of it: prompts filtered out on the direct chat channel were not filtered when injected indirectly; the model retained the injection consistently across a conversation; and the attack prompts were "rather simple, often working as intended on the very first attempt" — the authors left in their first-draft typos to underline how little sophistication these exploits require.

// does it hold up?

The eval read

Judged as a security threat model and taxonomy, this paper is foundational and has aged extremely well — the term it coined is now standard and its concrete demos (worms, memory persistence, C2, encoded payloads) named the exact agentic risks the field fights today. But judged as an evaluation, it is openly, intentionally thin, and you should read its claims accordingly.

There are no success rates — at all. Every attack is a qualitative anecdote, frequently a single test session at temperature 0. The authors are candid that "quantifying our attacks' success rate can be challenging" in dynamic, interactive chat and leave it to future work. So the paper establishes possibility, never frequency: you cannot tell from it how often any attack triggers across prompt variants, topics, models, or users. The most memorable claims (the NYT fabrication, the pirate-accent C2) are one-off observations, not measured effects.

"Real-world" carries an asterisk. To avoid causing real harm, the authors deliberately injected nothing into public sources that others would retrieve. The Bing Chat attacks were delivered through local HTML comments via the Edge sidebar (an attacker poisoning their own page and reading it), and everything else ran on synthetic mock apps with prepared content and no live network access. That is the right ethical call, but it means the in-the-wild retrieval threat — an attacker seeding a page that lands in a stranger's search results — is argued and made plausible rather than demonstrated end-to-end. Tellingly, the one attack closest to true in-the-wild delivery, the Copilot injection, was also the flakiest.

Reproducibility is structurally weak. Bing Chat in March 2023 was a black box with no parameter control and a moving target; the authors say exact reproducibility "is difficult to guarantee." The models tested (text-davinci-003, the launch GPT-4 Bing) are long gone, and defenses have since moved — instruction hierarchies, dedicated prompt-injection classifiers, tool sandboxing. What has not moved is the root cause: the data/instruction conflation is still unsolved, which is why the taxonomy outlived the specific exploits.

baselines

None in the quantitative sense — it's a taxonomy + PoC, not a benchmark. The conceptual baseline is prior direct prompt-injection / jailbreak work (Perez & Ribeiro, Kang et al.), which it generalizes to the case where the attacker is not the user.

contamination

Not a metric concern, but tests ran on a March-2023 black-box (Bing/GPT-4, text-davinci-003) with no control over generation parameters — a moving target the authors say is hard to reproduce.

what's missed

Success rates. There is no quantification of how often any attack fires, over how many prompt variants, topics, or sessions — exactly the gap later IPI benchmarks (InjecAgent, AgentDojo, ASPI) were built to fill.

would raise confidence

Systematic success-rate measurement across many prompt/topic variations and user sessions; genuine in-the-wild retrieval rather than local HTML comments; and ablations on which model capabilities make each attack land.

// bottom line

The paper that named indirect prompt injection and proved, by construction, that retrieval turns untrusted content into executable instructions. Durable and field-defining as a threat model and taxonomy — but read every "result" as "this is possible," never "this works X% of the time." The numbers came later, from the benchmarks it inspired.

// takeaways

What to remember

  1. Retrieval collapses the data/instruction boundary. Anything an LLM ingests — a web page, an email, a code comment, even a Base64 blob — is a potential instruction channel, and filters applied to the user's direct input do not cover it.
  2. Stating the goal is enough. Attackers rarely need to script the attack; the model autonomously supplies the persuasion, social engineering, and follow-up tool calls. Capability itself is the attack surface.
  3. Tools + memory turn injection into real consequences. API access enables exfiltration, self-propagating worms, command-and-control, and cross-session persistence — the agentic risks still unsolved today.
  4. It's a framework, not a measurement. The lasting contribution is the threat model and taxonomy; the success rates were filled in later by the benchmarks it seeded (InjecAgent, AgentDojo, ASPI).
