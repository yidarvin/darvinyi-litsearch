Show Your Work: Scratchpads for Intermediate Computation with Language Models

Teach a language model to write out its intermediate steps into a "scratchpad" before answering, and multi-step addition, polynomial evaluation, and Python execution all improve dramatically — the direct ancestor of chain-of-thought prompting.

Maxwell Nye, Anders Johan Andreassen, Guy Gur-Ari, Henryk Michalewski, Jacob Austin, David Bieber, David Dohan, Aitor Lewkowycz, Maarten Bosma, David Luan, Charles Sutton, Augustus Odena · MIT & Google Research

// the gap

What problem is this solving?

Large pre-trained Transformers are remarkably good at tasks that can be done "in one pass" — generating realistic text, synthesizing short programs — but they struggle badly with tasks that need unbounded multi-step computation. GPT-3 can't reliably add integers past a few digits in the few-shot setting; models that can write a working Python function often can't predict what that same function outputs when it's actually run on a given input; and even specialized recurrent and graph neural networks fail to systematically generalize when predicting the output of programs containing loops.

The paper's diagnosis is architectural: applying a Transformer to these tasks the standard way forces the entire answer out in a single forward pass, with a fixed number of layers and a fixed amount of computation regardless of how hard the specific input actually is. There's no mechanism for the model to spend more computation on a harder instance before committing to an output. Prior work — Neural Turing Machines, Adaptive Computation Time, Universal Transformers — attacks this by changing the network itself, adding recurrent external memory or a learned per-step halting mechanism. That fixes the adaptive-computation problem, but at the cost of a new architecture and training procedure, forfeiting the ability to just use an existing, pre-trained, few-shot-capable language model as-is.

The gap this paper targets: can you give an off-the-shelf, architecturally unmodified Transformer LM something functionally like adaptive computation time, purely by changing how you design the task — not the network?

// how it works

The method

The proposal is simple: let the model produce an arbitrary sequence of intermediate tokens — a scratchpad — before it produces the final answer, and train it on that expanded target with ordinary supervised likelihood. For long addition, the scratchpad literally writes out the grade-school carrying algorithm digit by digit; for executing a Python function, the scratchpad becomes the program's full execution trace: the sequence of source-code lines run, interleaved with the state of every local variable after each line, encoded as a JSON dictionary.

algorithm induction: D = {xi, f(xi)}Ni=1   ·   learning to execute: D = {(πi, xi, fi(xi))}Ni=1 // π_i = source code of program f_i; the scratchpad supervises the path from x_i to f_i(x_i), not just the endpoint

At training time the model sees input + scratchpad + target and is trained with standard likelihood; at test time it sees only the input and must generate the scratchpad and answer itself, via sampling or beam search. The authors argue the scratchpad buys three things at once: adaptive computation (the model can "think" for as many tokens as the problem needs, rather than a fixed budget), externalized state (intermediate values live in tokens the model can attend back to, rather than being compressed into hidden activations), and reduced error compounding (because each intermediate state is quantized to discrete token embeddings and sampled, rather than propagated as continuous recurrent state the way it is in a Neural Turing Machine).

The paper applies this idea at three levels of generality, using a family of dense decoder-only Transformer language models ranging from 2 million to 137 billion parameters (the same model family used in the companion MBPP program-synthesis paper). First, algorithm induction with hand-designed scratchpads: long addition (the carry is tracked explicitly after each digit) and polynomial evaluation (each term is computed and summed separately). Second, and more general, learning to execute arbitrary code by training the model to predict a program's full execution trace rather than hand-designing task-specific intermediate steps — this generalizes past addition and polynomials to any function expressible in Python, and is evaluated on both synthetic programs and the real human-written programs in the MBPP dataset.

// what they found

The results

On long addition (fine-tuned on 1–8 digit problems, tested in- and out-of-distribution on 9–10 digit problems), scratchpad models solve essentially all in-distribution problems once past a critical scale of roughly 10⁸ parameters, while baseline models trained without a scratchpad plateau around 32% accuracy even at the largest size tested. The gap is starkest out-of-distribution: on 9-digit addition the scratchpad model reaches 93% accuracy at the largest scale while the no-scratchpad baseline stays near 0% throughout; on 10-digit addition scratchpad reaches 52% while baseline again stays near 0%.

On polynomial evaluation (137B-parameter model, few-shot; 8B-parameter model fine-tuned for 2,000 steps), scratchpad again wins under both regimes:

ConditionFew-shotFine-tuning
Direct prediction8.8%31.8%
Scratchpad20.1%50.7%

On synthetic Python programs (simple while-loops and if-statements, 200 held-out test programs), the same pattern holds, though the few-shot scratchpad number required a modified scoring criterion (marked below) after the authors noticed models kept reusing the function's original return-variable name instead of the expected output label:

ConditionFew-shotFine-tuned
Direct prediction11%20%
Scratchpad26.5%*41.5%

The most demanding test is real, human-written MBPP programs — and it's the one that exposes how much scratchpad tracing depends on training data volume. Trained on only the 374 original MBPP training tasks, scratchpad actually loses to direct prediction (5.1% vs. 10.3% per-task execution accuracy). Augmenting the tracing data with 17,000 extra programs sampled from the model itself (MBPP-aug) more than triples scratchpad's own per-task execution accuracy (5.1% → 17.3%) while the identical augmentation actively hurts the direct-prediction baseline (10.3% → 5.1%). Layering in two further data sources — a 670,904-trace extraction from Project CodeNet and a 9-million-example single-line Python transformation dataset — pushes scratchpad's per-task execution accuracy to a best of 26.6%:

These MBPP numbers all come from a 212-task subset of the official 500-task test split — the tasks whose ground-truth execution trace fits inside the model's 512-token generation window across all three input-output examples. Combining CodeNet with the single-line dataset (rather than either alone) gives the best per-task execution accuracy (26.6%); the CodeNet-only variant is close behind on per-example trace accuracy (42.1%, the single best number in the table).

// does it hold up?

The eval read

The comparisons the paper does make are clean: direct-prediction and scratchpad models share the same base model family and the same fine-tuning budget at each scale, so the addition and polynomial results are genuine apples-to-apples ablations of "intermediate steps vs. none," not confounded by a stronger baseline recipe. That's the right control to run, and the addition experiment in particular — sweeping 2M to 1B parameters and reporting both in-distribution and two out-of-distribution digit lengths — is unusually thorough for a paper this early in the scratchpad/chain-of-thought line.

But the paper never benchmarks scratchpads against the adaptive-computation architectures it positions itself against in Related Work — Neural Turing Machines, Universal Transformers, PonderNet. The claim is that scratchpads get similar benefits without an architecture change; that's a real advantage if true, but the paper never actually measures scratchpad-equipped models against those purpose-built alternatives on the same tasks, so "we don't need a new architecture" is asserted, not demonstrated head-to-head.

The headline MBPP result also deserves a second look at what it's measured on. The 500-task official MBPP test split shrinks to 212 tasks (42%) once you keep only programs whose gold execution trace fits in the 512-token generation window — a constraint the authors are upfront about, but one that likely selects for shorter, simpler control flow than MBPP as a whole. Read literally, the strongest configuration still gets the final output right on only 26.6% of tasks (46.0% when counted per input-output example rather than requiring all three correct) — real Python execution remains mostly unsolved by this method, even after three different data-augmentation sources are stacked together. And the few-shot synthetic-tracing number (26.5%) uses a scoring rule the authors modified specifically for that one condition after observing the model's behavior, which the paper discloses with an asterisk but which means it isn't computed the same way as the other three numbers in that table.

baselines

Direct-prediction vs. scratchpad is a clean, same-model-same-budget ablation — but the paper never compares against the adaptive-computation architectures (NTM, Universal Transformer, PonderNet) it cites as the alternative approach to the same problem.

contamination

Not benchmark leakage in the usual sense — the model is deliberately trained on the target skill via scratchpad demonstrations — but there's no independent replication: MBPP was introduced by an overlapping author team and used here as the primary real-code eval.

what's missed

The strong Python-execution results use one fixed model size (137B) with no size sweep, unlike the addition experiments; the 512-token window cuts the MBPP eval to 42% of the official test set; everything is fully supervised on gold intermediate steps — the paper explicitly flags learning to use a scratchpad without direct supervision as unsolved future work; no code or data release is mentioned anywhere in the paper.

would raise confidence

A model-size sweep for polynomial evaluation and Python execution matching the rigor of the addition experiments; a direct comparison against Universal Transformers or PonderNet on the same tasks; evaluation on the full (unwindowed) MBPP test set now that context windows are far larger than 512 tokens; a released dataset/code artifact for independent replication.

// bottom line

The core idea — externalize intermediate computation as generated tokens instead of demanding it in one forward pass — is the paper's lasting contribution, and it's exactly the mechanism that chain-of-thought prompting and later reasoning-trace training would generalize from a supervised, task-specific scratchpad into an unsupervised, general-purpose one. But taken on its own evidence, this paper shows scratchpads reliably beat "no scratchpad" on the same model, not that they beat the architectural alternatives it cites, and its hardest, most realistic result — tracing real Python programs — still tops out under 50% per-example accuracy on a pre-filtered, easier-than-average subset of MBPP.

// takeaways

What to remember

  1. The core move — train a Transformer to emit intermediate computation steps into a "scratchpad" before its final answer, using only ordinary supervised fine-tuning on an unmodified architecture — is the direct supervised precursor to chain-of-thought prompting, which a few months later would show the same benefit without any fine-tuning at all.
  2. The effect scales: scratchpad addition models go from failing almost entirely out-of-distribution to 93% accuracy on 9-digit and 52% on 10-digit addition the model never trained on, while no-scratchpad baselines stay near 0% regardless of size — and the in-distribution gap only opens up past a critical model scale (~10⁸ parameters).
  3. On real, human-written Python programs (MBPP), scratchpad tracing is highly data-hungry: trained on only the original 374 tasks it actually loses to direct prediction (5.1% vs. 10.3%), and only pulls ahead — reaching 26.6% per-task, 46.0% per-example execution accuracy — after stacking three augmentation sources including 17,000 model-sampled programs, a 670,904-trace CodeNet extraction, and a 9-million-example single-line dataset.
