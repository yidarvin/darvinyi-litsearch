PaperBench: Evaluating AI's Ability to Replicate AI Research

Can an AI agent read an ML paper and reproduce it from scratch — write the codebase, run the experiments, match the numbers? OpenAI built a benchmark of 20 ICML 2024 papers, graded against 8,316 author-co-written criteria. The best agent scores 21.0%. ML PhDs still win.

Starace, Jaffe, Sherburn, Aung, Chan, Maksin, Dias, Mays, Kinsella, Thompson, Heidecke, Glaese, Patwardhan · OpenAI

What problem is this solving?

An AI agent that could read a fresh ML paper and autonomously reproduce it — understand the contributions, build the codebase from nothing, run the experiments, and recover the headline numbers — would be a genuine accelerant for research. It is also a capability worth watching closely: PaperBench is explicitly framed as a measure of model autonomy for OpenAI's Preparedness Framework, Anthropic's Responsible Scaling Policy, and Google DeepMind's Frontier Safety Framework.

The trouble is that no existing benchmark captures that full loop. CORE-Bench hands the agent the paper's repository and asks it to re-run the code — reproduction, not replication from scratch. MLE-bench, MLAgentBench, and DSBench score agents on Kaggle competitions, which are often dated, self-contained, and unrepresentative of modern research. RE-Bench poses 7 hand-crafted ML engineering tasks, but each is narrow and ships with a "scoring function" that gives a perfect signal on a single objective — exactly what you cannot write for "reproduce this whole paper."

And that is the deeper problem: how do you grade a from-scratch replication objectively? A paper has dozens of intertwined empirical claims. A single pass/fail is far too coarse, and asking an LLM for a holistic 0–100 score is unreproducible and unauditable. PaperBench's central bet is that the grading machinery — not the task framing — is where the hard work lives.

// how it works

The method

PaperBench contains 20 ICML 2024 Spotlight and Oral papers (plus 2 NeurIPS 2024 Workshop papers as a public dev set, and an internal held-out set). For each, the agent — the "candidate" — is given the paper as PDF and Markdown, an addendum of author clarifications, and task instructions. It must produce a repository containing all the code needed to reproduce the paper's empirical results, with a reproduce.sh at the root as the single entrypoint. The agent may browse the web, but a per-paper blacklist bars the authors' own repo and any other online replication.

Rubrics as hierarchical trees

The grading rubric is the paper's real contribution. Each paper's rubric is a tree that decomposes "replicate this paper" into progressively finer requirements, with the leaves being binary, individually gradable criteria. Across all 20 papers there are 8,316 leaf nodes — hundreds per paper. Crucially, each rubric was co-developed with one of the paper's original authors over multiple weeks, to keep the criteria accurate and realistic.

A leaf is one of three requirement types, which determine how it is graded:

  • Result Match — did the executed submission actually reproduce a specific quantitative result from the paper? (The hardest type; graded from reproduce.sh, its log, and the reproduction's output files.)
  • Execution — did a particular execution actually occur when reproduce.sh ran?
  • Code Development — does the source contain a correct implementation of some requirement, whether or not it was ever run?

A parent's score is the weighted average of its children's scores, propagating up to a single Replication Score for the paper:

Score(node) = Σi wi · Score(childi)Σi wi // leaves are graded 0/1; weighted averages roll up the tree to the paper's Replication Score

Grading at scale: SimpleJudge + JudgeEval

A rubric with hundreds of leaves is far too expensive to grade by hand on every submission, so PaperBench ships an LLM-based judge, SimpleJudge. It grades each leaf independently: prompted with the paper Markdown, the full rubric JSON, the specific leaf requirement, and the submission. Because a whole codebase rarely fits in context, the judge first ranks the files by relevance and keeps only the top ten. The default backend is o3-mini (reasoning effort high), at roughly $66 per paper (≈50M input + 2M output tokens, costed at o1 API pricing).

To know whether the judge can be trusted, the authors build a second benchmark — JudgeEval — by hand-grading partial replications of five papers (four from PaperBench plus one from the dev set, built from scratch or by modifying author code) and treating those human labels as ground truth. Each leaf is a binary classification, so judges are scored with standard metrics. o3-mini reaches an F1 of 0.83 at $66/paper; o1 is marginally better (0.84) but costs $830/paper, so o3-mini is chosen as the most cost-effective judge for the main results.

Finally, a lighter variant — PaperBench Code-Dev — grades only Code Development leaves, skipping execution and result-matching entirely. No GPU run is required and grading drops to ~$10/paper, at the cost of measuring "did the agent write plausible code" rather than "did the code work."

// what they found

The results

With the simple BasicAgent scaffold and a 12-hour limit, the best model is Claude 3.5 Sonnet (New) at 21.0%, with OpenAI o1 at 13.2%. Everything else lands in single digits. The headline takeaway is blunt: current agents cannot replicate ML papers end-to-end.

Inspecting the logs, the authors find a consistent failure mode: every model except Claude frequently quits early, declaring it had either finished or hit an unsolvable problem, and all models fail to budget their time over a long horizon. The bottleneck is not planning — models write fluent multi-step plans — but execution: actually carrying a plan through hours of iterative debugging.

Scaffolding changes the picture — and the ranking

A second scaffold, IterativeAgent, removes the agent's ability to stop early and prompts it to work piecemeal. This boosts o1 to 24.4% (and 26.0% with a 36-hour limit) and o3-mini to 8.5% — but hurts Claude, dropping it to 16.1%. The same prompt change moves o1 up ~11 points and Claude down ~5: the leaderboard is partly measuring scaffold-model fit, not raw capability. On the cheaper Code-Dev variant, o1 with IterativeAgent reaches 43.4% — far higher, because "wrote the code" is much easier than "ran it and matched the result."

Humans still win — by working longer

Eight ML PhDs form the human baseline. On a shared 3-paper subset, the best of 3 human attempts reaches 41.4% after 48 hours, versus 26.6% for o1 on the same subset. The dynamic behind this gap is shown in Figure 3 (plotted on a slightly larger 4-paper subset): o1 sprints ahead in the first hour, then plateaus; humans start slow (digesting the paper) but keep climbing and overtake the agent after the 24-hour mark — the same "fast start, early plateau" shape RE-Bench reported.

None of this is cheap to measure: a single o1 IterativeAgent 12-hour rollout runs ~$400 in API credits, or ~$8,000 to sweep all 20 papers, plus the $66/paper grading on top.

// does it hold up?

The eval read

PaperBench is the most realistic from-scratch replication benchmark to date, and the author-co-written rubrics are a real methodological advance. But three things should temper how you read the numbers.

First, the absolute score is judge-mediated. The o3-mini judge has an F1 of 0.83 on JudgeEval, not 1.0, so roughly one in six leaf judgments may be wrong — and the judge is non-deterministic, so the same submission can score differently on re-grade. The leaderboard's reported precision rides on a fallible, stochastic grader. Worse, JudgeEval validates the judge on separate partial replications, not on the actual leaderboard submissions, so judge accuracy on the graded runs is assumed, not measured.

Second, contamination is a built-in clock. All 20 papers are public ICML 2024 work whose author codebases are online; a model pretrained on a large web corpus may have internalized solutions. The per-paper blacklist stops live browsing of those repos but cannot scrub memorized code from the weights. The authors argue recency protects today's models — a fair point — but they also concede this is a time-bomb: the benchmark's validity decays as these papers age into future training sets, and there is no never-published held-out variant for the public 20.

Third, the headline metric conflates "wrote plausible code" with "reproduced the result." Result Match is the hardest leaf type, yet Code-Dev (code only) scores 43.4% where the full eval scores 13–24%, and the two correlate only weakly (the authors report Pearson r = 0.48 between them for o1) — so a large share of partial credit can come from code that was written but never run successfully. A single Replication Score buries that distinction.

baselines

Snapshot model set (late-2024/early-2025: Claude 3.5 Sonnet New, o1, o3-mini, DeepSeek-R1, GPT-4o, Gemini 2.0 Flash); Claude 3.7 was dropped over API rate limits. Scaffolds are deliberately minimal, and IterativeAgent shows scores swing ±5–11 points with prompt tuning — the authors are explicit these are "initial baselines," not capability ceilings.

contamination

All 20 papers are public with author code online; blacklist blocks live access but not memorization. Self-acknowledged decay risk as papers enter future pretraining corpora; no contamination probe (e.g. post-cutoff or perturbed tasks) is run.

what's missed

The % is only as trustworthy as a 0.83-F1, non-deterministic judge; Result Match vs Code Development is collapsed into one number; 20 ICML Spotlight/Oral papers is a small, prestige-skewed sample, not the messy median paper.

would raise confidence

Human-judge agreement measured on the actual graded submissions; adversarial-submission stress tests of the judge (flagged as future work); a contamination probe on held-out / perturbed papers; fuller reporting of run-to-run variance; and a larger, less prestige-biased paper set.

// bottom line

Read PaperBench as a capability-trend instrument — agents are good at fast early code and bad at long-horizon execution — rather than a precise scoreboard. The grading machinery is the lasting contribution; the absolute percentages ride on a fallible LLM judge and a contamination clock.

// takeaways

What to remember

  1. The best agent (Claude 3.5 Sonnet New, simple scaffold) replicates ICML papers at 21.0%; on the shared 3-paper subset, human ML PhDs hit 41.4% vs o1's 26.6% — agents can't yet replicate ML research end-to-end.
  2. The bottleneck is long-horizon execution, not planning: models write a lot of code fast then plateau after ~1 hour, while humans start slow and overtake them after 24 hours.
  3. The durable contribution is the grading stack — 8,316 author-co-written rubric leaves plus an LLM judge (o3-mini, F1 0.83, ~$66/paper) — which makes from-scratch replication objectively gradable, at the cost of a contamination clock and judge noise.
