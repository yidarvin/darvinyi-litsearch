DeepSeek-Coder: When the Large Language Model Meets Programming - The Rise of Code Intelligence

An open family of code models (1.3B–33B) trained from scratch on 2 trillion tokens, packed at the repository level and pre-trained with a fill-in-the-middle objective. The 33B-Instruct model became the first open code model to beat GPT-3.5-Turbo — and the pretraining backbone of the DeepSeek code lineage that runs through DeepSeekMath, V2/V3 and R1.

// the gap

What problem is this solving?

By late 2023 the best coding models — Codex / GPT-3.5 / GPT-4 — were closed. The open alternatives (StarCoder, CodeLlama, CodeGeeX2) trailed them by a wide margin, which throttled research that needed weights, training recipes, or a permissive commercial license. DeepSeek-Coder's goal is blunt: close the open-vs-closed gap on code with a model anyone can download and ship.

Two design gaps in the open models of the day shaped the recipe. First, prior code LLMs were pre-trained on file-level source code, shuffling files independently. That throws away the cross-file structure of real repositories — the import graph, the call graph — so the models struggled the moment a task spanned more than one file. Second, plain left-to-right next-token prediction never teaches a model to infill: completing a line given the code both before and after the cursor, which is exactly what an IDE autocomplete does. DeepSeek-Coder targets both: it builds training samples that respect a repository's dependency order, and it bakes a fill-in-the-middle task into pretraining.

// how it works

The method

DeepSeek-Coder is a decoder-only Transformer (RoPE, SwiGLU; the 33B uses grouped-query attention with group size 8 and FlashAttention-v2) released at 1.3B, 6.7B and 33B parameters, each in a base and an instruction-tuned variant. Every model is trained from scratch on 2 trillion tokens spanning 87 programming languages. The mix is 87% source code, 10% English code-related text (GitHub Markdown + StackExchange) and 3% Chinese natural language. The real contributions are in how the data is built and what objective the model is trained on.

Repository-level data construction

Public GitHub repos created before February 2023 are crawled, run through StarCoder-style rule filters (which alone cut the corpus to 32.8% of its raw size), then — the novel step — packed by dependency order. For each repository they parse invocation edges between files (regex on import / using / include), build a dependency graph, and run a topological sort so a file always appears after the files it depends on. A modified sort that picks minimal- (not zero-) in-degree nodes handles cyclic imports. Each ordered repository is concatenated into one training sample, with a path comment prepended to every file. After repo-level near-deduplication and a compiler/quality-model screen, the cleaned corpus is 798 GB across 603 million files. Decontamination is an n-gram filter: any file sharing a 10-gram (or an exact 3-gram for short strings) with HumanEval, MBPP, GSM8K or MATH is dropped.

Fill-in-the-Middle (FIM)

Alongside next-token prediction, half of the training examples use a fill-in-the-middle objective. A file is split into three pieces — prefix, middle, suffix — which are then reordered and joined with sentinel tokens so the model learns to generate the middle conditioned on both sides. DeepSeek-Coder uses the PSM (Prefix-Suffix-Middle) layout at a 0.5 FIM rate, applied at the document level before packing:

<|fim_start|> fpre <|fim_hole|> fsuf <|fim_end|> fmiddle <|eos|> // PSM-mode FIM training example: predict the masked middle from prefix + suffix

How much FIM? They ablate it on a 1.3B model over a Python subset, sweeping the rate across 0%, 50%, 100% and a 50% masked-span (MSP, T5-style) variant, scored on HumanEval-FIM (infilling) vs. HumanEval / MBPP (left-to-right generation). The trade-off is clean: 100% FIM maximizes infilling but gives the weakest code completion, while 0% can't infill at all. A 50% PSM rate also beats the 50% MSP alternative. So 0.5 PSM is the chosen balance — strong infilling without sacrificing generation.

Context, tuning, and a general-LLM variant

The context window is extended to 16K tokens by linear RoPE scaling (scale factor 1→4, base frequency 10000→100000) plus 1,000 extra steps — enough for repository-scale inputs. DeepSeek-Coder-Instruct is then produced by supervised fine-tuning on high-quality instruction data in the Alpaca format (2B tokens total). Finally, to test whether code skill is best built on a strong general model, they continue-pretrain DeepSeek-LLM-7B Base on 2T tokens of code+math+language to make DeepSeek-Coder-v1.5.

// what they found

The results

On the multilingual HumanEval + MBPP suite (re-running every baseline in one harness for fairness), DeepSeek-Coder-Base 33B sets a new open-source state of the art: an 8-language HumanEval average of 50.3% and MBPP 66.0% — about 9 and 11 points over CodeLlama-Base 34B (41.0% / 55.2%). Strikingly, the much smaller 6.7B base (44.7% / 60.6%) already surpasses CodeLlama-34B, a model five times its size. After instruction tuning, DeepSeek-Coder-Instruct 33B (69.2% HumanEval avg, 70.0% MBPP) edges past GPT-3.5-Turbo (64.9% / 70.8%) and narrows the gap to GPT-4 (76.5% / 80.0%). On Python HumanEval specifically, the 33B-Instruct hits 79.3%, between GPT-3.5-Turbo (76.2%) and GPT-4 (84.1%).

The same ordering holds across the other tasks: on DS-1000 (realistic data-science workflows) the 33B base scores 40.2% vs CodeLlama-34B's 34.3%; on single-line infilling the models top the table — the 1.3B (70.4% mean) already beats StarCoder-16B (69.7%) and the 33B reaches 81.2%; and on program-aided math (PAL across 7 benchmarks) the 33B base averages 65.8% vs 62.0%. The held-out LeetCode Contest benchmark (180 problems collected July 2023–January 2024, after the data cutoff) is the toughest test: DeepSeek-Coder-Instruct 33B scores 27.8% (28.9% with chain-of-thought), the only open model to beat GPT-3.5-Turbo (23.3%), though GPT-4-Turbo (40.6%) stays comfortably ahead.

Two ablations carry the paper's thesis. Repository-level pretraining pays off: on CrossCodeEval (cross-file completion, built from repos created after the cutoff so there's no leakage), dropping repo-level packing degrades Java, TypeScript and C#. And code models are best built on strong general models: DeepSeek-Coder-v1.5 7B, continued from DeepSeek-LLM-7B, gives up almost nothing on coding (HumanEval 44.7%→43.2%) while leaping on reasoning and language — GSM8K 43.2%→62.4%, MATH 19.2%→24.7%, MMLU 36.6%→49.1%, BBH 44.3%→55.2%. That finding is the seed of the DeepSeekMath / R1 line that followed.

// does it hold up?

The eval read

The headline — "open model beats GPT-3.5-Turbo on code" — is real, and the evaluation is unusually careful for a model report: every open baseline is re-run "using the same script and environment," which removes the usual apples-to-oranges noise of quoting numbers from other papers. But a few things deserve a skeptic's eye.

The most reassuring numbers are on the benchmarks the team can't have trained on — the self-built LeetCode Contest set (July 2023–January 2024) and CrossCodeEval (repos from March–June 2023), both strictly after the February-2023 cutoff. The least reassuring are HumanEval and MBPP, which the paper itself calls "straightforward programming tasks that may not accurately represent the kind of code most programmers typically write," and which have been mirrored on GitHub for years; n-gram decontamination shrinks but cannot eliminate that exposure, and the same risk applies to every baseline. The team is admirably candid here — they note GPT-4-Turbo and DeepSeek-Coder both scored higher on the July/August LeetCode problems and explicitly warn that contamination "cannot be entirely ruled out."

Two framing caveats. The flattering teaser numbers (e.g. the 28.9% LeetCode bar) are the chain-of-thought variant; the plain 33B-Instruct is 27.8%, and CoT helps unevenly — it lowers the 33B's easy-subset score (57.8%→53.3%) and the smaller models' hard-subset scores, even as it lifts overall pass@1. And the instruction-tuning recipe is a black box: "helpful and impartial human instructions," Alpaca-formatted, 2B tokens, with no dataset released — so the Instruct results (the ones that beat GPT-3.5) are the hardest to reproduce. Finally, all evals are single-sample greedy pass@1 on puzzle-style or single-function tasks; there is no execution-based, repository-editing agentic test (SWE-bench-style) — a gap the field, and DeepSeek's own later work, would soon fill.

baselines

Strong and fair: all open baselines (StarCoder, CodeLlama, CodeGeeX2, WizardCoder, Phind) re-run in one harness; closed comparisons are GPT-3.5-Turbo and GPT-4/-Turbo snapshots — current for early 2024, but a moving target.

contamination

Handled better than most: 10-gram decontamination + Feb-2023 cutoff, and the decisive evals (LeetCode, CrossCodeEval) are post-cutoff held-out sets. HumanEval/MBPP exposure is acknowledged but not quantified.

what's missed

Greedy pass@1 on single-function / competitive-puzzle tasks; no multi-sample pass@k, no agentic repo-editing eval. The instruction-tuning data is undescribed and unreleased.

would raise confidence

An execution-based SWE-bench-style benchmark, n-gram overlap statistics for HumanEval/MBPP, pass@k curves, and a documented (ideally released) instruction-tuning recipe.

// bottom line

A genuinely strong, well-evaluated open code model whose most load-bearing claims rest on properly held-out, post-cutoff benchmarks. The lasting contributions aren't the leaderboard wins — soon eclipsed by its own successors — but two recipes the field adopted: repository-level dependency packing and the finding that the best code LLMs are continued from strong general LLMs.

// takeaways

What to remember

  1. The open-vs-closed code gap got closed. DeepSeek-Coder-Instruct 33B was the first open model to beat GPT-3.5-Turbo on code, with a permissive license for commercial use — and the 6.7B base already matched CodeLlama-34B.
  2. Two recipes outlived the leaderboard. Pack pretraining data by repository dependency order (topological sort over the import graph), and train with a 0.5-rate fill-in-the-middle objective for IDE-style infilling.
  3. Build code models on strong general models. Continuing DeepSeek-LLM-7B into DeepSeek-Coder-v1.5 kept the coding skill while sharply raising math and reasoning — the thesis that seeded DeepSeekMath, V3 and R1.
