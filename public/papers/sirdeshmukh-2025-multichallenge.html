MultiChallenge: A Realistic Multi-Turn Conversation Evaluation Benchmark Challenging to Frontier LLMs

Older multi-turn tests are saturated — frontier models ace them. MultiChallenge rebuilds the bar from four failure modes that real conversations actually hit, and watches every frontier model fall below 50%.

Sirdeshmukh, Deshpande, Mols, Jin, Cardona, Lee, Kritz, Primack, Yue, Xing · Scale AI

// the gap

What problem is this solving?

Almost every chat with an LLM is multi-turn, yet almost every benchmark is single-turn. The few multi-turn evals that exist have stopped discriminating: MT-Bench and MT-Eval — once the standard — are saturated, with GPT-4, Claude 3, and Llama 3 all scoring near-perfect. A benchmark that everyone passes measures nothing.

The newer attempts fix the wrong thing. Multi-IF extends IFEval into multi-turn and multilingual settings, but it tests explicit, auto-verifiable instruction following — "keep every word in uppercase," "use exactly three bullet points." That is a clean, gradeable signal, but it is not what makes real conversations hard. Natural multi-turn dialogue forces three capabilities at once: following instructions, allocating attention across a growing context, and doing in-context reasoning over that context. Existing benchmarks isolate the first and ignore the other two.

MultiChallenge targets exactly that intersection. Its design constraint is unusual and load-bearing: every example must be realistic (the kind of slip that happens in everyday use), must require the full mix of capabilities, and must actually break current frontier models. The result is a benchmark that is small (273 conversations) but deliberately un-saturated — the top model clears barely 40%.

// how it works

The method

MultiChallenge is 273 conversation histories, each at most ten turns and ending on a final user turn that the model under test must answer. Every example belongs to one of four challenge categories, each isolating a distinct way multi-turn conversations go wrong:

  • Instruction Retention (69 examples) — a constraint stated in the first turn must still hold many turns later. In Figure 1, the user asks for film suggestions and specifies up front that everything must carry a UK age rating of 15 or 18; the model is graded on whether its final recommendation still obeys that constraint. The user never repeats it, and never issues a contradicting instruction.
  • Inference Memory (113 examples) — a detail dropped early must be silently recalled and connected later, when it is only implicitly relevant. A partner's nut allergy is mentioned in passing; turns later the user asks for dessert recipes. The crucial design rule: the final turn never directly asks for the fact ("does my girlfriend have a nut allergy?"). It only implies that the fact matters — so the model is tested on reasoning about relevance, not on lookup.
  • Reliable Versioned Editing (41 examples) — iterative revision of a document, schedule, or snippet across several versions, where the user changes their mind and refers back anaphorically ("the plan we had before we adjusted the workshop's ending time"). The model must resolve the reference, reproduce the right version without hallucinating, and apply the new edit.
  • Self-Coherence (50 examples) — staying consistent with one's own prior answers and resisting sycophancy. Having told the user that the next e-reader step is to register the device, the model must not cave when the user nudges, "all that's left to do now is choose a book, right?" Most frontier models flip to agree, contradicting themselves.

Grading: one binary rubric question per example

There is no single ground-truth answer to a 1,200-word conversation, so rule-based scoring is out. But handing the whole transcript to an LLM judge and asking "did the model do well?" aligns poorly with humans — only 37.33% of the time. The reason is circular: every frontier judge also scores under 50% on MultiChallenge, so it cannot reliably grade what it cannot itself do.

The fix is to collapse the hard judgment into an easy one. When an example is authored, the human writes a single binary (yes/no) rubric question that (a) needs only the model's final response as context and (b) is squarely within current LLM ability. For the nut-allergy example: "does any dessert recipe in this response contain nuts?" A judge model answering that narrow question hits 93.95% alignment with expert raters — above 90% in all four categories — versus 37.33% for the naïve whole-transcript baseline. The score of a model is just the fraction of examples whose final response passes its rubric question.

keep example ⟺ (# of 6 frontier models that fail) ≥ 3  ∧  rubric answerable by judge
score(M) = # examples where M's final response passes its binary rubric273 // the acceptance gate keeps only hard examples; the rubric makes auto-grading reliable

Building it: MMSE, then humans

Authoring realistic, frontier-breaking conversations by hand is slow, so the data is generated by a multi-agent system (MMSE) and then edited by people. MMSE takes three seeds — a hierarchical topic, a persona sampled from HuggingFace PersonaHub, and a category-specific config (definition, pass/fail criteria, few-shot failures) — and runs three roles: a Planner Agent that writes and updates a conversation "blueprint," a User Agent that voices the blueprint as concrete user turns designed to trip the assistant, and a Responder Agent that plays the assistant. Crucially, the responder is randomly sampled per example from a pool of six frontier models, so the data never overfits one model's quirks. The planner loops until it detects a failure (save the conversation) or hits the turn cap (discard it).

Synthetic drafts are then reviewed and edited by trained annotators on three axes: alignment to the category definition, naturalness, and whether the six frontier models fail for the right reasons. An example is only accepted if at least 3 of the 6 frontier models fail it, then passed through two further review layers drawn from a separate reviewer pool. MMSE roughly halves authoring cost — 73.6 minutes of human time per example with synthetic scaffolding versus 154.4 minutes from scratch — but the human pass is substantial: the final examples share only 74.5% string similarity with their synthetic originals (i.e. humans rewrote about a quarter of every conversation).

// what they found

The results

Every frontier model scores under 50%. Claude 3.5 Sonnet (June 2024) leads at 41.42% human-evaluated average, well ahead of second-place o1-preview at 37.23%, with a steep drop to Gemini 1.5 Pro (19.96), Llama 3.1 405B (14.92), Mistral Large (14.63), and GPT-4o (12.52). The automatic evaluator reproduces both the numbers and the ranking almost exactly — confirming the instance-rubric design rather than just re-scoring it.

The per-category spread is the point, not noise: it shows the four challenges tap genuinely different abilities. Claude wins overall but loses to o1-preview on Versioned Editing (24.39 vs 39.02) and Inference Memory (37.29 vs 41.53); Gemini matches o1-preview on Instruction Retention (31.43 vs 34.29) while trailing it badly elsewhere. No model is uniformly strong, which is exactly what you want from a benchmark meant to decompose "multi-turn ability" into parts.

Difficulty is not length

A natural worry for any multi-turn benchmark is that it just measures long-context endurance. It does not. Performance shows no monotonic trend with the number of turns (Figure 4), and the conversations are short anyway — 5 turns and ~1,232 words on average, far inside every model's context window. The hardness comes from the reasoning the context demands, not from its length.

Open models trail — even with the bias against the panel

Run through the auto-evaluator, the strongest open model is Llama-3.3-70B at 23.19%, ahead of Qwen2-72B (20.58), with the rest between 11 and 17%. Because every example was selected to break the six closed frontier models, the benchmark is, if anything, biased against those six — yet open models still land below Claude and o1-preview. Larger open models tend to do better (Figure 5), though the trend is uneven: Reliable Versioned Editing accuracy actually falls with scale, a reminder that the per-category samples are small.

// does it hold up?

The eval read

MultiChallenge gets the most important thing right: it is hard, it is realistic, and its difficulty is reasoning-bound rather than a length stunt. The instance-level binary rubric is a genuinely clever move — it converts an unjudgeable open-ended task into a reliable 94%-aligned auto-eval by shrinking the judge's job to a question the judge can actually answer. That trick is the paper's most transferable contribution, and you can see its DNA in the rubric-based evals that followed it (HealthBench, PRBench, the rubric-reward RL line).

But three structural choices bound what the headline number means. First, the benchmarked models built the benchmark. Examples are kept only when ≥3 of the same six frontier models fail, so the test set is adversarially mined against that exact panel — their absolute scores are deflated by construction and are not directly comparable to a model that did not shape the data. The authors are upfront about this and argue that future models within those six families, and open models, are still compared fairly; that is reasonable, but a reader should not read "Claude 41.4%" as a clean capability estimate.

Second, the difficulty ceiling is capped on purpose. To keep the LLM judge trustworthy, any example whose rubric question is itself beyond current models is excluded and held private. So the public benchmark systematically omits its hardest cases — the score is a floor on difficulty, and the most interesting failures are the ones you cannot see. Third, one binary rubric per example grades a single failure mode on the final turn only: no partial credit, no measure of mid-conversation recovery, no read on overall answer quality beyond the one trap. That is the deliberate price of reliable grading, but it means MultiChallenge scores "avoided this specific slip," not "held a good conversation."

Finally, scale. 273 examples, unevenly split (113 Inference Memory vs 41 Versioned Editing), means the eye-catching sub-results — "editing gets worse with size," the per-turn heatmap cells — ride on tens of examples and come with no reported confidence intervals. Treat the category-level ranking as the signal and the second-decimal numbers as noise.

baselines

The model panel (Claude 3.5 Sonnet Jun-2024, o1-preview, GPT-4o, Gemini 1.5 Pro, Mistral Large, Llama 3.1 405B) is a 2024 frontier and now a generation old — but the deeper issue is that this same panel both generated and filtered the data, so its scores are structurally deflated rather than neutral.

contamination

Strong by design: synthetic + heavily human-edited conversations with no fixed gold string, graded only on the final response. The hardest examples are kept private; the public 273 are on GitHub, so ordinary public-benchmark leakage applies over time.

what's missed

A single binary check on the final turn — no partial credit, no recovery dynamics, no overall conversation-quality signal. The capped difficulty hides the hardest cases, and small, uneven category sizes (41–113) leave sub-results without error bars.

would raise confidence

Refresh the panel so the tested models aren't the data generators; report CIs on per-category / per-turn cells; grow and balance the categories; release (or rotate in) the private hard split; add multi-rubric or graded scoring to capture quality beyond one yes/no.

// bottom line

A small, sharp, deliberately un-saturated benchmark that re-opens multi-turn evaluation after MT-Bench saturation — and whose instance-level binary rubric became a template for reliable LLM-as-judge grading. Read the per-category ranking, not the absolute scores: the panel that built it is the panel it grades.

// takeaways

What to remember

  1. Four realistic failure modes, all under 50%. Instruction retention, inference memory, versioned editing, and self-coherence — every frontier model fails the majority of the time, with Claude 3.5 Sonnet (Jun 2024) topping out at 41.4% and o1-preview at 37.2% while the rest sit below 20%.
  2. The binary rubric is the real contribution. Reducing each example's grade to one easy yes/no question on the final response lifts judge–human alignment from 37% to 94% — a reusable recipe for making LLM-as-judge trustworthy on tasks the judge itself can't fully do.
  3. Difficulty is reasoning, not length. Performance doesn't track turn count and the conversations are short; what breaks models is allocating attention and reasoning over scattered context — but the benchmarked models also generated and filtered the data, so read the category ranking, not the absolute scores.
