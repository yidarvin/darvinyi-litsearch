ZeroSearch: Incentivize the Search Capability of LLMs without Searching

Train an LLM to search by having a second LLM play the search engine — no real API calls during RL, and the resulting policy beats agents trained against live Google Search.

// the gap

What problem is this solving?

RL is the dominant recipe for teaching an LLM to use a search engine mid-reasoning — issue a query, read what comes back, decide whether to search again or answer. Search-R1 and R1-Searcher pioneered this by training against static local corpora (usually a Wikipedia dump), which is cheap but unrealistic: the model never learns to cope with the noise, redundancy, and formatting chaos of an actual search result. DeepResearcher and WebThinker closed that realism gap by training directly against live commercial search engines (Google via SerpAPI), and it shows — but it introduces two problems the ZeroSearch authors call out explicitly.

First, uncontrolled document quality: what a real search engine returns for a given query is unpredictable — sometimes gold, sometimes irrelevant — and that noise destabilizes RL training in ways the designer cannot control or schedule. Second, prohibitively high API costs: RL rollouts require repeated sampling (the paper's own runs generate on the order of 64,000 search requests for a single training run), and at commercial API pricing that becomes a real budget constraint, not just an inconvenience — the paper measures a Google Search bill of $586.7 for one experiment's worth of queries. Cost and instability aren't just implementation annoyances here; they directly cap how much RL search-training research anyone outside a well-funded lab can do. ZeroSearch's bet is that neither is actually necessary: an LLM that has absorbed enough of the internet during pretraining should already be able to play a search engine convincingly enough to teach another LLM how to search one.

// how it works

The method

ZeroSearch replaces the real search engine in the RL loop with a second, frozen LLM — the simulation LLM — that generates search results on demand. The policy model still emits the same <think> / <search> / <answer> interaction format used by Search-R1; the only difference is what answers the <search> call. Figure 1 shows the resulting training loop for both PPO and GRPO: the rollout module still contains a "Simulation LLM ↔ Policy Model" exchange, but the simulation LLM's parameters never update — it is a frozen prop, not a trainee.

Getting a raw LLM to sound like a search result takes a bit of work — directly prompting it produces text with an obvious style gap from real engine output. So the authors first collect real interaction trajectories (policy model against an actual search engine), extract the query→document pairs, and use an LLM-as-judge to label each document as useful (contains enough to answer the question) or noisy (doesn't). A lightweight supervised fine-tuning pass then teaches the simulation LLM to emit either kind on command — the prompt template literally has a [useful / noisy] slot, plus the ground-truth question and answer folded in to widen the simulator's effective knowledge. The practical payoff of this SFT step is precise, controllable document quality: the designer can now dial noise up or down at will, something no real search API offers.

That control is what powers the paper's second idea, a curriculum rollout: the probability of serving a noisy document is scheduled to rise over training,

pi = ps + bi/m − 1b − 1 (pe − ps) // pₛ, p_e: start/end noise probability · i/m: current/total training steps · b = 4 by default

Early in training the simulator mostly returns useful documents, so the policy first locks in the basic search-think-answer format; as i/m climbs, noisy documents become more common, forcing the policy to get better at recognizing and recovering from bad retrieval — exactly the skill live-search training provides "for free" but without the API bill or the loss of designer control over difficulty pacing.

Two more details matter for stability. The reward is token-level F1 overlap between prediction and ground truth rather than exact match — the authors found EM invited reward hacking (the policy padding answers with extra words to raise recall of a match). And during optimization, document tokens returned by the simulation LLM are masked out of the loss — gradients only flow through tokens the policy itself generated, since the retrieved text isn't something the policy actually chose to write. The framework is agnostic to the RL algorithm on top of this loop; the paper reports results with REINFORCE, PPO, and GRPO.

// what they found

The results

Across seven QA benchmarks (single-hop: NQ, TriviaQA, PopQA; multi-hop: HotpotQA, 2WikiMultiHopQA, MuSiQue, Bamboogle) and three model families (Qwen-2.5-7B, Qwen-2.5-3B, LLaMA-3.2-3B, each Base and Instruct), ZeroSearch beats every baseline it's compared against — including Search-R1, the one RL-search baseline the paper chooses to compare against directly (chosen, the authors say, because it "avoids complex reward design, data selection, or elaborate training pipelines," making the comparison cleaner). On Qwen-2.5-7B-Base, ZEROSEARCH-base averages 40.93 exact-match across the seven sets versus Search-R1-base's 39.51 — a real-search-trained agent losing to one that never queried a real search engine.

The reward curves in Figure 2 tell the more interesting story about how that happens. On Qwen-2.5-3B, ZeroSearch and Search-R1 climb together for the first ~50 steps, but ZeroSearch pulls ahead afterward and settles into a smoother, less-fluctuating trajectory than Search-R1's — the curriculum's gradually-hardening documents appear to buy training stability that unpredictable live search results don't. Panel (c) tracks a second signal on LLaMA-3.2-3B: interaction turns per rollout drop sharply in the first ~50 steps as the policy learns not to waste calls, then reward and turn-count both climb and plateau together once the model has locked in an efficient search strategy.

Scaling the simulator matters a lot. With Qwen-2.5-3B-Base as the policy (Table 4 in the paper), a fine-tuned 7B simulator already matches real Google Search (33.53 vs. 32.81 average EM), and a 14B simulator surpasses it (34.47 vs. 32.81) — while a merely prompted (not fine-tuned) simulator of the same sizes lags well behind both, confirming that the lightweight SFT step, not raw scale, is what closes the style gap. // does it hold up?

The eval read

The headline claim — a policy trained without ever touching a real search engine beats one that trained against live Google Search — is well-supported within the evaluation the paper chose to run. But that evaluation has a structural blind spot that maps precisely onto the method's own limitation, and the paper never tests it.

Every one of the seven benchmarks (NQ, TriviaQA, PopQA, HotpotQA, 2WikiMultiHopQA, MuSiQue, Bamboogle) asks about static, long-settled facts — the kind of thing that was already stable in 2019–2022 web snapshots and almost certainly sits inside the pretraining data of Qwen-2.5 and LLaMA-3.2 alike. That's exactly the regime where a simulation LLM's "knowledge frozen at pretraining" is indistinguishable from a real search engine's index: both already contain the answer. The paper's own selling point is that ZeroSearch works because the simulator has "acquired extensive world knowledge during large-scale pretraining" — which is precisely the property that would fail on anything time-sensitive (a live sports score, a just-announced product, a person who died last month). Live web search's whole comparative advantage is freshness, and this evaluation suite structurally cannot surface it, because none of the seven datasets contain fresh information. The paper frames itself as eliminating the need for "a real search engine," but it only demonstrates that on the subset of queries where a real search engine's freshness doesn't matter.

There's a second, related gap: no contamination analysis anywhere in the paper. NQ, TriviaQA, HotpotQA, and 2WikiMultiHopQA are years-old, heavily-studied benchmarks; if their answers are already latent in the policy model's pretraining weights, then "learning to search" on them partly measures how well RL elicits recall the model already had, not genuine information-seeking behavior. That confound applies to Search-R1 too, but it's especially pointed here, since ZeroSearch's core mechanism is explicitly "an LLM generating documents from its own world knowledge" — the paper never asks whether the policy is doing the same thing on the eval set, just faster.

On the comparison itself: restricting the RL-search baseline to Search-R1 alone (explicitly declining to also run R1-Searcher or ReSearch, both cited as directly comparable contemporaneous methods) is defensible for controlling confounds — the paper re-scores every RL method under the same F1 reward for a fair fight — but it does mean the claim "ZeroSearch beats RL-based search training" rests on one baseline, not the full field it name-checks in the related work.

baselines

Only Search-R1 is run as the RL-search comparison (R1-Searcher and ReSearch are cited but excluded, "as it avoids complex reward design"); all RL methods are re-scored under a shared F1 reward for fairness, which is good practice but narrows the baseline set the headline claim rests on.

contamination

No contamination or decontamination check anywhere. All seven QA sets are old, widely-used, Wikipedia/web-sourced benchmarks the base LLMs plausibly saw during pretraining — doubly relevant here since the simulator's entire mechanism is "recall pretraining knowledge and phrase it like a search result."

what's missed

Every eval set tests static, multi-year-old facts. Nothing measures the one scenario where a knowledge-frozen simulator should structurally fail and a real search engine should structurally win: fresh, post-cutoff information.

would raise confidence

A head-to-head run on a time-sensitive benchmark (e.g., a RealtimeQA-style set of current events) against real search, plus a decontamination pass on the existing seven datasets, would tell us whether the "simulator ≥ real search" finding is general or an artifact of testing only stale facts.

// bottom line

ZeroSearch is a genuinely clever, well-executed cost/stability fix for RL search-training, and the ablations (curriculum, loss masking, algorithm choice) are honest and convincing on their own terms. But its evaluation only tests the setting where its central simplification — "a pretrained LLM already knows enough to fake a search engine" — is true by construction. Until someone runs it on genuinely fresh information, the claim that it can replace real search is a claim about static-fact QA, not about search.

// takeaways

What to remember

  1. ZeroSearch replaces the real search engine in RL rollouts with a second, frozen LLM fine-tuned to emit "useful" or "noisy" documents on command — turning search-agent RL into a closed two-LLM loop with zero API calls.
  2. A curriculum schedule (pi = ps + bi/m−1/(b−1) · (pe − ps)) gradually raises the odds of noisy documents, letting the policy master format first and robustness second — and produces a smoother reward curve than training against an actual, uncontrollable search engine.
  3. Scale and economics both favor the simulator: a 14B fine-tuned simulation LLM edges out live Google Search on average accuracy (34.47 vs. 32.81 EM) while costing $70.8 against $586.7 for the same ~64,000-query run.
  4. The catch is evaluative, not architectural: every benchmark used tests static, pretraining-era facts, which is exactly the regime where a simulator built from pretraining knowledge can't be told apart from a real index — the method's advantage on fresh information remains untested.
