CORE-Bench: Fostering the Credibility of Published Research Through a Computational Reproducibility Agent Benchmark

Can an agent reproduce a published paper's own results, from its own code? CORE-Bench turns that question into 270 graded tasks across 90 papers in computer science, social science, and medicine, at three difficulty tiers — code output already given, a Dockerfile given, or nothing but a README. The best baseline agent, a lightly customized AutoGPT, clears 60% of the easiest tier and just 21% of the hardest, where it has to install its own dependencies and figure out how to run the code at all.

Zachary S. Siegel, Sayash Kapoor, Nitya Nadgir, Benedikt Stroebl, Arvind Narayanan · Princeton

// the gap

What problem is this solving?

Computational reproducibility — rerunning a study's own provided code and data and getting its own reported results back — is the most basic bar a computational finding can clear, and the paper's survey of the literature finds that bar gets missed constantly. Even when code and data are available, the paper tallies field after field where reproduction attempts fail at high rates: 1,480 of 2,000 studies surveyed by Trisovic et al. across disciplines, 472 of 549 NLP papers surveyed by Belz et al., and — closer to home for this paper's own field — only 18 of 28 papers in the 2022 ML reproducibility challenge were fully reproducible despite code and data being available, with 6 of those 28 remaining unreproducible even after participants corresponded directly with the original authors.

Meanwhile, coding agents were visibly getting better at software tasks in general — on SWE-bench, language models alone solved under 5% of real GitHub issues while agentic scaffolds pushed past 30% — which fed a wave of claims that AI could soon automate scientific research end to end. The authors' rejoinder is procedural: before an agent can be trusted to conduct novel research, it should first be able to reliably reproduce existing research, a narrower and more checkable capability. No benchmark measured that specific skill. Building one is harder than it sounds — verifying that a given paper's code is actually reproducible can take a human researcher hours per paper, which makes constructing a benchmark at any real scale a bottleneck in itself.

// how it works

The method

CORE-Bench sidesteps the hours-per-paper verification bottleneck by building entirely from CodeOcean.com capsules — packaged code+data environments that are already known to be reproducible with comparatively little effort. Each task hands an agent a capsule's code repository and a set of task questions, drops it into an isolated environment, and asks it to run the code, extract the answers from whatever the code produces, and submit a structured report.

Capsule selection runs through a documented ten-criterion filter (publicly tied to a real paper; CS, medical, or social science; written in Python or R; has a README; runs in under 45 minutes; needs only a simple Bash command; produces results labeled clearly enough to write task questions against; low result variance; under 10 GB; locally reproducible) applied to 5,090 initial CodeOcean capsules. That funnel narrows to 2,536 in-discipline capsules, then 1,657 written in Python or R, and finally 90 capsules that survive every criterion — 25 medical science, 28 social science, and 37 computer science (drawn from 355 randomly sampled CS candidates, since CS capsules were far more plentiful).

For each of the 90 papers (split 45/45 into train and test sets), the authors hand-wrote task questions about the outputs of a successful run — e.g. "report the test accuracy of the neural network after epoch 10," or a vision question asking an agent to read a correlation value off a specific plot. A task counts as solved only if every one of its questions is answered correctly, and every task has at least one open-ended, non-guessable question. Each of the 90 papers becomes three tasks — one per difficulty tier, sharing the same 181 underlying questions across the corpus — that vary only in how much scaffolding the agent is handed:

TierGiven to the agentRequired of the agent
CORE-Bench-EasyThe complete code output from a successful runPure information extraction — find the answer, don't run anything
CORE-Bench-MediumDockerfile + README with run instructionsExecute the Docker command, then extract results
CORE-Bench-HardREADME only, no DockerfileInstall dependencies, determine the run command, execute, then extract results

Two baseline agent families are evaluated, both built on AutoGPT: plain AutoGPT (unmodified except for a bolted-on query_vision_language_model tool for reading figures), and CORE-Agent, a task-specific variant that adds a programmatic check on the submitted report's format plus a short list of difficulty-tier-specific prompting hints distilled from qualitatively reading the training set's failure logs — e.g. "use pdftotext/pdftoppm for PDF results" on Easy, "use absolute paths instead of environment variables" on Medium, "determine and install dependencies before running code" on Hard. Both agents run on GPT-4o and GPT-4o-mini backends, capped at $4 of API spend per task. Every task runs in its own isolated virtual machine, orchestrated by a manager process that provisions VMs, launches the agent, and collects results in parallel — cutting total evaluation time for the full 270-task suite from an estimated 20+ days sequentially to a little over two hours.

Grading accounts for the fact that some code genuinely produces different outputs run-to-run: the authors manually reproduced every capsule three times and grade an agent's answers correct if they fall within a 95% prediction interval built from those three runs — deliberately more forgiving than a single fixed ground-truth value, though the authors note only 17 of the 181 task questions actually have stochastic answers.

// what they found

The results

CORE-Agent with GPT-4o is the top performer at every difficulty level on the held-out test set: 60.00% on Easy, 57.78% on Medium, and 21.48% on Hard. The same architecture with the smaller GPT-4o-mini backend scores 44.44% / 32.59% / 16.30%, while unmodified AutoGPT trails badly — 35.56% / 37.78% / 6.67% with GPT-4o, and just 8.89% / 2.22% / 2.22% with GPT-4o-mini. Task-specific tuning is disproportionately valuable for the weaker model: a few prompt hints and a report-format check took GPT-4o-mini's Easy-tier accuracy from 8.9% to 44.44%, a roughly five-fold jump, versus a smaller relative gain for GPT-4o.

Two findings point at why agents fail rather than just how often. First, vision questions are the harder modality: on the Easy test set, CORE-Agent with GPT-4o answered 87.88% of written questions correctly but only 59.26% of vision questions (reading a value off a plot or figure); with GPT-4o-mini the gap is even wider (81.81% written vs. 37.78% vision). Second, Python tasks are markedly easier than R tasks — R capsules more often emit results as full PDF manuscripts an agent has to parse, and R dependency installation tends to run longer than Python's; since computer-science capsules are disproportionately Python-based, this partly explains why CS was the most reproducible discipline of the three. Third, the $4 cost cap is not the binding constraint it looks like: raising it to $10 on the Hard-tier training set moved GPT-4o's accuracy only from 26% to 31% and left GPT-4o-mini essentially unchanged, because agents that eventually succeed tend to succeed cheaply (CORE-Agent+GPT-4o's successful Hard-tier runs average $0.54) while agents that fail get stuck repeatedly re-attempting the same broken dependency install and simply burn out the budget rather than making incremental progress toward it ($2.59 average for failed runs).

// does it hold up?

The eval read

The most careful part of this paper's design is also its most quietly important: grading against a 95% prediction interval from three independent manual runs, rather than a single brittle ground-truth string, is exactly the right response to code that has genuine run-to-run variance, and it's a level of grading rigor plenty of agent benchmarks skip entirely. That said, the headline 60% / 58% / 21% numbers deserve more scrutiny than the paper's confident framing invites. CORE-Agent's gains over AutoGPT come substantially from a short, hand-written list of difficulty-tier-specific prompting hints — "use pdftotext," "use absolute paths," "install dependencies before running" — distilled from reading training-set failure logs. That's a legitimate and honestly-labeled engineering exercise (the paper explicitly frames CORE-Agent as task-specific, not general-purpose), but it also means the reported numbers describe an agent partially reverse-engineered against this exact benchmark's own quirks. The 45/45 train/test split is the right instinct for checking that these hints generalize rather than overfit, but no train-vs-test accuracy gap is reported in the main text as an explicit validation of that generalization.

The benchmark's construction methodology also filters toward the easy end of real-world reproducibility by design: all 90 papers come from CodeOcean, a platform whose capsules are cited as already "reproducible with little effort," and the ten selection criteria explicitly exclude anything that takes over 45 minutes to run, has high result variance, needs a non-trivial command to execute, or produces poorly labeled output. That's a defensible simplification — verifying reproducibility "in the wild" is expensive, and the authors are transparent that this is a deliberate trade-off — but it means CORE-Bench-Hard's 21% ceiling is likely an optimistic estimate of how well agents would do reproducing an arbitrary paper pulled off GitHub with no README, an undocumented conda environment, and hours-long training runs, which describes a large share of the reproducibility failures the paper's own Table 1 catalogs.

Two smaller issues are worth flagging for anyone citing this paper's exact figures. The evaluation-harness diagram (Figure 5) still labels one baseline agent "CRAB-Agent" rather than "CORE-Agent" — an apparent leftover internal name not fully scrubbed even in the June 2026 v2 revision. And the main text states that task-specific tuning improved GPT-4o's Easy-tier accuracy "from 35.6% to 60.60%," while Table 5 reports CORE-Agent+GPT-4o's Easy accuracy as exactly 60.00% — a small prose/table rounding mismatch on the paper's single most-quoted headline number. Neither issue affects the paper's conclusions, but both are the kind of detail a careful reader (or a reproduction attempt of this very paper) would want reconciled against the released code.

baselines

Two AutoGPT-family agents (generic AutoGPT, task-tuned CORE-Agent) on GPT-4o / GPT-4o-mini — a reasonable choice for 2024, but not compared against a more modern ACI-style scaffold (e.g. SWE-agent's interface design), so the 21% Hard-tier ceiling may partly reflect AutoGPT's own brittleness rather than a hard limit on reproducibility as a task.

contamination

Not the SWE-bench sense of leakage (grading is against runtime outputs, not memorizable text), but the papers' own published result values are public — an agent could in principle recall a number from the paper's text instead of actually executing code, especially on Hard, and the harness doesn't check for that shortcut.

what's missed

A reported train→test accuracy gap to validate that CORE-Agent's hand-written hints generalize rather than overfit; a hint-by-hint ablation showing how much each prompting change contributes; and an acknowledgment, beyond the selection-criteria table, of how much CodeOcean's already-reproducible corpus likely flatters the headline numbers relative to reproducibility in the wild.

would raise confidence

The train/test gap made explicit; a memorized-answer control that masks the source paper's own reported numbers from the agent's context; a stronger, non-AutoGPT scaffold as a second baseline family; and reconciling the Figure 5 naming and the 60.00%/60.60% prose-table mismatch in a future revision.

// bottom line

CORE-Bench operationalizes a real and previously unmeasured capability — can an agent reproduce a paper's own results — with an unusually thoughtful grading design for handling stochastic code output. But its headline numbers are best read as "how good one hand-tuned AutoGPT variant is at the easiest slice of computational reproducibility," not "how close AI is to auditing science": the corpus is drawn from a platform selected for easy reproducibility, and a meaningful share of CORE-Agent's edge over plain AutoGPT comes from prompt hints written against this exact benchmark's training failures rather than from general gains in agentic reasoning.

// takeaways

What to remember

  1. 270 tasks, 90 papers, 3 disciplines, 3 tiers. Sourced from 5,090 CodeOcean capsules down to 90 via a ten-criterion filter (CS/medical/social, Python/R, runnable in under 45 minutes, low variance, under 10 GB); each paper becomes an Easy/Medium/Hard task sharing 181 underlying questions.
  2. Best result: 60% / 58% / 21%. CORE-Agent with GPT-4o leads every tier on the test set; the drop to 21% on Hard — install dependencies, find the run command, execute, extract results — shows where automated reproduction actually breaks.
  3. Task-specific tuning roughly doubles weak-model accuracy. A handful of prompting hints and a report-format check took GPT-4o-mini's Easy-tier score from 8.9% to 44.44% — but that also means the headline numbers are partly a study in prompt engineering against this specific benchmark.
  4. Vision-reading and dependency installation are the real bottlenecks — not raw budget. Doubling the cost cap from $4 to $10 barely moved Hard-tier accuracy, because failing agents get stuck rather than run out of money.
  5. This is the easy end of reproducibility by construction. CodeOcean-sourced, criteria-filtered capsules exclude the messiest real-world cases (no README, high variance, hours-long runs), so 21% on Hard likely overstates how well agents would do on an arbitrary GitHub repository.
