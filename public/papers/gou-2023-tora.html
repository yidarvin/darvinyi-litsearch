ToRA: A Tool-Integrated Reasoning Agent for Mathematical Problem Solving

A series of 7B–70B agents that interleave natural-language rationales with executable code in a single trajectory — trained by imitating 16k GPT-4-generated tool-use traces, then "shaping" the output space with self-sampled and teacher-corrected trajectories.

// the gap

What problem is this solving?

By late 2023 there were two competing recipes for teaching LLMs to do math, and each had a hole in it. Rationale-based methods — chain-of-thought prompting, Let's Verify Step by Step-style verification — reason entirely in natural language, but language models are unreliable at arithmetic and symbolic manipulation: a single wrong multiplication or a mis-simplified fraction sinks an otherwise correct derivation. Program-based methods (PAL, Toolformer) sidestep that by outsourcing computation to code, but a program alone struggles with the parts of a math problem that need genuine planning, case-splitting, or self-correction — the reasoning has to be squeezed into syntax before any computation can happen, and there's no natural way to reconsider a wrong turn mid-solution.

The paper's bet is that these two failure modes are complementary, not competing: language is good at planning and error recovery, code is good at execution. What was missing was a model trained to move fluidly between the two — reason in language until a step needs computation, hand that step to a real interpreter, read back the result, and keep reasoning — rather than a model locked into one modality for an entire solution.

// how it works

The method

ToRA's inference loop is a tight three-part cycle. Given a problem q, the model first generates a rationale r₁ in natural language; if that rationale already contains a boxed final answer, generation stops. Otherwise the model writes a program a₁ (guided by r₁) that a real Python interpreter executes, producing output o₁; that output is appended back into the context and the cycle repeats — rationale, program, execution, rationale — until the model emits an answer inside \boxed{} or a maximum of n rounds is reached. The resulting trajectory is denoted τ = r₁a₁o₁…rₙ₋₁aₙ₋₁oₙ₋₁rₙ. Figure 2 lays the three formats side by side on the same problem: rationale-only reasoning derives an algebraic identity but then miscomputes the final step; program-only code hits a Python syntax error it can't recover from; the tool-integrated format uses SymPy to solve the system exactly, catches the duplicate roots, and boxes the right set of values.

No off-the-shelf dataset pairs natural-language rationales with interactive tool use, so the authors build one. They prompt GPT-4 with few-shot tool-use exemplars and run it greedily over the training splits of GSM8k and MATH (n = 3 interaction rounds max), retaining trajectories that land on the correct final answer; where greedy decoding fails, they fall back to nucleus sampling (10 samples, up to 4 kept per question). This successfully annotates 98.2% of GSM8k and 83.1% of MATH training questions, and after filtering out trajectories with tool-execution errors or wrong answers, yields 16k annotated trajectories — ToRA-Corpus. Table 1 in the paper is explicit about how small this is next to prior math-reasoning corpora: RFT uses >100k annotations, WizardMath >96k, MathInstruct 260k — ToRA-Corpus is the only one that combines tool use and rationale interleaving, but it is roughly an order of magnitude smaller than the alternatives. Imitation learning then simply fine-tunes LLaMA-2 and CodeLLaMA (7B–70B) on ToRA-Corpus with a standard next-token loss over the trajectory, conditioned on the problem.

Because most training questions have only one recorded valid trajectory, a model fine-tuned on ToRA-Corpus alone has a narrow, brittle output distribution. The paper's second training stage, output space shaping, addresses this directly (Figure 3). The imitation-learned model samples 64 candidate trajectories per training question via nucleus sampling; trajectories that reach the correct answer with no tool errors are kept as-is (233k distinct valid samples across the whole training set). For trajectories that go wrong, the authors exploit an observation from prior work that errors tend to occur partway through an otherwise-plausible trajectory: they truncate a wrong trajectory at an arbitrary line, then have a fixed teacher model (CodeLLaMA-34B, itself trained on ToRA-Corpus) greedily complete the remainder — yielding 69k corrected trajectories. Up to 4 sampled-or-corrected trajectories per problem are merged with the original ToRA-Corpus, giving a final 69k-annotation training set that every ToRA model is retrained on.

// what they found

The results

Across 10 mathematical reasoning datasets (GSM8k, MATH, and 8 out-of-distribution sets spanning basic arithmetic to competition math), ToRA "consistently surpasses prior state-of-the-art open-source models across all scales, achieving 13% to 19% absolute improvements across 10 tasks," per the paper's own summary. The headline number: ToRA-Code-7B reaches 44.6% on the competition-level MATH benchmark — 22 points above the previous best open-source model, WizardMath-70B, at 22.7%, despite being an order of magnitude smaller. Scaling up, ToRA-Code-34B reaches 50.8% on MATH, the first open-source result to clear 50% on that benchmark; that beats GPT-4's own chain-of-thought result (42.5%) by 8.3 points and is competitive with GPT-4 augmented with code execution (GPT-4-Code, 51.8%). At 70B, ToRA reaches 84.3% on GSM8k and 49.7% on MATH — both comfortably above ChatGPT's CoT numbers (80.8% / 35.5%) in Table 2.

The ablations isolate why. Comparing the three reasoning formats head-to-head on the same training data (Figure 4), the tool-integrated format beats rationale-only and program-only training by 29.0% and 6.7% absolute respectively with LLaMA-2, and by 19.1% and 9.8% with GPT-4 few-shot prompting — confirming that neither modality alone captures what the combination gets. Output space shaping is worth a separate, smaller boost: it improves GSM8k by 3.4% and MATH by 4.0% on average, with sampling alone worth 2.7% on average and sampling-plus-correction worth up to 4.5%, and the gain persists even at 70B scale (47.3% → 49.7% on MATH). ToRA also reports fast inference — an average of just 1.02 tool-interaction rounds per problem — despite being trained on trajectories with up to 3 rounds.

// does it hold up?

The eval read

The headline "22% absolute over WizardMath-70B" comparison is worth reading carefully: it pits a 7B tool-augmented model against a 70B model with no tool access at all. That's exactly the point the paper wants to make — tool integration beats parameter count — but it also means the comparison isn't controlled for the one variable (tool access) that actually explains most of the gap; WizardMath-70B with PAL-style code access (row "CodeLLaMA (PAL) 70B" isn't reported, but the 34B CodeLLaMA-PAL baseline already reaches 23.9% on MATH, closer to ToRA territory) would have been a fairer apples-to-apples control. To its credit, the paper does include exactly this kind of matched comparison in its own ablation — rationale-only vs. program-only vs. tool-integrated on identical LLaMA-2 checkpoints and identical training data (Figure 4) — and that controlled comparison is where the method's real evidence lives, not the cross-scale WizardMath number that leads the abstract.

The more interesting soft spot is upstream of any benchmark table: ToRA-Corpus's 16k trajectories are entirely GPT-4-generated and filtered by one criterion — does the trajectory land on the correct final answer. GPT-4 successfully produces a boxed answer for 83.1% of MATH training questions and 98.2% of GSM8k, which is itself informative: MATH and GSM8k are public, several-years-old benchmarks by the time GPT-4 was trained, and a model that already "knows" the answer can back-fill a plausible-looking rationale-and-program trace without the tool use actually being load-bearing for getting the answer right. The paper never checks this — there's no ablation isolating how many ToRA-Corpus trajectories are genuinely necessary tool use versus post-hoc justification of a memorized answer, and no train/test contamination analysis for GPT-4 itself against MATH or GSM8k.

To the paper's credit, its own error analysis (Table 4, not reproduced here) is unusually candid: manually annotating 100 MATH failures, the authors attribute 38% to plain reasoning errors and a striking 21% to diagram misunderstanding — MATH's geometry figures are linearized into Asymptote source text, which ToRA has to parse without ever seeing an actual image, a structural limitation the aggregate accuracy numbers completely hide. Tool-usage failures (inappropriate tool use, syntax errors, runtime errors) account for another 28% combined. That's a real, if small-sample, admission that the headline MATH accuracy understates how much of the remaining error is about input modality and tool discipline rather than mathematical reasoning per se.

baselines

Broad and mostly well-chosen (GPT-4, ChatGPT, Claude-2, PaLM-2, plus open-source WizardMath/Platypus-2/RFT/Toolformer), but the marquee "beats WizardMath-70B by 22 points" claim compares across both scale and tool access at once — the paper's own same-data rationale/program/tool ablation (Figure 4) is the fairer evidence and shows a smaller but still real 6.7–9.8 point edge for tool integration alone.

contamination

Not addressed. ToRA-Corpus is generated by prompting GPT-4 on the MATH/GSM8k training splits and keeping only answer-correct trajectories, with no check on whether GPT-4 already "knew" those training answers from pretraining exposure to these public benchmarks — which would let a plausible-but-non-essential tool-use trace get rewarded purely because the final answer happened to match.

what's missed

The paper's own manual failure analysis (100 MATH examples) attributes 21% of errors to diagram misunderstanding (geometry figures linearized as Asymptote text) and 28% combined to tool-use mechanics — real, quantified weaknesses that the single aggregate MATH accuracy number doesn't surface.

would raise confidence

A decontamination check on GPT-4's exposure to MATH/GSM8k; an ablation that varies trajectory acceptance criteria beyond "correct final answer" (e.g., requiring the tool call to be non-trivial); and a larger-scale version of the Table 4 error taxonomy rather than 100 hand-labeled examples.

// bottom line

The controlled ablation is convincing — interleaving rationale and tool use genuinely beats either format alone on identical data and models. The cross-scale "beats a 70B model" framing that leads the abstract is less so, and the training data's single-criterion filtering (correct final answer, from a proprietary model that may already know these public benchmarks' answers) is a real, unexamined soft spot in an otherwise well-ablated paper.

// takeaways

What to remember

  1. Interleaving natural-language rationale with executable code beats either format alone, on a controlled ablation: +29.0%/+6.7% (LLaMA-2) and +19.1%/+9.8% (GPT-4) over rationale-only and program-only respectively.
  2. Training data doesn't need to be huge to work — ToRA-Corpus is 16k trajectories (later 69k after output-space shaping), an order of magnitude smaller than contemporaries like WizardMath (>96k) or MathInstruct (260k) — but every trajectory is GPT-4-generated and accepted on a single "did it land on the right answer" test, with no check for whether the tool use was actually necessary.
  3. ToRA-Code-34B was the first open model to clear 50% on MATH (50.8%), beating GPT-4's own CoT result — but the paper's own error analysis shows 21% of remaining failures come from misreading MATH's Asymptote-linearized geometry diagrams, a modality gap the headline number doesn't reveal.
