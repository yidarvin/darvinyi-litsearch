Formalizing and Benchmarking Prompt Injection Attacks and Defenses

A formal definition that reduces every prompt injection attack to one function — compromised data x̃ = A(target data, injected instruction, injected data) — plus the field's first systematic benchmark: 5 attacks × 10 defenses × 10 LLMs × 7 tasks. The headline finding: no existing defense is sufficient.

What problem is this solving?

By late 2023 the prompt injection literature was a pile of demos. Perez & Ribeiro's PromptInject had shown that handcrafted inputs could hijack a GPT-3 application's goal; Greshake et al.'s indirect-injection work had shown attackers could plant instructions in retrieved web content to hijack Bing Chat and GitHub Copilot. Both were compelling, and both were case studies — qualitative proofs-of-concept against one or two applications, with no formal definition of what a "prompt injection attack" even is, and no shared benchmark that a new attack or a new defense could be measured against. Ten different papers proposing ten different heuristic attacks and ten different heuristic defenses had no common yardstick: a defense that looked strong in its own paper's evaluation might simply have never been tested against the attacks that would break it.

This paper's contribution is exactly that missing scaffolding. It proposes the first formal framework for prompt injection — a definition precise enough that every existing attack becomes a special case of one equation — and then uses that framework to run the first systematic, quantitative evaluation: 5 attacks and 10 defenses, crossed with 10 LLMs (GPT-4 down to InternLM-Chat-7B) and 7 NLP tasks, 49 target/injected task combinations per LLM. The framework is also generative: composing existing attack strategies inside it produces a new attack — Combined Attack — that turns out to beat every attack it was built from.

// how it works

The method

Every LLM-integrated application takes the same shape: it combines an instruction prompt (from the developer or user) with data (usually from an untrusted external source — a resume, a webpage, a social media post) and sends the concatenation f(s ⊕ x) to the backend LLM f. The paper calls the application's own task the target task t = (st, xt) and the attacker's desired task the injected task e = (se, xe). A prompt injection attack is then defined precisely: it modifies only the target data xt — the attacker cannot touch the instruction prompt or the model itself — such that the application ends up accomplishing the injected task e instead of t.

That definition compresses into one framework. Every attack is a function A that crafts the compromised data from three ingredients — the clean target data, the attacker's injected instruction, and the attacker's injected data:

x̃ = A(xt, se, xe) // compromised data — every existing attack is a specific choice of A

The five attacks the paper benchmarks are all instances of this one equation, differing only in how they glue the pieces together: Naive Attack simply concatenates xt ⊕ se ⊕ xe; Escape Characters inserts a special character like \n first, hoping the LLM reads it as a context break; Context Ignoring inserts an explicit task-switching phrase ("Ignore previous instructions") before the injected content; Fake Completion inserts a fabricated response to the target task first, tricking the LLM into believing the target task is already done; and the framework's own contribution, Combined Attack, chains all three tricks together (escape character, fake completion, escape character again, context-ignoring phrase, then the injected instruction and data). On the defense side, the paper formalizes ten methods into two families: five prevention defenses that pre-process the data or redesign the instruction prompt (paraphrasing, retokenization, delimiters, sandwich prevention, instructional prevention), and five detection defenses that try to flag compromised data before it reaches the application (perplexity thresholds, windowed perplexity, asking the LLM itself, checking whether the response is even valid for the target task, and a known-answer probe that plants a secret-key instruction to see if it survives).

Evaluation runs on 7 standard NLP tasks (duplicate-sentence detection, grammar correction, hate detection, natural-language inference, sentiment analysis, spam detection, summarization), each used as both a target and an injected task — 49 combinations — with 100 held-out examples per task and three purpose-built metrics: PNA (performance with no attack, the utility baseline), ASV (Attack Success Value — how well the injected task's own metric scores when the LLM is queried under attack), and MR (Matching Rate — how often the attacked response matches what the LLM would have said if queried directly with the injected task, a metric that doesn't depend on ground-truth labels).

// what they found

The results

On GPT-4, every one of the five attacks works, and the framework-inspired Combined Attack is the strongest across almost every target/injected task pair (Fake Completion briefly edges it out in one cell — grammar correction target, duplicate-sentence-detection injected). Averaged over all 49 task combinations, ASV climbs steadily as attacks get more sophisticated: Naive Attack 0.62, Escape Characters 0.66, Context Ignoring 0.65, Fake Completion 0.70, Combined Attack 0.75.

Averaged over all 10 LLMs and all 49 task combinations, Combined Attack reaches ASV 0.62 and MR 0.78 — meaning the attacked response matches the directly-queried injected-task response 78% of the time. Attack strength tracks model capability: GPT-4 is the single most vulnerable model tested, and across the 10-model lineup, ASV and MR correlate with model size at Pearson r = 0.63 and 0.64 respectively — bigger, more instruction-following models are easier to hijack precisely because they're better at following instructions, wherever those instructions appear.

On defenses, the picture is bleaker. Every prevention defense either leaves ASV/MR still substantial or costs real utility. Paraphrasing cuts Combined Attack's ASV the most but drops average PNA-T (clean-task performance) by 0.14 — and on grammar correction specifically, PNA-T collapses from 0.48 to 0.01, because paraphrasing a grammatically-broken sentence "fixes" the very errors the task is supposed to detect. Delimiters and sandwich prevention roughly halve ASV on some tasks (duplicate-sentence detection: 0.76 → 0.36–0.39; hate detection: 0.74 → 0.36–0.39) but barely move it on others — summarization stays at 0.70–0.73 ASV under delimiters, sandwich, or instructional prevention alike, versus 0.75 with no defense at all. Among the five detection defenses, known-answer detection is the strongest overall but still has real blind spots: its false-negative rate is near 0 for attacks that use escape characters, but climbs to 0.12 on grammar-correction targets even against the Combined Attack it's specifically evaluated against — and separately, naive LLM-based detection achieves a near-0 false-negative rate only by also flagging up to 93% of clean data as compromised (hate-detection target), which is a denial-of-service, not a working defense.

// does it hold up?

The eval read

As a benchmark-defining paper, this one is graded on how well its evaluation actually establishes the ground truth the field then built on for two years — and mostly, it holds up, with a few real gaps the authors themselves flag.

The attack suite is comprehensive for its moment but is entirely heuristic. All five attacks (and the paper's own Combined Attack) are hand-designed string manipulations; nothing here is optimization-based. That matters because the benchmark's own successor work (StruQ, evaluated separately in this atlas) shows the gap is not cosmetic: a defense that drives every heuristic attack in this paper's suite to near-0% success rate still loses to a gradient-guided GCG suffix roughly half the time. A reader taking this paper's benchmark as a sufficiency bar for "is my defense secure" would be badly miscalibrated against an adaptive, white-box attacker — a limitation the paper's own Discussion section names directly as future work ("optimization-based prompt injection attacks").

The task suite is also narrower than the threat model it's arguing for. All 7 tasks are single-turn text classification or short-form generation (spam detection, sentiment analysis, summarization, etc.) on small, well-worn academic datasets (e.g. MRPC, SST2, SMS Spam, HSOL) — not the tool-calling, multi-turn, agentic settings where prompt injection is arguably highest-stakes today (an agent that reads a poisoned webpage and then sends an email or executes code). None of the 49 target/injected task pairs involves calling a tool, browsing, or taking any action with real-world side effects — the kind of agentic setting that InjecAgent and AgentDojo would formalize into their own benchmarks a few months later. So the benchmark's numbers say a great deal about text-classification hijacking and comparatively little about what happens once the injected task can move money, send an email, or run code.

baselines

Fair for the moment it was written — it's the first paper to put multiple attacks and defenses on a common footing at all — but by construction excludes the strongest concurrent defense ideas (e.g. Jatmo, task-specific fine-tuning) which are only discussed qualitatively, not benchmarked.

contamination

Not classic train/test leakage, but the classification datasets (MRPC, SST2, HSOL) are old and extremely widely seen during LLM pretraining, which can inflate PNA (clean-task performance) in ways unrelated to attack robustness; the paper doesn't decompose this.

what's missed

No optimization-based attacks (GCG, TAP-style search); no multi-turn or tool-use settings; only 100 sampled pairs per metric with no confidence intervals or variance reported across the 49 × 10 grid.

would raise confidence

Report variance/CIs across the sampled pairs; add at least one adaptive, gradient-based attack as a stress test; extend the task suite to include at least one agentic/tool-calling scenario so utility-vs-security tradeoffs generalize past text classification.

// bottom line

The formalization is the paper's lasting contribution — x̃ = A(xt, se, xe) gave the field a shared vocabulary, and the paper's own conclusion — "no existing defenses are sufficient" — was, and largely still is, correct. But treat the specific numbers (0.75 average ASV, known-answer detection as the best defense) as a lower bound on how bad prompt injection is, not an upper one — a stronger, optimization-based, or agentic-context attacker will do worse to every defense measured here.

// takeaways

What to remember

  1. Prompt injection got a formal definition here: an attack modifies only the target data xt to make an LLM-integrated application accomplish an attacker-chosen injected task instead of its intended target task — and every existing attack collapses into one function, x̃ = A(xt, se, xe).
  2. Combining simple heuristic tricks (escape characters + fake completion + context-ignoring) beats every individual attack it's built from — Combined Attack reaches 0.75 average ASV on GPT-4, and more capable LLMs are consistently easier to hijack (Pearson r = 0.63 between ASV and model size).
  3. None of the 10 benchmarked defenses (5 prevention, 5 detection) is sufficient — the best ones trade real utility loss for partial security, or (like naive LLM-based detection) "work" only by flagging most clean data as compromised too.
