LLM.int8(): 8-bit Matrix Multiplication for Transformers at Scale

A mixed-precision decomposition that keeps 99.9% of a transformer's values in 8-bit and isolates the sparse, systematic "outlier" feature dimensions that emerge past 6.7B parameters into a 16-bit matmul — cutting inference memory in half with, empirically, zero degradation up to 175B parameters.

// the gap

What problem is this solving?

By late 2022, serving a large language model was itself a memory problem: a 175B-parameter checkpoint needs roughly 350GB just to hold 16-bit weights, forcing multi-GPU, multi-node deployments even before a single token is generated. The obvious fix is quantization — store and multiply in 8-bit instead of 16 or 32-bit, which should roughly halve memory. That fix already existed and worked fine for models up to a few hundred million parameters. The paper's core empirical finding is that it silently breaks once models get large.

Naive 8-bit quantization schemes — including the authors' own prior vector-wise scheme, which held up cleanly through 2.7B parameters — collapse abruptly at 6.7B parameters: mean zero-shot accuracy on the EleutherAI evaluation harness (WinoGrande, HellaSwag, PIQA, LAMBADA) falls off a cliff for the 8-bit baseline exactly where the 16-bit and LLM.int8() curves keep climbing (Figure 1). The cause is a phenomenon the paper is the first to characterize precisely: at that scale, a small number of feature dimensions — at most 7 in every model they test, and often just 6 — start producing systematically large-magnitude activations (magnitude ≥ 6, versus the typical range for other features) in all transformer layers and roughly 75% of all sequence positions. These aren't noise: zeroing out these outlier dimensions alone drops top-1 attention softmax probability mass by more than 20% and degrades C4 validation perplexity by 600–1000%, despite the outliers being only about 0.1% of all feature dimensions. Zeroing the same number of random features instead costs at most 0.3% probability mass and about 0.1% perplexity — the outliers are doing a disproportionate amount of the model's actual computation, and a single tensor-wide (or even row/column-wise) quantization scale gets destroyed by trying to represent both these giant values and the ordinary ones in the same 8-bit range.

So the gap is specific: no existing post-training quantization scheme explicitly modeled the fact that a fixed, tiny, predictable set of dimensions needs full precision while everything else can safely go to 8-bit — and no one had shown that this problem is a genuine phase transition with model scale rather than a gradual degradation.

// how it works

The method

LLM.int8() is two techniques stacked together, both visible in Figure 2. The first, vector-wise quantization, is a refinement of ordinary per-tensor int8 quantization: instead of a single scale constant for an entire matrix, treat matrix multiplication as a sequence of independent row·column inner products and give each row of the activations X and each column of the weights W its own scaling constant (cx, cw). Quantize, run the multiplication in int8 (accumulating in int32), then dequantize by dividing by the outer product of those constants. This alone recovers full 16-bit performance up to 2.7B parameters — but not beyond, because it still forces every value in a row or column, including a rare 20x-larger outlier, through the same 8-bit range.

The second and more important piece is mixed-precision decomposition. The authors define, per input, the set of feature dimensions that contain at least one value with magnitude α ≥ 6.0 (empirically the threshold at which quantization degradation drops to near zero), that recur in at least 25% of transformer layers, and that appear in at least 6% of sequence positions — this three-part rule is what separates a truly systematic outlier feature from a one-off spike. Those dimensions (empirically never more than 7 of them, out of thousands) are pulled out of the input and weight matrices and multiplied directly in 16-bit floating point. The remaining >99.9% of the matrix still goes through 8-bit vector-wise quantization. The two partial products — a small 16-bit matmul over the outlier columns and a large 8-bit matmul over everything else — are simply added back together to reconstruct the output.

Cf16  ≈  Σh∈O Xf16hWf16h  +  Sf16 · Σh∉O Xi8hWi8h // O = the ≤7 outlier feature dimensions (fp16 term, left) · everything else (int8 term, right, denormalized by S)

Because the outlier set is so small and so stable, this costs almost nothing extra: for models up to 13B parameters the decomposition adds only about 0.1% additional memory. No retraining, calibration dataset, or quantization-aware fine-tuning is required — a pretrained 16-bit checkpoint can be loaded and converted directly.

What makes this tractable at all is that the outlier phenomenon is itself highly regular. Figure 3 shows the percentage of transformer layers and sequence positions affected by outlier features as a function of model size: a fairly gradual climb up to about 6B parameters, then a sharp phase shift right at 6.7B after which essentially all layers (100%) and about 75% of sequence positions are affected. The paper cross-validates this isn't a software artifact by reproducing it across three independent training codebases (OpenAI's GPT-2 models, Meta's Fairseq-trained OPT family, and EleutherAI's GPT-J on Mesh-TensorFlow) and two separate inference frameworks.

// what they found

The results

On C4 validation perplexity across dense models from 125M to 13B parameters (Table 1), every naive 8-bit scheme the paper tests — absmax, zeropoint, row-wise, and vector-wise quantization — degrades badly at scale, with 13B-parameter 8-bit perplexity actually worse than the same baseline's 6.7B result (a sign of the outlier collapse, not a data artifact). LLM.int8() (vector-wise + decomposition) instead tracks the 32-bit float reference almost exactly across the whole range: at 13B parameters it reports 12.46 perplexity against a 32-bit reference of 12.45. On zero-shot accuracy across OPT models from 125M to 175B, LLM.int8() overlaps the 16-bit baseline curve at every scale (Figure 1), while an 8-bit baseline without the decomposition falls from about 0.69 mean accuracy at 6.7B to roughly 0.31–0.33 by 13B–175B — near-random on multiple-choice tasks.

On memory, the headline claim is that a 175B-parameter 16/32-bit checkpoint can be converted to Int8 and run for inference with no measured degradation, roughly halving the memory footprint versus 16-bit — for BLOOM-176B specifically, the paper reports a 1.96× memory reduction. Table 3 (in the appendix) turns this into concrete hardware outcomes: on 8× A100 80GB, both 8-bit and 16-bit inference already fit OPT-175B/BLOOM; but on 8× A100 40GB or 8× RTX 3090 (24GB) — consumer/academic-server-class hardware — 16-bit inference caps out at OPT-66B, while 8-bit inference is the first to make OPT-175B/BLOOM fit at all. On a single Colab Pro GPU (15GB), 8-bit newly fits OPT-13B where 16-bit was limited to GPT-J-6B.

Runtime is a secondary result the paper is careful to frame correctly: because it prioritizes memory over speed, the quantize/dequantize overhead can actually slow down inference for models under about 6.7B parameters relative to a plain fp16 baseline (those models fit comfortably on a single GPU anyway, so the paper argues this matters less in practice). Where it does pay off is at the scale the method targets: raw int8 matmul in cuBLASLt becomes about 2× faster than 16-bit cuBLAS once the model dimension reaches roughly 5140 — the paper's 13B-parameter GPT-3-sized model, not 175B — and the paper is explicit that stacking the mixed-precision decomposition on top slows things down further, so only the 13B and 175B models end up with a net speedup at all once quantize/dequantize overhead is included (Table 5): 1.22× at 13B (model dimension 5140) and 1.81× at the full 175B scale (model dimension 12288), short of a clean 2× but still a real win at the scale the paper is built for.

// does it hold up?

The eval read

The core empirical claim — an 8-bit phase transition at 6.7B parameters that a fixed outlier threshold (α=6.0, ≥25% of layers, ≥6% of sequence positions) fixes with no measured degradation — is unusually well triangulated: three independent training codebases, two inference frameworks, both a perplexity metric and a four-task zero-shot accuracy suite, and models up to 175B parameters (OPT-175B, BLOOM-176B). That's a lot of surface area for a result this specific, and it's why the paper's outlier-threshold recipe (not just its headline number) became the thing later work builds directly on.

That said, the evaluation is narrower than the "at scale" framing suggests in a few ways worth flagging. The zero-shot suite is four tasks (WinoGrande, HellaSwag, PIQA, LAMBADA) — useful as a fast proxy but not the breadth a reader would want before trusting "no performance degradation" as a blanket claim; there's no coverage of generation quality, instruction-following, reasoning benchmarks, or human-judged output, all of which matter more for how these models are actually used today. The perplexity results also stop at 13B — the 175B/176B claims rest on the zero-shot suite and on the appendix's memory/runtime tables, not on a matched 175B perplexity comparison against a 16-bit reference. And it is explicitly an inference-time-only method: nothing here quantizes gradients or supports backpropagation through the int8 path, so it says nothing about training or fine-tuning cost — the natural follow-up question the field asked immediately (and that Dettmers's own next paper, QLoRA, answers by going to 4-bit weights with fp16-precision LoRA adapters for gradient flow).

baselines

Compared against the strongest contemporary 8-bit alternatives the authors implement themselves (absmax, zeropoint, row-wise, vector-wise, and a row-wise+decomposition ablation) rather than a strawman single baseline — a genuinely fair, apples-to-apples comparison since the paper's own vector-wise scheme is the strongest prior baseline it's improving on.

contamination

Not applicable in the usual sense — this is a systems/quantization paper, not a benchmark-contamination-sensitive capability claim. The main risk instead is scope creep in how "no performance degradation" gets quoted downstream, since it was measured on perplexity + 4 zero-shot tasks, not on the broader instruction-following/reasoning suites the field cares about today.

what's missed

Generation-quality and instruction-following evaluation; a matched 175B perplexity table; any accounting of training/fine-tuning cost (explicitly inference-only); and the runtime section's own admission that quantization overhead can be a net slowdown below 6.7B parameters, which the headline "faster inference" framing can undersell.

would raise confidence

A perplexity or instruction-following comparison at the full 175B/176B scale rather than only zero-shot accuracy; wall-clock, not just matmul-level, end-to-end latency numbers across the model-size range; and a direct head-to-head against the mixed-precision decomposition's main successors (GPTQ, SmoothQuant, AWQ) on the same hardware.

// bottom line

The phase-transition finding and the outlier-isolation fix are both real and unusually well-verified for a systems paper — that's why LLM.int8() (via the open-sourced bitsandbytes library) became a default way to run large open models on consumer hardware, and why it's the explicit background QLoRA builds on. Its limits are mostly about scope, not correctness: inference-only, a narrow eval suite for the "no degradation" claim, and — as later 4-bit weight-only methods (GPTQ, AWQ) and QLoRA itself demonstrate — not the last word on how far mixed/low-precision inference could be pushed.

// takeaways

What to remember

  1. Naive 8-bit quantization doesn't degrade transformers gradually with scale — it holds up fine to 2.7B parameters and then collapses sharply at 6.7B, because a handful of feature dimensions (≤7, ~0.1% of all dimensions) start producing systematic, large-magnitude ("outlier") activations in every layer that a single quantization scale can't represent alongside ordinary values.
  2. The fix (mixed-precision decomposition) is cheap because the outlier set is tiny and stable: isolate those ≤7 dimensions into a 16-bit matmul, keep the other >99.9% of values in 8-bit, and add the two partial products back together — for ~0.1% extra memory and no retraining.
  3. This bought roughly 2× memory savings with no measured accuracy loss up to 175B parameters (OPT-175B, BLOOM-176B), making models that previously needed 8× A100 GPUs runnable on consumer-grade 24GB cards — and it's an inference-only technique that QLoRA (2023), by the same lead author, directly extends toward efficient training via 4-bit weights.
