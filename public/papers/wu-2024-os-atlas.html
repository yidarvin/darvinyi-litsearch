OS-ATLAS: A Foundation Action Model for Generalist GUI Agents

Open VLMs lag GPT-4o badly at GUI grounding — turning "click the search bar" into an (x, y). OS-Atlas closes the gap not with a clever architecture but with a data engine: the largest open cross-platform screenshot corpus to date (13M+ elements) plus a unified action space that fixes the tap-vs-click naming clash. The result: an open 7B that grounds like GPT-4o and beats it on every out-of-distribution agent benchmark tested.

Zhiyong Wu, Zhenyu Wu, Fangzhi Xu, Yian Wang, Qiushi Sun, Chengyou Jia, Kanzhi Cheng, Zichen Ding, Liheng Chen, Paul Pu Liang, Yu Qiao · Shanghai AI Lab / SJTU / HKU / MIT

// the gap

What problem is this solving?

A GUI agent that works from screenshots — the way a person does, rather than parsing HTML or an accessibility tree — lives or dies on GUI grounding: converting a natural-language intent like "open the export menu" into a concrete screen coordinate to click. Practitioners overwhelmingly outsource this to closed VLMs (GPT-4o, Gemini Pro Vision) because open models are visibly worse at it, especially on screens they weren't trained on (out-of-distribution, OOD).

OS-Atlas pins the open-model weakness on two concrete causes:

  • No large open GUI corpus, and none with desktop data. Open VLMs are barely pre-trained on GUI screenshots. The few open grounding datasets are web/mobile-only and small (SeeClick ~3.3M elements, GUICourse ~10.7M, Ferret-UI <1M), and the big one (CogAgent, 70M) is closed. Desktop grounding data — Windows/macOS/Linux app screens — was essentially absent from open sources. Since all GUIs share design conventions, a broad multi-platform corpus should buy strong grounding and OOD transfer; nobody had built it openly.
  • Action-naming conflicts poison multitask training. The same physical act is labeled differently across datasets — "tap" on mobile vs. "click" on desktop, "press home" vs. "home", "type" vs. "input". Blindly mixing such sources for fine-tuning makes the model fight itself, and performance drops.

The goal is a single open foundation action model that grounds well across Windows, macOS, Linux, Android, and the web, generalizes to unseen apps, and can stand in for a commercial VLM when building agents.

// how it works

The method

OS-Atlas is trained in two consecutive phases on two backbones — InternVL-2-4B (no prior GUI data) and Qwen2-VL-7B (already trained with some GUI data) — yielding the 4B and 7B variants.

Phase 1 — GUI grounding pre-training → OS-Atlas-Base

Train on <screenshot, referring expression or instruction, coordinate> triplets: the model sees a screen and a request and predicts a point or bounding box. The real contribution is the data engine — an open, multi-platform synthesis toolkit and the resulting corpus: 2.3M+ unique screenshots and 13.58M GUI elements across five platforms (Table 1: ~1.9M web + 285K mobile + 54K desktop screenshots), the first open corpus to include desktop grounding data.

  • Web — crawl ~4M pages from FineWeb URLs, render each entire page and segment it into 1920×1080 shots (not just the top fold), extract every visible clickable element and its HTML-derived coordinate. Raw 3.7M shots / 37M elements → rule-filtered (drop broken renders, cap ≤10 elements/page) to 1.6M screenshots / 7.7M elements.
  • Desktop & mobile — real OS automation: AndroidEnv for Android, OSWorld for Linux, physical machines for Windows/macOS. Read the accessibility (A11y) tree (pyatspi / pywinauto / ApplicationServices) to get element boxes, then drive state changes by sampling actions via Depth-First Search and Random Walk exploration.
  • Two data flavors. Referring-expression grounding (REG, "click the Open button") is cheap to scale. Instruction grounding (IG, "delete the last file") needs reasoning and is scarce; OS-Atlas synthesizes it by having GPT-4o read before/after screenshots of trajectory datasets (Mind2Web, AMEX, AITZ) with Set-of-Mark prompting, plus existing IG from AndroidControl and Wave-UI.

Along the way they re-annotate the 11.32% of mislabeled samples they found in the popular ScreenSpot benchmark and release the corrected version as ScreenSpot-V2.

Phase 2 — Action fine-tuning → OS-Atlas

Multitask imitation on <screenshot, task, action history> → <thought, action type, parameters>. The key trick is a unified action space that standardizes every dataset into basic actions available on all platforms — click, type, scroll — plus extensible custom actions (open_app, drag, long_press, select) that let users add unseen behaviors at inference. This collapses the action vocabulary from 17 unique types down to 10, resolving the tap≡click / home≡press-home / type≡input conflicts that otherwise degrade training. Custom actions are what give OS-Atlas its OOD reach.

The finished model runs in three modes: Grounding (emit coordinates for an external planner like GPT-4o), Action (solve step-level tasks zero-shot, including OOD), and Agent (further SFT for one target app).

Grounding is scored with a deliberately simple rule — a hit if the predicted point lands inside the ground-truth box — which the authors themselves flag as too lenient, so they also report IoU:

Acc = 1N Σ 1[ x̂i ∈ Bi ]   vs.   IoU = | B̂ ∩ B || B̂ ∪ B | // point-in-box accuracy (lenient — a large target inflates it) vs. box-overlap IoU (strict)
// what they found

The results

Evaluated across six benchmarks on three platforms, OS-Atlas posts state-of-the-art among open models and overtakes GPT-4o on out-of-distribution agent tasks.

Grounding (ScreenSpot)

In the standard setting (no planner), OS-Atlas-Base-7B reaches 82.47 average accuracy — clear of concurrent best-open UGround-7B (73.30), SeeClick (53.40), and even Qwen2-VL-7B (42.89), the very backbone it builds on. With a GPT-4o planner it rises to 85.14 (vs. UGround's 81.40). Crucially, the +40-point jump over the Qwen2-VL-7B base shows the gain is the corpus, not the backbone.

As a drop-in grounding module (OSWorld)

Replace the coordinates a GPT-4o agent produces with OS-Atlas-Base's. On the fully interactive OSWorld testbed, GPT-4o alone solves 5.03% of tasks; +SeeClick 9.21%; +OS-Atlas-Base-7B 14.63% — roughly 3× the bare agent. Set-of-Mark prompting actually hurts (4.59%). Human performance is 72.36%.

Zero-shot OOD agent tasks (vs. GPT-4o)

On step-wise success rate, OS-Atlas-7B beats GPT-4o on all six OOD datasets — including OmniAct-Desktop, a domain it never saw during action fine-tuning. The gaps are largest where GPT-4o is weakest (GUI-Odyssey, ~5× improvement).

What actually drives it (ablations)

Two ablations isolate the levers. On grounding (Figure 4), removing instruction-grounding data barely moves accuracy — referring-expression data alone is nearly enough — but training on web data only collapses the other platforms: desktop text accuracy falls 72.16 → 32.99 and mobile 85.71 → 17.58. Web data does not transfer; the cross-platform corpus is the whole point. On OOD agent tasks (Figure 5), dropping grounding pre-training guts desktop (where almost no fine-tuning data exists), and dropping the unified action space hurts across the board. Pushing further, training on all 7 agent datasets (OS-Atlas-Pro) lifts 7B desktop SR 94.05 → 97.04 and mobile 72.79 → 78.13.

// does it hold up?

The eval read

The data work is genuinely strong and the open release is the lasting contribution. But three things temper the "open alternative to GPT-4o" headline.

The metric is lenient, and the eval set is partly self-made. Grounding accuracy counts a hit whenever the predicted point lands anywhere inside the target box, so large buttons inflate it — the authors say outright that it "does not capture more fine-grained errors" and that IoU correlates only weakly with it. The 82.47 should be read with that caveat; an IoU-headline or distractor-aware metric would be a sterner test. ScreenSpot-V2 is their own re-annotation (11.32% of labels rewritten); it's released and the baseline comparisons in Table 2 are on the original ScreenSpot, which is the fair call, but a model's authors curating the benchmark is worth noting.

The win rests on the 7B — and on its already-GUI-trained backbone. The strong model is OS-Atlas-7B, built on Qwen2-VL-7B, which had GUI data before OS-Atlas touched it. The 4B (InternVL-2, no prior GUI data) is materially weaker and actually loses to GPT-4o on several OOD agent tasks (OmniAct-Web SR 22.99 vs. 34.06; OmniAct-Desktop 26.94 vs. 50.67). To the authors' credit the +40-point ScreenSpot jump over the bare Qwen2-VL-7B shows the corpus adds most of the lift — but backbone choice is load-bearing for the flagship number.

"GUI agent" here mostly means "grounding module." Almost all agent-task numbers are step-wise success at sub-task granularity with gold action history fed in — errors can't compound, so SR overstates real autonomous completion. The one fully interactive test, OSWorld, lands at 14.63% vs. 72.36% human. Grounding is advanced enough to triple a GPT-4o agent, but long-horizon planning and execution remain the wall — exactly OSWorld's own finding that grounding, not reasoning, is the bottleneck. OS-Atlas pushes the grounding frontier; it does not claim to close the autonomy gap.

baselines

Strong open grounding lineup on ScreenSpot (SeeClick, CogAgent, Qwen2-VL, concurrent UGround). But the zero-shot OOD agent comparison is GPT-4o only — no open action-model OOD baseline — and the GPT-4o version/date isn't pinned (a moving target as of late 2024).

contamination

Reasonable hygiene: Wave-UI is scrubbed of ScreenSpot/Mind2Web/OmniAct entries, and the fine-tuning sets (Mind2Web, AMEX, AITZ) are kept distinct from the OOD web evals (GUI-Act-Web, OmniAct). But a 1.6M-screenshot FineWeb crawl vs. benchmark-site overlap audit isn't reported.

what's missed

Point-in-box grounding accuracy is lenient (their own IoU caveat); step-wise SR with gold history overstates autonomous task completion. The only interactive end-to-end test (OSWorld) is 14.63% vs. 72.36% human, and the flagship rests on the GUI-pretrained 7B backbone.

would raise confidence

Interactive end-to-end evaluation on every platform (not just OSWorld); IoU or distractor-aware grounding as the headline metric; an OOD comparison against other open action models; a pinned GPT-4o version and a crawl-vs-benchmark contamination check.

// bottom line

OS-Atlas's real product is the open data engine and corpus — 13M+ cross-platform elements (with the first open desktop data) plus a unified action space — which lifts an open 7B to GPT-4o-replacing grounding (82.47 on ScreenSpot) and beats GPT-4o on all six OOD agent benchmarks. But "GUI agent" still resolves to "grounding module": on the only fully interactive testbed it reaches 14.63% vs. ~72% human, and the headline leans on a GUI-pretrained backbone and a lenient point-in-box metric. A grounding breakthrough, not an agent one.

// takeaways

What to remember

  1. The data infrastructure is the contribution. An open multi-platform synthesis toolkit plus the largest open GUI grounding corpus to date — 2.3M+ screenshots, 13.58M elements across five OSes, including the first open desktop data — is the lever that lifts open models. The ablation is blunt: web-only data does not generalize to desktop or mobile.
  2. A unified action space is a cheap, reusable fix. Three standardized basic actions plus extensible custom actions collapse 17 conflicting action types to 10, killing the cross-platform tap-vs-click clash that silently degrades any agent trained on mixed-source data.
  3. Grounding is advancing faster than autonomy. OS-Atlas-Base-7B grounds at 82.47 on ScreenSpot (vs. UGround 73.30, SeeClick 53.40) and OS-Atlas-7B tops GPT-4o on every zero-shot OOD agent benchmark — yet on the only interactive testbed (OSWorld) the best config is 14.63% vs. 72.36% human. Grounding is solved enough to matter; end-to-end task completion is not.
