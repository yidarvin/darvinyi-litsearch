Insights Generator: Systematic Corpus-Level Trace Diagnostics for LLM Agents

Debugging an LLM agent still means opening a handful of traces by hand and guessing — so the failure modes that only surface across thousands of runs stay invisible. Insights Generator reframes trace debugging as a corpus-scale search: a scout/investigator agent system proposes behavioral hypotheses on a sample, validates each one statistically across the entire trace corpus through a stateful Python layer, and writes an evidence-backed report. Professional engineers handed an IG report improved their agent scaffold by 30.4 points — nearly double the next-best analysis tool.

// the gap

What problem is this solving?

An agentic system run over a benchmark produces a pile of long execution traces — interleaved reasoning steps, tool calls, and environment observations, each routinely tens of thousands of tokens. Today, debugging that pile is almost entirely manual: an evaluation harness flags the failed runs, a practitioner opens a small subset, forms an ad-hoc hypothesis, and iterates. That loop has a structural blind spot. Evaluations are defined over final task outcomes, so they yield aggregate pass/fail signals but reveal nothing about the intermediate behavior — and they never surface patterns that span the broader space of scenarios.

The patterns that matter most are exactly the ones aggregate metrics hide. A failure localized to a specific slice of inputs, task types, or execution contexts can be real and systematic yet too small to move the top-line number — invisible unless the practitioner already knew to look for it. Worse are silent failures: behavioral trends that quietly reduce reliability without ever triggering an explicit error signal. They never show up on an evaluation dashboard, because nothing crashed. The paper names the target object directly — a trace insight: a grounded natural-language statement characterizing a salient pattern across a group of traces, each one backed by trace-level evidence. Their running example:

// a representative trace insight

84% of incorrect Python-using traces exhibit "silent computation failures": the code executes without errors but implements the wrong mathematical model. Cross-checking fails to catch these because it verifies internal consistency within the wrong framework rather than challenging the framework itself. Observed in 94/112 incorrect Python-using traces (37.6% of all traces); 0/19 cross-checking traces detected the modeling error.

Surfacing claims like that at corpus scale runs into two walls. The first is scale and complexity: a trace corpus routinely exceeds any single model's context window, and the patterns of interest are semantic — silent failures, systematic inefficiencies, cohort-level behavioral differences — not anything you can catch with keyword matching or a fixed taxonomy. The second is that there is no evaluation standard: a trace insight is a free-form claim about population-level behavior, and whether it is correct, grounded in cited evidence, and actually useful to a downstream engineer can't be reduced to a single existing metric.

Prior work only chips at fragments of this. Single-trajectory debuggers (AgentDebug, AgenTracer, DoVer, AgentRx) reason about one run at a time; taxonomy classifiers (MAST, AgentFail, Who&When) can only label failure modes someone anticipated in advance; scaffold-centric optimizers treat the traces as secondary. None performs iterative pattern discovery directly over a corpus of traces — which is the gap Insights Generator is built to fill.

// how it works

The method

Insights Generator (IG) is a multi-agent analysis system built on one core separation: hypothesis discovery is split from hypothesis validation, and no agent ever loads raw trace content into its context. Instead, every agent reaches the traces only through callable tools — summarization, feature extraction, cohort comparison — that run inside a stateful Python data-processing layer and return only aggregated results. That single design choice is what lets IG operate over corpora far larger than any context window. Three typed roles divide the labor:

  • Orchestrator — the sole user-facing agent. It decomposes the analysis objective, dispatches subagents in parallel, judges whether the accumulated findings are sufficient, and synthesizes the final report. It carries only light read-only tools, deliberately preserving its context for coordination rather than heavy analysis.
  • Scout (Hypothesis) agent — optimized for breadth. It scans a representative sample (typically 50–200 traces) with LLM-driven summarization and extraction, and proposes diverse, testable hypotheses — including searches for omissions, steps the agent should have taken but didn't. It does not try to validate anything.
  • Investigator (Explore) agent — optimized for depth. Given one hypothesis, it validates it at corpus scale: computing distributional statistics, running 2–3 cohort comparisons on different samples, and returning a finding labeled confirmed / refuted / inconclusive with quantified prevalence and 8–10+ evidence items, each a trace id plus a verbatim quote plus the specific mechanism visible in that trace.

Everything is plumbed through a processed trace store. Traces and extracted features are linked by a short string trace identifier (cheaper than UUIDs and far less prone to hallucinated references), and a schema cache — per-column statistics, null rates, value distributions, notable correlations — is computed automatically and injected into every subagent's context at dispatch. A hybrid semantic-keyword vector store is exposed as a search tool. The 14 analysis primitives fall into trace inspection, LLM-driven analysis, and data management, with availability gated by role: the Orchestrator gets a coordination subset, the Scout adds extraction, and only the Investigator can run cohort comparison and persist a finding's cohort.

The whole thing runs as an orchestrator-driven loop (Algorithm 1). At each turn the orchestrator inspects the current candidate hypotheses and confirmed findings, then picks one action: dispatch a parallel batch of scouts to surface new hypotheses, dispatch a parallel batch of investigators to validate prioritized ones (keeping only those confirmed, then refreshing the schema cache), or submit the report. Scout and investigator rounds interleave in any order, for as many rounds as coverage demands:

report = Synth( { f ∈ Investigate(h, C) : f.status = confirmed }  for h ∈ Scout(Csample, 𝒮, Q) ) // scouts propose hypotheses ℋ over a sample; investigators validate each against the full corpus C; the orchestrator synthesizes confirmed findings F into a deduplicated report — repeated until no credible lead remains

The results

Because a trace insight has no off-the-shelf metric, IG is evaluated along two orthogonal axes — who evaluates (a calibrated LLM judge vs. a human expert) and what is measured (report quality as an artifact vs. downstream impact when someone acts on it) — yielding four settings. Two benchmarks supply the traces: SpreadsheetBench (SSB), real-world spreadsheet-manipulation tasks, and HLE (Humanity's Last Exam), expert-level academic questions. The comparison systems span a ladder of architectures: a Single-Agent Coding baseline (Claude Code), CC Subagents (Claude Code + Task subagents, the same scout/investigator decomposition), a Trace2Skill-style flat analyst fleet, and RLM (a Recursive Language Model at depth 2). Reassuringly, the rankings come out consistent across all four settings.

1 · LLM-as-a-judge: comparable breadth, higher depth

A Claude Opus 4.6 judge pools every system's findings into a union-gold reference set (50 canonical clusters per benchmark), then scores three layers. The first surprise is that coverage barely separates the field: all multi-agent systems cite a member trace for 87–98% of clusters, and IG actually sits below CC Subagents and Trace2Skill (91% avg vs. 95% and 97%). Where IG pulls ahead is the position-swapped round-robin pairwise win rate and the per-dimension quality rubric — it leads mechanism explanation (+0.16 over Trace2Skill) and specificity (+0.20). Detection is a tie; evidentiary depth is the edge.

The ablations — holding the toolkit and data layer fixed — locate where the win comes from. Disabling subagent dispatch entirely (Orchestrator-only) collapses win rate from 77.9% to 32.0%, a 45.9 pp drop; keeping parallel dispatch but replacing the typed Scout/Investigator roles with a single untyped subagent (Generic Subagents) drops a smaller 10.4 pp to 67.5%. So parallel dispatch does the heavy lifting, and typed role specialization adds a further, real, increment on top.

2 · Human-expert intervention: the headline result

The primary evaluation tests whether a report actually helps a person build a better agent. Professional software engineers were each randomly handed an IG report or a CC-Subagent report — blinded to which — and asked to modify a scaffold to improve its score on 250 held-out SpreadsheetBench tasks, with identical tools and a fixed time budget. Each participant's score is the mean of three runs (n = 6 per arm).

Engineers given an IG report lifted the scaffold by 30.4 pp over baseline (to 57.4%), versus 16.2 pp for the Claude Code report (43.2%) — a 14.2 pp gap in IG's favor, statistically significant at p ≈ 0.016 (two-sided Welch's t-test) and robust under leave-one-participant-out refits and an exact permutation test. IG's gain nearly doubles the next-best tool. IG-condition participants touched slightly more files (4.0 vs. 3.2) while making smaller median net edits (94 vs. 149 net lines), consistent with reports that point to more, better-targeted changes.

3 · Human-expert-as-a-judge: the disagreement

When 12 human experts instead scored 23 insights (11 IG, 12 CC) on a 1–5 rubric, IG and CC came out statistically indistinguishable — 4.25 vs. 4.22 overall. IG led on depth (+0.17) and evidence (+0.06); CC led on correctness (+0.14); actionability was a wash. The authors read the gap between this and the LLM judge's strong IG preference as a limit of human evaluators — that people can't effectively assess insights spanning large sets of traces, claims, and evidence within a fixed time budget.

4 · Iterative patcher loop & cost

In a fully-autonomous loop, a Claude Code (Opus 4.6) patcher edits the scaffold from each round's regenerated report until saturation. All three report-equipped systems drive the held-out pass rate to 0.81–0.84 (from 0.41 unmodified) — IG, RLM, and CC Subagents all converge to roughly the same place. The instructive contrast is the no-report Pure-Patcher: it posts a strong round-1 gain (0.80) then reverses to 0.58 by round 3, inventing a prompt instruction that contradicts the grader's exact-match contract (12 of 13 round-2 regressions). The regenerated analysis layer, not the patcher, is what sustains improvement. Cost-wise IG is the most expensive option at ~$76 and 48 minutes per run, versus $23/20 min (RLM), $26/37 min (Trace2Skill), and $38/34 min (CC Subagents).

// does it hold up?

The eval read

The architecture is a clean answer to a real bottleneck, and the headline downstream number is the kind of result that's hard to fake: engineers handed an IG report improved a held-out scaffold by 30.4 pp, nearly double the next-best tool, blinded to condition. That's a genuine, near-causal demonstration that report quality translates into engineering effectiveness. But the paper's case rests on a stack of model-graded comparisons, and the cracks are where those comparisons disagree with each other.

The sharpest tension is internal. The LLM judge ranks IG far ahead on pairwise quality — 77.9% vs. 62.4% — but when human experts scored the same kind of reports, IG and the CC baseline were a dead heat (4.25 vs. 4.22). The authors resolve this in IG's favor by arguing humans simply can't evaluate at this scale. That's plausible, but it's equally consistent with the opposite reading: that the Claude Opus 4.6 judge over-rewards the deep, verbose, citation-dense style IG is built to produce, and humans — who can tell — don't see the same gap. The concern compounds because the "union-gold" reference set the coverage and quality scores are measured against is itself clustered by the same judge family, and IG's own reasoning agents all run Claude Opus 4.6. Model and judge are entangled; a different-family or human-anchored gold would be far more convincing.

Two more caveats temper the framing. First, the flagship 30.4 pp result rests on n = 6 per arm on a single benchmark (SpreadsheetBench); the authors are commendably upfront — they run robustness checks, flag the small sample, and note the edit-size gap is sensitive to one +1112-line participant — but generalization to harder, longer-horizon domains (HLE, SWE-Bench Pro, Terminal-Bench) is explicitly left to future work, and the human study never ran on HLE. Second, IG's advantage is specifically a human-in-the-loop one: under fully-autonomous patching, IG ties RLM and CC at 0.81–0.84 — and RLM gets there at roughly a third of IG's cost ($23 vs. $76). The paper doesn't foreground that an analysis layer, not IG specifically, is what the autonomous loop needs.

baselines

A thoughtful ladder — Single-Agent Coding, CC Subagents, Trace2Skill, RLM — that cleanly isolates tool surface, multi-agent topology, and typed roles. But every system runs on Claude / Claude Code and the judge is Claude Opus 4.6, the same family IG's own agents use, so model and judge are entangled.

contamination

Not a benchmark, so leakage is moot — but the evidence base is narrow: traces from GPT-5.2 / 5.4 on just SpreadsheetBench and HLE. Whether IG's findings hold across other scaffolds and model families is untested, and the gold reference is LLM-clustered, not human.

what's missed

The LLM judge's 77.9% preference isn't echoed by human judges (4.25 vs. 4.22, n.s.); IG trails on raw coverage (91 vs. 95 / 97); the 30.4 pp headline is n = 6 on one benchmark; and IG costs 2–3× more while tying a 3×-cheaper RLM under autonomous patching.

would raise confidence

A cross-family or human-anchored judge to rule out same-family bias; the intervention study replicated on a harder benchmark with larger n; a token/$-matched comparison; and a direct reconciliation of the LLM-vs-human judge disagreement (e.g. correlating judge scores with realized intervention lift).

// bottom line

Insights Generator is a well-motivated reframing — from per-trace inspection to corpus-scale hypothesis testing — and its scout/investigator split routed through a stateful Python layer is a sound answer to the context-window wall. Its strongest evidence is the blinded, held-out human-intervention study: a 30.4 pp scaffold lift, nearly double the next tool. The honest caveats are that IG's measured "quality" edge comes from a same-family LLM judge that human experts pointedly did not corroborate, the flagship study is n = 6 on one benchmark, and under fully-autonomous patching a cheaper recursive baseline matches it — so the win is real but narrowest exactly where it's measured most rigorously.

// takeaways

What to remember

  1. Corpus-level, not per-trace. IG formalizes "trace insights" — grounded population-level claims with cited evidence — and discovers them with a scout (propose hypotheses on a sample) / investigator (validate at corpus scale) loop that routes all trace access through a stateful Python layer, so it works on corpora bigger than any context window.
  2. The result that matters is downstream and human-in-the-loop. Professional engineers handed an IG report lifted a held-out SpreadsheetBench scaffold by 30.4 pp over baseline (57.4% vs. 27.0%), nearly doubling the 16.2 pp from the next-best report — p ≈ 0.016, albeit n = 6 on one benchmark.
  3. The win is depth; the caveat is the judge. A Claude Opus 4.6 judge scores IG far ahead on pairwise quality (77.9% vs. 62.4%), but human experts rate IG and the CC-Subagent baseline a wash (4.25 vs. 4.22), and under fully-autonomous patching a 3×-cheaper recursive baseline matches IG's 0.81–0.84 terminal pass rate.
