Code Llama: Open Foundation Models for Code

Meta turns Llama 2 into a family of open code models by stacking specialization stages — 500B code tokens, infilling, 100k-token contexts, and instruction tuning. The result sets the open-model state of the art (up to 67.8% HumanEval, 65.6% MBPP), ships a permissive commercial license, and becomes the base model under the first generation of open coding agents.

// the gap

What problem is this solving?

By mid-2023 the strongest code models were either closed (Codex, GPT-3.5, GPT-4) or code-only open models trained from scratch (AlphaCode, InCoder, StarCoder). Both left gaps. Closed models couldn't be inspected, fine-tuned, or shipped commercially. Code-only open models started from random weights, so every dollar of compute went to learning code from zero rather than building on a strong general language prior — and they were missing capabilities developers actually use day to day.

Three of those missing capabilities mattered most. First, infilling: plain autoregressive models can only continue a prompt, but an IDE needs to fill a hole given the code on both sides of the cursor. Second, long contexts: Llama 2 topped out at 4,096 tokens, far too short for repository-level reasoning rather than single-function completion. Third, instruction following that is both helpful for programming and safe.

Code Llama's bet is that you get the best code model not by training on code alone but by specializing a strong general foundation model — Llama 2, already trained on 2T tokens — through a cascade of additional training stages, and then bolting on infilling, long context, and instruction tuning as explicit steps. The paper's own ablation backs the premise: initializing from Llama 2 beats the same architecture trained on code only for a fixed compute budget, to the point that Llama 2 70B is roughly equivalent to Code Llama 7B on Python benchmarks.

// how it works

The method

Code Llama is a specialization pipeline, not a single training run. Starting from Llama 2 (7B/13B/34B/70B), Meta applies a cascade of fine-tuning stages, each annotated with the number of tokens it sees:

  • Code training (500B tokens; 70B variant: 1T). Continued training on a near-deduplicated dataset that is 85% code, 8% natural language about code, and 7% general text — the small NL slice is there to keep language understanding from regressing.
  • Python specialization (+100B tokens). A separate Code Llama - Python branch trains another 100B tokens on a Python-heavy mix to study single-language specialization.
  • Long-context fine-tuning (LCFT, +20B). Extends the usable context from 4,096 to 16,384 training tokens and up to 100,000 at inference.
  • Instruction fine-tuning (+~5B). The Code Llama - Instruct branch for safe, helpful, promptable models.

Infilling (fill-in-the-middle)

The 7B, 13B, and 70B models train with a causal-masking objective: a document is split into prefix / middle / suffix at random character positions, the middle is moved to the end, and the reordered sequence is predicted autoregressively. Half the examples use prefix-suffix-middle (PSM) ordering, half suffix-prefix-middle (SPM); the transformation is applied with probability 0.9. Four new special tokens mark the prefix, middle, suffix, and end of the infilled span. This is what lets the model complete code at the cursor with full surrounding context — docstrings, missing function bodies, real-time IDE completion.

Long context via RoPE base-period scaling

Rather than interpolate the rotary position embedding frequencies linearly (as position-interpolation methods do), Code Llama simply increases the base period the frequencies are derived from, from 10,000 to 1,000,000, then fine-tunes on 16k-token sequences:

θi = θ−2i/d,   θ: 10,000 → 1,000,000 // larger base period = slower rotation = less bias toward short-range attention, enabling stable extrapolation past the 16k fine-tuning length

The larger period reduces the model's bias toward short-distance attention and lets it extrapolate well beyond the 16,384-token training length — staying stable out to 100,000 tokens.

Instruction tuning without expensive human code labels

Code Llama - Instruct mixes three data sources. (1) The proprietary "RLHF V5" dataset from Llama 2 (supervised fine-tuning plus millions of rejection-sampling examples scored by a reward model), inheriting Llama 2's instruction-following and safety. (2) A machine-generated self-instruct set built with execution feedback instead of human annotation: prompt Llama 2 70B for ~62,000 interview-style questions, dedup to ~52,000, then for each have Code Llama 7B write unit tests and ten candidate solutions, and keep the first solution that passes its tests — yielding ~14,000 question–tests–solution triplets. (3) A rehearsal slice (6% code, 2% natural language) to prevent regression on general ability.

// what they found

The results

On the standard Python benchmarks — HumanEval (zero-shot) and MBPP (3-shot), pass@1 with greedy decoding — Code Llama set a new state of the art among open models. The headline numbers: Code Llama - Instruct 70B reaches 67.8% on HumanEval (essentially matching GPT-4's reported 67.0%), and Code Llama - Python 70B reaches 65.6% on MBPP. Specialization compounds at every step: Code Llama - Python 7B (38.4% HumanEval / 47.6% MBPP) beats Llama 2 70B (30.5% / 45.4%) at one-tenth the size.

Two effects drive the gains. Specialization: the extra 100B Python tokens add between 4.3 and 8.3 points of HumanEval pass@1 and 1.2 to 6.4 points of MBPP over the general code model. Scale: on MBPP pass@1, going 7B→13B adds 5.6 points, 13B→34B adds ~8, and 34B→70B adds ~7. On the harder, prompt-heavy APPS benchmark, Code Llama 34B reaches 56.3% / 24.3% / 15.4% pass@100 on the introductory / interview / competition tiers.

Long context actually works

The long-context claims hold up empirically. Perplexity on large (≥50 kB) source files keeps decreasing well past the 16,384-token fine-tuning length and only ticks up slightly after 100k tokens — avoiding the usual blow-up when transformers run past their training length. On a synthetic key-retrieval task at ~16k tokens, the larger models stay near 100% accuracy across positions, while the 7B drops to ~54% when the key sits at the very start (relative position 0.0) and GPT-3.5-Turbo-16k dips around position 0.2. LCFT does cost a little on short sequences — standard short-context benchmark scores slip slightly — but the trade buys genuine repository-scale context.

Instruction tuning trades a little code skill for a lot of safety

Code Llama - Instruct improves markedly on safety benchmarks at moderate cost to raw code generation. For the 34B model, TruthfulQA (truthful & informative) rises from 34.64 to 47.37, and ToxiGen toxicity drops from 17.62% to essentially 0.00% — the least toxic among all compared models. Red teaming with 25 Meta employees across three exercises (offensive security, malware, responsible AI) focused on "dual-intent" prompts; reviewers found only limited evidence of false refusals, and several questioned how much real-world uplift current LLMs give to malicious-code production at all.

// does it hold up?

The eval read

The open-model SOTA claim is well supported, but the headline "up to 67%" deserves scrutiny. That number is Code Llama - Instruct 70B on HumanEval — a single best-case variant-and-size cell, computed with greedy pass@1. The same Instruct family is non-monotonic: the 34B Instruct scores 41.5% HumanEval, below its own 13B Instruct (42.7%) and far below the 34B base model (48.8%), a sign that the instruct fine-tune can dent raw code skill and that single-seed pass@1 carries real variance the paper doesn't bound with error bars.

The closed-model comparisons (GPT-4 67.0%, GPT-3.5 48.1%, PaLM-2) are quoted from those models' own reports, not re-run under Code Llama's harness, and GPT-4's MBPP is simply absent — so "matches GPT-4" rests on one number measured by someone else. HumanEval and MBPP are also small, public, and years old; with training data scraped from GitHub, contamination is a live risk the paper doesn't probe (contrast the contamination analyses later normalized by LiveCodeBench and MLE-bench). And the strongest 34B result in the paper, Unnatural Code Llama (62.2% HumanEval), was trained on a GPT-4-distilled "unnatural instructions" set and never released — a teaser, not a shipped capability.

Finally, the benchmarks measure function-level synthesis. They say little about the repository-level software engineering Code Llama's long context was built for — a gap made vivid two months later when SWE-bench fine-tuned SWE-Llama from Code Llama and resolved just 0.70% of real GitHub issues (13k-context, BM25 retrieval) — barely above zero, and still under 1.5% even on the easier SWE-bench Lite split.

baselines

Fair for the open-model SOTA claim (StarCoder, code-cushman, code-only models). But closed baselines (GPT-4/3.5, PaLM-2) are quoted from their own reports, not re-run, and GPT-4 MBPP is missing.

contamination

No leakage analysis despite GitHub-scraped training data and a small, public HumanEval/MBPP — exactly the gap later code benchmarks were built to close.

what's missed

Greedy pass@1 with no variance bars hides the non-monotonic Instruct 34B; "up to 67%" cherry-picks one cell; the best 34B number (Unnatural Code Llama) was never released; function-level evals ignore repo-scale SWE.

would raise confidence

Contamination probes, re-running closed baselines in-harness, pass@1 error bars across seeds, a repo-level evaluation, and releasing the Unnatural recipe or model.

// bottom line

A landmark engineering result more than a benchmark surprise: Code Llama packaged infilling, 100k-token context, and safe instruction following into permissively licensed open models that genuinely led the open field. The reported "matches GPT-4" leans on a single greedy pass@1 cell and an un-rerun external number, and the function-level evals flatter a model whose real test — repository-scale work — was still ahead of it.

// takeaways

What to remember

  1. Specialize, don't restart. A cascade of fine-tuning on Llama 2 (+500B code, +100B Python, +LCFT, +instruct) beats training a code model from scratch for a fixed budget — Llama 2 70B ≈ Code Llama 7B on Python, and each stage's gain is measured.
  2. The long-context recipe the field copied. Bumping the RoPE base period from 10,000 to 1,000,000 and fine-tuning at 16k tokens gives stable extrapolation to 100k — long context bought cheaply, as a fine-tuning stage rather than full retraining.
  3. Execution feedback replaces human code labels. Self-instruct (~14k question–tests–solution triplets kept only if the generated tests pass) plus instruction tuning drove ToxiGen toxicity to ~0% — at a small, sometimes non-monotonic, cost to raw code accuracy. Released under a permissive commercial license, Code Llama became the base for the first open coding agents (SWE-Llama).
