OpenHands: An Open Platform for AI Software Developers as Generalist Agents

Formerly OpenDevin: an open-source (MIT) platform that lets an AI agent act like a human developer — writing code, running a sandboxed bash shell and Jupyter server, and browsing the web — all coordinated through an event-stream architecture, with a shared agent hub and 15 built-in benchmarks.

// the gap

What problem is this solving?

By mid-2024 the agent literature had a reproducibility problem hiding inside a benchmarking problem. The headline coding-agent numbers everyone quoted — SWE-agent on SWE-bench, a web agent on WebArena, a tool agent on GAIA — each came wrapped in its own bespoke harness: its own action format, its own sandbox, its own prompt scaffolding, its own evaluation glue. Comparing two systems meant comparing two entire stacks, and reproducing a result often meant reverse-engineering a research codebase that was never meant to be re-run by anyone else.

The existing open frameworks (AutoGPT, LangChain, MetaGPT, AutoGen) each filled some of this gap, but none offered the full set a developer-agent actually needs at once: a graphical UI for human-in-the-loop steering, a standardized, reusable tool library, a built-in sandbox for executing arbitrary agent-generated code, a real web browser, multi-agent delegation, an agent hub of ready baselines, and an integrated evaluation framework. Worse, the most powerful action interface — letting the agent simply write code rather than emit constrained JSON tool-calls — needs a genuinely safe execution environment to be usable at all, which is exactly the piece research prototypes tend to skip.

OpenHands (f.k.a. OpenDevin) is the bet that all of this should live in one permissively-licensed platform: a common substrate where new agents are a few dozen lines of Python, the runtime is a reproducible Docker sandbox, and 15 benchmarks are wired in so that any agent can be measured the same way. The paper is as much an infrastructure release as a research result — its claim is that a single generalist agent on this substrate can be competitive across software, web, and miscellaneous tasks without per-benchmark tuning.

// how it works

The method

OpenHands is built around three components — an Agent abstraction, an Event Stream, and a Runtime — and one organizing idea: everything an agent does or sees is an event, appended to a single chronological log that is the agent's state. The agent reads that history and emits an action; the runtime turns the action into an observation; the observation is appended; the loop repeats.

st = [ a1, o1, a2, o2, … , at-1, ot-1 ]
at = Agent(st)   ·   ot = Runtime(at) // the event stream s is the state; the Agent maps history → action, the Runtime maps action → observation

An agent is just a class with a step(state) method that returns the next action — implementing a new one is roughly the few-dozen-line body the paper prints as a minimal example. The power comes from the action space, which is borrowed from CodeAct: instead of a fixed menu of JSON tools, the core primitives are IPythonRunCellAction (run arbitrary Python), CmdRunAction (run a bash command), and BrowseInteractiveAction (drive a browser through the BrowserGym DSL from WorkArena). Because actions are code, any tool a developer would reach for — a REST call, a Python function, a CLI — is expressible, and the agent can even write new tools for itself when no API exists. This is also a superset of JSON tool-calling: you can wrap a Python function as a callable tool and still get function-calling ergonomics.

The runtime: a Docker sandbox you can bring your own image to

Letting an agent run arbitrary code is only safe if it runs somewhere disposable. For each session OpenHands spins up an isolated Docker container and talks to it through an action-execution REST API running inside the container. That API server holds three live capabilities the agent shares with a human engineer: a persistent bash shell, a Jupyter IPython server for interactive Python, and a Playwright-driven Chromium browser that returns rich observations (HTML, DOM, the accessibility tree, screenshots, open tabs). A user's working directory is mounted in, and — crucially for evaluation — OpenHands can take an arbitrary user-provided Docker image and install its execution API into it, so an agent can be dropped into any software environment.

Skills, delegation, and a hub of agents

On top of the raw primitives sits an AgentSkills library — Python utilities auto-imported into the IPython environment. Its inclusion philosophy is deliberately narrow: don't re-teach the model things it already knows (it can already use pandas), only add a skill when writing the code directly is hard (precise line-based file editing, adapted from SWE-agent and Aider) or when it needs an external model (e.g. parse_image via a vision model, parse_pdf). Agents can also cooperate: an AgentDelegateAction lets the generalist CodeActAgent hand a subtask to a specialist, e.g. delegating browsing to the BrowsingAgent. All of this lives in an AgentHub of 10+ community agents (CodeActAgent, BrowsingAgent, a GPTSwarm agent, and lightweight "micro agents" that are just shared prompts), each usable as a baseline.

// what they found

The results

The headline isn't a single state-of-the-art number — it's generality. The same unmodified CodeActAgent, with no per-benchmark prompt engineering, is competitive across three task families at once. On SWE-Bench Lite (300 real GitHub issues, no hint text) CodeActAgent with claude-3.5-sonnet resolves 26.0% at $1.10/instance — within range of the SWE-specialist scaffolds it's measured against (SWE-agent 18.0, AutoCodeRover 19.0, Aider 26.3, Moatless 26.7, Agentless 27.3). On HumanEvalFix (Python) the same lineage hits 79.3% 0-shot with gpt-4o — almost double StarCoder2-15B's 48.6%, and behind SWE-agent's 87.7% only because that figure is 1-shot to OpenHands' 0-shot.

The breadth holds up across the other 11 benchmarks. On GPQA-diamond (198 graduate-level questions) tool-augmented CodeActAgent reaches 52.0% with claude-3.5-sonnet — above GPT-4 chain-of-thought's 38.8% and well past the 21.9% non-expert-human baseline, though still short of the 81.3% domain experts. It tops the baseline agents on the AgentBench OS subset (57.6% vs GPT-4's 42.4%), on MINT math (77.3%) and code (50.0%), and posts 47.3% on BIRD text-to-SQL, 76.5% on ML-Bench, 47.2% on ToolQA, and 78.8% on ProofWriter. On the web side the BrowsingAgent reaches 14.8–15.5% on WebArena (competitive with the gpt-4-turbo WebArena agent's 14.4%) and 40.8% on MiniWoB++ — though the latter sits far below the 91.1% of CC-NET, a specialist trained with RL and behavioral cloning. Cost is reported throughout (running the full 2,294-instance SWE-bench is estimated at ~$6.9k), and the platform's adoption is itself part of the result: MIT-licensed, 32K+ GitHub stars, and 2.1K+ contributions from 188+ contributors at publication.

// does it hold up?

The eval read

OpenHands is a strong platform paper, and its evaluation should be read for what it is: a breadth demonstration, not a controlled study. The generality claim — "one agent, three task families" — is the right thing to argue, but the evidence is softer than the framing suggests. The numbers in the cross-category table are not a single configuration: SWE-Bench Lite and GPQA come from CodeActAgent v1.8, the starred GPQA gpt-4o number is back-filled from v1.5, WebArena is the BrowsingAgent reached via delegation, and GAIA is a different agent (GPTSwarm) entirely. Each cell also floats across models (gpt-4o-mini, gpt-4o-2024-05-13, claude-3.5-sonnet). "The same agent" is doing real work, but the table quietly mixes agent versions, scaffolds, and backbones, so it under-supports the strict reading.

Two structural caveats. First, every result is a single pass@1 run with no seeds, no variance, and no error bars — and agent trajectories are notoriously high-variance, so a 26.0 vs 26.7 ordering against a baseline carries little signal. Second, almost everything is run on subsets: SWE-Bench Lite (300) not the full 2,294; GAIA L1 only (53 of 466); GPQA-diamond only; the ML-Bench quarter split; ToolQA easy. The numbers are also pinned to mid-2024 model snapshots and have aged out — the platform is the durable artifact, the leaderboard is not. And Table 1's framework comparison, where OpenHands earns a checkmark in every column, is a self-authored feature matrix, not a measurement.

On contamination the paper is honest where it's cheap (SWE-bench is always run without hint text) but silent where it matters: SWE-bench and HumanEvalFix solutions are on public GitHub, GPQA answers are searchable, and a tool-using, web-browsing agent blurs the line between "reasoning" and "retrieval" precisely on the benchmark (GPQA) where that distinction is the point. None of this sinks the contribution — the platform's value was vindicated downstream — but it does mean the headline scores should be treated as existence proofs, not rankings. Tellingly, the later papers that do run the controlled experiment OpenHands skipped found scaffold choice swings results by an order of magnitude (MLE-bench: AIDE 8.7% vs OpenHands 4.4% on the same model; Terminal-Bench: moving a model off OpenHands changed scores by ~17 points), which is exactly why holding the scaffold fixed is the experiment that would have raised confidence here.

baselines

Mostly fair — open-source, reproducible, no benchmark-specific prompt engineering. But comparisons mix agent versions (v1.5/v1.8), models, and even different agents per cell, and some baselines are trained specialists (CC-NET 91.1% on MiniWoB++) that aren't apples-to-apples with a zero-shot generalist.

contamination

Under-examined. SWE-bench is run without hints (good), but no train-cutoff/leakage analysis for public benchmarks whose solutions are on the web, and tool-use + web search muddies "reasoning vs retrieval" on GPQA specifically.

what's missed

Single-run pass@1, no seeds or variance; subset evaluations throughout; numbers tied to fast-aging 2024 model snapshots. The headline contribution is infrastructure, which a benchmark snapshot can't fully capture, and 32K stars is adoption, not capability.

would raise confidence

Fix one model + one agent version across all 15 benchmarks; report multiple seeds with variance; run full sets not subsets; add a contamination/cutoff probe; and ablate the scaffold (swap OpenHands ↔ SWE-agent ↔ Agentless on a fixed model) to isolate the platform's own contribution.

// bottom line

As a research claim the evaluation is a breadth demo, not a controlled comparison — read the scores as existence proofs, not a leaderboard. As an artifact, OpenHands is the rare paper whose contribution outran its own metrics: the event-stream-plus-Docker-sandbox design became the default open scaffold that later, more rigorous benchmarks (MLE-bench, SWE-Bench Pro, Terminal-Bench) build on and measure against.

// takeaways

What to remember

  1. One event stream is the whole abstraction. State = a chronological list of actions and observations; the Agent maps history→action, the Runtime maps action→observation. That single loop backs a CLI, a web UI, IDE plugins, human-in-the-loop steering, and multi-agent delegation.
  2. Code is the action space. Borrowing CodeAct, agents act by writing Python/bash/browser code inside a per-session Docker sandbox reached through a REST execution API — strictly more expressive than JSON tool-calling, and safe enough to be usable because it's disposable and isolated.
  3. Generality over peak score. The same CodeActAgent reaches 26.0% SWE-Bench Lite, 15.3% WebArena, and 52.0% GPQA-diamond without per-task tuning — competitive everywhere, best nowhere — and the MIT-licensed platform (formerly OpenDevin) became the de-facto open scaffold the coding-agent field now benchmarks on.
