AndroidWorld: A Dynamic Benchmarking Environment for Autonomous Agents

Google DeepMind's mobile answer to OSWorld: a fully functional Android emulator that hands agents durable, execution-based reward on 116 tasks across 20 real apps — each one randomly re-parameterized into millions of variants, so a benchmark score finally reflects whether an agent is robust, not lucky.

// the gap

What problem is this solving?

Most UI-agent evaluation in 2024 graded an agent by comparing each of its actions to a pre-recorded human demonstration — the AitW / AndroidControl / Mind2Web "step-accuracy" template. That is misleading, the authors argue, because in a live environment there are many valid paths to a goal, the environment behaves non-deterministically, and a good agent recovers from its own mistakes — none of which a single golden trajectory can credit.

The better paradigm is online evaluation with a ground-truth outcome reward: did the message actually get sent, does the calendar event actually exist? A handful of environments did this — WebShop, WebArena, OSWorld, MiniWoB++ — but they had two gaps. First, they were desktop/web only: there was no comprehensive reward-providing environment for mobile, despite Android being the most-used OS on Earth, the most popular way people reach the web, and an open world of 2M+ apps. Second, they relied on static test specifications — a fixed start state and a hand-coded success check tied to that state. Change the task's parameters and the reward signal breaks.

The mobile-specific benchmarks that did exist were tiny and shallow: B-MoCA validated 6 tasks across 4 apps with regular expressions ("Call 911", "turn on airplane mode"); Mobile-Env offered 13 task templates for a single app. AndroidWorld is built to be the mobile counterpart to OSWorld — real apps, real OS, execution-based reward — but with one feature none of its predecessors had: every task is dynamically re-parameterized, and the reward logic is engineered to stay correct across all those instantiations.

// how it works

The method

AndroidWorld is a fully functional Android environment: a Pixel 6 emulator running Android 13, driven through the AndroidEnv library and the Android Debug Bridge (adb). It is deliberately lightweight — 2 GB of memory and 8 GB of disk, runnable on a laptop with no specialized hardware. On top of it sit 116 hand-built programmatic tasks across 20 real apps: open-source apps pinned to fixed F-Droid versions plus built-in system apps (Settings, Contacts), all chosen to need no login and to store their data on-device so the state is fully controllable.

The agent sees a full-resolution screenshot and an accessibility (UI) tree; it acts through a human-like space — tap, swipe, type, go-home, go-back — plus a small set of function-calling APIs (e.g. send_text_message) that Android, as an OS, can expose where a web browser can't. Every run pins the device clock to October 15 2023, 15:34 UTC so time-dependent behavior is reproducible.

Durable rewards from system state

The core trick is how the reward is computed. Each task ships its own initialize_task / is_successful / teardown logic, and success is determined by inspecting the device's actual system state through adb — the file system, app SQLite databases, and system settings — using the same mechanisms the apps themselves use to persist data. For an app backed by SQLite (a common pattern), the evaluator checks for newly-inserted or deleted rows; for a file task, whether the file exists at the expected path. This is far more durable than matching superficial UI changes, and it is reusable across unrelated apps: one file_exists check serves file-management, note-taking, and media tasks alike. Where system state can't be inspected, the environment falls back to validating on-screen UI elements.

R = 1k  Σi=1..k 1[ state(s) ⊨ checki ] // composite-task reward: average over k sub-checks read from adb system state — partial credit for partial completion

Because success is inferred from state rather than a fixed screenshot, tasks compose: "create a calendar event and text the details to a contact" is two standalone tasks merged, and the agent earns partial reward for each half it completes (the (a + b)/2.0 validators), which smooths the optimization landscape.

Automatic task parameterization

The signature feature: each task is instantiated from a random seed into fresh parameters — the date, title and length of an event; the expense category; the file name — yielding a practically infinite set of initial conditions and success criteria (millions of unique goals). The hard part isn't generating random parameters; it's writing success-checking logic that remains valid across every instantiation, which AndroidWorld's careful state management is what makes possible. This is the same idea that made MiniWoB++ useful for RL — and indeed the authors port MiniWoB++ into the same TaskEval interface as MobileMiniWoB++ (92 of the original 114 tasks; 12 dropped for rendering/touch/timing issues), rendering HTML inputs as native Android widgets (e.g. a date-picker) and dropping the DOM in favor of the same accessibility-tree observation.

The reference agent: M3A

To prove the environment is usable, the authors build M3A (Multimodal Autonomous Agent for Android), a zero-shot agent that fuses ReAct-style reasoning-and-acting with Reflexion-style self-reflection. It runs in two stages. Act: given the available action types, operating guidelines, the accessibility tree's leaf nodes, and a screenshot annotated with Set-of-Marks (numbered bounding boxes over each UI element), it emits a JSON action plus its reasoning. Reflect: after executing, it observes the before/after UI states and writes a short summary — what it intended, whether it worked, why it might have failed — which becomes the action history that conditions the next step. A text-only variant drops the screenshot and reads only the accessibility tree; M3A-Simple strips the guidelines and reflection to measure what minimal prompting buys; and a SeeAct baseline ports a strong web agent to Android.

// what they found

The results

The best agent — M3A on GPT-4 Turbo with accessibility-tree input — completes 30.6% of AndroidWorld's tasks, far below the 80.0% human success rate. Two findings stand out. First, multimodality didn't help on Android: adding the Set-of-Marks screenshot lowered GPT-4 M3A from 30.6% to 25.4%, because native Android accessibility trees are already rich. (On MobileMiniWoB++, whose accessibility trees are sparse, Set-of-Marks did help — 67.7% vs 59.7%.) Second, scaffolding matters: M3A-Simple drops to 19.8% on Android versus M3A's 30.6%, but the two tie at 67.7% on the easier MobileMiniWoB++ tasks — reflection and operating guidelines earn their keep only on the harder mobile workflows.

The web-trained SeeAct collapses on mobile (15.5% vs M3A's 30.6%): it can't handle long-presses and swipes and, caching only actions rather than outcomes, falls into repetitive behavior like endless scrolling. Across the board, failures are not about high-level reasoning but about operating the phone — perceptual errors (missing a visual cue), wrong action ordering, missing operational knowledge (deleting notes one-by-one vs all-at-once), poor grounding on text fields and sliders, and an inability to "remember" content across steps for transcription or counting. The agents are also slow: M3A averages 3.9 minutes per task (2.5 min text-only), roughly 3× longer than humans.

The robustness punchline

The most important experiment is methodological. Run the same strong agent (M3A + GPT-4) across three random seeds and the scores are 27.6%, 26.3%, and 33.2% (mean 29.0%) — yet Table 3's headline reports a single-seed 30.6% "for consistency with existing literature." Drill into individual tasks and it's starker: with a fixed seed the agent can look completely incapable of a task (0%) purely from a bad parameter draw, while across different seeds it solves the same task ~45% of the time. Even fixed-seed runs vary, exposing model non-determinism. Changing the device entirely — Pixel 5 on Android 12 — barely moved the score (28.4%), confirming the environment itself is stable; it's the task parameters that drive the variance.

// does it hold up?

The eval read

The environment is the contribution, and it's a strong one — but the headline number is the weakest part of the paper, by the paper's own evidence. The 30.6% is a single-seed point estimate of 2024-vintage, zero-shot scaffolds (GPT-4 Turbo, Gemini 1.5 Pro, Gemma-2-27B), with no fine-tuned mobile agents, no GUI-grounding specialists, and no self-improvement loop. So the ceiling describes those baselines, not a fundamental wall — and the absolute number is already stale. The telling detail is that Set-of-Marks multimodal input underperformed text-only accessibility-tree input on Android (25.4% vs 30.6%): the score is being carried by accessibility-tree quality more than by visual grounding, so "can agents see a phone screen?" is partly answered by "how good is the a11y tree?", not the model's eyes.

On the other hand, the reward design is exactly right for the contamination problem. Because success is read from system state via adb and the exact (state, success-criterion) pair is freshly generated from a random seed, there is no static answer key to memorize — parameterization defends the labels. The caveat is that it doesn't defend the task distribution: the 116 templates and the open-source apps are public on GitHub, so an agent can still be tuned to the task structures. And the durable-state story has a soft edge the paper doesn't quantify: "where state inspection is impractical" it falls back to checking on-screen UI elements — precisely the brittle signal the design set out to avoid — and we're never told what fraction of the 116 tasks rely on that fallback.

baselines

Zero-shot ReAct+Reflexion on 2024 models (GPT-4 Turbo, Gemini 1.5 Pro, Gemma-2-27B) plus a ported web agent (SeeAct). No fine-tuned, grounding-specialized, or self-improving agents — the 30.6% ceiling is a scaffold property, not a capability limit.

contamination

Low by construction: execution-based reward + per-seed parameterization means no static label to leak. But task templates and apps are open-source on GitHub, so the task distribution is learnable even if the answers aren't.

what's missed

The headline is single-seed (30.6%) when the paper's own data says the honest number is mean-across-seeds (29.0% ± ~3pt, per-task swinging 0%→45%). The signature parameterization is stress-tested on only 5 tasks × 20 trials; MobileMiniWoB++ uses a 62-task subset; the UI-fallback reward fraction is undisclosed.

would raise confidence

Report mean ± CI over many seeds as the primary metric; disclose how many tasks score via system-state vs UI fallback; add modern + fine-tuned + grounding agents; widen the parameter stress-test; and detail the human-80% baseline (how many annotators, what agreement).

// bottom line

The 30.6% will age out within a model generation; the durable, parameterized, execution-based reward on a real OS won't. AndroidWorld's lasting move is making mobile agents trainable and honestly measurable — and proving that a fixed-seed benchmark score can be luck. It became a standard mobile computer-use environment for exactly that reason.

// takeaways

What to remember

  1. AndroidWorld is OSWorld for mobile: a real, lightweight Pixel-6/Android-13 emulator with execution-based reward on 116 tasks across 20 apps, where success is read from OS/app system state through adb (files, SQLite rows, settings) — accurate, durable, and reusable across apps instead of matching screenshots.
  2. Every task is re-parameterized from a random seed into millions of variants with success-checking logic that stays valid — the real engineering feat — which is what enables robustness testing and online/RL training that static benchmarks can't support. (It also ports MiniWoB++ in as 92 MobileMiniWoB++ web tasks.)
  3. The best 2024 agent, M3A on GPT-4 Turbo, reaches 30.6% vs 80.0% human; multimodal Set-of-Marks input underperformed text-only accessibility-tree input, and failures cluster on UI grounding, memory, and operational knowledge — not high-level reasoning.
  4. The methodological lesson outlasts the score: the same agent scores 27.6 / 26.3 / 33.2% across three seeds and can swing 0%→45% per task, so single-seed benchmark numbers can be luck — report the mean across parameterizations.
