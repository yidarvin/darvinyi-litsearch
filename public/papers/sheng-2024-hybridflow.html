HybridFlow: A Flexible and Efficient RLHF Framework

The systems paper behind verl. It reframes RLHF as a dataflow graph and combines a single-controller brain for coordinating models with multi-controller muscle for distributed compute — then adds a zero-redundancy 3D resharding engine. The result: 1.53×–20.57× higher RLHF throughput, and the open framework that the RL-for-reasoning wave was built on.

Guangming Sheng, Chi Zhang, Zilingfeng Ye, Xibin Wu, Wang Zhang, Ru Zhang, Yanghua Peng, Haibin Lin, Chuan Wu · HKU & ByteDance

// the gap

What problem is this solving?

PPO-style RLHF is not one model — it is a small federation of LLMs passing data around. The canonical setup runs four: an actor (the policy being aligned), a critic, a frozen reference policy, and a reward model. Each iteration walks three stages — generation (the actor autoregressively samples responses), preparation (critic values, reference log-probs, and rewards, each a single forward pass), and training (actor + critic updated by Adam). Variants reshuffle the cast: Safe-RLHF bolts on a cost model, ReMax deletes the critic. The natural abstraction is a dataflow graph: nodes are model computations, edges are data dependencies.

The trouble is that in RLHF every node is itself a distributed program spanning many GPUs (tensor/pipeline/data parallelism), and every edge is a many-to-many multicast — a "resharding" of data between two models that may use entirely different parallelism layouts. Existing frameworks were forced to pick one of two programming models, and each pick hurts:

  • Single-controller (classic RL systems like RLlib): one central brain dispatches every operation. Flexible — you can express any dataflow from a single script — but dispatching billions of operators to accelerators from one controller drowns in control overhead.
  • Multi-controller (every existing RLHF system — DeepSpeed-Chat, OpenRLHF, NeMo-Aligner): each GPU runs its own copy of the program with point-to-point messaging. Near-zero dispatch overhead, but inflexible: computation and communication are nested together, so changing one model's data dependency means rewriting every dependent model. These systems support essentially one algorithm (PPO) and one hard-coded model placement.

Two structural facts make this expensive. First, the actor does both training (compute-bound, wants a large model-parallel size) and generation (memory-bound, wants a large data-parallel size) — optimal layouts differ, but switching between them means resharding the actor's weights every iteration. For a 70B actor that is 140 GB of weights, up to 36.4% of an iteration when train and generation live on different devices. Second, models have heterogeneous footprints and dependencies, so the best GPU placement (colocate everything? split actor and critic apart?) changes with model size and cluster scale — and existing systems hard-wire exactly one. Actor training + generation alone is 58.9% of total RLHF time, so getting this wrong is costly.

// how it works

The method

HybridFlow's thesis is that you do not have to choose. The RLHF dataflow has only a handful of nodes, so dispatching control to them from a single controller is cheap — but the computation inside each node is enormous, so it wants multi-controller efficiency. So they go hybrid: a single-controller coordinates models at the inter-node level (flexible dataflow, data resharding, placement), while each model runs its distributed compute under the multi-controller paradigm at the intra-node level. This one move decouples "what data goes where" from "how each model computes," so an algorithm change touches a few lines of orchestration and never the distributed kernels.

Hierarchical APIs

Each model is a class — ActorWorker, CriticWorker, etc. — inheriting a base (3DParallelWorker, or FSDPWorker / ZeROWorker) that wraps distributed init, forward/backward, generation, and optimizer steps, reusing the kernels of existing engines (Megatron-LM, FSDP, DeepSpeed for compute; vLLM for generation). Between models, an @register decorator binds each operation to one of 8 transfer protocols (3D_PROTO, DP_PROTO, ONE_TO_ALL, …). A protocol is just a collect + distribute pair that hides the many-to-many resharding: the source model gathers its outputs to the controller as futures, the destination scatters them to its own parallel groups, and the actual bytes move GPU-to-GPU with no central bottleneck. A ResourcePool virtualizes GPU sets, so colocating two models (share a pool) vs. splitting them (separate pools) is a one-line change. The payoff is concrete: PPO is ~8 lines; Safe-RLHF adds ~5 lines; ReMax just removes the critic calls and adds one generation call.

3D-HybridEngine: zero-redundancy resharding

The actor's train↔generate transition is the hot path. HybridFlow keeps a single copy of actor weights on one set of Na GPUs and reshards in place. Training uses parallel groups p·t·d (pipeline, tensor, data); generation uses pg·tg·dg·d with smaller tensor/pipeline sizes and a larger "micro" data-parallel size for throughput:

Na = p·t·d = pg·tg·dg·d,    dg = p·tpg·tg // generation gets d_g extra data-parallel replicas per training replica

The trick is how the generation groups are formed. The naive version — call it HybridFlow-V — reuses the standard grouping and must all-gather every parameter across all GPUs, then keep redundant training weights around on the GPUs where train and generation layouts don't overlap. HybridFlow instead picks generation TP/PP ranks at strided intervals so that each GPU's generation shard overlaps the training shard it already holds. Now the all-gather happens only within a small micro-DP group, training weights are reused directly, and memory redundancy drops to zero:

Auto device mapping

Finally, HybridFlow searches for the best GPU placement. Given the dataflow and cluster, it enumerates placement plans — for PPO's four models that is the 15 partitions of a 4-set (a Bell number) — finds the minimum GPU allocation that avoids OOM for each colocated set, then uses a latency simulator to pick the best parallelism per model and a cost model to estimate end-to-end iteration time. Caching parallelism strategies across placements keeps the search to at most half an hour, negligible against days of training. The whole stack is ~12k lines of Python (1.8k for the APIs, 2.4k for the engine, 1.9k for the mapper).

// what they found

The results

On a cluster of 128 A100-80GB GPUs (16 machines, 600 GB/s NVLink, 200 Gbps inter-machine), running PPO, ReMax, and Safe-RLHF on Llama models from 7B to 70B, HybridFlow beats all three state-of-the-art baselines at every scale. The headline span is 1.53×–20.57× throughput (tokens/s). For PPO specifically, the average speedups are:

PPO baselineavg speedupmax
DeepSpeed-Chat3.67×7.84×
OpenRLHF3.25×5.93×
NeMo-Aligner12.52×20.57×

The biggest wins land on the largest models: the highest average speedup, 9.64×, comes at 70B, where HybridFlow's resharding cuts the train↔generate transition overhead by up to 71.2% vs. DeepSpeed-Chat and 89.1% vs. OpenRLHF (both pay heavy inter-machine cost from ZeRO-3). NeMo-Aligner is the weakest baseline because it lacks a KV-cache in generation, so generation alone can be up to 81.2% of its iteration time — which is why the speedups over it run highest.

It also scales: 66.8% strong-scaling efficiency averaged over algorithms and sizes, and even in the hardest case — a 7B model on 128 GPUs, where the fixed global batch starves each worker — it still beats the best baseline (OpenRLHF) by 1.68×/1.53×/1.71× on PPO/ReMax/Safe-RLHF. The auto-mapper earns its keep: the optimal placement genuinely shifts with scale — colocate all models on 16–64 GPUs, split at 96–128 GPUs, standalone for 13B on 128 GPUs — and Algorithm 1 finds the best one in every case.

The 3D-HybridEngine ablation isolates the resharding win directly: HybridFlow cuts transition time by 55.2% (11.7s) on average and up to 89.1% (78.2s) at 70B versus the baselines, while staying flat across cluster sizes. And using a smaller generation tensor-parallel size than training (e.g. tg=2 for 7B, tg=4 for 13B) cuts generation latency by 60.3% and 36.4% respectively — confirming the case for decoupling train and generation layouts.

// does it hold up?

The eval read

As a systems paper the engineering is clean and the win is real — but read the benchmark like a benchmark. The entire evaluation is throughput (tokens/s); there is not a single learning curve, reward trajectory, or downstream-accuracy number anywhere. The implicit claim is that HybridFlow computes the same math as the baselines, so quality is unchanged — almost certainly true, but "almost": resharding, mixed precision, and different parallel layouts can perturb numerics, and the paper never shows a parity run proving the faster system converges to the same place. For a framework whose whole point is to train models, the absence of any model-quality check is the conspicuous gap.

The 20.57× headline also deserves an asterisk. That maximum is against NeMo-Aligner, a baseline the paper itself diagnoses as crippled (no generation KV-cache, generation = up to 81.2% of its time). Strip that out and the honest story is the ~3.3–3.7× average over DeepSpeed-Chat and OpenRLHF, with a minimum of 1.53×. The range "1.53×–20.57×" is accurate, but its top end is set by a near-straw baseline; the modal, fair-fight number is closer to 3×.

A few more setup choices shape the result. To "fairly" compare, the authors fix every response to 1024 tokens because some baselines lack continuous batching — but that disables a real-world generation optimization and standardizes away the most variable, KV-cache-stressing part of the workload. Baselines are pinned to specific versions (DeepSpeed-Chat 0.14.0, OpenRLHF 0.2.5, NeMo-Aligner 0.2.0); this space moves monthly, so the deltas are a 2024 snapshot, not a standing verdict. Everything runs on one homogeneous, NVLink-rich A100 fabric — on slower interconnects the resharding all-gathers could behave differently — and the ceiling is 70B / 128 GPUs, below where the reasoning-RL wave it enabled now operates. Finally, the auto-mapper is validated only circumstantially (its picks match the best hand-tried placement in §8.3); the simulator's actual prediction error is never quantified.

baselines

Three real SOTA systems (DeepSpeed-Chat, OpenRLHF, NeMo-Aligner) at fixed versions — fair, but the 20.57× max is against NeMo-Aligner's KV-cache-less generation; the representative number is ~3×.

contamination

N/A for a systems paper — but the flip side is no model-quality / convergence validation at all; throughput parity is asserted, never measured.

what's missed

No reward/accuracy parity run; fixed 1024-token responses disable continuous batching; single A100/NVLink testbed; ≤70B, ≤128 GPUs; simulator accuracy unquantified.

would raise confidence

A convergence-parity curve vs. a baseline; a slower-interconnect testbed; reported simulator prediction error; scaling past 70B / 128 GPUs.

// bottom line

The speedups are genuine and the design is the right one — but the fair-fight figure is ~3×, not 20×, and "the trained model is identical, only faster" is assumed rather than shown. The strongest validation isn't in the paper at all: open-sourced as verl, it became the de-facto RLHF/RL substrate (DAPO and much of the open RL-for-reasoning wave build on it), which is a far harder test than any benchmark table.

// takeaways

What to remember

  1. Hybrid beats either pure paradigm. Put the single-controller where the graph is small (inter-node coordination, data resharding, placement) and the multi-controller where the compute is huge (inside each model). Decoupling data dependencies from computation is what makes a new RLHF algorithm an 8-line script instead of a system rewrite.
  2. The actor's train↔generate reshard is the hot path. Keeping one weight copy and forming generation parallel groups that overlap the training shards drives memory redundancy to zero and the all-gather into a tiny micro-DP group — cutting transition time up to 89% and most of the 70B speedup.
  3. verl is the real legacy. The 1.53×–20.57× numbers are a 2024 snapshot, but the open-source release became the standard training backbone for the open RL-for-reasoning era — the infrastructure layer under DAPO and many descendants on this map.
