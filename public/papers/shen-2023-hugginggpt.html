HuggingGPT: Solving AI Tasks with ChatGPT and its Friends in Hugging Face

Language as the universal interface: ChatGPT plans a task list from a user request, picks a Hugging Face model for each subtask by reading its model card, dispatches execution, and stitches the results into an answer. The idea seeded the LLM-as-orchestrator pattern — but the paper's own numbers show its controller decomposing a request correctly, end to end, only 63% of the time even with GPT-3.5.

Yongliang Shen, Kaitao Song, Xu Tan, Dongsheng Li, Weiming Lu, Yueting Zhuang

// the gap

What problem is this solving?

By early 2023, LLMs like ChatGPT were fluent at language but structurally boxed in: text in, text out. They couldn't natively segment an image, transcribe speech, or generate a picture, and even where a unified multimodal model existed (Flamingo, BLIP-2, Kosmos-1), it had to be trained end-to-end for every new capability — an expensive, closed way to add skills. Meanwhile, thousands of narrow expert models that already solved individual vision, speech, and language sub-tasks well were sitting on Hugging Face, unused by the LLM, because nothing connected the two.

The paper's framing: rather than train one model to do everything, or bolt tools onto an LLM one integration at a time (as Toolformer and Visual ChatGPT had begun to do), what if the LLM's job were narrowed to coordination — parse the request, decide which existing expert models to call and in what order, and let each model do what it already does well? The open question is whether "language as the interface" — feeding the LLM nothing but natural-language model descriptions — is enough glue to make that coordination reliable.

// how it works

The method

HuggingGPT casts an LLM (ChatGPT) as a controller that never touches raw pixels or audio itself — it only ever reads and writes text, including structured JSON — and delegates every actual modality-specific computation to expert models hosted on Hugging Face. The workflow runs in four stages:

  • Task Planning. ChatGPT parses the user request into a structured list of tasks (JSON objects with a task name, id, dependency ids, and arguments), using a specification-based prompt plus in-context demonstrations. Tasks can be single, sequential, or a dependency graph — e.g. "generate a new image, then describe it in speech" requires the image step to finish before captioning can start.
  • Model Selection. For each planned task, ChatGPT is shown a filtered shortlist of candidate Hugging Face models (filtered by task type, then ranked by download count to fit the context window) along with their text descriptions, and picks one — a single-choice selection made purely from the model card's natural-language description, not from running the model.
  • Task Execution. The selected models run on hybrid endpoints (Hugging Face Inference Endpoints or a local deployment), with independent tasks executed in parallel where dependencies allow.
  • Response Generation. ChatGPT integrates the structured outputs (bounding boxes, captions, audio file paths, class probabilities) from every executed task into a single, human-readable summary answering the original request.

The whole system is training-free: no component is fine-tuned. The only thing HuggingGPT does at inference time is repeatedly call the same LLM (temperature 0, with a logit_bias nudge toward valid JSON) on carefully engineered prompts with different few-shot demonstrations for planning, selection, and response generation. Task planning is evaluated as three separate problem types — single, sequential, and graph tasks — each with its own metric: F1/accuracy for single tasks, F1 plus normalized edit distance for sequential tasks, and a GPT-4-as-judge score ("GPT-4 Score") for graph tasks, since a DAG can have more than one valid topology.

// what they found

The results

The clearest, least ambiguous finding: task-planning quality scales hard with the controller LLM's own capability, and open instruction-tuned models of that era were nowhere close to strong enough. On the human-annotated planning dataset (46 hand-labeled complex requests), accuracy climbs from near-zero for Alpaca-7b/Vicuna-7b to GPT-3.5 to GPT-4:

Even at the top of that table, GPT-4 — the strongest controller tested — gets barely 4 in 10 sequential-task decompositions exactly right against human-annotated ground truth, and the paper says so itself: "even though GPT-4 outperforms other LLMs, there still remains a substantial gap when compared with human annotations." The separate end-to-end human evaluation (130 diverse requests rated by human experts across passing rate, rationality, and success rate) tells a similar story for the deployed system: GPT-3.5 reaches a 63.08% success rate — i.e. more than a third of user requests are not ultimately resolved correctly — versus 15.64% for Vicuna-13b and 6.92% for Alpaca-13b.

An ablation on demonstration variety shows planning quality is sensitive to prompt engineering, not just model scale: going from 2 to 10 task types represented in the few-shot demonstrations lifts GPT-3.5's single-task F1 from 48.29 to 53.70 and GPT-4's from 67.08 to 71.05 — a real but modest effect that plateaus past roughly 4 demonstrations. Qualitatively, the paper shows HuggingGPT correctly chaining multi-stage, cross-modality requests (pose detection → conditional image generation → captioning → text-to-speech) end to end when everything goes right, which is the demonstration most readers remember — but it sits alongside quantitative numbers that are considerably more modest.

// does it hold up?

The eval read

HuggingGPT is an influential proof of concept — "language as the interface" is a real and durable idea — and to its credit, the paper's own Limitations section (Section 5) is candid about most of what follows. But the gap between the demo and the numbers is wide enough to be the main thing worth taking away.

The headline capability, decomposed correctly, fails more often than it succeeds. The paper's own human evaluation puts end-to-end success at 63.08% for GPT-3.5 — the model configuration actually built into the system (GPT-4 was not run through the same 130-request human evaluation, only the smaller planning-accuracy tables). A controller architecture is only as good as its planning step, and on the human-annotated ground truth, even GPT-4 tops out at 41.36% exact-match accuracy on sequential tasks. Read together, HuggingGPT's central mechanism — LLM-driven task decomposition — is shown by the paper's own evaluation to be the system's weakest link, not a solved sub-problem.

Most of the evaluation grades against a pseudo-label the same model family produced. Tables 3–5 (single/sequential/graph task planning) are scored against pseudo-labels generated by GPT-4 itself, and graph-task correctness is judged by a "GPT-4 Score" — GPT-4 grading whether a plan is acceptable. Only the smaller 46-example human-annotated subset (Table 6) checks against real human labels, and that is precisely where the "substantial gap" the authors flag actually shows up — the self-graded numbers elsewhere in the paper are, by construction, more forgiving.

The open-model baselines are the weakest available at the time, with no strong contemporaneous agent baseline. Alpaca-7b/13b and Vicuna-7b/13b are early (Feb–March 2023) instruction-tuned LLaMA derivatives specifically known for weak tool-use and format-following — a low bar that makes GPT-3.5/GPT-4's advantage look larger than a comparison against a purpose-built tool-use model (e.g. Toolformer, which the paper cites but never runs head-to-head) would show.

baselines

Alpaca-7b/13b and Vicuna-7b/13b only — both notoriously weak at structured tool-calling. No comparison to Toolformer, ReAct, or Visual ChatGPT, all cited in Related Work but never benchmarked against directly.

contamination

Not discussed. The 46-example human-annotated planning set and the 130-request human-evaluation set are original to this paper, which sidesteps classic contamination — but neither dataset's size or collection protocol (beyond "we invite some annotators") is detailed enough to assess selection bias.

what's missed

The most-cited demonstrations (Figures 1, 2, and the case studies) are qualitative successes; the quantitative tables tell a harder story — 63% end-to-end success for the shipped GPT-3.5 configuration, 41% exact planning accuracy for GPT-4 against human ground truth.

would raise confidence

Full human-annotated evaluation (not just 46 examples) for every table, not only the smallest one; a head-to-head run against a contemporaneous tool-use agent; and an error breakdown of the 37% of GPT-3.5 failures — planning, selection, or execution — the paper doesn't provide.

// bottom line

The architectural idea — route a request through an LLM controller to a marketplace of expert models, using only natural-language model descriptions as glue — genuinely shaped the LLM-agent-as-orchestrator pattern that followed. But the paper's own most trustworthy number, the 46-example human-annotated planning accuracy, and its own end-to-end human evaluation both say the same thing: even the strongest controller tested gets the plan wrong close to half the time, and the shipped GPT-3.5 system resolves fewer than two in three real user requests. The demos work because they were chosen to work; the tables show what happens on average.

// takeaways

What to remember

  1. Language as the universal interface. HuggingGPT routes a user request through an LLM controller to a marketplace of expert models, using nothing but natural-language model-card descriptions to decide which model to call — no fine-tuning, no hard-coded tool schema per model.
  2. Four stages, one model wearing four hats. Task planning → model selection → task execution → response generation, all driven by the same LLM at different prompts — an early instance of the plan-then-act agent loop that later frameworks formalized.
  3. Planning is the bottleneck, by the paper's own numbers. Even GPT-4 reaches only 41.36% sequential-task accuracy and 58.33% graph-task accuracy against human-annotated ground truth; the deployed GPT-3.5 configuration resolves user requests end to end only 63.08% of the time.
  4. Most quantitative evaluation is GPT-4-graded, not human-graded. Only a 46-example subset checks against real human labels — and that subset is exactly where the reported gap between LLM self-grading and human judgment is largest.
