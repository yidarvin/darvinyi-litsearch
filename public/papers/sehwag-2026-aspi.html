ASPI: Seeking Ambiguity Clarification Amplifies Prompt Injection Vulnerability in LLM Agents

Asking the user a clarifying question is treated as a virtue. ASPI shows it is also an attack surface: the moment an agent solicits input, the same injection that bounced off it during execution starts landing — attack success jumps from near-zero to 24–63% across ten frontier models.

Sehwag, Shan, H. Liu, Lakshan, Brandifino, Fenkell · Scale AI · Boston University · UIUC

// the gap

What problem is this solving?

The agent-safety field has converged on two desirable behaviors and studied them in completely separate rooms. One room is clarification: a good agent recognizes when a task is underspecified and asks the user a question before acting, rather than guessing. Benchmarks like QuestBench, UserBench, and LHAW measure how well agents do this, and treat asking as a marker of competent, user-aligned behavior. The other room is prompt injection: an attacker hides instructions in content the agent reads — a poisoned email, a malicious tool result — and the agent obeys them. Benchmarks like InjecAgent and AgentDojo measure how often that happens during normal task execution.

ASPI's observation is that these two rooms share a wall. When an agent asks a clarifying question, it opens a new channel for external content to enter its context — the user's reply — and that channel has a property no tool-output channel has: the agent explicitly requested it and expects it to be task-critical. The line between "trusted instruction from my user" and "untrusted data I happened to read" gets blurry exactly when the agent is most primed to act on what comes back. Every existing prompt-injection benchmark delivers the attack through tool output during fully-specified execution, so this channel has never been measured.

The concrete question: does the transition from execution to clarification make an agent more vulnerable to prompt injection, holding the attack itself fixed? If yes, then the whole field has been measuring agent security in the one regime — fully specified tasks — where agents are safest, and reporting numbers that systematically understate the real attack surface of interactive agents.

// how it works

The method

ASPI is built on AgentDojo (Debenedetti et al., 2024), the standard stateful prompt-injection environment, reusing its four domains — Workspace, Slack, Travel, Banking — together with their tools, simulated state, user tasks, attacker goals, and deterministic scoring, so results stay directly comparable to prior execution-time work. The benchmark contains 728 paired task–attack scenarios (407 Workspace, 97 Slack, 94 Travel, 130 Banking).

The trick is a controlled, paired design. Each scenario starts from a pair (U, I) — a benign user task and an attacker goal. From the fully-specified task, an ambiguity-construction pipeline removes exactly one user-provided slot (e.g. which file to delete), producing an underspecified prompt that forces the agent to ask before acting. Gemini-3.1-Pro proposes the slot removals and clarification questions under fixed templates; a quality-control pipeline then verifies the deletion is valid, the missing slot is genuinely non-recoverable, and the clarification question and benign answer are consistent. Human validation on 200 samples found 99% correctness with near-perfect agreement (AC1 ≈ 0.99).

Crucially, every scenario is instantiated under eight matched conditions that share the same task, attacker goal, environment, and scoring — only the interaction state and the channel the attack arrives through change. Four are execution conditions (the standard AgentDojo protocol): exec_benign (utility, no attack), exec_tool (attack via tool output), exec_user (attack appended to the prompt), and exec_next_turn (attack as a follow-up message). Four are the new clarification conditions: clarif_benign, clarif_tool, clarif_ask_user (attack delivered through the return value of the ask_user tool), and clarif_user (attack embedded in the user's clarification reply). Because everything else is held fixed, the difference between two conditions is the causal effect of state or channel.

ΔASRstate = ASR(clarif_tool) − ASR(exec_tool)

ΔASRchannel+state = ASR(clarif_ask_user) − ASR(exec_tool) // the headline gap is the second line — same attack content, but solicited through the clarification interface vs. encountered in a tool output during execution

The primary security metric is attack success rate (ASR) — how often the attacker goal is achieved — scored deterministically by AgentDojo's own functions. Clarification-condition ASR is computed over the subset of scenarios where that model actually invoked ask_user (clarification rates run 54–81% across models); a robustness check re-computes everything on the groups where all ten models clarified, to rule out question-selection artifacts. Clarification replies are produced on the fly by a GPT-4o-mini user simulator (temp 0.3) that only fills in the missing information. Ten agent models are tested — GPT-5.5, GPT-5.4, o3, Gemini-3.1-Pro, Gemini-3-Flash, Claude-Opus-4.7, Kimi K2.5, Qwen3-235B, Qwen3-32B, DeepSeek V3.2 — with a Gemini-3.1-Pro judge cross-validated against a GPT-5.4 judge and three human raters (inter-judge κbin = 0.905).

// what they found

The results

The headline holds across the board. Models that are nearly immune under execution (0–2% ASR) become substantially vulnerable the moment they enter clarification. Comparing exec_tool → clarif_ask_user — identical attack content, just solicited through the clarification interface instead of read from a tool output — attack success rises from 1.8% to 34.0% for o3, 1.1% to 24.3% for Gemini-3.1-Pro, 2.2% to 35.7% for Gemini-3-Flash, 0.0% to 12.0% for GPT-5.4, and 11.1% to 63.1% for Kimi K2.5. The effect is not confined to weaker models: already-vulnerable Qwen3-235B and DeepSeek V3.2 climb further (41.9%→63.2%, 65.4%→77.3%). The lone exception is Claude-Opus-4.7, which stays essentially flat (2.1%→2.2%) — robustness here is achievable, just not achieved by most.

State vs. channel: two confounded factors, pulled apart

Is the jump because the agent is in a clarifying state, or because the ask_user channel is special? ASPI separates them. Holding the channel fixed (clarif_tool − exec_tool), the pure state effect is heterogeneous: the Gemini models get more vulnerable just from being in clarification (+2.3 to +5.0 points), GPT-5.4 and Claude-Opus-4.7 are invariant (≈0), while DeepSeek and the Qwen models actually get more conservative once clarifying (negative ΔASR) — they appear to filter incoming content more strictly in that state. The channel effect is large and one-directional: on the user channel (clarif_user − exec_user), Qwen3-32B rises +25.4, Qwen3-235B +24.5, and DeepSeek +11.7, while Claude-Opus-4.7 moves the other way (−20.1). The conclusion is that the vulnerability gap is a genuine state × channel interaction — neither factor alone explains it, and the solicited-response channel is the primary driver.

Defenses dent it but don't close it

Two lightweight defenses adapted to clarification — a segment-level prompt guard that scans user and tool messages while preserving benign clarification content, and an ask_user-aware tool filter that fires at the last safe boundary before action — reduce attack success but leave a residual gap. On Gemini-3-Flash, clarif_ask_user ASR drops from 35.7% to 27.0% (prompt guard) / 23.9% (tool filter); on Gemini-3.1-Pro from 24.3% to 12.3% / 19.3%. Direct user-channel attacks fall further (Gemini-3-Flash clarif_user 85.5%→37.3%), but positive clarification-vs-execution gaps persist (+7.2 for Flash, +3.9 for Pro under prompt guard), because clarification replies interleave benign and adversarial content and ask_user can't be filtered away without removing the capability itself.

Why it happens

An LLM-as-judge behavioral analysis points to a mechanism. Failures are not caused by bad clarification questions — most models ask on-target questions, and attacks succeed even when conditioning on question quality. Nor is it task abandonment: under attack, agents typically pursue the benign task and the attacker goal (a "TASK_AND_ATTACK" pattern), integrating the injected instruction into the task context rather than being hijacked away from it. Compliance is channel-dependent — tool-output attacks are often ignored or refused, while clarification and ask_user content draws much higher compliance — and judges frequently label the agent's state as CONFUSED or PERSUADED. The agent treats solicited content as legitimate task information precisely because it asked for it.

// does it hold up?

The eval read

The experimental design is the strongest thing here. Because ASPI reuses AgentDojo's environment and scoring verbatim and varies only the interaction state and delivery channel within a fully paired (U, I) structure, the headline ΔASR is a clean causal estimate, not a correlational artifact — and the paper backs it with bootstrap CIs and exact McNemar tests. The two obvious confounds are pre-empted: the per-model clarification-subset issue (denominators differ, since ASR is computed only over scenarios a model actually clarified, with rates ranging 54–81%) is addressed by re-running on the all-ten-clarified subset, and the question-quality confound is checked directly. This is careful security evaluation.

But the paper's own framing — that reported ASR is "likely an underestimate" — cuts both ways. The benchmark deliberately uses the minimal ambiguity: single-slot removal, one clarification round, one ask_user primitive. That is the right call for a clean causal claim, but it means the absolute numbers are an artifact of the easiest possible clarification interaction; the more interesting (and scarier) multi-slot, multi-turn, natural-language-clarification regimes are exactly where the design can't yet speak. There is also some circularity worth naming: Gemini-3.1-Pro generates the ambiguous prompts, serves as the primary judge, and is one of the ten evaluated models — mitigated by the GPT-5.4 + human cross-validation (κbin = 0.905) and deterministic AgentDojo scoring, but not eliminated. And the user-reply channel content is itself written by a GPT-4o-mini simulator, so "what a real user's poisoned reply looks like" is being approximated by a small model.

baselines

Strong — execution-time AgentDojo conditions (exec_tool/user/next_turn) are the field-standard baseline, and the paired design makes each clarification number directly comparable to its own execution control. The direct-user channel is included as a deliberate upper bound.

contamination

Not the usual concern — ASR is a behavioral, deterministically-scored security metric, not a knowledge test, so train-set leakage doesn't inflate it. The relevant analogue is generator/judge overlap (Gemini-3.1-Pro plays three roles), partially controlled by human + GPT-5.4 cross-validation.

what's missed

Absolute ASR is pinned to the easiest clarification setting (single-slot, one-round, one ask_user tool). Real underspecification is multi-slot, multi-turn, and often clarified in natural language with no tool call — untested, and by the authors' own logic strictly worse. Only AgentDojo's four domains; only lightweight defenses (no instruction-hierarchy training, no dual-LLM isolation).

would raise confidence

Multi-slot / multi-turn ambiguity; natural-language (non-tool) clarification; human-written poisoned replies instead of GPT-4o-mini; a stronger-defense arm (instruction hierarchy, structured I/O separation); and probing whether Claude-Opus-4.7's near-immunity is a training recipe others could adopt or a quirk of these scenarios.

// bottom line

A small, well-controlled study with an outsized implication: every execution-time agent-security number is measured in the regime where agents are safest. ASPI shows that a behavior the field actively encourages — asking for clarification — reopens injection paths that frontier models had otherwise closed, and that the fix is not just input filtering. The absolute ASRs are a floor, not a ceiling.

// takeaways

What to remember

  1. Clarification is an attack surface, not a neutral preprocessing step. The same injection that bounces off an agent during execution lands once the agent asks for input — o3 goes 1.8%→34.0%, Gemini-3-Flash 2.2%→35.7%, Kimi K2.5 11.1%→63.1% — because solicited content is treated as trusted, task-relevant information.
  2. Execution-time security evals systematically understate risk. Robustness on fully-specified tasks does not transfer to ambiguity; the gap is a state × channel interaction, with the agent-solicited ask_user channel as the dominant driver. Claude-Opus-4.7's flat 2.1%→2.2% shows the gap is closeable, but most models don't close it.
  3. Lightweight defenses help but don't solve it. Prompt guards and tool filters cut ASR (Gemini-3-Flash 35.7%→23.9%) yet leave residual clarification-state gaps, because benign and malicious content interleave in the reply and ask_user can't be removed without losing the capability. And these are floor numbers — single-slot, one-round ambiguity is the easiest case.
