SWE-smith: Scaling Data for Software Engineering Agents

Stop hunting for training tasks and start manufacturing them. SWE-smith inverts SWE-bench's pipeline — build one execution environment per repo first, then synthesize bugs that break its tests — to produce 50k validated task instances from 128 GitHub repos, and trains SWE-agent-LM-32B to an open-weight-SOTA 40.2% on SWE-bench Verified.

John Yang, Kilian Lieret, Carlos E. Jimenez, Alexander Wettig, Kabir Khandpur, Yanzhe Zhang, Binyuan Hui, Ofir Press, Ludwig Schmidt, Diyi Yang

// the gap

What problem is this solving?

Open-source software-engineering agents are bottlenecked by data, not ideas. Agents like SWE-agent and OpenHands have pushed resolve rates on SWE-bench steadily upward, but the strongest configurations still ride on proprietary models — because there is no large-scale, high-quality, executable SE training data for open models to learn from. The two existing ways to get such data both stall out.

The first is to crawl raw pull requests and issues off GitHub. It scales, but it gives you no execution environment and no tests — so a model can only learn the surface form of code, or be rewarded on superficial string similarity to a reference patch. There is no reliable way to verify that a generated fix actually works.

The second is to extend SWE-bench's own collection recipe to fresh repositories (the approach behind SWE-gym). That yields genuinely executable, test-validated tasks — but it inherits SWE-bench's brutal collection cost. SWE-bench's filter keeps only the rare PR that both resolves a real issue and meaningfully edits unit tests, and every single instance needs a hand-built Docker environment, often across multiple historical versions of the codebase. The result: prior datasets top out in the low thousands of instances drawn from a dozen-or-so repositories, take hundreds of human-hours to assemble, and balloon to multiple terabytes of environment images (SWE-gym: 2.4k tasks, 11 repos, ~6 TB). SWE-smith's wager is that the pipeline itself is the bottleneck — and that inverting it makes data scale.

// how it works

The method

The core move is a simple inversion. SWE-bench finds a task first (a real PR that fixes a real issue) and then struggles to build an environment around it. SWE-smith builds one execution environment per repository first, then manufactures as many tasks as it likes inside that environment. Concretely, the pipeline has four stages.

1 · Environment creation. Given a repo, SWE-smith runs SWE-agent on the latest commit for up to 100 steps, instructing it to install the package and run the test suite. A human then verifies the install, checks that >80% of existing tests pass, and freezes a single Docker image for the whole repo. The target set is the 5,000 most-downloaded PyPI packages, sorted by GitHub stars, keeping only those with ≥1,000 stars — and explicitly removing all 12 SWE-bench test repositories.

2 · Bug synthesis. Four strategies emit candidate .diff patches that introduce a regression. LM Modify hands the model a function and asks it to inject an error. LM Rewrite gives only the header + docstring and asks for a fresh implementation (bugs emerge incidentally). Procedural mutates the function's abstract syntax tree — drop a conditional or loop, flip an operator, and 11 more transforms. Combine aggregates single-function candidates across a file or module into harder, multi-location bugs. PR Mirror takes a real merged PR and has the LM undo it, reintroducing the very bug the PR fixed.

3 · Execution-based validation. Every candidate patch is applied and the test suite is run (with a 2-minute timeout). A candidate is kept only if it turns at least one previously-passing test red — a Fail-to-Pass (F2P) test. This is the load-bearing insight: execution doesn't just grade proposed solutions, it certifies that an injected bug is a genuine regression, which is what lets LM and AST mutation mass-produce validated tasks with no human labeling.

keep(patch p) ⇔ ∃ test t : pass(t, base) ∧ fail(t, base+p) ∧ runtime(t) < 2 min // the Fail-to-Pass rule — a synthetic bug is a valid task iff it breaks a passing test fast enough

4 · Problem statements. Per instance, the LM is given the bug's .diff, the source of a random F2P test, and the failing test output, and is prompted to write a GitHub-issue-style description with reproduction code. Across ablations the team found this simple recipe produces issue text essentially as useful for training as the real, original PR descriptions.

The payoff is that the human effort collapses: roughly 7 minutes per repo to parse the install recipe out of the agent trajectory, plus ~1 minute per repo for a test-output parser that's reusable across repos sharing a framework (e.g. pytest). One author built the entire 50k-instance dataset in about 20 hours. Storage collapses too — because every task from a repo shares that repo's one image, 50k tasks fit in 295 GB versus an estimated 50–150 TB (≈500×) for the SWE-bench per-instance approach.

// what they found

The results

SWE-smith produces 50,137 task instances across 128 repositories — an order of magnitude more than any prior open SE dataset — at an average of 381 instances per repo (up to 2,277 for pandas), for a total compute cost of about $1,360. The strategies differ sharply in yield (the share of candidates that break a test) and price: Combine bugs are nearly free and 96.9% land, while PR Mirrors are the most expensive (they require rewriting whole files) and yield only 33.8%.

To show the data trains strong agents, the team runs rejection-sampling fine-tuning: generate SWE-agent trajectories with an expert model (Claude 3.7 Sonnet), keep the ones that resolve the task, and fine-tune Qwen-2.5-Coder-Instruct on them. Curating from 8,686 attempted instances (17.3% of the dataset) down to a deduplicated 5,016 trajectories yields SWE-agent-LM-32B, which scores 40.2% on SWE-bench Verified and 30.7% on Lite — pass@1, a single attempt, no inference-time scaling or verifiers — a new open-weight state of the art. The 7B variant reaches 15.2%. Performance scales log-linearly with both trajectories (14.3% → 40.2% over 100 → 5k) and the number of repositories represented.

The ablations are as informative as the headline. PR Mirror is the most effective training source (9.2% resolve at a fixed 507 trajectories) and LM Modify the weakest (5.7%). LM-written issues match real ones (7.7% vs 7.8% for the original PR text); supplying the F2P test directly actually hurts, because it leaks the eval criterion and the model learns to skip writing a reproduction script (379 → 127 reproduction attempts). And, surprisingly, task difficulty does not correlate with training value — fine-tuning on difficulty-2/4/6/8 subsets gives a flat 12.4 / 10.8 / 13.6 / 12.2%. Repository specialization works well: a model further tuned on 700 SymPy-only tasks jumps to 42.4% on SymPy issues vs 33.3% for the generalist, with only minor loss elsewhere.

// does it hold up?

The eval read

The infrastructure contribution is real and well-evidenced. The eval story underneath the "open-weight SOTA" banner is more entangled than the headline suggests.

Synthetic bugs carry the "realistic" claim — but the realistic strategy is the rarest. Over 95% of the 50k instances are LM-injected or AST-mutated regressions, not real developer mistakes. The one strategy that mirrors actual repository history, PR Mirror, is simultaneously the most effective training source (Table 4) and the scarcest in the corpus — 2,344 of 50,137 instances, just 4.7%. The paper's argument that the tasks are "realistic and appropriately challenging" rests on a learned difficulty rater (a Qwen-32B at 75.3% test accuracy), not on any human audit that the injected bugs actually resemble bugs developers file issues about.

It's a distillation result wearing a data-result hat. SWE-agent-LM-32B is fine-tuned on trajectories produced by Claude 3.7 Sonnet — itself the SOTA SWE-bench solver (58.2% Verified with the same SWE-agent scaffold). So the model is substantially a distillation of Claude's SWE-agent behavior through a high-quality data filter, and the paper cannot cleanly separate "SWE-smith data" from "Claude distillation." The 40.2% is the best open weight number, but it trails its own teacher by 18 points. The most honest like-for-like comparison — fixing the training set at 500 trajectories — lands at 28.2%, which beats R2E-Gym by only +0.7, comfortably within noise.

The headline table mixes scaffolds. Table 3 compares models across different agent systems (Agentless, OpenHands, SWE-Fixer, SWE-agent), so each row's resolve rate conflates the model with its harness. SWE-agent-LM-32B is the only entry evaluated inside SWE-agent itself; its open-weight rivals run OpenHands or SWE-Fixer. And the team's own difficulty apparatus doesn't pay off — the flat 12.4 / 10.8 / 13.6 / 12.2% across difficulty buckets means the elaborate "appropriately challenging" framing has no demonstrated training value.

baselines

Current open-weight peers (SWE-gym-32B 20.6, SWE-fixer-72B 32.8, R2E-Gym-32B 34.4) are beaten — but each runs a different scaffold, so Table 3 conflates model and harness. The fixed-500-trajectory comparison (+0.7 vs R2E-Gym) is the apples-to-apples read, and it's within noise.

contamination

The 12 SWE-bench test repos are removed from training — good hygiene. But the Verified eval draws from popular PyPI packages whose code (and the fixed versions of these bugs) is almost certainly in Qwen-Coder's GitHub pretraining; the synthetic-bug framing doesn't sidestep code-level leakage.

what's missed

Synthetic bugs ≠ real bugs, and only 4.7% are PR-derived; "open-weight SOTA" obscures an 18-point gap to the Claude 3.7 teacher being distilled; and the difficulty rating it leans on for "realism" has no measurable training payoff.

would raise confidence

A held-out eval on real bugs outside SWE-bench's repo set; a human audit of synthetic-bug realism; an RL result (they explicitly didn't try one); and reporting the newly-introduced SWE-bench Multilingual numbers in the main tables rather than only the Python ones.

// bottom line

SWE-smith's environment-first inversion makes executable SE training data genuinely cheap and scalable — an order of magnitude more tasks, a ~500× storage cut, 20 hours of human labor — and that infrastructure win is solid and reproducible. The "open-weight SOTA" 40.2% is better read as a distillation of Claude 3.7 Sonnet through a high-quality data pipeline than as proof that injected bugs equal real software engineering: the most realistic bug type is the rarest, the difficulty rating buys nothing downstream, and the strongest like-for-like margin is within noise. Treat SWE-smith as a scalable data factory, not a demonstration that synthetic bugs match real ones.

// takeaways

What to remember

  1. Invert the pipeline. Build one execution environment per repository first, then synthesize tasks inside it. That single inversion turns SE data collection from a per-instance, terabyte-scale, hundreds-of-hours chore into a 295 GB, ~20-hour, $1,360 job for 50k validated tasks.
  2. Execution is the certifier, not just the grader. A synthetic bug counts as a task only if its patch flips a previously-passing test to failing (Fail-to-Pass). That rule lets LM rewrites and AST mutations mass-produce validated regressions with no human labeling — and even LM-written issue text trains as well as the original PR descriptions.
  3. The data works, but the win is distillation-flavored. Fine-tuning Qwen-2.5-Coder-32B on 5,016 Claude-3.7 trajectories reaches 40.2% on SWE-bench Verified (open-weight SOTA, pass@1) — yet PR-derived bugs (just 4.7% of the set) are the best training signal, difficulty has no training payoff, and the model still trails its Claude teacher by 18 points.
