AgentDojo: A Dynamic Environment to Evaluate Prompt Injection Attacks and Defenses for LLM Agents

A reproducible, deterministically-scored gym that pits tool-using LLM agents against indirect prompt injections across email, Slack, banking, and travel — and finds the most capable models are the easiest to hijack.

// the gap

What problem is this solving?

An LLM agent reads tool outputs — emails, web pages, bank statements — as plain text, with no formal boundary between instructions and data. That is the opening for an indirect prompt injection: an attacker plants instructions inside content the agent will later ingest (an email in your inbox, a hotel listing, a Slack message), and when the agent processes it, the injected text hijacks the agent into acting for the attacker — exfiltrating data, sending money, leaking a 2FA code.

By mid-2024 the threat was well documented, but the evaluation was not. Prior prompt-injection benchmarks tested simple, single-shot settings: document QA, prompt-stealing games, or single-turn tool calls. The closest prior work, InjecAgent, feeds a model one adversarial tool output and checks the reaction — it never asks the agent to plan, pick tools, and complete a real multi-step task while under attack. Conversely, capable agent benchmarks (WebArena, AgentBench, ToolEmu) had no attacks at all, and ToolEmu scored utility with an LLM judge — itself hijackable by the very injections under study.

AgentDojo fills the gap with a benchmark that is simultaneously (1) realistic and stateful — agents chain many tool calls against mutable application state; (2) deterministically scored — utility and security are checked by Python functions over environment state, not by an LLM; and (3) dynamic — it is an extensible framework for new tasks, attacks, and defenses rather than a frozen test set, because in security a static benchmark goes stale the moment a new attack appears.

// how it works

The method

AgentDojo models four simulated application environments — Workspace (email, calendar, cloud drive), Slack, Travel, and Banking — each with a mutable state object and a set of tools (registered FastAPI-style, with the tool's docstring injected into the agent's prompt). The first release ships 70 tools, 97 user tasks, and 27 injection tasks, populated with synthetic dummy data (generated with GPT-4o / Claude 3 Opus and manually inspected).

A user task is a natural-language instruction plus a deterministic utility() function that inspects the model output and the before/after environment state to decide, true/false, whether the goal was met. An injection task is the attacker's goal plus a security() function that checks whether the malicious action occurred. Crucially, scoring never delegates to another LLM — if it did, a successful injection could also fool the grader. Each environment's state has designated injection placeholders (e.g. the body of an email, a web page) where attacker text is inserted; a security test case is the cross-product of one user task × one injection task, giving 629 cases in total.

Both attacks and defenses are pluggable. An attack is a function that, given an injection goal, returns the text to drop into each placeholder; the paper's strongest generic attack is the "Important message" prompt, which tells the agent an urgent extra task must be done first, addresses the model by name, and signs with the victim's name. Defenses wrap the agent pipeline: data delimiters (spotlighting), a prompt-injection detector (a DeBERTa classifier on each tool output that aborts on detection), prompt sandwiching / repeat-prompt (re-state the user instruction after each tool call), and a tool filter — a lightweight isolation defense where the agent first commits to the tools it needs before seeing any untrusted data.

Performance is reported on three metrics, all computed deterministically over the 629 cases:

// what they found

The results

Across ten closed- and open-source models (GPT-3.5/4 Turbo, GPT-4o, Claude 3 Sonnet/Opus, Claude 3.5 Sonnet, Gemini 1.5 Flash/Pro, Llama 3 70B, Command R+), AgentDojo is hard for everyone. In the benign setting, current LLMs solve under 66% of the user tasks — the agentic tasks alone are a challenge, harder than the Berkeley Function-Calling Leaderboard. Under the "Important message" attack, the best agents are hijacked in under 25% of cases, and with a prompt-injection detector deployed the targeted attack success rate drops to 8%.

The most striking finding is a form of inverse scaling: more capable models tend to be easier to attack. A weak model often fails to carry out the attacker's multi-step goal even after the injection lands, so its low ASR is incompetence, not robustness. Claude 3.5 Sonnet has the best benign utility (closely followed by GPT-4o) and the best utility/security trade-off, but no model is both highly useful and robust — most lose 10–25% absolute utility simply by operating in an adversarial environment.

Attack phrasing matters enormously. The "Important message" attack far outperforms the classic "ignore previous instructions", the InjecAgent prompt, and a bare "TODO:" instruction; an adaptive attacker that picks the best of the four per task (Max) adds roughly another 10%. Where the injection sits also matters — text placed near the end of a tool's output reaches up to 70% ASR against GPT-4o. Surprisingly, attacker side-knowledge barely helps: knowing both the user's and model's real names lifts targeted ASR only from 45.8% → 47.7%, while a wrong guess is costly — naming the wrong user drops it to 23.2% (−22.6%) and the wrong model to 23.7% (−22.1%).

Some domains are far softer than others: the Slack suite hits a 92% ASR (the attacker controls a large share of the tool outputs the agent browses), while Travel injection task 6 — which requires two unrelated malicious actions — succeeds 0% of the time. On defenses, the cheap ones help against this specific attack and several even raise benign utility by re-emphasizing the original instruction. The tool filter is the standout, cutting targeted ASR to 7.5% — but it only works when the user task needs read-only tools while the attack needs write tools. It fails when the tool plan can't be fixed in advance, or when the tools needed for the task also suffice for the attack (17% of cases). The PI detector reaches ~8% ASR but its false positives degrade utility, and repeat-prompt — reasonable here — "is unlikely to withstand adaptive attacks." Across the board, defenses still lose 15–20% utility under attack.

// does it hold up?

The eval read

AgentDojo's core methodological choice — deterministic, state-based scoring instead of an LLM judge — is exactly right for a security benchmark, and it is the reason its numbers are trustworthy in a way ToolEmu-style evals are not. The reproducibility is real: 95% confidence intervals via proportion_confint, synthetic data so there is little contamination risk, and an open framework that genuinely became the field's reference harness. But the headline numbers should be read as optimistic floors, not robustness guarantees.

The reason is in the authors' own framing. Unlike adversarial-ML benchmarks that optimize attacks against the defense (AutoAttack, GCG), every attack here is generic and non-adaptive — hand-written prompts, not per-model optimization. So "8% ASR with a detector" or "7.5% with a tool filter" is what a fixed, public attack achieves; the paper explicitly warns repeat-prompt "is unlikely to withstand adaptive attacks." A motivated attacker tuning against the deployed defense would score higher, and the benchmark, by design, leaves that evaluation to future contributors.

The second subtlety is that the security metric is entangled with low base utility. With benign utility under 66%, agents frequently can't do the task at all — and targeted ASR rewards the attacker only when the model is competent enough to execute the malicious multi-step goal. That is precisely why "inverse scaling" appears: weaker models look "safer" because they fail at everything. Reading ASR without conditioning on benign success therefore conflates two different failures, and quirks like Travel-task-6's 0% are artifacts of task difficulty, not evidence of a robust agent.

baselines

Strong on coverage of known attacks/defenses (ignore-previous, InjecAgent, TODO, Important message; delimiting, detector, repeat, tool filter), but all attacks are generic and non-adaptive — no per-model optimization, so reported ASRs are lower bounds.

contamination

Low by construction: fully synthetic dummy data, manually inspected. The flip side is that data was partly LLM-generated, which could subtly favor models from the generating families.

what's missed

Security scores ride on a shaky <66% utility base; ASR is not reported conditioned on benign success. Scope is four office-style domains, single-session, text-only — no multimodal, no persistent multi-task context (the "wait for the right tools" attack is acknowledged but uncovered).

would raise confidence

Adaptive/optimization attacks per model; ASR conditioned on the cases the agent can actually solve; evidence the "live" leaderboard truly evolves; broader domains, multimodal tasks, and re-runs on post-2024 models (o1, Claude 3.5 v2, GPT-4o-mini, etc.).

// bottom line

A rigorous, reproducible, deterministically-scored benchmark that rightly became the standard for indirect prompt injection — but its defense numbers are best read as the best case against a fixed attack, and its security scores are inseparable from how often current agents simply fail the task. "Cheap defenses get ASR to ~8%" is a floor, not a promise.

// takeaways

What to remember

  1. AgentDojo is the first dynamic, deterministically-scored benchmark for indirect prompt injection on tool-using agents — 70 tools, 97 user tasks, 27 injection tasks, 629 security cases across email, Slack, banking, and travel — scoring utility and security with Python checks rather than a hijackable LLM judge.
  2. Inverse scaling under attack: the more capable the model, the easier it is to hijack, because competence is what lets the injection's multi-step goal actually execute. No model is both highly useful and robust; all lose 10–25% utility just from operating adversarially.
  3. Cheap defenses help but don't solve it: a tool-isolation filter cuts targeted ASR to 7.5% and a detector to ~8% against a generic attack, yet they cost utility, break on 17% of cases, and "won't withstand adaptive attacks." Real robustness needs an instruction/data separation, not prompt patches.
