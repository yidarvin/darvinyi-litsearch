Checklists Are Better Than Reward Models For Aligning Language Models

Turn every instruction into a weighted checklist of yes/no requirements, grade responses against it with an LM judge plus a confidence-gated verifier program, and use the satisfaction score as the RL reward. The result — RLCF — is the only method to improve Qwen2.5-7B-Instruct on all five instruction-following benchmarks, while the reward models that top RewardBench actively hurt.

Viswanathan, Sun, Ma, Kong, Cao, Neubig, Wu · Carnegie Mellon University & Apple

// the gap

What problem is this solving?

Reinforcement learning is the workhorse for adapting a model to follow instructions, but it only works cleanly where the reward is cheap and unambiguous. On verifiable tasks — math, code, format constraints — a program can score the answer, so RL has a clean target. On the open-ended instructions people actually type ("list Airbnbs in Singapore for 2 people under a budget, and add tips if listings aren't available"), there is no answer key. The field's standard fix, RLHF, optimizes a single learned reward model against a fixed notion of "helpfulness" and "harmfulness".

The paper's argument is that every existing source of automatic feedback for these non-verifiable instructions is structurally limited:

  • Verifiable-answer-only RL can reinforce only the narrow slice of behavior a program can check, ignoring subjective constructs like topicality, tone, or completeness.
  • Reward models are flexible, but their notion of reward is arbitrary and opaque — a single scalar that compresses everything — which is exactly what makes them reward-hackable.
  • Distilling preferences from a larger prompted model forces that model to silently infer which aspects to grade, collapsing the "generator–verifier gap" that gives RL something to learn.
  • Even multi-criterion prompts like UltraFeedback — which scores four global principles (instruction-following, helpfulness, truthfulness, honesty) — apply the same fixed rubric to every prompt, so the criteria are rarely comprehensive for any given instruction.

The gap the authors target: a grading signal that is automatic (no human annotation), flexible (covers all aspects of quality), intuitive (tracks perceptible differences between responses), and applicable to any instruction. Their bet is that the missing ingredient is not a better scalar reward model but a per-instruction rubric.

// how it works

Reinforcement Learning from Checklist Feedback

RLCF reduces "is this a good response?" to a list of yes/no questions. A checklist is a sequence of requirements, each phrased so it can be answered against a candidate response, such that a response is acceptable if and only if it answers "yes" to every item. To make that biconditional hold, the authors want checklists that are comprehensive and natural (entailed by the instruction), and — because false-positive rewards hurt RL more than false negatives — objective and atomic (one aspect of quality per item).

1. Generating the checklist

Two recipes are compared. Direct simply prompts an LM to extract a checklist from the instruction — simple, but it tends to parrot the instruction back, which limits coverage and objectivity. Candidate-based instead treats a requirement as "any aspect of an instruction that, when absent, causes a response to fail": it first samples responses of varying quality, then prompts the LM to write a checklist of all their possible failure modes, each with an importance weight from 0 to 100. Judged on InFoBench, candidate-based checklists are consistently better on objectiveness, atomicity, and overall preference (56.0% vs 38.0% preferred by humans), trading a little naturalness for a lot of objectivity. The resulting rubric set, mined from WildChat instructions, is released as WildChecklists (~130k instructions).

2. Scoring a response

Each checklist item is graded by two graders that cover each other's blind spots. An LM judge (Qwen2.5-72B-Instruct) returns a 0–100 score per item, averaged over 25 samples to reduce variance. Because LLMs are unreliable on hard discrete constraints ("does the response contain the letter R at least three times?"), a verifier program is also generated — but only when the model is confident it can check the item exactly; it abstains on subjective items like "is the sentence coherent?". The program score (when present) is averaged with the judge score.

The per-item scores are combined into one reward by an importance-weighted average:

score(x, y) = Σi wi · si(x, y)Σi wi // wi = item importance (0–100); si = ½·(25-sample LM-judge score + verifier-program score, when available)

3. Turning scores into a policy

For each instruction the base policy samples a response pair (temperature 1.3, top-p 0.9), each is scored against the checklist, and only the 40% of pairs with the largest gap on at least one criterion are kept — pairs that are too similar give no usable preference signal. The kept pairs become preference data for DPO. Notably, because RLCF is just a drop-in evaluator over directly-sampled responses, it does not require the perturbed/controlled candidates earlier work relied on.

The conceptual claim

The unifying frame (Figure 3): automatic evaluators differ in how many explicit criteria they expose. A classic reward model is one implicit criterion; UltraFeedback is four; Constitutional AI is a handful. Checklist feedback sits at the far end — an "extreme mixture-of-evaluators" where the space of prompted evaluators is effectively unbounded and a unique subset is selected for every instruction. That per-instruction specificity is the whole pitch.

// what they found

The results

The headline is consistency. Across five widely-used instruction-following benchmarks — IFEval, InFoBench, FollowBench, AlpacaEval, and Arena-Hard — RLCF is the only automatic-feedback method that improves Qwen2.5-7B-Instruct on every one. Concretely it delivers a +6.0 point gain on InFoBench overall (78.1 → 84.1, a 6.9% relative requirement-following improvement), a +3.9 point gain in average hard-satisfaction-rate on FollowBench (71.4 → 75.3, with CSL rising 3.05 → 3.30), and a +3.3 point rise in Arena-Hard win rate (51.3 → 54.6). On IFEval — the most program-checkable benchmark — gains are smaller (2.8–3.0% relative on loose metrics, roughly neutral on strict).

The contrast with reward models is the point. Skywork-Reward-Gemma-2-27B and ArmoRM are top of the RewardBench leaderboard, yet as RL teachers they are inconsistent: they help on InFoBench but hurt on IFEval and FollowBench (Skywork drops FollowBench HSR to 69.5, below the 71.4 baseline). The only row below that beats the base model on all three axes at once is RLCF:

Why reward-model accuracy lies. On RewardBench itself, the checklist-based reward is worse at picking the better response (90.0 Chat / 80.7 Chat-Hard / 71.4 Safety / 88.5 Reasoning vs Skywork's 96.1 / 89.9 / 93.0 / 98.1) — yet it teaches better. That echoes prior findings that reward-model "accuracy" is poorly correlated with RLHF efficacy. The qualitative failure modes (Table 8) make it tangible: Skywork scores a perfect Spanish translation only 25.5/100 and gives 33.1 vs 8.0 to two semantically-identical "dense" sentences — reward models are too sensitive to surface form. Checklists give stable, interpretable scores instead.

It generalizes, with caveats. Candidate-based checklists beat direct ones after RL (+2 on IFEval, +2–3 on FollowBench). The recipe works off-policy: training Llama 3.1 8B Instruct and OLMo 2 7B Instruct on preference pairs collected from Qwen still improves them (Llama InFoBench 83.1 → 84.2, FollowBench HSR 68.0 → 72.3). RLCF helps most on "content" constraints (FollowBench HSR 60.0 → 66.4), suggesting checklists push the model to attend to the whole instruction rather than a few salient spans. And the 25-sample judge — the cost bottleneck — can be cut to 5 samples while retaining most of the gain, saving 55% of wall-clock time (40 vs 92 hours).

// does it hold up?

The eval read

The empirical core is clean and the baselines are current — DPO from two top RewardBench models, UltraFeedback, a rubric-free AI judge, and SFT distillation, all sharing the same 72B teacher so the checklist contribution is isolated rather than confounded with "bigger teacher". The consistency result is real and well-supported. But the title makes a universal claim that the evidence narrows considerably.

One policy, one algorithm. Every headline number is Qwen2.5-7B-Instruct trained with DPO. There is no GRPO/PPO, no second base policy at the headline scale, no larger model. The off-policy Llama/OLMo runs and the from-base Qwen runs help, but the sweeping "checklists are better than reward models" rests on a single 7B + DPO configuration — a point the authors themselves flag as a limitation.

The framing is benchmark-relative. Reward models actually beat checklists on RewardBench and match or beat them on InFoBench in some configs; checklists win specifically on the RL-teaching axis. The honest reading is "checklists are a better RL teacher for instruction following here," not "checklists are better evaluators." Notice too that Skywork posts the highest Arena-Hard win rate in the table (55.1) — RLCF's edge is breadth, not dominance on any single metric.

Shared-judge risk. The same Qwen2.5-72B family writes the checklists, grades the items, and serves as the AI-judge baseline — and two of the five eval benchmarks (Arena-Hard, AlpacaEval) are themselves LLM-judged. Gains on LLM-judged benchmarks could partly reflect judge agreement rather than genuine quality. It is at least consistent with this worry that the smallest gains land on IFEval, the one benchmark scored by a program rather than a model.

Specialization tax. Because WildChat is ~75.5% daily-assistance and writing, RLCF nudges the model toward that distribution: GSM8K and TruthfulQA each drop ~1–1.5 points, and the safety profile shifts (XSTest false refusals improve 92.0 → 95.6 but true refusals slip 83.0 → 81.0). It is not a safety-alignment method, and it inherits its training prompts' blind spots. The cost is also non-trivial: grading 130k instructions takes ~4 days on 8×H100, halvable but still heavy.

baselines

Strong and current (top RewardBench RMs, UltraFeedback, AI judge, SFT) with a shared 72B teacher — but all comparisons are one base policy + DPO only.

contamination

No train/test leakage concern, but a shared Qwen-72B judge across generation, grading, and two LLM-judged eval benchmarks invites self-preference bias.

what's missed

"Better evaluator" is really "better RL teacher": RMs still win on RewardBench and on raw Arena-Hard; gains shrink on the program-checked benchmark.

would raise confidence

GRPO/PPO results, multiple policies and scales, an independent non-Qwen judge, and a held-out instruction set not graded by an LLM.

// bottom line

A genuinely useful result with an over-broad title. The durable contribution is mechanistic: an instruction-specific, weighted, yes/no rubric — graded by a judge plus a confidence-gated verifier — is a denser, harder-to-hack RL signal than a single scalar reward model, and RewardBench accuracy is a bad proxy for how well an evaluator teaches. Whether "checklists beat reward models" generalizes past 7B + DPO is still open.

// takeaways

What to remember

  1. RLCF reframes open-ended alignment as a per-instruction checklist of weighted yes/no requirements — an "extreme mixture-of-evaluators" that hands RL a denser, less hackable reward than a single learned scalar.
  2. It is the only method that improves Qwen2.5-7B-Instruct on all five instruction-following benchmarks (+6.0 InFoBench, +3.9 FollowBench HSR, +3.3 Arena-Hard); reward models that top RewardBench (Skywork-27B, ArmoRM) actively hurt on IFEval/FollowBench — static RM accuracy ≠ RL-teaching quality.
  3. Two design choices carry the win: candidate-based checklist generation (enumerate the failure modes of varying-quality responses) beats prompting the instruction directly, and pairing an LM judge with a confidence-gated verifier program lets each cover the other's blind spots.
  4. The scope is narrow — one 7B policy + DPO, a shared Qwen-72B judge, ~1-point regressions on GSM8K/TruthfulQA, and 4 days of 8×H100 grading — so read it as "a better RL teacher for instruction following," not a universal verdict on reward models.
